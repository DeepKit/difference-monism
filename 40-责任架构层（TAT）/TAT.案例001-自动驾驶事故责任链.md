# TAT.案例001 · 自动驾驶事故责任链

> 定位：TAT 端到端走通案例——从事件发生到责任闭合的完整路径。展示五合同模型和闭合度谱系如何在真实场景中运作。
> 作者：Yi Fu（付毅，ODDFounder）

---

## 场景

一辆开启 L3 自动驾驶的乘用车，在城市道路上以 40km/h 行驶时，未能识别一名横穿马路的行人，发生碰撞。行人重伤。

### 涉事方

| 角色 | 位置 |
|------|------|
| 车主/驾驶员 | 坐在驾驶位，当时未主动驾驶 |
| 汽车制造商 | 生产和销售了该车辆 |
| AI 软件供应商 | 提供了自动驾驶感知和决策系统 |
| 安全监管机构 | 批准了该 L3 系统上路 |
| 行人（受害方） | 横穿马路（非人行横道） |

---

## 第一步：责任识别（TAT 五层责任分层）

### 因果责任
- **直接因果**：AI 感知系统未能识别行人 → 制动未触发
- **共因**：行人选择了非人行横道穿越
- **系统因果**：L3 系统的识别率在设计时已知对"部分遮挡的横穿行人"有盲区

### 角色责任
- **制造商**：角色是"将安全产品投放市场"——应确保系统在可预见的危险场景下能安全降级
- **AI 供应商**：角色是"提供满足安全规格的感知系统"——应标注已知盲区
- **监管机构**：角色是"确保 L3 上路前验证充分"——审批流程是否包含了此类场景
- **驾驶员**：角色是"在系统请求时接管"——事故前系统是否发出了接管请求？

### 认识责任
- **AI 供应商早知道**：内部测试报告显示"部分遮挡横穿行人"的识别率低于安全阈值。这份报告是否被披露？
- **制造商早知道**：同型号车辆此前已有 3 起"未识别横穿行人"的近失报告。这些报告是否被处理？
- **监管机构可能不知道**：如果供应商和制造商都未上报该盲区，监管机构的信息是断裂的

### 制度责任
- **谁有正式追责接口**：制造商的瑕疵担保责任（产品责任法）；AI 供应商的合同违约责任（对制造商）
- **谁没有**：行人只有一条追责路径——起诉制造商。行人无法直接追责 AI 供应商（无合同关系）
- **制度缺口**：监管机构的"审批通过"是否构成责任豁免？目前法律对此模糊

### 规范责任
- 在因果/角色/认识/制度责任被压清之后：制造商和 AI 供应商如果明知盲区而未行动，应受更高程度的责备
- 行人的违规横穿虽然有过错，但不应承担与"被系统未识别"同等程度的责备——因为系统被设计为应能处理"可预见的行人违规"

---

## 第二步：责任门槛判断（TAT 五合同模型）

### 应用五合同模型

| 合同 | 判断 |
|------|------|
| **主体合同** | 制造商是第一责任主体（它把产品投放市场）。AI 供应商是第二责任主体（它的软件是事故的直接技术原因）。驾驶员在未收到接管请求时不承担主体责任。 |
| **接口合同** | 制造商的 OTA 更新接口 + AI 供应商的盲区报告接口 + 监管机构的事故报告接口——这三个接口在本次事故中全部失效（盲区未被报、近失未被处理、事故才触发追查） |
| **证据合同** | 需要固定：AI 系统的事故前 30 秒传感器日志、内部测试报告的时间戳、近失报告的提交和审批记录、监管机构审批文件 |
| **熔断合同** | 应立即：暂停该型号 L3 系统在类似场景的使用（OTA 降级为 L2）；启动对所有已售车辆的同场景复测 |
| **补偿合同** | 制造商承担行人的医疗费+赔偿。制造商可向 AI 供应商追偿（如果合同约定）。监管机构不应被豁免——其审批失误对公共安全有系统性影响 |

---

## 第三步：闭合度判断

该事故涉及**高不可逆伤害（重伤）+ 高公共风险（所有同型号车主）+ 系统性制度缺口**。

→ 应适用 **高闭合**。

这意味着：
- 责任主体必须被明确指定（不能是"整个行业"）
- 证据链必须完整且可被独立复核
- 熔断必须是强制性的（不能只发"安全建议"）
- 补偿必须有确定的赔付路径和金额

---

## 第四步：熔断与补偿执行

### 熔断动作

```yaml
freeze_action:
  target: "同型号 L3 系统城市道路自动驾驶功能"
  scope: "全局——所有已售车辆"
  method: "OTA 远程降级为 L2（驾驶员必须监控）"
  trigger: "单起重伤事故 + 事前已知盲区 + 近失报告未处理"
  authority: "监管机构强制令 或 制造商主动召回"
  human_review_required: true
```

### 补偿路径

```yaml
compensation:
  primary_payer: "制造商"
  amount: "医疗费 + 收入损失 + 康复费 + 精神损害"
  secondary_recovery: "制造商向 AI 供应商追偿合同违约金"
  systemic: "监管机构建立 L3+ 事故公共补偿基金（由制造商和供应商按市场份额缴纳）"
```

### 证据固化

```yaml
evidence_seal:
  - sensor_log_30s_before_crash     # 不可篡改的事故传感器数据
  - internal_test_report_blind_spot # 标注时间戳——证明"早知道"
  - near_miss_reports_x3            # 证明"早有机会处理"
  - regulatory_approval_documents   # 证明"谁批的"
  - ota_update_log                  # 证明"何时熔断"
```

## 第五步：责任闭合确认

如果以下全部满足，本案例可被标记为"责任闭合"：

- [x] 制造商被正式指定为第一责任主体
- [x] AI 供应商被追偿
- [x] 监管机构的审批漏洞被写入修订程序
- [x] 同型号车辆被熔断（OTA 降级）直至盲区修复
- [x] 行人获得全额赔偿
- [x] 证据链固化，可供后续类似事故引用
- [ ] 盲区修复后重新验证通过——L3 功能恢复（仍在等待中）

**当前闭合度：高闭合（除最后一项修复验证外，其余均已完成或正在执行）。**
