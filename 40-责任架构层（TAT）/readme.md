# TAT 责任架构理论
## The Accountability Threshold

> **当前公开口径**：`v1.0 公开征评包 / 预印本支撑文档组`
>
> **定位**：TAT 是建立在 `DM -> ASTO -> ECET` 之上的责任架构层。
> 它不负责取代上游哲学地基或文明治理框架，而负责把“谁承接高影响后果”这一问题压缩成责任主体、责任接口、证据固化、熔断执行与补偿承接的最小结构要求。
>
> **作者 / Author**：Yi Fu（付毅，ODDFounder，fuyi.it@live.cn）
>
> **文件名说明**：目录中保留的 `RM.TAT.*` 文件名前缀属于早期命名阶段的路径保留，用于兼容既有引用；当前对外主品牌与层级口径仍以 `DM -> ASTO -> ECET -> TAT` 为准。

## 当前最稳定位

TAT 当前最适合被公开描述为：

> **一个面向高影响系统的责任门槛与责任架构框架，其主文档组已经适合公开征评与预印本式发布，但仍不是完成全球制度实施或大规模经验验证的闭合终稿。**

这意味着：

- 可以公开发布、公开比较、公开征评
- 可以作为预印本平台的占位与引用对象
- 不应包装为已经完成全球法律实施的成熟制度方案

## 统一发布模板

### 适用范围

- 用于处理高影响系统中的责任主体、责任门槛、责任证据链、熔断执行与补偿承接问题
- 用于把 `DM -> ASTO -> ECET` 的上游判断压缩成责任闭合度谱系与最小责任结构要求
- 用于公开征评、预印本支撑文档组和制度比较讨论

### 不是什么

- 不是取代 DM、ASTO、ECET 的平行终极理论
- 不是已经获得全球法律承认的成熟制度架构
- 不是脱离工程证据链和执行条件即可独立运作的责任口号

### 边界

- 哲学地基问题优先回引 DM，应用桥接问题优先回引 ASTO，文明治理约束优先回引 ECET
- TAT 负责责任主体、责任门槛与责任闭合度谱系的结构设计，不直接替代工程验证和产出封存流程
- 当前主包适合公开征评，不适合包装成已完成现实法制部署的终稿

### 与上游关系

- TAT 建立在 `DM -> ASTO -> ECET` 之上，是五层公开主链中的第四层
- ODD 继续把 TAT 的责任要求压成工程执行、验证、留痕与追责闭环
- 若脱离上游层直接谈 TAT，容易误读成孤立制度宣言

## DM 风险防火墙

### 最小依赖

即使不先接受 `DM` 的强版本，`TAT` 至少仍可建立在以下弱前提上：

1. 高影响行动会产生可外溢的后果
2. 后果需要可追溯的责任承接
3. 责任门槛需要异议、复审、熔断与补偿接口
4. 权力配置会影响责任逃逸与责任闭合

### `DM` 受争议时如何阅读本层

若 `DM` 的强形而上学或强规范推导受争议，
`TAT` 当前最稳的降级阅读方式是：

- 先把它读成高影响系统的责任门槛框架
- 先把它读成责任主体、责任接口与责任补偿的程序协议
- 不把它继续写成“责任从本体论直接推出”的唯一结果

### 本层可独立成立的部分

- 责任门槛
- 五合同模型
- 复审、熔断、补偿、回滚接口
- 责任逃逸识别

### 本层仍强依赖 `DM` 的部分

- 从最小本体论直接推出规范正当性的强说法
- 用 `DM` 为所有责任裁决提供最终哲学兜底的写法

### 本层防火墙句

> **即使 `DM` 受争议，`TAT` 仍可作为高影响系统的责任门槛与责任闭合框架成立；本层首先由可追溯、可异议、可补偿的程序正当性站住，而不是只由上游本体论站住。**

## 责任分层与闭合度谱系

### 责任分层

为避免把“责任”混成单一口号，TAT 当前固定区分五层：

1. `因果责任`
谁实际造成、共同造成或放任了高影响后果。

2. `角色责任`
谁在组织或制度结构中本应承担预警、复核、审批、熔断、补偿等职责。

3. `认识责任`
谁知道、应知道或有能力知道风险、阈值越界与后果外溢。

4. `制度责任`
谁拥有可追责、可留痕、可熔断、可申诉、可补偿的正式接口。

5. `规范责任`
在前四层被压清之后，才讨论谁应受更高程度的责备、谴责与正当化压力。

TAT 当前直接处理的是：

- `角色责任`
- `认识责任`
- `制度责任`

并把 `因果责任` 作为重要输入，但不把 `制度可追责` 直接偷换成 `道德可责`。

### 责任闭合度谱系

TAT 不再把“责任闭合”表述成所有场景都追求的单一绝对状态，而改用 `闭合度谱系`：

1. `高闭合`
适用于高不可逆伤害、高公共风险、高规模外溢场景。
默认要求责任主体、证据接口、熔断权、补偿承接与复审链条尽量清晰。

2. `中闭合`
适用于一般组织治理与中等风险决策。
允许有限分工与阶段转介，但不能切断审计轨迹与升级路径。

3. `受控模糊`
适用于创新探索、危机快速响应或高不确定早期试探。
允许局部职责先模糊后压实，但必须保留最低护栏。

### 受控模糊的最低护栏

只要进入 `受控模糊`，至少仍要保留：

1. `审计轨迹`
谁做了什么判断、何时升级、何时冻结，必须能回放。

2. `触发阈值`
什么情况下从模糊转向高闭合，不能事后补写。

3. `升级与复审接口`
不能把模糊当成永久挡箭牌，必须允许上提、复核与申诉。

4. `补偿与退出接口`
不能让成本只留在弱势节点，必须写明谁承接损害与撤回成本。

### 明确禁止

以下做法不属于“受控模糊”，而属于 `责任逃逸`：

- 用战略性模糊规避可追责主体
- 用外包链条切断后果承接
- 用“集体决策”抹平个体审批权
- 用高风险试验绕过冻结、申诉与补偿接口

## 五合同模型与程序化门槛

为避免 `TAT` 被误读成只有原则、没有结构，

当前入口固定把本层最小职责压成五类合同：

1. `主体合同`
谁是最终承接者，谁不能通过分包、壳层或角色漂移消失。

2. `接口合同`
哪些高影响动作必须经过责任接口，哪些绕过通道必须被封死。

3. `证据合同`
哪些记录必须留存、固化并可由外部复核。

4. `熔断合同`
谁有权暂停、降级、撤回、冻结与召回。

5. `补偿合同`
事故后谁按什么路径赔付、托底与追偿。

与之配套，

门槛本身也必须程序化，而不能只停留在口头上。

当前最低程序要求是：

1. `谁设门槛`
默认由承担授权、承接与复审责任的主体设定，
不得由只享受扩张收益的一方单独决定。

2. `谁审 R=1`
至少应有不受直接扩张收益驱动的独立复核方参与。

3. `谁能提出异议`
受后果影响、承担补偿、承担审计或承担运维后果的一方，
都应有异议入口。

4. `谁能冻结或回滚`
必须存在清楚的冻结、限制扩张或撤回部署接口。

5. `谁复审门槛设定者`
门槛制定者自身也要接受复核与申诉，
否则门槛结构本身会退化为权力豁免结构。
