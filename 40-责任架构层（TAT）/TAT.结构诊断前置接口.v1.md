# TAT.结构诊断前置接口 v1

> **定位**：本文把 `TAT` 进入责任门槛判断之前，必须接收的 `ASTO` 前置结构诊断压成最小接口。  
> **用途**：防止 `TAT` 直接从事后归责起步，而跳过系统当前处于何种成熟度、节律与操作位点的结构判断。

---

## 一、为什么需要这个前置接口

`TAT` 负责的是责任门槛、责任锚定与责任闭合。

但责任问题并不总是从事故发生那一刻才开始。

很多责任漂移、责任真空与责任割裂，
在更早阶段就已经形成：

- 结构还停留在 `共识态`，却被冒充成已可外放的 `物化态`
- 系统已进入 `流变 / 脉冲`，却仍沿用旧责任接口
- 当前动作明明还在 `设计` 或 `回溯`，却提前宣称责任结构已经闭合

因此，`TAT` 进入正式责任门槛判断前，
应先读取 `ASTO` 的前置结构编码。

---

## 二、最小前置字段

在进入 `R = 0 / R = 1` 判断前，至少填写：

| 字段 | 说明 |
|:---|:---|
| `state_form` | 当前处于 `自在 / 共识 / 编码 / 物化 / 定向` 的哪一态 |
| `dynamic_stage` | 当前处于 `混沌 / 秩序 / 流变 / 脉冲 / 崩解 / 归元` 的哪一阶 |
| `action_step` | 当前主要动作位于 `七序` 的哪一步 |
| `boundary_status` | 当前责任边界是否清晰、可执行、可申诉 |
| `exception_flag` | 是否已触发例外、红线或高阶人工裁决需求 |
| `responsibility_flow_pattern` | 当前责任流向主模式 |

---

## 三、到 TAT 变量的映射

### 1. `state_form`

- `共识 / 编码`：责任结构往往仍偏表面，`r` 不宜高估
- `物化`：可以开始检查责任接口是否已真正不可绕过
- `定向`：应检查系统是否已形成默认外部化倾向

### 2. `dynamic_stage`

- `秩序`：适合检查责任闭合的稳态质量
- `流变`：优先检查责任接口是否随变迁同步升级
- `脉冲 / 崩解`：优先检查熔断权、补偿承接与复审入口

### 3. `action_step`

- `设计 / 物化`：优先检查前置责任架构
- `回溯`：优先检查证据链与责任回接
- `消解`：优先检查退出、交接与后果封存

### 4. `boundary_status`

- `模糊 / 可争议`：`R` 不应轻易视为成立
- `不可绕过`：才有资格继续讨论责任闭合强度

### 5. `responsibility_flow_pattern`

- `下滑 / 蔓生 / 沉积`：优先检查责任是否被默认推向低位承压点
- `引爆`：优先检查事故时的集中归责是否伪装成闭合
- `割裂`：优先检查权责是否被结构性切开

---

## 四、最小输出格式

进入 `TAT` 审议前，
建议先输出一张结构诊断卡：

```text
对象：
state_form：
dynamic_stage：
action_step：
boundary_status：
exception_flag：
responsibility_flow_pattern：

进入 TAT 的理由：
当前最疑似缺口：
是否允许进入 R 判断：
```

---

## 五、最短句

> **`TAT` 不应只问“谁来负责”，还应先问“这个系统现在处于什么结构状态，以至于责任会这样流”。**
