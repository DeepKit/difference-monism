# Progee 英文系统期刊默认执行模板

## 作用

本文件不是最终 venue 锁定书，而是在最终目标尚未拍板前，先把 `Progee` 压成一套更接近“英文系统期刊 + supplement / artifact”路线的默认执行模板。

## 当前默认假设

1. 正文以英文主稿为底稿。
2. 送审使用匿名版。
3. supplement / artifact 只提供最小可复现链，不把整仓直接打包进投稿包。
4. 评估主张只覆盖“状态机治理、证据优先持久化、最小可复现路径”，不额外扩性能宣称。

## 默认交付物

1. `paper/en/Progee-System-Paper.en.md`
   作为英文主稿稳定入口。
2. `paper/blind/Progee-System-Paper.blind.md`
   作为匿名送审入口。
3. `submission/投稿信模板.md`
   作为英文系统期刊路线的基础投稿信。
4. `materials/Artifact-最小执行模板.md`
   作为 supplement / appendix / AE 的最小交付骨架。
5. `materials/Artifact-交付目录草案.md`
   作为对外交付目录与附件打包骨架。
6. `materials/结果与主张映射表.md`
   作为主张、脚本与输出物之间的最短映射层。
7. `materials/审稿人最小复核路径.md`
   作为 AE / 审稿人最短执行路线。
8. `materials/附录与实现锚点表.md`
   作为正文、附录与仓库实现之间的索引。
9. `code/README.md`
   作为代码主位、复现入口与脚本入口说明。

## 默认 artifact 结构

建议后续按以下最小形态整理：

1. `Artifact README`
   说明仓库主位、目标平台、运行边界与输出物。
2. `Environment`
   Windows + PowerShell + PostgreSQL test DB + 已构建测试执行器。
3. `Run`
   运行 `scripts/run_e2e_tests.ps1`，必要时再导出 `eval/*.csv`。
4. `Outputs`
   至少给出 `bin/dunitx-results.xml`；如需表图，再导出 `eval/*.csv`。
5. `Mappings`
   用 `结果与主张映射表` 与 `附录与实现锚点表` 连接正文、代码与输出物。
6. `Review Path`
   用 `审稿人最小复核路径` 指导不同深度的审稿执行。
7. `Boundaries`
   写明 destructive DB tests 的安全保险丝，不伪装成“零配置即可运行”。

## 默认执行顺序

1. 先冻结一个可引用代码版本或提交快照。
2. 再确认 `Replication.md`、`run_e2e_tests.ps1` 与包内说明三处口径一致。
3. 再把 artifact 说明压成“环境前提 + 最小步骤 + 预期输出 + 边界”四段。
4. 最后才按真实 venue 决定是否补 appendix、AE 表单或 DOI。

## 当前最小提交口径

- 我们提交的是一篇可复现系统研究论文，而不是性能竞赛稿。
- artifact 的目标是支撑“关键主张可复核”，不是声称仓库所有路径都已外部稳定复现。
- 若审稿人只执行最小路径，仍应能得到 XML 测试输出，并理解系统评估如何与论文主张对应。
