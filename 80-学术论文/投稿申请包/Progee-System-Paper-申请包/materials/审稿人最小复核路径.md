# Progee 审稿人最小复核路径

## 目标

给审稿人一条由浅入深的最短复核路径，使其不必先理解整仓，仍能判断稿件是否兑现了最小可复现承诺。

## 路线 A：只读补充材料

适用：时间极短，只想判断 supplement 是否自洽。

1. 先看 `结果与主张映射表.md`。
2. 再看 `Artifact-最小执行模板.md`。
3. 最后看 `附录与实现锚点表.md`。

最少应确认：

- 论文主张与输出物存在一一对应关系。
- artifact 边界被明写，没有把仓库能力夸成外部有效性闭合。

## 路线 B：只核复现链

适用：愿意核脚本与运行边界，但不立即执行整套环境。

1. 阅读 `docs/Replication.md`。
2. 检查 `scripts/run_e2e_tests.ps1` 与 `scripts/export_eval_metrics.ps1`。
3. 阅读 `code/README.md` 与本目录下的交付目录草案。

最少应确认：

- 运行前提写清。
- destructive tests 的安全保险丝被明确写出。
- 输出物与论文表图口径可接上。

## 路线 C：最小可运行复核

适用：愿意真正执行一次最小 artifact。

1. 准备 PostgreSQL test DB。
2. 设置 `PROGEE_ALLOW_DESTRUCTIVE_TESTS=1` 等环境变量。
3. 运行 `scripts/run_e2e_tests.ps1`。
4. 如需表图，再运行 `scripts/export_eval_metrics.ps1`。
5. 核对 `bin/dunitx-results.xml` 与 `eval/*.csv`。

最少应确认：

- 测试脚本可运行。
- 输出物确实生成。
- 论文中的最小评估叙述可以在 supplement 中被追索。

## 不应误推的结论

1. 最小复现路径可运行，不等于整仓全部路径均已跨平台稳定复现。
2. 评估输出可导出，不等于外部有效性已经完全闭合。
3. 有 artifact，不等于稿件已转化为性能竞赛或 benchmark 论文。
