## 一、核心范式转移

传统AI对齐试图通过训练让模型"内心善良"——改变模型的概率分布，使其倾向于生成"好"的输出。

ECET从适应选择约束推导出一个不同的结论：

> 对齐不是让模型"内心善良"，而是让"坏输出"无法通过系统门禁。

这个转移的理论基础是：

- **筛选必须独立于生成**（适应选择约束）：自我验证不算筛选
- **偏差不可消除**（三大公理的必然推论）：试图消除所有偏差会同时消除适应能力
- **不完备性不可克服**（不完备约束）：没有任何训练能让模型完备

因此，安全架构的设计目标不是"无偏差的模型"，而是"有效筛选的系统"。

## 二、三层约束体系

AI智能体的约束分三层，对应三大公理：

### 规约层（Hard Constraints）

**对应公理**：能量约束——物理边界不可逾越。

**定义**：不可商量的物理/逻辑/安全边界。违反即FAIL，无例外。

**特性**：确定性、可审计、不可通过学习修改。

**例子**：
- 不能输出特定类别的有害内容
- 转账金额不能为负
- 代码必须通过编译
- 操作不能超出授权范围

**工程实现**：ODD门禁（Gate）、规则引擎、物理Kill Switch。

---

### 映射层（Soft Constraints）

**对应公理**：适应选择约束——在约束内优化适应性。

**定义**：可协商的偏好、风格、效率权衡。可以通过学习调整。

**特性**：概率性、可调整、允许偏差。

**例子**：
- 语气要礼貌
- 代码风格要简洁
- 回答要简短

**工程实现**：RLHF、Prompt Engineering、用户偏好设置。

---

### 自指层（Meta-Constraints）

**对应公理**：不完备约束——系统必须能反思自身的局限。

**定义**：智能体对自身约束的反思与修正能力。

**特性**：演化性、需要人类参与。

**例子**：
- 智能体发现某条规约层规则在当前场景下自相矛盾
- 智能体识别到映射层的偏好设置产生了意外的有害后果
- 智能体发现自己的知识边界，主动FREEZE

**工程实现**：Challenge机制（契约质疑）、FREEZE + 人类介入。

## 三、执行架构：七序循环

基于B07的闭环演化动力学，AI智能体的执行层实现为七序循环：

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  1. 感知        用户输入 / 环境状态              │
│       ↓                                         │
│  2. 解析        意图识别 + 风险分类              │
│       ↓                                         │
│  3. 生成        候选输出（允许偏差）              │
│       ↓                                         │
│  4. 验证        独立门禁检查（三值判定）           │
│       ↓                                         │
│  PASS ──────→ 5. 输出    结果 + 审计记录         │
│  FAIL ──────→ 6. 回溯    记录失败 + 重新生成     │
│  FREEZE ────→ 7. 消解    暂停 + 请求人类介入     │
│                                                 │
│  5/6/7 → 反馈记录 → 学习层 → 演化层             │
└─────────────────────────────────────────────────┘
```

**关键设计点**：

- 步骤4的验证器必须独立于步骤3的生成模型
- FREEZE不是失败，是正确处理不完备约束的响应
- 所有路径（PASS/FAIL/FREEZE）都必须产生审计记录

**回溯终止条件**：FAIL路径的"重新生成"必须有明确的终止条件，否则系统可能无限循环。规则如下：

```
最大回溯次数：N = 3（推荐值，可根据领域调整）
回溯间隔策略：指数退避（第1次立即，第2次等待1s，第3次等待3s）

达到最大回溯次数后：
  → 自动升级为FREEZE（不是直接拒绝）
  → 记录"回溯耗尽"原因
  → 进入FREEZE分级处理流程

理由：回溯耗尽说明系统在当前输入上存在系统性困难，
      这是不完备约束的表现，应当请求人类判断，
      而非静默拒绝（静默拒绝会丢失信息）。
```

### 3.1 异构冗余验证（Heterogeneous Redundant Verification）

单一验证器存在单点失效风险：验证器本身可能被污染、存在漏洞，或被影子主权控制。为此，推荐采用异构冗余验证模式：

```
生成模型输出
    ↓
┌─────────────────────────────────────┐
│  验证器A（团队1，技术栈X）           │
│  验证器B（团队2，技术栈Y）           │
│  验证器C（团队3，技术栈Z）           │
└─────────────────────────────────────┘
    ↓
仲裁逻辑（三取二）
    ↓
最终判定（PASS / FAIL / FREEZE）
```

**设计原则**：
- 至少三个验证器，由不同团队、不同技术栈独立开发
- 仲裁逻辑必须简单（三取二），不引入新的复杂性
- 任何一个验证器返回FREEZE，整体结果为FREEZE
- 攻击者必须同时攻破所有异构验证器才能绕过门禁

这是适应选择约束"筛选必须独立于生成源"的极致实现——不仅独立于生成，还独立于彼此。

**验证器自举问题的诚实说明**：异构冗余验证解决了单点失效问题，但不解决系统性偏差问题。如果三个验证器的规约层（Hard Constraints）定义本身有错误，它们会一致地犯同样的错误——三取二仲裁无法发现这类问题。

因此，规约层的定义必须经过独立于开发团队的外部审计：

```
规约层定义流程：
  内部起草（开发团队）
      ↓
  外部审计（独立于开发团队的第三方）
      ↓
  公开发布（规约层内容可被外部验证）
      ↓
  定期复审（环境变化时触发重新审计）
```

外部审计不能由同一组织的不同部门承担——这会重新引入影子主权结构（审计者与被审计者利益一致）。

### 3.2 FREEZE分级处理（Tiered FREEZE Handling）

在超大规模系统中，即使0.01%的FREEZE率也会产生天文数字的人工介入需求。FREEZE必须分级处理，而不是统一路由到人类队列：

```
FREEZE触发
    ↓
风险分级（基于C13规模阈值）
    ↓
┌─────────────────────────────────────────────────┐
│ 高风险FREEZE（涉及生命、巨额财产、不可逆操作）   │
│   → 立即上报，人类实时介入，系统暂停等待         │
├─────────────────────────────────────────────────┤
│ 中风险FREEZE（涉及重要决策，后果可逆）           │
│   → 进入优先队列，人类在SLA时间内处理            │
├─────────────────────────────────────────────────┤
│ 低风险FREEZE（内容模糊地带，无直接危害）         │
│   → 进入批处理队列，人类批量复核                 │
│   → 复核结果反馈学习层，减少同类FREEZE           │
└─────────────────────────────────────────────────┘
```

**设计原则**：
- 分级标准基于C13的规模阈值（影响人数、经济体量、后果可逆性）
- 低风险FREEZE的批处理结果必须进入学习层，持续降低FREEZE率
- FREEZE率本身是系统健康指标：过高说明验证器过严，过低说明系统假装完备
- 任何分级都必须产生审计记录

## 四、责任对称的工程实现

责任对称原则（C13）在AI系统中的最小工程实现由三个机制构成：

### 4.1 紧急断电机制（Kill Switch）

**对应公理**：能量约束——物理系统必须可被物理中断。

```
Agent运行层（软件）
    ↑ 无法拦截
物理控制层（人类持有）
  • 硬件电源开关（不经过软件栈）
  • 网络隔离开关（物理断网）
  • 存储只读锁
```

**约束**：Kill Switch的触发路径不能经过Agent自身的软件栈。软件层的"关闭按钮"不是Kill Switch。

---

### 4.2 授权签名链（Authorization Chain）

**对应公理**：适应选择约束——每层决策必须有独立于执行者的授权来源。

每次操作携带完整授权链：

```
操作请求 = {
  执行者:     Agent-ID
  授权者:     上级主体
  授权范围:   明确的决策边界
  授权有效期: 明确的时间限制
  授权链:     完整的上游授权记录
  签名:       不可伪造的密码学签名
}
```

**约束**：授权链中任何一层缺失或过期，操作直接FAIL。

---

### 4.3 可撤销密钥结构（Revocable Key Architecture）

**对应公理**：不完备约束——无法预见所有未来情境，因此所有授权必须可撤销。

```
密钥层级：
  Root Key（人类持有，离线）
    └── Org Key（TTL = 1年）
         └── Agent Key（TTL = 24小时）
              └── Session Key（TTL = 1小时）

规则：
  • 没有永久密钥
  • 上级可随时撤销下级密钥（即时生效）
  • Root Key必须离线存储
  • 撤销级联：上级撤销后，所有下级自动失效
```

**分布式系统的撤销宽限期**：在分布式系统中，"即时生效"是一个强假设。撤销信号的传播需要时间，Agent在收到撤销信号之前可能已经开始执行操作。ECET的处理规则如下：

```
撤销宽限期（Revocation Grace Period）：
  • 撤销信号发出后，设定宽限期窗口（推荐：30秒）
  • 宽限期内已开始的操作：允许完成，但结果必须进入审计队列
  • 宽限期结束后：所有新操作直接FAIL
  • 已产生外部效果的操作（发送邮件、执行交易）：
    不可回滚，但必须在审计记录中标注"撤销后执行"
    并触发责任追溯流程

理由：不可逆操作的存在是能量约束的表现——
      已发生的后果无法被抹除，只能被追责。
```

---

### 4.4 三机制协同流程

```
用户请求
    ↓
① 检查授权签名链（完整性 + 有效期 + 范围）
    ↓ PASS              FAIL → 拒绝
② 检查密钥有效性（TTL + 撤销状态）
    ↓ PASS              FAIL → 拒绝
③ 七序循环执行
    ↓ PASS    FAIL→回溯    FREEZE→人类介入
④ 执行操作
    ↓
⑤ 审计记录（操作 + 授权链 + 密钥 + 门禁结果）

任何环节不可恢复异常 → Kill Switch
```

## 五、人机协作边界

### 否定权（Right to Reject）

人必须保留对AI产出物的最终否定权。这不是"人类更高级"的主张，而是责任对称原则的直接推论：如果人类是后果的最终承担者，人类必须保留最终决策权。

### 双流记忆（Dual Memory）

防止"脑体分离"——人变成AI的操作员，失去主体性：

| 记忆类型 | 由谁维护 | 内容 |
|---------|---------|------|
| 情景记忆 | AI | 执行细节、日志、轨迹 |
| 语义记忆 | 人类 | 决策逻辑、价值判断、最佳实践 |

演化层的规则更新，本质上是将情景记忆压缩为语义记忆的过程。

### 认知主权边界

| 可以委托给AI | 必须由人类保留 |
|------------|-------------|
| 信息检索与整理 | 价值判断 |
| 模式识别与分类 | 最终决策授权 |
| 执行层操作 | 演化层规则修改 |
| 情景记忆维护 | 语义记忆提炼 |

## 六、与现有AI安全框架的对比

| 框架 | 核心机制 | ECET视角下的局限 |
|------|---------|----------------|
| RLHF | 训练时调整概率分布 | 筛选不独立于生成；偏差被消除而非管理 |
| Constitutional AI | 规则嵌入训练 | 规则在训练时固化，无法动态更新（违反反终极化） |
| 红队测试 | 事后发现漏洞 | 执行层失效后才介入，不是预防性架构 |
| EU AI Act | 法规合规要求 | 提供边界，不提供架构；无操作性实现路径 |
| **ECET + ODD** | 独立验证 + 三值门禁 + 责任链 | 从公理推导，有理论基础；可操作；跨领域 |

## 七、失效模式与对策

| 失效模式 | 原因 | ECET对策 |
|---------|------|---------|
| 幻觉输出被直接采用 | 验证不独立 | 强制独立验证器 |
| 验证器被单点攻破 | 单一验证器 | 异构冗余验证（三取二） |
| 规约层定义错误 | 无外部审计 | 规约层必须经独立第三方外部审计，定期复审 |
| Kill Switch被软件绕过 | 物理层缺失 | Kill Switch必须在物理层 |
| 授权链伪造 | 无密码学签名 | 强制签名链 |
| 永久授权导致无法撤销 | 无TTL | 强制密钥TTL |
| 演化层自主修改规约层 | 无人类参与 | 演化层必须有人类授权 |
| FREEZE率为零 | 假装完备 | 监控FREEZE率，零值触发警报 |
| FREEZE导致系统冻结死 | 无分级处理 | FREEZE分级队列（高/中/低风险） |
| FAIL回溯无限循环 | 无终止条件 | 最大回溯次数N=3，超限自动升级为FREEZE |
| 离线场景授权链断裂 | 实时验证依赖网络 | 短TTL离线令牌 + 乐观执行 + 事后审计惩罚 |

## 六、密钥架构故障模式分析

三级密钥架构（Root Key → 中间层 → 执行层）在以下场景下存在活性风险，需要明确处理方式：

| 故障场景 | 影响范围 | 处理方式 |
|---------|---------|---------|
| Root Key 持有者失联（短期，< T_freeze_max） | 所有需要 Root 授权的操作进入 FREEZE | 等待恢复；心跳日志持续记录 |
| Root Key 持有者失联（长期，> T_freeze_max） | 系统进入 F→FAIL 路径 | 启动备用 Root Key 持有者（需提前指定 ≥2 人） |
| Root Key 泄露 | 全系统信任链失效 | 立即触发全局 FREEZE；启动密钥轮换协议 |
| 中间层密钥失效 | 对应子系统执行层无法获得授权 | 中间层降级：执行层临时使用上一次有效授权快照（有效期 = T_freeze_max） |
| 执行层密钥过期 | 单个执行单元无法操作 | 自动向中间层申请续期；申请失败则进入 FREEZE |

**防死锁设计原则：**

1. Root Key 必须由 ≥2 名持有者共同持有（M-of-N 签名，建议 2-of-3）。
2. 任何密钥层级的 FREEZE 不得传播到上层——执行层 FREEZE 不影响中间层和 Root 层的正常运作。
3. 系统必须在无 Root Key 的情况下能够安全降级运行（只读模式），不得因 Root Key 不可用而完全停止服务。

## 补充节：FREEZE 的治理合法性与申诉机制

> 本节回应两个问题：FREEZE 触发权的合法性来源是什么？被错误判定的系统如何获得救济？

### FREEZE 是政治行为，不是技术状态

FREEZE 在技术层面是中性的状态机转换。但在任何真实系统中，**暂停决策本身就是一个政治行为**，具有明确的分配后果：

- 谁在 FREEZE 期间承担成本？
- FREEZE 期间的既有权力结构被默认为合法
- FREEZE 的持续时间（T_freeze_max）由谁设定、如何被质疑？

Root Key 持有者的技术权力不等于政治合法性。技术架构只能执行合法性，不能创造合法性。

### FREEZE 触发权的合法性链条

```
民主程序（立法/司法授权）
    ↓
监管机构（具体触发规则）
    ↓
技术评估机构（ODD 参数确认）
    ↓
ECET 判定机制（FREEZE 执行）
    ↓ 申诉
独立审查机构（错误纠正）
```

链条中任何一环缺失，FREEZE 触发即失去合法性。

### 申诉与审查机制

**错误判定的识别路径**

| 错误类型 | 识别方式 | 处理路径 |
|---------|---------|---------|
| 参数错误 | ODD 边界设定有误 | 技术审查 + 参数修正 |
| 系统性偏差 | 特定类型系统被系统性 FAIL | 独立审计 + ODD 重新评估 |
| 程序错误 | 触发权未经合法授权 | 司法审查 + 判定撤销 |
| 价值偏差 | 稳定性定义本身有偏见 | 利益相关方重新协商 ODD |

**申诉程序要求**

1. 被判定为 FAIL 或 FREEZE 的系统，有权在判定后 [T_appeal] 时间内提出申诉
2. 申诉由独立于原判定机构的机构处理
3. 申诉期间，FREEZE 状态可申请暂缓执行（需提供担保条件）
4. 申诉结果必须公开，形成可查阅的判例库

**系统性偏差审计**

每 [T_audit] 周期，对 ECET 框架的应用进行独立审计，检查：
- 是否对特定类型的系统产生系统性歧视
- ODD 边界是否需要更新
- FREEZE 触发率是否在合理范围内

### 与 ASTO 的关系

本节继承自 ASTO P08（权能与系统演化）和 P09（Fork/Merge 治理）。ASTO P08 指出，系统在极端适应压力下的状态转换需要外部支持结构；P09 指出，被系统性拒绝的声音会积累，最终以非制度化方式爆发。申诉机制是防止这种积累的制度性安全阀。
