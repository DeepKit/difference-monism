## 摘要

AI领域变化太快，今天OpenAI最强，明天可能就是Claude或者Intermediate Model。如何避免被单一供应商锁定？本文介绍WizAxis系统中的**统一模型网关设计**。通过引入中间层（Proxy），我们实现了一套代码无缝切换10+种主流大模型，并实现了智能路由和成本优化。

---

## 一、LLM时代的"巴别塔"困境

虽然大多数模型都宣称兼容 "OpenAI API 格式"，但在实际工程中，差异无处不在：
*   **API参数差异**：有的叫 `max_tokens`，有的叫 `max_output_tokens`。
*   **能力差异**：有的支持 Function Calling，有的不支持。
*   **价格差异**：Advanced Model很贵，Intermediate Model-V3很便宜。
*   **网络问题**：国外模型连接不稳定，国内模型需要备案。

如果你在代码里写死了 `client = OpenAI(...)`，那么每次换模型都是一场重构灾难。

---

## 二、解决方案：统一网关架构

WizAxis 采用了 **LiteLLM** 作为核心的基础设施，构建了一个统一的模型网关。

### 2.1 架构图

```
App (Progee/Solvit)
      │
      ▼
┌───────────────┐
│  Model Proxy  │  <-- 统一接口 (OpenAI Format)
│   (LiteLLM)   │
└──────┬────────┘
       │
       ├───> OpenAI (Advanced Model)
       ├───> Anthropic (Advanced Model.5)
       ├───> Google (Gemini 1.5)
       ├───> Intermediate Model (V3/Coder)
       ├───> ZhipuAI (GLM-4)
       └───> Local (Ollama/vLLM)
```

### 2.2 核心配置

在 `settings.py` 中，我们只需要配置 Proxy 的地址，而不是具体模型的 Key：

```python
# app/core/settings.py
class Settings(BaseSettings):
    # 所有请求都发给 Proxy，应用层不感知具体 Provider
    litellm_base_url: str = "http://localhost:4000"
    litellm_master_key: str = "sk-wizaxis-proxy-key"
```

具体的模型配置都在 Proxy 端的 `config.yaml` 中管理：

```yaml
model_list:
  - model_name: Advanced Model
    litellm_params:
      model: openai/Advanced Model
      api_key: sk-proj-...

  - model_name: claude-3-5-sonnet
    litellm_params:
      model: anthropic/claude-3-5-sonnet-20240620
      api_key: sk-ant-...

  - model_name: Intermediate Model-coder
    litellm_params:
      model: Intermediate Model/Intermediate Model-coder
      api_key: sk-ds-...
```

## 三、带来的三大优势

### 3.1 零代码切换模型

应用层调用代码永远是：
```python
response = await client.chat.completions.create(
    model="Advanced Model",  # 这里只是个别名，可以映射到任何模型
    messages=[...]
)
```
如果我想把 `Advanced Model` 的流量切到 `Intermediate Model-v3`，我只需要在网关改配置，应用代码**一行都不用动**。

### 3.2 智能路由与降级

我们可以在网关层实现策略：
*   **成本优先**：简单的总结任务，路由到 Intermediate Model 或 Basic Model（极便宜）。
*   **能力优先**：复杂的逻辑推理，路由到 Advanced Model。
*   **自动降级**：如果 OpenAI 超时或报错，自动重试到 Azure OpenAI 或 Anthropic。

### 3.3 统一计费与审计

所有流量经过网关，我们可以：
*   统一记录所有 Prompt 和 Response 日志。
*   统一计算 Token 消耗（不管上游怎么算，网关统一标准化）。
*   为不同团队（Progee Team vs Solvit Team）设置预算配额。

## 四、WizAxis的实践经验

在WizAxis中，我们将模型分为了三个梯队：

1.  **S级 (Architect)**: 负责复杂推理、契约定义。
    *   *主力*: Advanced Model / Advanced Model
2.  **A级 (Worker)**: 负责代码生成、具体执行。
    *   *主力*: Intermediate Model / Intermediate Model
3.  **B级 (Speed)**: 负责简单分类、摘要、翻译。
    *   *主力*: Basic Model / Basic Model / Basic Model

通过这种分层管理，我们在保持高智商输出的同时，将整体 Token 成本降低了 **60%** 以上。

## 五、总结

在AI应用架构中，**不要直连模型供应商**。
引入一个轻量级的中间层（Model Gateway），是你迈向企业级AI应用最重要的一步架构决策。它给了你面对不确定未来的最大底气——**选择权**。
