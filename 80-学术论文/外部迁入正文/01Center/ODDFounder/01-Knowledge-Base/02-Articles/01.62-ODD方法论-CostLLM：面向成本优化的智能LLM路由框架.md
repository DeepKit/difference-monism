## 摘要

随着组织部署多种能力和成本各异的大型语言模型，一个关键挑战浮现：**如何为每个任务自动选择最优模型**。对简单查询使用Advanced Model等昂贵模型浪费50-90%成本，而对复杂任务使用廉价模型导致质量下降和重试成本。本文介绍**CostLLM**，一个智能路由框架，自动评估任务复杂度并选择成本最优模型。

核心贡献包括：(1) **双层评估架构**，结合零成本快速评估（<5ms）与精确LLM评估；(2) **多因子复杂度评分**算法，考虑消息模式、关键词、领域和历史数据；(3) **自学习反馈环**，持续改进路由决策；(4) **成本感知熔断器**，防止预算超支。在WizAxis平台部署后，CostLLM实现了**67%成本降低**，同时保持96%的质量对等。

## 1. 引言

### 1.1 模型选择困境

LLM市场选择爆炸：Advanced Model Turbo（$0.01/1K输入）、Claude-3 Advanced Model（$0.015/1K）、Basic Model Turbo（$0.0005/1K）、Intermediate Model（$0.0001/1K）。每个模型在能力、延迟和成本间有不同权衡。

**朴素方法的失败**：
- **总用最好的模型**：简单任务浪费50-90%预算
- **总用最便宜的模型**：质量受损，重试成本累积
- **手动任务分类**：不可扩展，易出错

| 任务 | 最优模型 | 用Advanced Model成本 | 最优成本 | 节省 |
|------|----------|-------------|----------|------|
| "2+2等于几？" | Intermediate Model | $0.01 | $0.0001 | 99% |
| "翻译'hello'" | Basic Model | $0.01 | $0.0005 | 95% |
| "设计微服务架构" | Advanced Model | $0.03 | $0.03 | 0% |

**核心洞察**：任务复杂度差异巨大，但模型选择往往是静态的。

### 1.2 研究问题

1. **如何自动评估任务复杂度**而不需昂贵的LLM调用？
2. **如何将复杂度映射到最优模型选择**，考虑成本、质量和延迟？
3. **如何通过反馈持续改进路由决策**？

## 3. 双层复杂度评估

### 3.1 快速评估：零成本

快速评估使用规则启发式估计复杂度，无需LLM调用：

```python
class FastEval:
    """零成本复杂度评估，延迟<5ms"""
    
    def evaluate(self, request: Request) -> EvalResult:
        scores = []
        
        # 因子1: 消息数量 (权重: 0.15)
        msg_score = self._eval_message_count(request.messages)
        scores.append(msg_score * 0.15)
        
        # 因子2: 消息长度 (权重: 0.20)
        len_score = self._eval_message_length(request.messages)
        scores.append(len_score * 0.20)
        
        # 因子3: 关键词提示 (权重: 0.30)
        kw_score = self._eval_keywords(request.messages)
        scores.append(kw_score * 0.30)
        
        # 因子4: 历史相似度 (权重: 0.20)
        hist_score = self._eval_historical(request)
        scores.append(hist_score * 0.20)
        
        # 因子5: 领域检测 (权重: 0.15)
        dom_score = self._eval_domain(request.messages)
        scores.append(dom_score * 0.15)
        
        complexity = sum(scores)
        confidence = self._aggregate_confidence()
        
        return EvalResult(
            complexity_score=complexity,
            confidence=confidence,
            recommended_model=self._map_to_model(complexity)
        )
```

### 3.3 精确评估：LLM辅助

当快速评估置信度低于70%时，调用精确评估：

```python
class PreciseEval:
    """LLM复杂度评估，成本~$0.001/次"""
    
    EVAL_PROMPT = """
    分析以下任务的复杂度（1-10分）：
    
    任务：{messages}
    
    考虑：推理深度、领域专业性、输出复杂度、歧义程度
    
    JSON输出：{"complexity": <1-10>, "tier": "basic|standard|advanced|expert"}
    """
    
    def evaluate(self, request: Request) -> EvalResult:
        response = self.llm.call(
            model="Basic Model-turbo",  # 用便宜模型评估
            messages=[{"role": "user", "content": self.EVAL_PROMPT}],
            max_tokens=100
        )
        result = json.loads(response.content)
        return EvalResult(
            complexity_score=result["complexity"],
            confidence=0.90
        )
```

def select_model(
    complexity: float,
    budget_status: str,   # "sufficient" | "tight" | "critical"
    user_tier: str,       # "free" | "pro" | "enterprise"
    latency_req: str      # "normal" | "fast" | "realtime"
) -> str:
    tier = get_tier_from_complexity(complexity)
    
    if budget_status == "critical":
        strategy = "cost_first"
    elif user_tier == "enterprise":
        strategy = "quality_first"
    else:
        strategy = "balanced"
    
    return MODEL_TIERS[tier]["models"][strategy]

class FeedbackCollector:

            "token_efficiency": "Token使用效率",

    def collect(self, request_id: str) -> Feedback:
        explicit = self._collect_explicit(request_id)
        implicit = self._collect_implicit(request_id)

class RuleUpdater:
    """基于反馈更新路由规则"""
    
    def update(self, feedbacks: List[Feedback]):
        # 分析失败案例
        failures = [f for f in feedbacks if f.quality_score < 0.7]
        
        for failure in failures:
            original = self.get_routing_decision(failure.request_id)

## 6. 成本感知熔断器

### 6.1 预算保护

```python
class CostCircuitBreaker:
    """防止预算超支的熔断器"""
    
    def __init__(self, config: BudgetConfig):
        self.daily_budget = config.daily_budget
        self.hourly_limit = config.hourly_limit
        self.state = "closed"
    
    def check(self, estimated_cost: float) -> Decision:
        current_daily = self.get_daily_spend()
        current_hourly = self.get_hourly_spend()
        
        # 检查日预算
        if current_daily + estimated_cost > self.daily_budget * 0.9:
            return Decision(
                allowed=False,
                reason="approaching_daily_limit",
                fallback_model="Intermediate Model-chat"
            )
        
        # 检查小时限制
        if current_hourly > self.hourly_limit:
            self.state = "open"
            return Decision(
                allowed=False,
                reason="hourly_limit_exceeded",
                retry_after=self._next_hour()
            )
        
        return Decision(allowed=True)
```

### 6.2 降级策略

| 预算状态 | 策略 | 行为 |
|----------|------|------|
| 充足 (< 70%) | 正常 | 按复杂度选择最优模型 |
| 紧张 (70-90%) | 保守 | 优先选择成本较低模型 |
| 临界 (> 90%) | 降级 | 强制使用最便宜模型 |
| 超支 | 熔断 | 拒绝请求或排队 |

- **自学习**：持续改进路由决策
- **67%成本降低**：显著节省运营成本
