## Abstract

Large Language Models (LLMs) face a fundamental constraint: finite context windows that limit their ability to maintain coherent long-term interactions. While context windows have expanded from 4K to 200K tokens, real-world agent systems serving thousands of users over months still encounter the "teacup trying to hold an ocean" problem. This paper introduces **SlideFold**, a multi-level sliding window compression architecture that theoretically supports unlimited compression levels while preserving critical information. Our approach combines three key innovations: (1) **N-Level Sliding Compression** with configurable thresholds at each level, (2) **Hot/Warm/Cold Tiered Storage** for cost-effective memory access, and (3) **Episodic-to-Semantic Transformation** that converts specific events into abstract knowledge. Experimental results on the WizAxis platform demonstrate 73% token reduction while maintaining 94% information retention rate. SlideFold provides a practical, production-ready solution for building "infinite memory" systems within finite context constraints.

## 1. Introduction

### 1.1 The Context Window Bottleneck

When discussing the capability boundaries of Large Language Models, attention often focuses on parameter scale, training data, and reasoning ability. However, in real-world agent systems, a more subtle yet critical bottleneck exists: **the finite nature of context windows**.

Advanced Model's 128K tokens, Claude's 200K tokens—these numbers appear substantial. But in an agent system running continuously for months and serving tens of thousands of users, they become inadequate containers trying to hold an ocean of information.

Consider a typical scenario:
- A conversation lasting 100 turns
- Average 500 tokens per turn
- Total: 50,000 tokens
- This exceeds most models' effective context windows
- Even if within limits, the cost becomes prohibitive

This leads to our core research question: **How can we build an "infinite" memory system within finite context windows?**

### 1.2 From Chat History to Memory System

Traditional dialogue systems require only a simple `chat_history` array storing messages chronologically. Agent systems, however, need an **actively managed knowledge system**.

| Dimension | Chat History | Agent Memory |
|-----------|--------------|--------------|
| Organization | Linear time sequence | Hierarchical, classified, indexable |
| Intelligence | None (pure storage) | Active (summarization, compression, forgetting) |
| Lifecycle | Disappears after session | Cross-session persistence |
| Core Question | "What happened?" | "What does this mean?" |

This transformation represents a fundamental shift: **memory evolves from passive logging to active cognitive storage**.

### 1.3 Contributions

This paper makes the following contributions:

1. **N-Level Sliding Window Architecture**: A theoretically unlimited compression hierarchy where each level applies configurable compression ratios
2. **Tiered Storage Model**: Hot/Warm/Cold classification optimizing access patterns and costs
3. **Episodic-to-Semantic Transformation**: Automatic conversion of specific events to abstract knowledge
4. **Production Implementation**: Validated architecture deployed on WizAxis platform

Recent work has focused on extending context windows through architectural modifications. Longformer (Beltagy et al., 2020) introduced sparse attention patterns. ### 2.3 Summarization-Based Approaches

Recursive summarization (Wu et al., 2021) and hierarchical summarization (Liu et al., 2023) compress conversations through LLM-generated summaries. SlideFold extends these approaches with **multi-level sliding windows** and **type-aware compression strategies**.

## 3. SlideFold Architecture

### 3.1 Design Principles

SlideFold is built on three core principles:

1. **Compression over Deletion**: Preserve information through transformation, not removal
2. **Recency Bias**: Recent information is more valuable than old information
3. **Type-Aware Processing**: Different content types require different compression strategies

### 3.2 N-Level Sliding Window Model

The core innovation of SlideFold is its N-level compression hierarchy:

```
Level 0 (Raw):     [T1][T2][T3][T4][T5][T6][T7][T8][T9][T10]...
                          ↓ (threshold: 5 turns)
Level 1 (Summary): [S1: T1-T5 summary][T6][T7][T8][T9][T10]...
                          ↓ (threshold: 3 summaries)
Level 2 (Meta):    [M1: S1-S3 meta-summary][S4][T9][T10]...
                          ↓ (threshold: N summaries)
Level N:           [Ultra-compressed knowledge nugget]
```

**Key Properties**:
- Each level has independent threshold and compression ratio
- Compression is triggered when threshold is exceeded
- Theoretically unlimited levels (practical limit: 5-7 levels)
- Information density increases at higher levels

### 3.3 Compression Trigger Conditions

```python
class CompressionTrigger:
    def __init__(self, config: CompressionConfig):
        self.turn_threshold = config.turn_threshold      # e.g., 20 turns
        self.token_threshold = config.token_threshold    # e.g., 0.7 (70%)
        self.context_window = config.context_window      # e.g., 128000
    
    def should_compress(self, memory_state: MemoryState) -> CompressionLevel:
        # Condition 1: Turn count exceeds threshold
        if len(memory_state.turns) > self.turn_threshold:
            return CompressionLevel.SLIDING_WINDOW
        
        # Condition 2: Token usage approaching saturation
        usage_ratio = memory_state.total_tokens / self.context_window
        if usage_ratio > self.token_threshold:
            return CompressionLevel.AGGRESSIVE
        
        # Condition 3: Multi-level cascade
        for level in range(1, memory_state.max_level):
            if memory_state.count_at_level(level) > self.level_threshold(level):
                return CompressionLevel.CASCADE
        
        return CompressionLevel.NONE
```

**Two Critical Thresholds**:
- **Turn Threshold** (e.g., 20 turns): Triggers sliding window compression
- **Token Threshold** (e.g., 70%): Triggers aggressive compression

### 3.4 Sliding Window Algorithm

```python
def sliding_window_compress(
    memory: List[MemoryUnit],
    window_size: int = 5,
    preserve_recent: int = 3
) -> List[MemoryUnit]:
    """
    Compress oldest entries while preserving recent ones.
    
    Args:
        memory: List of memory units at current level
        window_size: Number of units to compress together
        preserve_recent: Number of recent units to keep uncompressed
    
    Returns:
        Compressed memory list
    """
    if len(memory) <= preserve_recent + window_size:
        return memory  # Not enough to compress
    
    # Separate: [to_compress | preserved_recent]
    to_compress = memory[:-preserve_recent]
    preserved = memory[-preserve_recent:]
    
    # Compress oldest window_size units
    compress_window = to_compress[:window_size]
    remaining = to_compress[window_size:]
    
    # Generate summary using LLM
    summary = llm.summarize(
        content=[unit.content for unit in compress_window],
        strategy=get_strategy(compress_window),
        max_tokens=calculate_target_tokens(compress_window)
    )
    
    # Create compressed unit at next level
    compressed_unit = MemoryUnit(
        level=compress_window[0].level + 1,
        content=summary,
        source_ids=[unit.id for unit in compress_window],
        timestamp_range=(compress_window[0].timestamp, compress_window[-1].timestamp),
        compression_ratio=len(compress_window)
    )
    
    return [compressed_unit] + remaining + preserved
```

### 3.5 Folding Mechanism

The "Fold" in SlideFold refers to the recursive nature of compression:

```
Initial State (10 raw turns):
[R1][R2][R3][R4][R5][R6][R7][R8][R9][R10]

After First Fold (compress R1-R5):
[S1][R6][R7][R8][R9][R10]

After Second Fold (compress R6-R10):
[S1][S2]

After Third Fold (compress S1-S2):
[M1]

Each fold reduces token count by ~60-80%
```

### 4.2 Context Loading Strategy

```python
def load_context_with_budget(
    query: str,
    token_budget: int = 4000
) -> List[ContextUnit]:
    """
    Load context respecting token budget with tiered priority.
    """
    context = []
    used_tokens = 0
    
    # Priority order for loading
    priorities = [
        ("system_prompt", 500),      # Fixed allocation
        ("hot_tier", 1500),          # Recent turns (full)
        ("warm_tier", 1000),         # Summaries
        ("cold_tier", 1000),         # Semantic retrieval
    ]
    
    for source, max_tokens in priorities:
        content = fetch_tier(source, query, max_tokens)
        
        if used_tokens + count_tokens(content) > token_budget:
            # Budget exceeded: aggressive compression
            content = aggressive_summarize(
                content, 
                target_tokens=token_budget - used_tokens
            )
        
        context.append(content)
        used_tokens += count_tokens(content)
        
        if used_tokens >= token_budget:
            break
    
    return context
```

## 5. Type-Aware Compression Strategies

### 5.1 Content Type Classification

Different conversation types require different compression strategies:

| Content Type | Compression Strategy | Retention Period | Example |
|--------------|---------------------|------------------|---------|
| Casual Chat | Extract emotion/preference changes | 7 days | "User was in good mood today" |
| Task Execution | Preserve decision chain + tool calls | Archive after completion | "Steps 1-5 completed, results..." |
| Technical Discussion | Extract knowledge + user skill level | Permanent | "User proficient in Python, interested in Rust" |
| Sensitive Information | Immediate summary, delete original | Per privacy policy | "User provided payment info (processed)" |

## 6. Episodic-to-Semantic Transformation

### 6.1 The Transformation Concept

This is the most sophisticated aspect of SlideFold. **Episodic memory records specific events; semantic memory stores abstract knowledge.**

```python
# Episodic Memory (specific event)
episodic = {
    "type": "episodic",
    "content": "On 2024-12-15, user asked how to configure Nginx reverse proxy. "
               "I provided complete configuration. User expressed satisfaction.",
    "timestamp": "2024-12-15T10:30:00Z",
    "tokens": 156
}

# Transformed to Semantic Memory (abstract knowledge)
semantic = {
    "type": "semantic",
    "content": "User is familiar with Nginx configuration and has practical "
               "needs for reverse proxy setup.",
    "confidence": 0.85,
    "source_episodes": ["episodic_memory_xxx"],
    "tokens": 24
}
```

**Key Insight**: This transformation preserves critical information (user skill profile) while discarding unnecessary details (specific time, full conversation).

### 6.2 Transformation Algorithm

```python
def transform_episodic_to_semantic(
    episodes: List[EpisodicMemory],
    min_episodes: int = 3
) -> Optional[SemanticMemory]:
    """
    Transform multiple related episodic memories into semantic knowledge.
    
    Requires minimum episodes to establish pattern confidence.
    """
    if len(episodes) < min_episodes:
        return None
    
    # Extract common patterns
    patterns = extract_patterns(episodes)
    
    # Generate semantic summary
    semantic_content = llm.generate(
        prompt=EPISODIC_TO_SEMANTIC_PROMPT,
        context={
            "episodes": [e.content for e in episodes],
            "patterns": patterns,
            "time_span": calculate_time_span(episodes)
        }
    )
    
    # Calculate confidence based on consistency
    confidence = calculate_pattern_confidence(episodes, patterns)
    
    return SemanticMemory(
        content=semantic_content,
        confidence=confidence,
        source_episodes=[e.id for e in episodes],
        created_at=datetime.now(),
        memory_type="derived_semantic"
    )
```

def extract_patterns(episodes: List[EpisodicMemory]) -> List[Pattern]:

### 7.2 Compression Pipeline

```python
class CompressionPipeline:
    def __init__(self, config: PipelineConfig):
        self.trigger = CompressionTrigger(config)
        self.strategy_factory = CompressionStrategyFactory()
        self.transformer = EpisodicSemanticTransformer()
        self.storage = TieredStorage(config)
    
    async def process(self, memory_state: MemoryState) -> MemoryState:
        """
        Main compression pipeline.
        """
        # Check if compression needed
        compression_level = self.trigger.should_compress(memory_state)
        
        if compression_level == CompressionLevel.NONE:
            return memory_state
        
        # Apply appropriate compression
        if compression_level == CompressionLevel.SLIDING_WINDOW:
            memory_state = await self._sliding_compress(memory_state)
            
        elif compression_level == CompressionLevel.AGGRESSIVE:
            memory_state = await self._aggressive_compress(memory_state)
            
        elif compression_level == CompressionLevel.CASCADE:
            memory_state = await self._cascade_compress(memory_state)
        
        # Check for episodic-to-semantic opportunities
        memory_state = await self._check_semantic_transform(memory_state)
        
        # Update storage tiers
        await self.storage.rebalance(memory_state)
        
        return memory_state
    
    async def _cascade_compress(self, state: MemoryState) -> MemoryState:
        """
        Multi-level cascade compression.
        """
        for level in range(state.max_level):
            units_at_level = state.get_units_at_level(level)
            
            if len(units_at_level) > self.trigger.level_threshold(level):
                # Compress and promote to next level
                compressed = await self._compress_level(units_at_level, level)
                state.replace_units(units_at_level, compressed)
        
        return state
```

## 9. Discussion

### 9.1 Optimal Level Configuration

Our experiments suggest the following guidelines:

- **Levels 1-2**: Sufficient for most applications (80% of use cases)
- **Levels 3-4**: Recommended for long-running agents (months of history)
- **Levels 5+**: Rarely needed; diminishing returns after level 4

### 9.2 Content Type Distribution

Analysis of 45,892 sessions revealed:
- Casual Chat: 45%
- Task Execution: 30%
- Technical Discussion: 20%
- Sensitive Information: 5%

Type-aware compression improved overall retention by 8% compared to uniform strategies.

### 9.3 Limitations

1. **Compression Latency**: Each level adds ~50-100ms latency
2. **LLM Dependency**: Quality depends on summarization model capability
3. **Information Loss**: Inevitable at higher compression levels
4. **Cold Start**: New users lack compression history

### 9.4 Future Work

1. **Learned Compression**: Train specialized compression models
2. **User-Adaptive Thresholds**: Personalize compression based on user patterns
3. **Cross-User Knowledge**: Share semantic knowledge across users (with privacy)
4. **Streaming Compression**: Real-time compression during conversation

## 10. Conclusion

SlideFold presents a practical solution to the fundamental challenge of managing LLM context within finite windows. By combining multi-level sliding window compression, tiered storage, and episodic-to-semantic transformation, we achieve 73% token reduction while maintaining 94% information retention.

The key insight is that **compression is not about losing information—it's about distilling information**. Just like human memory, we don't remember every detail, but we remember the patterns, lessons, and wisdom that shape our understanding.

SlideFold is deployed in production on the WizAxis platform, serving over 1,200 users daily. The architecture is open for adoption and extension by the research community.
