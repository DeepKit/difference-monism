# 04.07 多租户计费：跨应用边界的访问控制设计

> **作者**: Yi Fu ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-11
> **标签**: 多租户, SaaS, BPL, 权限控制, Billing

---

## 摘要

SaaS 产品的核心是**多租户 (Multi-Tenancy)**。一个企业客户可能包含 50 个账号，他们共享一个钱包，共享一套权限策略。**Billing** 系统通过 **BPL (Business Policy Logic)** 插件架构，实现了灵活的跨应用边界控制，让"企业版"的权限管理变得简单。

---

## 一、什么是 BPL？

传统的权限控制是硬编码的：`if user.is_admin then allow`。
但在 SaaS 中，规则千奇百怪：
*   "A公司允许员工使用 Advanced Model，但每人每天限 10 次。"
*   "B公司禁止员工在周末使用系统。"
*   "C公司只允许特定 IP 段访问。"

Billing 引入了 **BPL (Business Policy Logic)**，类似于 AWS IAM Policy。

---

## 二、策略模型

```json
{
  "Version": "2026-01-01",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["llm:generate"],
      "Resource": ["model:Advanced Model"],
      "Condition": {
        "NumericLessThan": { "usage:daily_count": 10 }
      }
    },
    {
      "Effect": "Deny",
      "Action": ["*"],
      "Condition": {
        "DateWeekend": true
      }
    }
  ]
}
```

### 2.1 动态求值
每次请求到达 Billing 时，网关会加载当前租户的所有 Policy，进行动态求值。
这个过程必须极快（< 1ms），因此我们在内存中缓存了编译后的策略树。

---

## 三、租户隔离与继承

### 3.1 组织架构
*   **Root Organization**: 集团。
*   **Department**: 部门。
*   **Team**: 小组。

### 3.2 策略继承
子组织自动继承父组织的策略（通常是取交集或优先 Deny）。
这使得管理员可以在 Root 层级配置"全员禁止使用 DALL-E"，所有下级部门自动生效。

---

## 四、跨应用边界

Billing 是一个独立服务，Progee 和 Solvit 是它的 Client。
如何保证 Progee 的用户不能用 Solvit 的额度？

### 4.1 App Context
请求中携带 `App-ID` 和 `App-Secret` 签名。
Billing 验证签名后，加载该 App 的专属策略。

### 4.2 资源隔离
在 Policy 中，可以限制资源范围：
`"Resource": ["app:progee:*"]`。
这确保了 Token 泄露也不会导致跨应用的盗刷。

---

## 五、总结

多租户计费的本质是**"资源归属权"**与**"访问使用权"**的分离。
通过 BPL 策略引擎，Billing 系统将复杂的业务规则从代码中剥离出来，变成了可配置的数据。这不仅满足了企业客户的定制化需求，也让系统的扩展性得到了质的飞跃。
