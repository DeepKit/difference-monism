# 01.68 缓存预热策略：LLM应用的冷启动优化

> 为什么AI第一次回复总是慢？因为它在"热身"。学会预热，让AI随时在线。

---

## 一、冷启动的痛

你有没有注意到：

```
第1次对话：AI需要3秒才回复
第2次对话：AI只需要1秒
第3次对话：AI秒回
```

这就是**冷启动问题**。

第一次对话时，AI需要：
- 加载用户上下文
- 检索相关知识
- 构建完整提示词
- 等待模型响应

后续对话因为有了缓存，所以快很多。

**问题**：用户最在意的恰恰是第一次体验！

## 二、冷启动 vs 热启动

| 类型 | 场景 | Token消耗 | 延迟 |
|------|------|-----------|------|
| 冷启动 | 新会话/新用户 | ~6000 | 3-5秒 |
| 热启动 | 连续对话 | ~2000 | 1-2秒 |

**差距**：Token消耗3倍，延迟2-3倍！

## 四、策略1：登录时预热

用户登录的那几秒，正好用来预热：

```python
async def on_user_login(user_id):
    # 用户看到的：加载中...
    # 后台做的：预热缓存
    
    await asyncio.gather(
        # 1. 加载用户偏好
        load_user_preferences(user_id),
        
        # 2. 加载最近上下文
        load_recent_context(user_id),
        
        # 3. 预热常用知识
        warm_up_knowledge(user_id),
        
        # 4. 预编译提示词模板
        precompile_prompts(user_id)
    )
    
    # 标记缓存就绪
    cache.set(f"warm:{user_id}", True, ttl=3600)
```

**效果**：用户第一次对话就能享受"热启动"体验。

## 五、策略2：定时预热

对于高活跃用户，提前预热：

```python
# 每天凌晨4点执行
@scheduled(cron="0 4 * * *")
async def daily_warmup():
    # 获取昨日活跃用户Top 100
    active_users = get_top_active_users(limit=100)
    
    for user_id in active_users:
        await warm_up_user_cache(user_id)
        
    log(f"预热完成: {len(active_users)} 用户")

async def warm_up_user_cache(user_id):
    # 预加载用户数据
    user_context = await build_user_context(user_id)
    
    # 存入缓存
    cache.set(
        key=f"context:{user_id}",
        value=user_context,
        ttl=86400  # 24小时
    )
```

**效果**：早高峰时，活跃用户都是"热"的。

## 六、策略3：预测性预热

根据用户行为预测下一步：

```python
class PredictiveWarmer:
    # 行为-预热映射
    PREDICTIONS = {
        "open_project": ["load_project_context", "load_recent_files"],
        "start_coding": ["load_code_style", "load_api_docs"],
        "enter_debug": ["load_error_patterns", "load_debug_history"],
    }
    
    async def on_user_action(self, user_id, action):
        # 预测下一步需要什么
        predictions = self.PREDICTIONS.get(action, [])
        
        # 后台预热
        for prediction in predictions:
            asyncio.create_task(
                self.warm_up(user_id, prediction)
            )
```

**示例**：
- 用户打开项目 → 预热项目上下文
- 用户开始写代码 → 预热代码规范
- 用户进入调试 → 预热错误模式库

## 七、缓存层次设计

```
┌─────────────────────────────────────────┐
│  L1: 内存缓存 (Redis)                    │
│  延迟: <10ms | 容量: 有限 | 成本: 高      │
├─────────────────────────────────────────┤
│  L2: 本地缓存 (SQLite)                   │
│  延迟: <50ms | 容量: 中等 | 成本: 低      │
├─────────────────────────────────────────┤
│  L3: 数据库 (PostgreSQL)                 │
│  延迟: <200ms | 容量: 大 | 成本: 中       │
└─────────────────────────────────────────┘
```

```python
class MultiLevelCache:
    def get(self, key):
        # L1: 内存
        value = self.redis.get(key)
        if value:
            return value
        
        # L2: 本地
        value = self.local.get(key)
        if value:
            self.redis.set(key, value, ttl=300)  # 回填L1
            return value
        
        # L3: 数据库
        value = self.db.get(key)
        if value:
            self.local.set(key, value)  # 回填L2
            self.redis.set(key, value, ttl=300)  # 回填L1
            return value
        
        return None
```

## 八、上下文增量更新

不要每次都重建完整上下文：

```python
class IncrementalContextBuilder:
    def build_context(self, user_id, is_cold_start):
        if is_cold_start:
            # 冷启动：构建完整上下文
            return self.build_full_context(user_id)
        else:
            # 热启动：只加载增量
            return self.build_delta_context(user_id)
    
    def build_full_context(self, user_id):
        return {
            "global": self.load_global_context(),      # ~1000 tokens
            "app": self.load_app_context(user_id),     # ~2000 tokens
            "user": self.load_user_context(user_id),   # ~2000 tokens
            "session": self.load_session_context(),    # ~1000 tokens
        }  # 总计 ~6000 tokens
    
    def build_delta_context(self, user_id):
        # 只加载变化的部分
        return {
            "session_delta": self.load_session_delta(),  # ~500 tokens
            "recent_messages": self.load_recent(limit=3) # ~1500 tokens
        }  # 总计 ~2000 tokens
```

**节省**：热启动节省67% Token！

## 九、预热效果监控

```python
class WarmupMetrics:
    def record(self, user_id, is_warm, latency):
        # 记录预热命中率
        if is_warm:
            self.warm_hits += 1
        else:
            self.cold_hits += 1
        
        # 记录延迟分布
        self.latencies.append({
            "user_id": user_id,
            "is_warm": is_warm,
            "latency": latency,
            "timestamp": now()
        })
    
    def report(self):
        return {
            "warm_hit_rate": self.warm_hits / (self.warm_hits + self.cold_hits),
            "avg_warm_latency": avg([l["latency"] for l in self.latencies if l["is_warm"]]),
            "avg_cold_latency": avg([l["latency"] for l in self.latencies if not l["is_warm"]]),
        }
```

**目标指标**：
- 预热命中率 > 80%
- 热启动延迟 < 1.5秒
- 冷启动延迟 < 3秒

## 十一、总结

缓存预热的核心思想：

> **不要等用户问了才准备，要在用户问之前就准备好**

三个关键策略：
- **登录预热**：利用登录时间预加载
- **定时预热**：活跃用户提前准备
- **预测预热**：根据行为预测需求

让你的AI应用告别"冷启动"，随时保持最佳状态。
