# 填空题模型：让AI输出更可控的秘诀

> **作者**: Progee Team
> **日期**: 2026-01-10
> **标签**: Prompt Engineering, AI可控性, ODD

---

## 摘要

AI生成的代码"差不多对"但细节不符合预期？本文介绍Progee的"填空题模型"，通过将开放式任务转化为结构化的填空题，显著提升AI输出的可控性和准确性。

---

## 1. 问题：AI的"自由发挥"

### 1.1 作文题的困境

```
你：实现一个用户登录功能

AI的理解可能是：
  A) 一个简单的用户名密码表单
  B) 带OAuth的社交登录
  C) 带2FA的企业级登录
  D) 带生物识别的移动端登录
  
结果：AI自由发挥，可能不是你想要的
```

### 1.2 根本原因

**开放式问题 = 无限可能 = 不可控**

```
作文题："写一篇关于春天的文章"
  → 可以写花、写鸟、写心情...无限可能

填空题："春天来了，___开了，___叫了"
  → 答案范围明确，可控
```

---

## 2. 解决方案：填空题模型

### 2.1 核心思想

> **把开放式任务转化为结构化的填空题**

```
作文题（不可控）：
  "实现用户登录功能"

填空题（可控）：
  artifact_type: ___________  (从698种中选择)
  name: ___________
  input_spec:
    - username: ___________
    - password: ___________
  output_spec:
    - success: ___________
    - token: ___________
  acceptance_criteria:
    - Given ___________ When ___________ Then ___________
```

### 2.2 填空的类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **枚举填空** | 从预定义列表选择 | artifact_type: pg_function |
| **格式填空** | 按指定格式填写 | name: fn_{动词}_{名词} |
| **模板填空** | 按模板结构填写 | Given-When-Then |
| **约束填空** | 满足约束条件 | lines < 100 |

---

## 3. ODD的填空题结构

### 3.1 任务规格模板

```yaml
task:
  # 枚举填空：从698种产出物类型中选择
  artifact_type: ___________
  
  # 格式填空：符合命名规范
  artifact_name: ___________
  
  # 结构填空：输入参数列表
  input_spec:
    - name: ___________
      type: ___________
      required: ___________
      
  # 结构填空：输出参数列表
  output_spec:
    - name: ___________
      type: ___________
      
  # 列表填空：副作用声明
  side_effects:
    - ___________
    
  # 模板填空：验收标准
  acceptance_criteria:
    - Given ___________
      When ___________
      Then ___________
```

### 3.2 实际示例

```yaml
task:
  artifact_type: pg_function
  artifact_name: fn_login
  
  input_spec:
    - name: p_username
      type: VARCHAR(50)
      required: true
    - name: p_password
      type: VARCHAR(100)
      required: true
      
  output_spec:
    - name: success
      type: BOOLEAN
    - name: token
      type: VARCHAR(200)
    - name: error_msg
      type: TEXT
      
  side_effects:
    - UPDATE users SET last_login_at = NOW()
    - INSERT INTO login_logs
    
  acceptance_criteria:
    - Given 存在用户admin/123456
      When 调用fn_login('admin', '123456')
      Then 返回success=true, token非空
    - Given 存在用户admin/123456
      When 调用fn_login('admin', 'wrong')
      Then 返回success=false, error_msg='密码错误'
```

## 4. 填空题的约束力

### 4.1 类型约束

```yaml
# artifact_type必须是698种之一
artifact_type: pg_function  ✓
artifact_type: my_custom    ✗ (不在列表中)

# 数据类型必须是数据库支持的
type: VARCHAR(50)  ✓
type: string       ✗ (不是SQL类型)
```

### 4.2 格式约束

```yaml
# 函数名必须以fn_开头
artifact_name: fn_login     ✓
artifact_name: login        ✗ (不符合命名规范)

# 验收标准必须是Given-When-Then格式
acceptance_criteria:
  - Given ... When ... Then ...  ✓
  - 登录应该成功                  ✗ (格式不对)
```

### 4.3 完整性约束

```yaml
# 必填字段不能为空
artifact_type: ___  ✗ (必填)
input_spec: []      ✓ (可以为空列表)

# 输出必须与验收标准对应
output_spec: [success, token]
acceptance_criteria: "Then 返回success=true"  ✓ (对应)
acceptance_criteria: "Then 返回result=ok"     ✗ (不对应)
```

## 5. 减少不确定性

### 5.1 不确定性来源

| 来源 | 作文题 | 填空题 |
|------|--------|--------|
| 产出物类型 | AI猜测 | 明确指定 |
| 命名风格 | AI自由发挥 | 格式约束 |
| 参数设计 | AI设计 | 预定义结构 |
| 验收标准 | 模糊 | Given-When-Then |
| 边界条件 | 可能遗漏 | 必须列出 |

### 5.2 量化效果

```
实验：让AI实现同一个功能

作文题模式：
  - 第1次：80%符合预期
  - 修改后：90%符合预期
  - 再修改：95%符合预期
  - 总对话轮次：5轮

填空题模式：
  - 第1次：98%符合预期
  - 总对话轮次：1轮
  
效率提升：5倍
```

## 6. 实践技巧

### 6.1 从模糊到精确

```
第一步：识别产出物类型
  "实现登录" → 需要哪些产出物？
  → pg_table, pg_function, delphi_form, unit_test

第二步：定义输入输出
  fn_login需要什么输入？输出什么？
  → input: username, password
  → output: success, token, error_msg

第三步：写验收标准
  怎么验证fn_login是正确的？
  → Given正确密码 When调用 Then返回token
  → Given错误密码 When调用 Then返回error
```

### 6.2 常见填空模板

```yaml
# 数据库表
artifact_type: pg_table
columns:
  - name: id, type: UUID, primary: true
  - name: ___, type: ___, nullable: ___
indexes: [___]
constraints: [___]

# API端点
artifact_type: fastapi_endpoint
method: ___  # GET/POST/PUT/DELETE
path: /___
request_body: {___}
response: {___}
status_codes: [200, 400, 401, 404, 500]

# React组件
artifact_type: react_component
name: ___
props: [{name: ___, type: ___, required: ___}]
state: [{name: ___, type: ___, initial: ___}]
events: [___]
```

## 7. 与Prompt Engineering的关系

### 7.1 填空题是结构化Prompt

```
传统Prompt：
  "你是一个专业的程序员，请帮我实现用户登录功能，
   要求使用PostgreSQL数据库，返回JWT token..."
   
填空题Prompt：
  "请根据以下规格生成代码：
   artifact_type: pg_function
   name: fn_login
   input_spec: [username, password]
   output_spec: [success, token, error_msg]
   ..."
   
区别：
  - 传统Prompt：自然语言，有歧义
  - 填空题Prompt：结构化，无歧义
```

### 7.2 填空题的优势

| 维度 | 传统Prompt | 填空题Prompt |
|------|------------|--------------|
| 歧义性 | 高 | 低 |
| 可验证 | 难 | 易 |
| 可复用 | 低 | 高 |
| 一致性 | 低 | 高 |

---

## 8. 总结

填空题模型的核心思想：

> **用结构化约束替代自然语言描述**

通过填空题模型：
- **减少歧义**：明确的类型和格式约束
- **提高一致性**：相同模板产出相似结果
- **便于验证**：结构化的验收标准
- **提升效率**：一次生成，精准交付

这是让AI输出可控的关键技术。
