# 06.10 A03: 698种产出物分类法详解 (The Taxonomy of 698 Artifacts)

> **版本**: v1.0
> **类型**: 技术标准 / 开源规范
> **适用受众**: 架构师、工具开发者、AI 研究员

---

## 1. 为什么我们需要这个分类？

在软件工程中，我们长期缺乏一种**通用的语言**来描述"产出物"（Artifact）。
*   当你说"写代码"时，你是指写一个 `Interface`，还是一个 `Class`，还是一个 `Unit Test`？
*   当你说"做设计"时，你是指画 `Figma`，还是写 `API Spec`，还是建 `Database Schema`？

对于人类，这种模糊是可容忍的。但对于 **AI**，这种模糊是灾难性的。
AI 需要精确的指令。
**ODD 产出物分类法**（ODD Artifact Taxonomy）就是为了解决这个问题而生的。它定义了 **14 大类**、**698 种** 具体的产出物类型，为 AI 辅助开发提供了一张精确的地图。

---

## 2. 14 大类概览

| 类别代码 | 名称 | 描述 | 包含示例 |
| :--- | :--- | :--- | :--- |
| `code` | 代码 | 具体的编程语言源代码 | `python_module`, `ts_react_component`, `delphi_unit` |
| `database` | 数据库 | 数据持久化对象 | `pg_table`, `redis_key`, `mongo_collection` |
| `config` | 配置 | 系统运行参数 | `env_file`, `yaml_config`, `toml_settings` |
| `infrastructure` | 基础设施 | 运行环境定义 | `dockerfile`, `k8s_deployment`, `terraform_plan` |
| `document` | 文档 | 给人类阅读的说明 | `prd_markdown`, `api_docs`, `user_manual` |
| `test` | 测试 | 质量保证代码与数据 | `unit_test`, `e2e_script`, `test_fixture` |
| `behavior` | 行为 | 运行时产生的副作用 | `email_sent`, `log_entry`, `metric_emitted` |
| `api` | 接口 | 系统间契约 | `openapi_spec`, `graphql_schema`, `grpc_proto` |
| `security` | 安全 | 访问控制与加密 | `ssl_certificate`, `rbac_policy`, `firewall_rule` |
| `design` | 设计 | 视觉与交互规范 | `figma_file`, `ui_mockup`, `design_token` |
| `asset` | 资源 | 静态非代码资源 | `svg_icon`, `font_file`, `3d_model` |
| `ml` | AI/ML | 模型、数据与提示词 | `prompt_template`, `model_weights`, `dataset_csv` |
| `build` | 构建 | 编译打包产物 | `binary_exe`, `docker_image`, `npm_package` |
| `data` | 数据 | 数据迁移与初始化 | `migration_script`, `seed_data`, `etl_job` |

---

## 3. 分类的设计原则

### 3.1 互斥性 (Mutually Exclusive)
一个产出物只能属于一个类别。例如，`SQL Migration` 虽然是代码，但其目的是数据操作，因此归类为 `data` 或 `database`（视具体侧重），而在 ODD 中我们将其归为 `data` 类中的 `migration_script`。

### 3.2 机器可读性 (Machine Readable)
每个类型都有一个唯一的 `type_code`（如 `pg_table`）。这使得 AI Agent 可以直接理解意图：
*   **Prompt**: "Create a `pg_table` named `users`."
*   **AI Action**: 能够直接调用 PostgreSQL 的 `CREATE TABLE` 模板，而不是去写 Python 的 SQLAlchemy Model。

### 3.3 验证驱动 (Verification Driven)
分类不仅仅是为了归档，更是为了**验证**。
*   `code` 类 -> 验证策略：**Compile** / **Lint**
*   `test` 类 -> 验证策略：**Run Test**
*   `document` 类 -> 验证策略：**Review**
*   `behavior` 类 -> 验证策略：**Effect Check** (Mock/Spy)

## 4. 如何使用这份分类法？

### 4.1 对于 Agent 开发者
将 `A03_The_Taxonomy_of_698_Artifacts.json` 加载到你的 Agent 的系统提示词或知识库中。
当用户输入模糊指令时，让 Agent 先**分类**，再**执行**。

### 4.2 对于团队管理者
使用这套分类法来标准化团队的交付物。不要只验收"代码"，要验收"代码+文档+测试+配置"。

### 4.3 对于 ODD 实践者
这是你们的字典。在编写**契约 (Contract)** 时，`Artifact Type` 字段必须严格遵循此规范。
