# 01.11 层上下文工程：让AI真正理解你的代码

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-10
> **标签**: 上下文工程, AI开发, Prompt Engineering

---

## 摘要

使用AI编程时，最大的挑战不是AI的能力，而是**如何让AI理解你的项目**。本文介绍Progee的17层上下文工程设计，通过分层注入上下文，让AI从"什么都不知道"变成"了解项目全貌"，显著提升代码生成质量。

---

## 1. 问题：AI的"失忆症"

### 1.1 每次对话都是新开始

```
你：帮我写一个用户登录函数
AI：好的，这是一个登录函数...（用了你项目里没有的框架）

你：不对，我们用的是Delphi
AI：抱歉，这是Delphi版本...（命名风格和项目不一致）

你：我们的命名规范是...
AI：明白了...（又忘了数据库结构）
```

**问题根源**：AI没有项目的"记忆"，每次都需要重新告知上下文。

### 1.2 上下文的困境

| 困境 | 说明 |
|------|------|
| **信息过载** | 把所有信息都塞给AI，超出token限制 |
| **信息不足** | 只给任务描述，AI缺乏背景知识 |
| **信息错位** | 给了很多信息，但不是AI当前需要的 |

---

## 2. 解决方案：17层上下文架构

### 2.1 设计理念

```
核心思想：
  - 不是一次性注入所有上下文
  - 而是按需、分层、动态注入
  - 不同任务需要不同层级的上下文
```

### 2.2 17层定义

| 层级 | 名称 | 说明 | 注入时机 |
|:----:|------|------|----------|
| **L1** | 安全硬边界 | 敏感数据规则、权限模型 | 始终 |
| **L2** | 架构硬边界 | 模块隔离、禁止的依赖 | 始终 |
| **L3** | 流程硬边界 | 锁规则、审计要求 | 始终 |
| **L4** | 系统级规范 | 全局约定、宪法原则 | 始终 |
| **L5** | 产品目标 | 软件做什么、不做什么 | 契约生效 |
| **L6** | 用户意图 | 用户故事、业务目标 | 契约生效 |
| **L7** | 功能树索引 | 系统→模块→功能→产出物 | 按需查询 |
| **L8** | 技术栈 | 语言/框架/数据库/目录结构 | 契约生效 |
| **L9** | 代码风格 | lint规则/命名/注释规范 | 任务执行 |
| **L10** | 契约规格 | 契约定义、模块边界 | 车间启动 |
| **L11** | 依赖关系 | 任务依赖、产出物依赖 | 任务分配 |
| **L12** | 车间知识库 | 代码结构、Bug模式、最佳实践 | 车间启动 |
| **L13** | 资源锁状态 | 已锁/待锁/冲突 | 任务执行 |
| **L14** | 任务规格 | input/output/side_effects | 任务执行 |
| **L15** | 验收标准 | acceptance_criteria、测试策略 | 任务执行 |
| **L16** | 执行上下文 | 任务类型、工作循环 | 任务执行 |
| **L17** | 纠正反馈 | 失败原因、返工历史 | 返工时 |

## 3. 各层详解

### 3.1 硬边界层（L1-L3）：不可违反

```yaml
L1 安全硬边界:
  - 密码必须加密存储（bcrypt/argon2）
  - 禁止在日志中输出敏感信息
  - API必须验证权限
  强制级别: API级拒绝，不可绕过

L2 架构硬边界:
  - UI层不能直接访问数据库
  - 模块A不能import模块B的内部实现
  - 必须通过接口通信
  强制级别: 编译/检查时拒绝

L3 流程硬边界:
  - 修改文件前必须获取锁
  - 关键操作必须记录审计日志
  - 状态转换必须符合状态机
  强制级别: 流程拒绝
```

**为什么重要**：这三层是"宪法"，AI无论如何不能违反。

### 3.2 规范层（L4-L6）：项目级约定

```yaml
L4 系统级规范:
  - 13条宪法原则
  - 24条编码指南
  - 全局命名约定

L5 产品目标:
  - Progee是什么：AI驱动的开发管理系统
  - Progee不做什么：不是IDE，不是代码编辑器
  - 目标用户：需要AI辅助开发的团队

L6 用户意图:
  - 用户故事：As a 开发者, I want 自动执行任务, So that 提高效率
  - 业务目标：让AI成为开发团队的一员
```

### 3.3 结构层（L7-L9）：技术架构

```yaml
L7 功能树索引:
  Progee
    └── 用户管理
          └── 登录功能
                ├── users表
                ├── fn_login函数
                └── FrmLogin窗体

L8 技术栈:
  - 语言: Delphi (FireMonkey)
  - 数据库: PostgreSQL 15
  - 目录结构: DeBorn零目录模式
  - 架构: 单机多线程

L9 代码风格:
  - 命名: PascalCase for types, camelCase for variables
  - 注释: 只在复杂逻辑处添加
  - 格式: 2空格缩进
```

### 3.4 任务层（L10-L12）：契约与知识

```yaml
L10 契约规格:
  contract: 用户登录功能
  artifacts:
    - pg_table: users
    - pg_function: fn_login
    - delphi_form: FrmLogin

L11 依赖关系:
  fn_login depends_on users表
  FrmLogin depends_on fn_login

L12 车间知识库:
  - 这个模块之前有过的Bug：密码比较时忘记trim
  - 最佳实践：使用pgcrypto的crypt函数
  - 已封版代码：可以参考fn_register的实现
```

### 3.5 执行层（L13-L17）：运行时上下文

```yaml
L13 资源锁状态:
  - users表: 已锁定（被当前任务）
  - fn_login: 待创建
  - FrmLogin: 未锁定

L14 任务规格:
  artifact_type: pg_function
  name: fn_login
  input_spec: {username, password}
  output_spec: {success, token, error_msg}

L15 验收标准:
  - Given 正确密码 When 调用 Then 返回token
  - Given 错误密码 When 调用 Then 返回error

L16 执行上下文:
  task_type: 新功能
  work_cycle: develop → test → review

L17 纠正反馈:
  rework_number: 2
  previous_failure: "忘记处理用户不存在的情况"
  similar_bugs: ["fn_register也有过类似问题"]
```

## 4. 上下文注入策略

### 4.1 冷启动 vs 热启动

```
冷启动（首次执行）：
  注入: L1-L6 + L8-L10 + L14-L16
  token消耗: 高
  场景: 新契约、新车间

热启动（同契约内）：
  注入: L13-L16（增量）
  token消耗: 低
  场景: 同一契约的后续任务
```

### 4.2 按需查询

```
L7 功能树：
  - 不是一次性注入整棵树
  - 而是根据当前任务，查询相关分支
  - 例如：开发登录功能时，只查询"用户管理"分支

L12 车间知识库：
  - 不是注入所有历史
  - 而是查询与当前任务相关的知识
  - 例如：开发fn_login时，查询"登录相关的Bug记录"
```

### 4.3 动态注入

```
L17 纠正反馈：
  - 只在返工时注入
  - 包含上次失败的原因
  - 包含相似Bug的修复记录
  
  效果：AI不会重复犯同样的错误
```

## 5. 实际效果

### 5.1 对比实验

```
场景：让AI实现用户登录功能

无上下文：
  - AI使用了项目中没有的框架
  - 命名风格不一致
  - 忘记了安全要求
  - 需要5轮对话修正
  
有17层上下文：
  - AI使用正确的技术栈
  - 命名符合项目规范
  - 自动应用安全规则
  - 1轮对话完成
```

### 5.2 token效率

| 策略 | 首次token | 后续token | 总token（10任务） |
|------|-----------|-----------|-------------------|
| 全量注入 | 8000 | 8000 | 80000 |
| 17层分层 | 6000 | 2000 | 24000 |
| **节省** | - | - | **70%** |

## 6. 实践建议

### 6.1 最小可用上下文

```
如果你只能注入3层，选择：
  L8 技术栈：让AI知道用什么语言/框架
  L9 代码风格：让AI的代码风格一致
  L14 任务规格：让AI知道要做什么
```

### 6.2 渐进式增强

```
第一步：技术栈 + 任务规格（基础）
第二步：+ 代码风格 + 验收标准（质量）
第三步：+ 安全边界 + 架构边界（安全）
第四步：+ 知识库 + 纠正反馈（智能）
```

### 6.3 常见错误

```
❌ 错误1：一次性注入所有上下文
   问题：超出token限制，AI反而困惑
   正确：按需分层注入

❌ 错误2：只注入任务描述
   问题：AI缺乏背景，生成的代码难以集成
   正确：至少注入技术栈和代码风格

❌ 错误3：忽略纠正反馈
   问题：AI重复犯同样的错误
   正确：返工时注入L17，告诉AI上次为什么失败
```

---

## 7. 总结

17层上下文工程的核心思想：

> **不是给AI更多信息，而是给AI正确的信息**

通过分层设计，我们实现了：
- **按需注入**：不同任务需要不同层级
- **动态查询**：功能树和知识库按需查询
- **增量更新**：热启动时只注入变化的部分
- **纠正反馈**：返工时注入失败原因

这让AI从"什么都不知道"变成"了解项目全貌"，显著提升代码生成质量。
