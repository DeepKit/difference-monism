# 01.19 车间模型：像管理工厂一样管理AI开发

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-10
> **标签**: 车间模型, AI管理, 软件工程

## 摘要

如何管理多个AI同时工作？如何避免AI之间的冲突？如何在AI故障时不丢失进度？本文介绍Progee的"车间模型"，将软件开发类比为工厂生产，用工业化思维管理AI开发团队。

---

## 1. 问题：AI开发的管理困境

### 1.1 单AI的局限

```
场景：一个中型项目，需要同时开发多个功能

单AI模式：
  - 任务A → 任务B → 任务C（串行）
  - 一个任务卡住，所有任务等待
  - 效率低下
```

### 1.2 多AI的混乱

```
多AI模式（无管理）：
  - AI-1 修改了文件X
  - AI-2 也在修改文件X
  - 冲突！覆盖！数据丢失！
```

### 1.3 我们需要什么

```
理想状态：
  ✓ 多个AI并行工作
  ✓ 互不干扰，资源隔离
  ✓ 故障可恢复，不丢失进度
  ✓ 有人监控，异常告警
```

---

## 2. 解决方案：车间模型

### 2.1 工厂类比

```
传统工厂                    Progee车间模型
─────────────────────────────────────────────
工厂                        Progee系统
车间                        AI执行容器
工人                        AI（Claude/GPT）
订单                        契约（Contract）
产品                        产出物（Artifact）
车间主任                    经理（Manager）
质检员                      验证器（Validator）
```

### 2.2 核心概念

**契约 ≈ 功能模块 ≈ 车间（1:1:1绑定）**

```
契约: 用户登录功能
  │
  └── 车间 #1（专门执行这个契约）
        │
        ├── 任务1: 创建users表
        ├── 任务2: 创建fn_login函数
        ├── 任务3: 创建登录界面
        └── 任务4: 创建单元测试
```

### 2.3 角色体系

```
人类
  └── 架构师（设计契约）
        └── 经理（监控调度）
              └── 车间（执行容器）
                    └── AI工人（执行任务）
```

| 角色 | 职责 | 不做什么 |
|------|------|----------|
| **人类** | 决策、确认、异常处理 | 不写代码 |
| **架构师** | 需求澄清、契约设计、任务分解 | 不执行任务 |
| **经理** | 车间启停、心跳监控、告警 | 不干预执行 |
| **车间** | 进程隔离、资源限制、会话管理 | 不做决策 |
| **AI工人** | 开发、测试、封版 | 不做架构决策 |

## 3. 车间生命周期

### 3.1 状态机

```
         ┌─────────────────────────────────────┐
         │                                     │
         ▼                                     │
      [idle] ──分配契约──▶ [starting]          │
         ▲                    │                │
         │                    ▼                │
         │               [active] ◀──返工──┐   │
         │                    │            │   │
         │                    ▼            │   │
         │              [任务执行中]────失败──┘   │
         │                    │                │
         │                    ▼                │
         └──释放────── [completed] ─────────────┘
                           │
                      故障 │
                           ▼
                      [crashed]
                           │
                      回收 │
                           ▼
                      契约回队列
```

### 3.2 关键流程

**启动流程**：
```
1. 经理发现空闲车间
2. 从队列取优先级最高的契约
3. 分配契约给车间
4. 车间加载上下文（冷启动/热启动）
5. 车间开始执行任务
```

**执行流程**：
```
1. 车间按依赖顺序执行任务
2. 每个任务：开发 → 测试 → 验证
3. 验证通过 → 封版
4. 验证失败 → 返工（最多N次）
5. 所有任务完成 → 契约完成
```

**故障恢复**：
```
1. 经理检测到车间无心跳
2. 标记车间为crashed
3. 契约回到队列（优先级提升）
4. 新车间接手，从上次状态继续
```

## 4. AI工人角色切换

### 4.1 全能工人

AI工人是**全能的**，在车间内按需切换角色：

```
┌─────────────────────────────────────────────────┐
│  AI工人（同一个AI，不同角色）                     │
│                                                 │
│  [开发工人] ──▶ [测试工人] ──▶ [集测工人] ──▶ [封版工人]
│                                                 │
│  编写代码      编写单测      集成测试      代码封版
│  pending →    developing →  unit_tested → sealed
│  developing   unit_tested   int_tested          │
└─────────────────────────────────────────────────┘
```

### 4.2 交叉审核

```
规则：集测/封版时，不能审核自己开发的代码

车间A的AI开发了fn_login
  → 车间B的AI来做集成测试
  → 车间C的AI来做封版审核

好处：避免"自己检查自己"的盲区
```

## 5. 资源管理

### 5.1 车间池

```
┌─────────────────────────────────────────────────┐
│  车间池（有限资源）                               │
│                                                 │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐           │
│  │ 车间#1  │ │ 车间#2  │ │ 车间#3  │ ...       │
│  │ 契约A   │ │  空闲   │ │ 契约C   │           │
│  │ active  │ │  idle   │ │ active  │           │
│  └─────────┘ └─────────┘ └─────────┘           │
│                                                 │
│  max_workshops = N（可配置，受硬件限制）          │
└─────────────────────────────────────────────────┘
```

### 5.2 契约队列

```
┌─────────────────────────────────────────────────┐
│  契约队列（按优先级排序）                         │
│                                                 │
│  [契约A-P0] [契约B-P1] [契约C-P1] [契约D-P2]     │
│                                                 │
│  优先级：P0 > P1 > P2 > P3                      │
│  故障回收的契约：优先级自动提升                   │
└─────────────────────────────────────────────────┘
```

### 5.3 锁机制

```
问题：多个车间可能同时修改同一个文件

解决：Mandatory Lease Lock

车间A要修改users.pas：
  1. 申请锁：LOCK users.pas FOR 30min
  2. 获得锁：开始修改
  3. 完成后：释放锁
  
车间B也要修改users.pas：
  1. 申请锁：发现已被锁定
  2. 等待或跳过（取决于依赖关系）
```

## 6. Delphi实现架构

### 6.1 多线程设计

```
┌─────────────────────────────────────────────────┐
│  Delphi 主进程                                   │
├─────────────────────────────────────────────────┤
│  主线程 (UI)                                     │
│  ├── TTimer: 任务列表轮询                        │
│  └── 用户交互事件                                │
├─────────────────────────────────────────────────┤
│  经理线程 (TManagerThread)                       │
│  ├── 心跳检测                                    │
│  ├── 超时处理                                    │
│  └── 告警发送                                    │
├─────────────────────────────────────────────────┤
│  车间线程池 (TWorkshopThread[])                  │
│  ├── 车间0: 调用CLI → 执行任务                   │
│  ├── 车间1: 调用CLI → 执行任务                   │
│  └── 车间N: ...                                 │
└─────────────────────────────────────────────────┘
```

### 6.2 关键代码

```pascal
// 经理线程
TManagerThread = class(TThread)
  procedure Execute; override;
  procedure CheckHeartbeats;  // 心跳检测
  procedure CheckTimeouts;    // 超时处理
  procedure SendAlert;        // 告警
end;

// 车间线程
TWorkshopThread = class(TThread)
  FWorkshopId: Integer;
  FCurrentRole: TWorkerRole;  // 当前角色
  procedure Execute; override;
  procedure SwitchRole(NewRole: TWorkerRole);
  function RunCLI(const Prompt: string): string;
end;

// AI工人角色
TWorkerRole = (wrDevelop, wrTest, wrIntegrationTest, wrSeal);
```

## 7. 实践效果

### 7.1 并行效率

```
场景：10个独立功能模块

串行模式（单AI）：
  10个模块 × 2小时/模块 = 20小时

车间模式（5个车间）：
  10个模块 ÷ 5车间 × 2小时 = 4小时
  
效率提升：5倍
```

### 7.2 故障恢复

```
场景：车间执行到一半崩溃

无车间模式：
  - 进度丢失
  - 需要从头开始
  - 浪费时间和token

车间模式：
  - 状态持久化在数据库
  - 新车间接手
  - 从上次状态继续
  - 零进度丢失
```

## 8. 总结

车间模型的核心思想：

> **用工业化思维管理AI开发**

通过车间模型，我们实现了：
- **并行执行**：多个车间同时工作
- **资源隔离**：车间之间互不干扰
- **故障恢复**：崩溃不丢失进度
- **角色分工**：人类决策，AI执行

这让AI开发从"手工作坊"升级为"现代工厂"。
