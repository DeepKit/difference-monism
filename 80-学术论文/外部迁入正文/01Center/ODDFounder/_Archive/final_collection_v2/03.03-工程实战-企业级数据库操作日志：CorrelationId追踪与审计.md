# 03.03 企业级数据库操作日志：CorrelationId追踪与审计

> **标签**: 日志, 审计, CorrelationId, 分布式追踪, UniBase

---

## 摘要

当系统出现 Bug 时，我们最想知道的是："这笔数据是谁改的？什么时候改的？对应的 API 请求是什么？" 普通的数据库日志（Binlog）只有数据变化，没有业务上下文。**UniBase** 通过 **Correlation ID (相关性ID)** 技术，将应用层的请求上下文穿透到数据库层，实现了全链路的可追踪审计。

---

## 一、断裂的链路

通常的日志是割裂的：
*   **Web Log**: `[Req-101] POST /update_user`
*   **DB Log**: `UPDATE users SET name='...'`

当 DB 出现慢查询或异常数据时，DBA 很难反推是哪个 API 请求导致的。

---

## 二、Correlation ID 上下文传递

### 2.1 生成与传递
1.  **Gateway**: 接收请求时，生成唯一的 `Correlation-ID`（如 `req-a1b2c3`）。
2.  **App Service**: 将该 ID 放入 `ThreadLocal` 或上下文对象 `Ctx` 中。
3.  **doQry**: 在执行 SQL 时，将该 ID 携带下去。

### 2.2 注入数据库 (`uDoQry.pas`)

doQry 有一个巧妙的设计：它会在每条 SQL 前加上**注释**。

```pascal
function DoQryBuildSql(const SQL, CorrelationId: string): string;
begin
  Result := Format('/* cid:%s */ %s', [CorrelationId, SQL]);
end;
```

生成的 SQL 变成：
`/* cid:req-a1b2c3 */ SELECT * FROM users ...`

### 2.3 数据库端的捕获
在 PostgreSQL 的慢查询日志（Slow Query Log）或审计插件（pgAudit）中，这个注释会被完整记录。
DBA 看到这条慢 SQL 时，复制出 `cid:req-a1b2c3`，去 ELK/Splunk 搜一下，立马就能找到对应的 Web 请求日志和完整的堆栈信息。

---

## 三、结构化审计表

除了注释，UniBase 还提供了一个统一的 `audit_logs` 表。
doQry 在执行 `INSERT/UPDATE/DELETE` 时，会自动记录：
*   `table_name`
*   `record_id`
*   `operation`
*   `old_value` / `new_value` (JSON)
*   `correlation_id`
*   `operator_id`

这构成了应用层的**数据时光机**，让我们可以回溯任何数据的变更历史。

---

## 四、总结

**Correlation ID 是分布式系统的红线。**
通过简单地将 ID 注入 SQL 注释，UniBase 低成本地打通了应用与数据库之间的监控壁垒。这是企业级系统可观测性（Observability）的重要一环。
