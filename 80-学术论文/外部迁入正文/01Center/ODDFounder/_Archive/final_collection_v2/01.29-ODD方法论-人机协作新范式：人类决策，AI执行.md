# 01.29 人机协作新范式：人类决策，AI执行

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-10
> **标签**: 人机协作, AI开发, 软件工程

---

## 摘要

AI能写代码了，人类还需要做什么？本文介绍Progee的人机协作模型，明确人类和AI的分工边界：人类负责决策，AI负责执行。这不是取代，而是增强。

---

## 1. 问题：人机边界模糊

### 1.1 两种极端

```
极端A：AI万能论
  "AI可以做一切，人类只需要提需求"
  问题：AI缺乏业务判断力，容易走偏

极端B：AI工具论
  "AI只是辅助，人类还是要写代码"
  问题：没有发挥AI的真正价值
```

### 1.2 我们需要什么

```
理想状态：
  - 人类做人类擅长的事（决策、判断、创意）
  - AI做AI擅长的事（执行、重复、规模化）
  - 边界清晰，协作高效
```

---

## 2. Progee的人机分工模型

### 2.1 角色体系

```
人类
  └── 架构师（设计）
        └── 经理（监控）
              └── 车间（容器）
                    └── AI工人（执行）
```

### 2.2 职责矩阵

| 角色 | 核心职责 | 不做什么 |
|------|----------|----------|
| **人类** | 产品决策、契约确认、异常处理 | 不写代码 |
| **架构师** | 需求澄清、契约设计、任务分解 | 不执行任务 |
| **经理** | 车间启停、心跳监控、告警 | 不干预执行 |
| **车间** | 进程隔离、资源管理、会话 | 不做决策 |
| **AI工人** | 开发、测试、封版 | 不做架构决策 |

### 2.3 决策 vs 执行

```
人类决策：
  ✓ 做什么功能（产品方向）
  ✓ 用什么技术（技术选型）
  ✓ 什么优先级（资源分配）
  ✓ 是否接受（验收确认）
  ✓ 异常处理（超出AI能力）

AI执行：
  ✓ 写代码（按规格实现）
  ✓ 写测试（按标准验证）
  ✓ 写文档（按模板生成）
  ✓ 重复劳动（批量处理）
  ✓ 7x24工作（不知疲倦）
```

## 3. 人类的不可替代性

### 3.1 业务判断

```
场景：用户说"我要一个登录功能"

AI能做：
  - 生成登录代码
  - 生成数据库表
  - 生成测试用例

AI不能做：
  - 判断是否需要OAuth
  - 判断是否需要2FA
  - 判断密码策略
  - 判断是否符合合规要求

这些需要人类的业务判断
```

### 3.2 异常处理

```
场景：AI连续3次生成的代码都编译失败

AI的局限：
  - 可能陷入重复错误
  - 缺乏"跳出来看"的能力
  - 不知道何时该求助

人类的价值：
  - 识别根本原因
  - 调整任务定义
  - 决定是否换方案
  - 提供额外上下文
```

### 3.3 创意和方向

```
AI擅长：
  - 在已知框架内工作
  - 按规格实现功能
  - 优化已有方案

人类擅长：
  - 定义新的框架
  - 发现新的需求
  - 做出权衡取舍
  - 承担责任
```

## 4. AI的独特价值

### 4.1 规模化执行

```
人类：1天写500行代码
AI：1小时写5000行代码

但关键是：
  - AI的5000行是否正确？
  - 需要人类定义"正确"的标准
  - 需要人类验收结果
```

### 4.2 不知疲倦

```
人类：8小时工作，需要休息
AI：24小时工作，不需要休息

应用场景：
  - 夜间执行批量任务
  - 周末处理积压工作
  - 并行执行多个任务
```

### 4.3 一致性

```
人类：状态波动，风格变化
AI：始终一致，按规格执行

应用场景：
  - 代码风格统一
  - 测试覆盖完整
  - 文档格式一致
```

## 5. 协作流程

### 5.1 标准流程

```
┌─────────────────────────────────────────────────┐
│  1. 人类：提出需求                               │
│     "我需要一个用户登录功能"                     │
└─────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────┐
│  2. 架构师（AI）：澄清需求、设计契约              │
│     "需要OAuth吗？需要2FA吗？"                   │
│     → 人类确认                                   │
└─────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────┐
│  3. 架构师（AI）：分解任务                        │
│     任务1: users表                              │
│     任务2: fn_login函数                         │
│     任务3: 登录界面                             │
│     → 人类确认                                   │
└─────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────┐
│  4. AI工人：执行任务                             │
│     自动执行，无需人类干预                        │
└─────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────┐
│  5. 人类：验收结果                               │
│     确认功能符合预期                             │
│     或：提出修改意见                             │
└─────────────────────────────────────────────────┘
```

### 5.2 异常流程

```
正常：AI执行 → 验证通过 → 封版
异常：AI执行 → 验证失败 → 重试（最多N次）→ 人类介入

人类介入时机：
  - 连续失败超过阈值
  - 成本超过预算
  - 遇到未知错误
  - AI主动求助
```

## 6. 实践建议

### 6.1 人类应该做的

```
✓ 清晰定义需求（ODD的产出物规格）
✓ 设定验收标准（acceptance_criteria）
✓ 及时响应异常（AI求助时）
✓ 定期审查结果（抽查质量）
✓ 持续优化流程（改进效率）
```

### 6.2 人类不应该做的

```
✗ 微观管理AI（让AI自主执行）
✗ 手动写代码（除非AI无法完成）
✗ 忽略异常告警（及时处理）
✗ 跳过验收环节（质量保证）
```

### 6.3 信任但验证

```
原则：Trust but Verify

信任：
  - 让AI自主执行任务
  - 不干预执行过程
  - 给AI足够的上下文

验证：
  - 自动化测试
  - 验收标准检查
  - 定期抽查审查
```

## 7. 未来展望

### 7.1 人类角色演变

```
过去：程序员 = 写代码的人
现在：程序员 = 指导AI写代码的人
未来：程序员 = 定义问题的人

核心能力转变：
  - 从"如何实现"到"实现什么"
  - 从"写代码"到"定义规格"
  - 从"执行者"到"决策者"
```

### 7.2 AI角色演变

```
现在：AI = 代码生成器
未来：AI = 开发团队成员

能力提升：
  - 更好的上下文理解
  - 更强的自主决策
  - 更少的人类干预
```

---

## 8. 总结

人机协作的核心原则：

> **人类决策，AI执行**

这意味着：
- **人类**：定义做什么、验收结果、处理异常
- **AI**：执行任务、保证质量、规模化交付

这不是AI取代人类，而是人类借助AI实现更大的价值。
