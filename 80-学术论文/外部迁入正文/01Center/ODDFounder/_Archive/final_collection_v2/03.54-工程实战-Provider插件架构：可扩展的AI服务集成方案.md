# 03.54 Provider插件架构：可扩展的AI服务集成方案

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-11
> **标签**: 插件架构, 依赖注入, 扩展性, Touchstone, JavaScript注入

---

## 摘要

AI 模型层出不穷，每个月都有新平台（Kimi, Intermediate Model, Claude...）。如果每次支持新模型都要改代码、发新版，开发团队会累死。**Touchstone** 设计了一套**基于配置和脚本的 Provider 插件架构**。用户（或社区）只需要编写一个 JSON 配置文件和一个 JS 脚本，就能让 Touchstone 支持任意新的 AI Web 界面。

---

## 一、硬编码 vs 插件化

*   **硬编码**：在 Delphi 代码里写 `if Provider = 'ChatGPT' then ...`。扩展性差。
*   **插件化**：将差异逻辑剥离。主程序只负责"加载插件"和"执行通用逻辑"。

---

## 二、插件结构 (`ModelProvider.pas`)

一个 Provider 插件包含三个部分：

### 2.1 `manifest.json` (元数据)
```json
{
  "id": "kimi",
  "name": "Kimi Chat",
  "url": "https://kimi.moonshot.cn",
  "logo": "kimi.svg"
}
```

### 2.2 `selectors.json` (DOM 选择器)
定义了如何操作该网站（详见《智能DOM查找器》）。
```json
{
  "input": "div[contenteditable='true']",
  "send_btn": ".send-button",
  "reply_container": ".chat-message"
}
```

### 2.3 `script.js` (注入脚本)
处理特殊的逻辑，比如绕过某些反爬虫检测，或者格式化输出。
```javascript
// 钩子函数
window.TouchstoneHooks = {
  beforeSend: function(text) { ... },
  parseOutput: function(dom) { return dom.innerText; }
};
```

---

## 三、加载与执行

Touchstone 启动时，扫描 `data/plugins/` 目录。
*   **动态加载**：读取 JSON，注册到 Provider 列表。
*   **运行时注入**：当用户打开 Kimi 窗口时，WebView2 自动注入对应的 `script.js`。
*   **沙箱隔离**：每个插件运行在各自的 WebView 进程中，互不影响。

---

## 四、社区驱动

这种架构最大的好处是**社区力量**。
当 Kimi 改版了，社区里立刻有人更新 `selectors.json` 并分享出来。
当出了一个新的 AI（比如 Grok），用户可以自己写一个插件包，马上就能在 Touchstone 里用起来，不需要等官方更新。

---

## 五、总结

**开放性是工具生命力的源泉。**
Touchstone 的 Provider 架构，将"集成新模型"的成本从"开发工作"降低到了"配置工作"。通过标准化接口和脚本注入能力，它构建了一个开放的 AI 聚合生态。
