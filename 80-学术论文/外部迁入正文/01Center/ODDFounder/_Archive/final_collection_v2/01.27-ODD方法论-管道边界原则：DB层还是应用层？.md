## 摘要

业务逻辑应该放在数据库还是应用层？这是一个经典的架构问题。本文介绍Progee的"管道边界原则"，提供一个清晰的决策框架，帮助你做出正确的选择。

---

## 1. 问题：逻辑放哪里？

### 1.1 ODD的管道模型

```
ODD核心模型：
  DB(状态A) → 管道 → DB(状态B)

问题：管道可以在两个地方执行
  1. DB内（SQL/存储过程/触发器）
  2. 应用层（Python/Delphi/外部调用）
```

### 1.2 两种极端

```
极端A：全放DB
  优点：性能好、事务简单
  缺点：难以调用外部服务、难以测试

极端B：全放应用层
  优点：灵活、易测试
  缺点：性能差、事务复杂
```

## 2. 边界黄金法则

### 2.1 第一性原理

```
PostgreSQL能做的事 ✓：
  - 数据存储和检索
  - 事务和一致性（ACID）
  - 集合操作（JOIN/GROUP BY）
  - 约束和触发器
  - 并发控制
  - 数据校验

PostgreSQL做不了的事 ✗：
  - 调用外部HTTP API
  - 发送邮件/短信/推送
  - 文件系统操作
  - AI推理/机器学习
  - 与用户交互
```

### 2.2 黄金法则

```
┌─────────────────────────────────────────────────┐
│              边界黄金法则                        │
│                                                 │
│    能在DB完成的 → 放DB                          │
│    必须出DB的 → 放应用层                        │
│                                                 │
│    DB擅长：数据操作                             │
│    应用层擅长：外部交互                          │
└─────────────────────────────────────────────────┘
```

---

## 3. 决策树

### 3.1 五问决策法

```
问题1: 是否需要调用外部系统？
  ├── 是 → 应用层
  │   （HTTP API、消息队列、邮件服务）
  └── 否 → 继续判断

问题2: 是否是纯数据变形？
  ├── 是 → DB层
  │   （SELECT/INSERT/UPDATE/DELETE）
  └── 否 → 继续判断

问题3: 业务逻辑复杂度？
  ├── 简单规则 → DB层
  │   （CASE WHEN、CHECK约束）
  └── 复杂逻辑 → 应用层
      （多步推理、AI判断）

问题4: 是否需要人工干预？
  ├── 是 → 应用层
  │   （审批流程、人工确认）
  └── 否 → DB层

问题5: 数据量和性能要求？
  ├── 大数据量集合操作 → DB层
  └── 单条记录处理 → 都可以
```

### 3.2 决策矩阵

| 场景 | DB层 | 应用层 | 原因 |
|------|:----:|:------:|------|
| 数据校验 | ✓ | | CHECK约束更可靠 |
| 状态转换 | ✓ | | 事务保证一致性 |
| 聚合统计 | ✓ | | DB擅长集合操作 |
| 发送邮件 | | ✓ | 需要外部服务 |
| 调用API | | ✓ | 需要HTTP客户端 |
| 文件处理 | | ✓ | 需要文件系统 |
| 审批流程 | | ✓ | 需要人工交互 |
| AI推理 | | ✓ | 需要ML框架 |

## 4. 管道类型分类

### 4.1 纯DB管道

```sql
-- 示例：用户登录验证
CREATE FUNCTION fn_login(p_username VARCHAR, p_password VARCHAR)
RETURNS TABLE(success BOOLEAN, token VARCHAR, error_msg TEXT)
AS $$
BEGIN
  -- 纯数据操作，适合放DB
  IF NOT EXISTS (SELECT 1 FROM users WHERE username = p_username) THEN
    RETURN QUERY SELECT false, NULL, '用户不存在';
    RETURN;
  END IF;
  
  IF NOT (SELECT password_hash = crypt(p_password, password_hash) 
          FROM users WHERE username = p_username) THEN
    RETURN QUERY SELECT false, NULL, '密码错误';
    RETURN;
  END IF;
  
  -- 更新登录时间
  UPDATE users SET last_login_at = NOW() WHERE username = p_username;
  
  -- 生成token
  RETURN QUERY SELECT true, gen_random_uuid()::VARCHAR, NULL;
END;
$$ LANGUAGE plpgsql;
```

### 4.2 应用层管道

```python
# 示例：用户注册（需要发送邮件）
async def register_user(username: str, email: str, password: str):
    # 数据操作：调用DB
    user_id = await db.execute(
        "SELECT fn_create_user($1, $2, $3)",
        username, email, hash_password(password)
    )
    
    # 外部调用：发送邮件（必须在应用层）
    await email_service.send_welcome_email(email, username)
    
    # 外部调用：发送统计事件
    await analytics.track("user_registered", {"user_id": user_id})
    
    return user_id
```

### 4.3 混合管道

```python
# 示例：订单支付（DB + 应用层）
async def pay_order(order_id: str, payment_info: dict):
    async with db.transaction():
        # Step 1: DB层 - 锁定订单
        order = await db.execute(
            "SELECT fn_lock_order_for_payment($1)", order_id
        )
        
        # Step 2: 应用层 - 调用支付网关
        payment_result = await payment_gateway.charge(
            amount=order.amount,
            card=payment_info
        )
        
        # Step 3: DB层 - 更新订单状态
        if payment_result.success:
            await db.execute(
                "SELECT fn_complete_payment($1, $2)",
                order_id, payment_result.transaction_id
            )
        else:
            await db.execute(
                "SELECT fn_fail_payment($1, $2)",
                order_id, payment_result.error
            )
    
    # Step 4: 应用层 - 发送通知
    await notification.send_payment_result(order_id, payment_result)
```

## 5. 常见误区

### 5.1 误区：全放应用层

```python
# 错误：简单的数据校验放应用层
def create_user(username, email):
    # 这些校验应该放DB的CHECK约束
    if len(username) < 3:
        raise ValueError("用户名太短")
    if "@" not in email:
        raise ValueError("邮箱格式错误")
    
    db.execute("INSERT INTO users ...")

# 正确：用DB约束
# CREATE TABLE users (
#   username VARCHAR(50) CHECK (length(username) >= 3),
#   email VARCHAR(100) CHECK (email ~ '^[^@]+@[^@]+$')
# );
```

### 5.2 误区：全放DB

```sql
-- 错误：在DB里调用外部API（PostgreSQL不擅长）
CREATE FUNCTION fn_send_email(p_to VARCHAR, p_subject VARCHAR)
RETURNS BOOLEAN AS $$
  -- 试图用plpython3u调用HTTP...
  -- 复杂、难维护、难测试
$$ LANGUAGE plpython3u;

-- 正确：邮件发送放应用层
```

## 6. 实践建议

### 6.1 默认策略

```
1. 数据校验 → DB（CHECK约束）
2. 状态转换 → DB（存储过程）
3. 业务规则 → DB（简单）或应用层（复杂）
4. 外部调用 → 应用层
5. 通知发送 → 应用层
```

### 6.2 边界清晰

```
DB层职责：
  - 数据完整性
  - 事务一致性
  - 业务规则（简单）
  
应用层职责：
  - 外部集成
  - 用户交互
  - 复杂业务逻辑
  - 异步处理
```

---

## 7. 总结

管道边界原则的核心：

> **能在DB完成的放DB，必须出DB的放应用层**

通过五问决策法：
1. 需要外部系统？→ 应用层
2. 纯数据变形？→ DB层
3. 逻辑复杂度？→ 简单DB，复杂应用层
4. 需要人工干预？→ 应用层
5. 大数据量？→ DB层

这个原则帮助你做出清晰的架构决策。
