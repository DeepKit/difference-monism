## 摘要

"乱码"是程序员的噩梦。GBK, Big5, Shift-JIS, EUC-KR... 面对一个没有 BOM 头的文件，如何准确判断它的编码？**TransSuccess** 是一款文件转码工具，其核心竞争力在于一套**高精度的启发式编码检测算法**。本文深入 `ControllerEncoding.pas`，揭秘如何精准区分中日韩（CJK）文本。

---

## 一、为什么检测编码这么难？

计算机里没有"编码"这个元数据。文件只是一堆 `0x` 字节。
*   `0xC4 0xE3` 在 GBK 里是 "你"。
*   `0xC4 0xE3` 在 ISO-8859-1 (西欧) 里是 "Äã"。

如果不靠猜（Heuristic），你永远不知道它是哪个。
这就是为什么打开旧代码经常看到乱码的原因。

## 二、TransSuccess 的检测流水线

TransSuccess 设计了一个漏斗式的检测流程，逐层排除。

### 2.1 第一层：BOM 检测 (The Easy Path)
最简单的情况。如果文件头是 `EF BB BF`，那是 UTF-8。如果是 `FF FE`，那是 UTF-16LE。
*   `TEncodingBOMDetector` 负责这一层。命中率 30%。

### 2.2 第二层：严格 UTF-8 验证
UTF-8 的字节序列有严格的规律（如多字节必须以 `10xxxxxx` 开头）。
如果一个文件**完全符合** UTF-8 规则，且包含一定数量的多字节字符，我们有 99.9% 的把握说它是 UTF-8。
*   `TUTF8EncodingDetector` 负责这一层。命中率 40%。

### 2.3 第三层：CJK 启发式竞速 (The Hard Path)
这是最难的部分。当文件既没有 BOM，也不是合法的 UTF-8 时，它可能是 ANSI (Windows默认编码)，具体是哪种 ANSI？

TransSuccess 引入了三个专门的检测器：
1.  **ChineseDetector** (GBK/GB18030)
2.  **JapaneseDetector** (Shift-JIS/EUC-JP)
3.  **KoreanDetector** (EUC-KR)

它们同时对文件的前 4KB 字节进行打分。

#### 打分逻辑 (Scoring)
以 GBK 为例：
*   **合法性检查**：GBK 的双字节范围是 `81-FE` + `40-FE`。落在这个范围内的给分。
*   **常用字检查**：如果解码出的字符落在"常用汉字区"，给高分。
*   **非法序列**：如果出现未定义的字节组合，扣分（甚至直接判死刑）。

### 2.4 决策仲裁
最终，控制器会比较三者的置信度（Confidence）。
```pascal
if (Chinese.Score > 0.8) and (Chinese.Score > Japanese.Score) then
  Result := 'GBK'
else if (Japanese.Score > 0.8) then
  Result := 'Shift-JIS'
else
  Result := 'ANSI'; // 放弃治疗，交给系统默认页
```

## 三、工程优化细节

在 `ControllerEncoding.pas` 中，我们做了一些关键优化：

### 3.1 采样 (Sampling)
不要读取整个文件。对于 1GB 的日志文件，读取前 **4KB** 通常就足够判断编码了。这让检测速度与文件大小无关。

### 3.2 异常捕获
读取文件时可能遇到从权限不足到文件被锁定等各种问题。代码中严密的 `try...except` 块保证了批量处理时，一个文件的错误不会中断整个任务。

### 3.3 安全网 (Safety Net)
如果所有检测器的置信度都很低，我们**绝不瞎猜**。我们会默认它可能是系统 ANSI，或者标记为 "Unknown"，提示用户手动选择。
**宁可不转，不可转错。** 转错（乱码再次转码）会导致数据永久损坏。

## 四、总结

字符编码检测没有银弹，它本质上是一个**概率游戏**。
TransSuccess 通过组合多种策略（BOM > 规则 > 统计），将识别准确率从通用的 80% 提升到了 **98%+**（特别是在 CJK 环境下）。

这套逻辑不仅用于转码工具，也是任何处理遗留数据的系统（如日志分析、代码迁移）所必须的基础设施。
