# 02.01 层上下文管理 vs 启发式裁剪

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-11
> **标签**: 上下文工程, RAG, 提示词工程, Progee

---

## 摘要

AI的上下文窗口（Context Window）虽然在变大（128k, 1M），但"注意力"是稀缺资源。塞入过多无关信息会导致AI"变傻"（Lost in the Middle）。现有的RAG方案多采用"启发式裁剪"（Heuristic Pruning），而Progee采用了确定性的**17层上下文管理架构**。本文对比这两种方案，解析为什么"结构化分层"是大型工程的唯一解。

---

## 一、启发式裁剪的陷阱

目前的AI辅助编程工具（如Cursor, Copilot）在处理大型项目时，通常采用以下策略来构建Prompt：

1.  **最近原则 (Recency)**：保留最近打开的5个文件。
2.  **相关性检索 (RAG)**：用向量搜索找最相似的20个代码块。
3.  **启发式压缩 (Heuristic)**：如果超长了，随机丢弃一些注释或中间行。

**这种策略的问题：**
*   **不确定性**：你永远不知道AI这次"看到"了什么。也许它看到了函数定义，但没看到上面的`@deprecated`注解。
*   **上下文漂移**：随着对话进行，最早期的核心约束（如"禁止使用第三方库"）可能被挤出窗口。
*   **注意力分散**：塞入大量低相关代码，稀释了关键指令的权重。

## 二、Progee的17层架构

Progee不依赖运气。我们设计了一个**17层上下文栈 (L1-L17)**，每一层都有明确的定义、优先级和注入时机。

### 2.1 架构全览

| 层级 | 名称 | 内容示例 | 注入策略 |
| :--- | :--- | :--- | :--- |
| **L1** | **硬边界 (Hard Rules)** | "绝对禁止明文存密码" | **Always** (置顶) |
| **L2** | **架构边界** | "UI层不能直连DB" | **Always** |
| **L3** | **流程边界** | "修改前必须加锁" | **Always** |
| **L4** | **系统规范** | 命名约定、Error定义 | **Global** |
| **L5** | **产品目标** | "做一个轻量级工具" | **Contract Scope** |
| **L6** | **用户意图** | 用户原始Prompt | **Task Scope** |
| **L7** | **功能树索引** | 模块结构图 | **Dynamic** |
| **L8** | **技术栈** | Python 3.11, Vue 3 | **Contract Scope** |
| **L9** | **代码风格** | Pylint规则 | **Task Scope** |
| **L10** | **契约规格** | Input/Output Schema | **Mandatory** |
| **L11** | **依赖关系** | 引用的Table/Class定义 | **Dependency Graph** |
| **L12** | **车间知识库** | 类似Bug的历史教训 | **Semantic Search** |
| **L13** | **资源锁状态** | 哪些文件只读 | **Runtime** |
| **L14** | **任务规格** | 具体要做什么 | **Mandatory** |
| **L15** | **验收标准** | Given-When-Then | **Mandatory** |
| **L16** | **执行上下文** | 当前尝试次数 | **Runtime** |
| **L17** | **纠正反馈** | 上次报错信息 | **On Retry** |

### 2.2 为什么这么分？

这17层不是为了炫技，而是为了**模拟人类专家的思维模型**。

*   **L1-L3 (宪法层)**：像人类的道德底线，任何时候都不能打破。它们被放在System Prompt的最前面，权重最高。
*   **L10/L14/L15 (任务层)**：这是当前工作的核心，必须完整注入，**一个字符都不能剪**。
*   **L7/L11/L12 (参考层)**：这是资料库。只有这部分会使用RAG或引用裁剪。

### 2.3 动态压栈算法

Progee有一个专门的 **Context Assembler（上下文组装器）**：

```python
def assemble_context(task, max_tokens=128000):
    context = []
    
    # 1. 必选层 (L1-L3, L10, L14, L15)
    # 这些是"硬骨头"，必须完整保留
    mandatory = load_layers([1, 2, 3, 10, 14, 15])
    remaining_tokens = max_tokens - count_tokens(mandatory)
    
    # 2. 动态层 (L11 依赖代码)
    # 根据依赖图谱，按深度优先加载
    dependencies = load_dependencies(task, limit=remaining_tokens * 0.6)
    
    # 3. 补充层 (L12 知识库)
    # 用剩下的空间放RAG检索结果
    knowledge = vector_search(task, limit=remaining_tokens * 0.4)
    
    return merge(mandatory, dependencies, knowledge)
```

## 三、实战效果对比

**任务**：修改 `User.login` 方法，增加验证码校验。

*   **启发式裁剪 (Copilot)**：
    *   可能注入了 `User.login` 代码。
    *   可能漏掉了 `CaptchaService` 的定义（因为它在另一个文件夹，且最近没打开）。
    *   **结果**：AI 幻觉出一个不存在的 `verify_code()` 函数。

*   **17层架构 (Progee)**：
    *   **L10**: 明确注入 `User` 和 `CaptchaService` 的契约定义。
    *   **L11**: 自动解析依赖，注入 `CaptchaService.validate` 的真实签名。
    *   **L1**: 注入"登录失败必须记录审计日志"的安全规则。
    *   **结果**：AI 生成了正确的调用代码，并自动加上了审计日志。

## 四、总结

上下文管理是AI工程化的深水区。
*   **启发式裁剪** 是把AI当聊天机器人，赌它能猜对。
*   **17层架构** 是把AI当工程师，给它提供精准的图纸和工具。

在Progee中，我们坚信：**Garbage Context In, Garbage Code Out。** 只有确定性的上下文，才能带来确定性的交付。
