# 01.35 为什么功能树比向量索引更精准

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-11
> **标签**: RAG, 向量检索, 功能树, 上下文管理

---

## 摘要

RAG（检索增强生成）是目前让AI理解私有代码库的主流方案。大多数RAG基于**向量索引（Vector Index）**。但在Progee的实践中，我们发现向量索引在代码检索上存在严重的精度问题。本文介绍Progee采用的替代方案——**功能树索引（L7 Functional Tree Index）**，解析为什么结构化导航比模糊搜索更适合软件工程。

---

## 一、向量索引的"迷雾"

向量检索（Vector Search）的原理是将代码片段转换成高维向量，然后计算余弦相似度。这对于自然语言文档（如Wiki）很有效，但对于代码，它经常失效。

### 1.1 "Lost in the Middle"（迷失在中间）
向量检索往往只能找到"语义相似"的片段，但无法理解代码的**结构关系**。
*   **用户问**："修改用户登录逻辑。"
*   **向量检索**：找出了 `login.py`，但也找出了 `login_test.py`，甚至 `login_button.tsx`。
*   **结果**：塞给AI一堆杂乱的上下文，AI不知道哪个才是核心逻辑。

### 1.2 缺乏全局视野
向量检索是**碎片化**的。它看不到：
*   这个函数被谁调用了？
*   这个类继承自谁？
*   这个模块属于哪个子系统？

代码是严密的逻辑结构，不是散落的文本碎片。用处理文本的方式处理代码，是降维打击（贬义）。

---

## 二、Progee的解决方案：L7 功能树

Progee 采用了 **Functional Tree（功能树）** 作为核心索引机制。这不是什么高深的黑科技，而是回归了软件工程的本质。

### 2.1 什么是功能树？

功能树是项目结构的**逻辑映射**，而不是文件系统的物理映射。

```text
Project Progee
├── System: 用户管理
│   ├── Module: 认证服务
│   │   ├── Function: 登录 (Login)
│   │   │   ├── Artifact: fn_login (PG函数)
│   │   │   ├── Artifact: User (Model)
│   │   │   └── Artifact: AuthService (Class)
│   │   └── Function: 注册 (Register)
│   └── Module: 权限服务
└── System: 任务管理
```

### 2.2 确定性导航 vs 概率性搜索

*   **向量检索**是**概率性**的：它猜测你可能需要 `login.py`。
*   **功能树**是**确定性**的：
    *   当我要修改"登录"功能时，我直接定位到 `System:用户管理 -> Function:登录` 节点。
    *   该节点下挂载的所有 Artifacts（表、函数、测试、文档）**百分之百**都是相关的，而且是**完整**的。

### 2.3 上下文注入的精准度

在 17层上下文架构 中，L7层就是功能树索引。

当 AI 开始执行一个任务时，它不需要去向量库里"大海捞针"。它只需要：
1.  确定当前任务归属的功能节点（Function Node）。
2.  提取该节点下的所有 Artifacts。
3.  (可选) 提取该节点的父级 Module 的公共 Artifacts。

这种方式注入的上下文，**信噪比（Signal-to-Noise Ratio）极高**。AI 看到的是一个完整的、结构化的功能切片，而不是一堆乱七八糟的代码碎片。

---

## 三、对比总结

| 维度 | 向量索引 (Vector Index) | 功能树索引 (Functional Tree) |
| :--- | :--- | :--- |
| **检索原理** | 语义相似度 (模糊) | 结构化导航 (精确) |
| **上下文完整性** | 碎片化，容易遗漏 | 完整切片，逻辑闭环 |
| **维护成本** | 需要不断Re-index | 随开发过程自然生长 |
| **适用场景** | 开放域问答，文档检索 | 软件开发，代码修改 |
| **AI 表现** | 容易幻觉，逻辑断裂 | 逻辑严密，修改准确 |

## 四、结论

软件工程是严谨的科学。在代码世界里，**结构（Structure）大于语义（Semantics）**。

Progee 选择功能树而不是盲目依赖向量检索，是因为我们深知：
**要让AI写对代码，首先要让AI看懂代码的骨架。**

向量检索可以作为辅助（例如搜索"那个处理日期的函数叫什么"），但核心的上下文管理，必须回归结构化。
