## 摘要

在追逐 Next.js, Rust, Vector DB 的浪潮中，我的技术栈显得格格不入：**Delphi, PostgreSQL, INI文件**。这听起来像是一个2000年的项目。但我正是用这些"土"技术，构建了最前沿的 AI Agent 系统 (Progee/WizAxis)。本文分享我的"实用主义"技术哲学。

---

## 一、被误解的"土"

大家觉得 Delphi 土，觉得 SQL 土，觉得 INI 土。
但这通常是因为混淆了 **"Old" (旧)** 和 **"Obsolete" (过时)**。

*   **过时**：指技术已经无法解决现代问题（如 Basic Model, IE）。
*   **旧**：指技术经过了时间的考验，变得稳定、高效、无处不在。

我选择的技术，都是**"Old but Gold"**。

### 1.1 Delphi vs Electron

现在做桌面应用，首选 Electron。但我选 Delphi (FireMonkey)。
*   **Electron**: 一个 Hello World 就要 100MB 内存，启动慢，依赖重。
*   **Delphi**: 编译出单一 EXE，几 MB 大小，**0.5秒启动**，原生性能。

做工具软件（如 TwoKeyRun, OmniSync），**快**就是最高的用户体验。用户不关心你用什么写的，只关心点一下是不是马上出来。

### 1.2 PostgreSQL vs 向量数据库

做 AI 应用，大家第一反应是 Pinecone, Milvus, Chroma。
我坚持用 **PostgreSQL + pgvector**。
*   **理由**：数据的一致性。
*   我的业务数据（用户、任务）在 PG 里。如果向量在另一个库，我就要维护分布式一致性，还要搞定备份、事务。
*   **All-in-PG** 让我可以在一个事务里同时更新业务数据和向量索引。这种**架构上的简单性**，省去了无数的运维麻烦。

### 1.3 INI vs YAML/JSON

我的很多配置依然用 INI。
*   **JSON**: 括号地狱，不支持注释。
*   **YAML**: 缩进地狱，容易手滑。
*   **INI**: 简单，扁平，支持注释，甚至非程序员都能看懂。

在 `EasyConfig` 项目中，我甚至让 INI 支持了层级引用。简单到了极致，就是强大。

## 二、唯快不破：开发效率的真相

我是一个人开发 20+ 个项目。我没有时间去配置复杂的 Webpack，去调试 npm 依赖冲突，去等待 Rust 编译。

**Delphi 的 RAD (Rapid Application Development) 模式** 是我保持高产的秘密。
*   **所见即所得**：拖控件，改属性，立马能跑。
*   **编译极快**：百万行代码几秒钟编译完。
*   **向后兼容**：10年前的代码，今天在新版 IDE 里依然能编译。

这种**确定性**和**速度**，让我可以把精力集中在**业务逻辑**和**AI 架构**上，而不是消耗在"配环境"上。

---

## 三、AI 时代的"土"技术优势

有趣的是，AI 似乎更喜欢这些"老"技术。

*   **SQL**: LLM 写 SQL 的能力远强于写 MongoDB 查询语法。因为 SQL 语料最丰富。
*   **Pascal**: 语法结构严谨，强类型，且自然语言可读性极强（Begin/End）。我发现 AI 生成 Pascal 代码的 Bug 率甚至低于 Python（因为 Python 动态类型太容易这就错了）。

ODD 方法论强调 **Definite (确定性)**。
老技术通常更加**显式、严格、确定**。这恰恰是控制 AI 幻觉的良药。

---

## 四、结语

时尚是轮回的。
*   Serverless 某种程度上回归了 CGI。
*   HTMX 让 HTML 回归了超文本。
*   AI 编程让 RAD 回归了主流。

我坚持用"土"技术，不是因为守旧，而是因为**透视了技术的本质**。
工具的目的是解决问题。如果一把旧锤子比新电钻更能精准地敲进钉子，那我为什么不用锤子呢？

用最朴实的代码，构建最智能的系统。这就是我的**"土味朋克"**。
