## 摘要

软件开发产出的不仅是代码。本文介绍Progee的698种产出物分类体系，覆盖代码、数据库、配置、文档、测试等14大类别，为AI辅助开发提供精确的"词汇表"。

---

## 1. 问题：产出物的模糊性

### 1.1 常见对话

```
你：帮我实现用户登录
AI：好的，这是登录代码...

问题：
  - 是要数据库表？还是API？还是界面？
  - 是要单元测试？还是集成测试？
  - 是要配置文件？还是文档？
```

### 1.2 根本原因

**缺乏精确的"产出物词汇表"**

就像化学有元素周期表，软件开发也需要一个完整的产出物分类体系。

## 3. 产出物属性

### 3.1 核心属性

每个产出物都有以下属性：

```yaml
artifact:
  # 分类
  category: code          # 14大类之一
  type_code: delphi_unit  # 698种之一
  language: delphi        # 编程语言
  platform: windows       # 目标平台
  
  # 标识
  name: CtrlUser          # 产出物名称
  file_path: CtrlUser.pas # 文件路径
  
  # ODD规格
  input_spec: {...}       # 输入规格
  output_spec: {...}      # 输出规格
  side_effects: [...]     # 副作用
  
  # 验证
  acceptance_criteria: [...] # 验收标准
  test_strategy: compile     # 验证策略
```

### 3.2 验证策略

| 策略 | 适用类型 | 验证方式 |
|------|----------|----------|
| compile | 代码类 | 编译通过 |
| execute | SQL/脚本 | 执行成功 |
| test | 有测试的 | 测试通过 |
| review | 文档/设计 | 人工审核 |
| effect | 行为类 | 检查副作用 |

## 4. 行为类产出物

### 4.1 特殊性

行为类产出物是**无形的**，但可以验证：

```
有形产出物：代码文件、配置文件、文档
无形产出物：状态转换、消息发送、API调用
```

### 4.2 7个子类别

| 子类别 | 示例 | 验证方式 |
|--------|------|----------|
| 数据操作 | INSERT/UPDATE/DELETE | 检查数据变化 |
| 消息通知 | 邮件/短信/推送 | 检查发送记录 |
| 外部调用 | HTTP API调用 | 检查调用日志 |
| 缓存操作 | 缓存写入/失效 | 检查缓存状态 |
| 权限审计 | 权限变更/审计日志 | 检查日志 |
| 任务调度 | 定时任务/队列任务 | 检查任务状态 |
| 弹性行为 | 熔断/限流/重试 | 检查触发记录 |

## 6. 实际应用

### 6.1 精确的任务定义

```yaml
# 模糊定义（不好）
task: 实现用户登录

# 精确定义（好）
tasks:
  - artifact_type: pg_table
    name: users
    
  - artifact_type: pg_function
    name: fn_login
    
  - artifact_type: delphi_form
    name: FrmLogin
    
  - artifact_type: unit_test
    name: test_login
```

### 6.2 AI理解更准确

```
模糊指令：
  "创建一个用户模块"
  → AI：是类？是表？是API？

精确指令：
  "创建 artifact_type=pg_table, name=users"
  → AI：明确知道要创建PostgreSQL表
```

## 7. 总结

698种产出物分类体系的价值：

> **为软件开发提供精确的"词汇表"**

通过这个体系：
- AI理解更准确（知道要产出什么）
- 验证更明确（每种类型有对应验证策略）
- 追溯更完整（产出物→任务→契约）
- 复用更方便（按类型查找相似产出物）

这是ODD方法论的基础设施。
