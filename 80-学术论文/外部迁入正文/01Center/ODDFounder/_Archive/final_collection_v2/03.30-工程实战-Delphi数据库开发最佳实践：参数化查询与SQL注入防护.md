# 03.30 Delphi数据库开发最佳实践：参数化查询与SQL注入防护

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )

> **标签**: Delphi, FireDAC, 安全, SQL注入, 最佳实践

---

## 摘要

在 Delphi 开发中，拼接 SQL 字符串是新手最常犯的错误，也是最大的安全隐患。**UniBase** 框架通过强制推行 **"零拼接"** 政策，建立了一套基于参数化查询的安全防线。本文分享 doQry 模块中的安全设计与最佳实践。

---

## 一、SQL 注入的原理与危害

```pascal
// 危险代码！
SQL := 'SELECT * FROM Users WHERE Name = ''' + EditName.Text + '''';
```
如果用户输入 `O'Malley`，SQL 就会报错。
如果用户输入 `' OR '1'='1`，他就能看到所有用户的数据。
这就是 SQL 注入。在 2026 年，写出这种代码是不可原谅的。

---

## 二、参数化查询 (Parameterized Query)

FireDAC 提供了完善的参数支持。

### 2.1 正确写法
```pascal
Query.SQL.Text := 'SELECT * FROM Users WHERE Name = :Name';
Query.ParamByName('Name').AsString := EditName.Text;
```
数据库引擎会将 `:Name` 视为一个**值**，而不是 SQL 代码的一部分。无论用户输入什么，它永远只会被当成字符串处理。

### 2.2 doQry 的封装
`uDoQry.pas` 将这一实践封装到了极致：
```pascal
DoQryExecSelect('GetUser', '{"Name": "' + EditName.Text + '"}', ...);
```
在底层，`uDoQryJsonParams` 会自动解析 JSON，并创建对应的 `TFDParam`。开发者根本没有机会去拼接 SQL。

---

## 三、防御性编码规范

### 3.1 强制使用 :Param
在 UniBase 项目中，我们启用了代码扫描工具，禁止在 `SQL.Text` 赋值语句中出现 `+` 号（除非是换行连接）。

### 3.2 动态表名/列名的处理
参数化查询不支持动态表名（如 `SELECT * FROM :TableName`）。
如果业务必须动态表名，我们使用**白名单验证**：
```pascal
if not IsValidTableName(TableName) then
  raise Exception.Create('Invalid table name');
SQL := Format('SELECT * FROM %s', [TableName]);
```

---

## 四、总结

**安全不是一个功能，而是一种习惯。**
通过 doQry 的强制参数化设计，我们从**机制**上杜绝了 SQL 注入的可能性。
记住：**永远不要信任用户的输入。**
