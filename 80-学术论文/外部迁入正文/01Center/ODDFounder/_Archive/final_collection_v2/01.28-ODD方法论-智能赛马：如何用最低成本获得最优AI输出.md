## 摘要

AI API调用成本是实际应用中的重要考量。本文介绍Progee的"智能赛马"机制，通过动态选择AI模型、失败升级策略、成本监控等手段，在保证质量的前提下最小化成本。

---

## 1. 问题：AI成本困境

### 1.1 模型定价差异

| 模型 | 输入价格 | 输出价格 | 能力 |
|------|----------|----------|------|
| Advanced Model | $30/1M | $60/1M | 强 |
| Claude-3-Advanced Model | $15/1M | $75/1M | 强 |
| Basic Model | $0.5/1M | $1.5/1M | 中 |
| Claude-3-Basic Model | $0.25/1M | $1.25/1M | 中 |

**价格差距：100倍以上**

### 1.2 常见误区

```
误区1：总是用最强模型
  问题：成本爆炸，简单任务杀鸡用牛刀

误区2：总是用最便宜模型
  问题：复杂任务失败率高，反复重试更贵

误区3：人工选择模型
  问题：效率低，难以规模化
```

---

## 2. 解决方案：智能赛马机制

### 2.1 核心思想

```
不是选择"最好的"模型
而是选择"够用的"模型

简单任务 → 便宜模型
复杂任务 → 强力模型
失败重试 → 升级模型
```

### 2.2 三层策略

```
┌─────────────────────────────────────────────────┐
│  Layer 1: 任务分级                               │
│  根据任务复杂度，预选模型等级                     │
├─────────────────────────────────────────────────┤
│  Layer 2: 失败升级                               │
│  便宜模型失败 → 自动升级到更强模型                │
├─────────────────────────────────────────────────┤
│  Layer 3: 成本监控                               │
│  实时监控成本，超预算告警                         │
└─────────────────────────────────────────────────┘
```

---

## 3. 任务分级策略

### 3.1 复杂度评估

| 因素 | 低复杂度 | 高复杂度 |
|------|----------|----------|
| 产出物类型 | 简单配置、SQL | 复杂算法、架构设计 |
| 代码行数 | <50行 | >200行 |
| 依赖数量 | 0-2个 | >5个 |
| 返工次数 | 0 | >2 |
| 历史失败率 | <10% | >30% |

### 3.2 模型选择矩阵

```
任务复杂度    推荐模型          备选模型
─────────────────────────────────────────
简单         Basic Model/Basic Model     Sonnet
中等         Sonnet/Advanced Model-mini  Advanced Model/Advanced Model
复杂         Advanced Model/Advanced Model        -
关键路径     Advanced Model/Advanced Model        人工审核
```

### 3.3 自动分级规则

```yaml
rules:
  - name: 简单配置
    conditions:
      - artifact_type in [env_config, app_config]
      - estimated_lines < 30
    model: Basic Model
    
  - name: 标准代码
    conditions:
      - artifact_type in [delphi_unit, python_module]
      - estimated_lines < 100
      - dependencies < 3
    model: sonnet
    
  - name: 复杂逻辑
    conditions:
      - artifact_type in [delphi_class, algorithm]
      - estimated_lines > 100
      - or: rework_count > 1
    model: Advanced Model
```

## 4. 失败升级策略

### 4.1 升级路径

```
Basic Model ──失败──▶ Sonnet ──失败──▶ Advanced Model ──失败──▶ 人工介入
  │              │               │
  └──成功────────┴───成功────────┴───成功──▶ 完成
```

### 4.2 升级触发条件

| 条件 | 说明 |
|------|------|
| 编译失败 | 生成的代码无法编译 |
| 测试失败 | 单元测试不通过 |
| 验收失败 | 不满足acceptance_criteria |
| 超时 | 执行时间超过阈值 |
| 重复错误 | 连续2次相同错误 |

### 4.3 升级决策

```pascal
function DecideUpgrade(Task: TTask; Failure: TFailure): TModel;
begin
  // 已经是最强模型，转人工
  if Task.CurrentModel = Advanced Model then
    Exit(HumanReview);
    
  // 编译错误，可能是简单问题，先重试
  if (Failure.Type = CompileError) and (Task.RetryCount < 2) then
    Exit(Task.CurrentModel);
    
  // 逻辑错误，升级模型
  if Failure.Type in [TestFailed, AcceptanceFailed] then
    Exit(UpgradeModel(Task.CurrentModel));
    
  // 重复错误，升级模型
  if Task.SameErrorCount >= 2 then
    Exit(UpgradeModel(Task.CurrentModel));
end;
```

## 6. 实际效果

### 6.1 成本对比

```
场景：100个混合复杂度任务

策略A（全用Advanced Model）：
  100任务 × $2/任务 = $200

策略B（全用Basic Model）：
  100任务 × $0.05/任务 = $5
  但：30%失败需要人工，实际成本更高

策略C（智能赛马）：
  60简单 × $0.05 = $3
  30中等 × $0.50 = $15
  10复杂 × $2.00 = $20
  总计: $38
  
节省：81%（相比全用Advanced Model）
```

### 6.2 质量保证

```
智能赛马的质量指标：
  - 一次成功率: 81%
  - 最终成功率: 97%（含升级）
  - 人工介入率: 3%
  
关键：便宜模型失败时自动升级，不牺牲质量
```

## 7. 实践建议

### 7.1 起步配置

```yaml
# 保守配置（质量优先）
default_model: sonnet
upgrade_threshold: 1  # 失败1次就升级
budget_alert: $100/day

# 激进配置（成本优先）
default_model: Basic Model
upgrade_threshold: 2  # 失败2次才升级
budget_alert: $20/day
```

### 7.2 持续优化

```
1. 收集数据：记录每个任务的模型、成本、成功率
2. 分析模式：哪类任务用便宜模型就够？
3. 调整规则：优化任务分级规则
4. 迭代改进：持续降低成本，保持质量
```

---

## 8. 总结

智能赛马的核心思想：

> **不是选最好的，而是选够用的**

通过三层策略：
- **任务分级**：简单任务用便宜模型
- **失败升级**：失败时自动升级
- **成本监控**：实时监控，超预算告警

实现了**成本降低80%，质量不打折**的目标。
