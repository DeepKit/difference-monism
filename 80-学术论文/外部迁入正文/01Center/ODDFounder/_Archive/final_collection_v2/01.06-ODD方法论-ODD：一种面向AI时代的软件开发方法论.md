# 01.06 ODD：一种面向AI时代的软件开发方法论

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-10
> **标签**: ODD, 方法论, AI开发, 软件工程

---

## 摘要

在AI辅助编程日益普及的今天，传统的软件开发方法论（TDD、BDD、DDD）面临新的挑战。本文介绍一种新的开发范式——**ODD（Output-Driven Development，输出驱动开发）**，其核心理念是"先定义产出物，再执行开发"。ODD特别适合人机协作场景，能够显著提升AI生成代码的质量和可控性。

---

## 1. 问题：AI时代的开发困境

### 1.1 传统方法论的局限

当我们使用AI（如Claude、GPT）辅助开发时，常常遇到这些问题：

```
❌ AI生成的代码"差不多对"，但细节不符合预期
❌ 反复修改提示词，效率反而更低
❌ AI不知道项目的整体架构，生成的代码难以集成
❌ 缺乏明确的验收标准，不知道什么时候算"完成"
```

传统方法论在AI时代的问题：

| 方法论 | 核心驱动 | AI时代的问题 |
|--------|----------|--------------|
| **TDD** | 测试用例 | AI写测试容易，但测试本身可能有问题 |
| **BDD** | 用户行为 | Given-When-Then太抽象，AI理解有偏差 |
| **DDD** | 领域模型 | 领域知识难以完整传递给AI |

### 1.2 根本原因

问题的根源在于：**我们告诉AI"怎么做"，却没有清晰定义"做出什么"**。

```
传统思路：需求 → 设计 → 编码 → 测试 → 交付
         （过程导向，AI容易在每一步偏离）

ODD思路：产出物定义 → 任务分解 → AI执行 → 验证交付
         （结果导向，AI有明确的目标）
```

## 2. ODD核心理念

### 2.1 一句话定义

> **ODD = 先定义产出物，再执行开发**

在写任何代码之前，先回答三个问题：
1. **产出什么？**（artifact_type：数据表？API？组件？）
2. **输入什么？**（input_spec：需要什么数据/依赖）
3. **如何验证？**（acceptance_criteria：怎么算完成）

### 2.2 ODD vs 传统方法

```
┌─────────────────────────────────────────────────────────────┐
│  传统开发                                                    │
│  "实现用户登录功能"                                          │
│  → AI理解：可能是表单？可能是API？可能是OAuth？               │
│  → 结果：反复沟通，效率低下                                  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  ODD开发                                                     │
│  任务1: artifact_type=pg_table, name=users                  │
│         output_spec={id, username, password_hash, ...}      │
│         acceptance=表存在且有正确字段                        │
│                                                             │
│  任务2: artifact_type=pg_function, name=fn_login            │
│         input_spec={username, password}                     │
│         output_spec={success, token, error_msg}             │
│         acceptance=正确密码返回token，错误密码返回error      │
│                                                             │
│  任务3: artifact_type=delphi_form, name=FrmLogin            │
│         input_spec={调用fn_login}                           │
│         output_spec={登录界面，含用户名密码输入框}           │
│         acceptance=界面可编译，调用API正确                   │
│                                                             │
│  → AI理解：每个任务都有明确的输入、输出、验收标准            │
│  → 结果：一次生成，精准交付                                  │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 ODD的"填空题"模型

ODD的精髓是把开发任务变成"填空题"而非"作文题"：

```
作文题（传统）：
  "请实现一个用户登录功能"
  → AI自由发挥，结果不可控

填空题（ODD）：
  artifact_type: ___pg_function___
  name: ___fn_login___
  input_spec: 
    - username: VARCHAR(50)
    - password: VARCHAR(100)
  output_spec:
    - success: BOOLEAN
    - token: VARCHAR(200)
    - error_msg: TEXT
  acceptance_criteria:
    - Given 正确的用户名密码 When 调用fn_login Then 返回success=true
    - Given 错误的密码 When 调用fn_login Then 返回success=false
  
  → AI只需"填空"，结果可控
```

## 3. ODD核心概念

### 3.1 产出物（Artifact）

产出物是ODD的核心概念，指软件开发过程中产生的、可验证的、有价值的交付物。

**产出物分类**（14大类，698种）：

| 类别 | 示例 | 是否有形 |
|------|------|:--------:|
| 代码 | 类、函数、组件 | ✓ |
| 数据库 | 表、视图、存储过程 | ✓ |
| 配置 | 环境变量、应用配置 | ✓ |
| 基础设施 | Dockerfile、K8s配置 | ✓ |
| 文档 | 设计文档、API文档 | ✓ |
| 测试 | 单元测试、集成测试 | ✓ |
| **行为** | 状态转换、消息发送 | ✗ |
| 构建 | 可执行文件、Docker镜像 | ✓ |
| 安全 | 证书、密钥、策略 | ✓ |
| API | OpenAPI、GraphQL Schema | ✓ |
| 设计 | UI设计、架构图 | ✓ |
| 资源 | 图片、字体、图标 | ✓ |
| AI/ML | 模型、数据集、Prompt | ✓ |
| 数据 | 迁移脚本、种子数据 | ✓ |

### 3.2 契约（Contract）

契约是对一组相关产出物的承诺：

```yaml
contract:
  name: 用户登录功能
  description: 实现完整的用户登录流程
  artifacts:
    - type: pg_table, name: users
    - type: pg_function, name: fn_login
    - type: delphi_form, name: FrmLogin
    - type: unit_test, name: test_login
  acceptance:
    - 用户可以通过界面登录
    - 错误密码显示提示信息
    - 登录成功后跳转到主界面
```

### 3.3 任务（Task）

任务是生产单个产出物的工作单元：

```yaml
task:
  artifact_type: pg_function
  artifact_name: fn_login
  
  input_spec:
    p_username: VARCHAR(50)
    p_password: VARCHAR(100)
  
  output_spec:
    success: BOOLEAN
    token: VARCHAR(200)
    error_msg: TEXT
  
  side_effects:
    - 更新users.last_login_at
    - 插入login_logs记录
  
  preconditions:
    - users表已存在
    - pgcrypto扩展已安装
  
  acceptance_criteria:
    - Given 存在用户admin/123456
      When 调用fn_login('admin', '123456')
      Then 返回success=true, token非空
    - Given 存在用户admin/123456
      When 调用fn_login('admin', 'wrong')
      Then 返回success=false, error_msg='密码错误'
```

## 4. ODD工作流程

### 4.1 五步循环

```
┌─────────────────────────────────────────────────────────────┐
│  ODD 核心循环                                                │
│                                                             │
│  1. 定义（Define）                                          │
│     定义产出物类型、输入输出规格、验收标准                    │
│     ↓                                                       │
│  2. 分解（Decompose）                                       │
│     将契约分解为多个任务，明确依赖关系                        │
│     ↓                                                       │
│  3. 执行（Execute）                                         │
│     AI按任务规格生成产出物                                   │
│     ↓                                                       │
│  4. 验证（Verify）                                          │
│     按验收标准检查产出物                                     │
│     ↓                                                       │
│  5. 封版（Seal）                                            │
│     验证通过后封版，不可更改                                 │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 验证策略

ODD定义了5种验证策略：

| 策略 | 适用场景 | 验证方式 |
|------|----------|----------|
| **compile** | 代码类 | 编译通过 |
| **execute** | 脚本/SQL | 执行成功 |
| **test** | 有测试的产出物 | 测试通过 |
| **review** | 文档/设计 | 人工审核 |
| **effect** | 行为类 | 检查副作用 |

## 5. ODD vs 其他方法论

### 5.1 对比表

| 维度 | TDD | BDD | DDD | ODD |
|------|-----|-----|-----|-----|
| **驱动因素** | 测试 | 行为 | 领域 | 产出物 |
| **核心产物** | 测试+代码 | 规范+代码 | 模型+代码 | 明确的产出物 |
| **AI友好度** | 中 | 中 | 低 | **高** |
| **验收明确性** | 高 | 中 | 低 | **高** |
| **适用范围** | 代码 | 功能 | 架构 | **全类型** |

### 5.2 ODD的独特优势

1. **AI友好**：填空题模型减少AI的不确定性
2. **全类型覆盖**：不仅是代码，还包括数据库、配置、文档等
3. **可验证**：每个产出物都有明确的验收标准
4. **可追溯**：产出物→任务→契约，完整追溯链

## 6. 实践建议

### 6.1 从小处开始

```
第一步：定义产出物类型
  - 这个任务要产出什么？代码？配置？文档？

第二步：明确输入输出
  - 输入：需要什么数据/依赖？
  - 输出：产出物的具体内容是什么？

第三步：写验收标准
  - 怎么验证产出物是正确的？
  - 用Given-When-Then格式
```

### 6.2 常见误区

```
❌ 误区1：产出物定义太模糊
   错误："创建一个用户模块"
   正确："创建pg_table:users，包含id/username/password_hash字段"

❌ 误区2：跳过验收标准
   错误：只定义产出物，不定义如何验证
   正确：每个产出物都有对应的acceptance_criteria

❌ 误区3：一个任务多个产出物
   错误：一个任务同时创建表+函数+界面
   正确：一个任务一个产出物，通过契约组织
```

---

## 7. 总结

ODD（输出驱动开发）是一种面向AI时代的软件开发方法论，其核心理念是：

> **先定义产出物，再执行开发**

通过将开发任务从"作文题"变成"填空题"，ODD能够：
- 提高AI生成代码的准确性
- 建立明确的验收标准
- 覆盖全类型产出物（不仅是代码）
- 实现可追溯的开发过程

在人机协作日益普及的今天，ODD为我们提供了一种更高效、更可控的开发范式。
