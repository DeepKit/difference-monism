# 01.12 契约驱动开发：做什么/不做什么/前置/后置

## 摘要

"契约"（Contract）是ODD方法论的灵魂。但ODD的契约不仅仅是API接口定义。它包含四个核心维度：**Do（做什么）、Don't（不做什么）、Pre（前置条件）、Post（后置条件）**。本文基于Progee的实践，深入解析如何写出一份AI"不得不遵守"的完美契约。

---

## 一、不仅是接口定义

传统的API文档（如Swagger）只告诉我们：输入A，返回B。
这对人类程序员可能够了，因为人类有常识。但对AI来说，这远远不够。

AI缺乏常识。如果你不告诉它"不要删除原文件"，它可能在处理完数据后顺手把源文件删了。如果你不告诉它"必须检查用户余额"，它可能直接扣成负数。

因此，ODD的契约必须是**防御性**的。

---

## 二、契约的四维结构

一个完整的ODD任务契约包含以下四个部分：

### 2.1 Do (做什么)
这是核心功能的正向描述。
*   *例*："创建一个名为 `users` 的PostgreSQL表。"
*   *要求*：包含 `Artifact Type`, `Name`, `Schema`。

### 2.2 Don't (不做什么)
这是**负向约束（Negative Constraints）**。这是防止AI"自作聪明"的关键。
*   *例*：
    *   "不要创建任何索引（Index），以后会单独创建。"
    *   "不要使用 `SERIAL` 类型，必须使用 `IDENTITY`。"
    *   "不要在表中包含 `deleted_at` 软删除字段。"
*   *价值*：AI往往倾向于生成"大而全"的代码，Don't 约束能有效地剪除这些冗余，保持设计的一致性。

### 2.3 Pre (前置条件)
这是任务执行的前提。
*   *例*：
    *   "必须确保 `public` schema 已经存在。"
    *   "必须确保 `uuid-ossp` 扩展已安装。"
*   *价值*：如果前置条件不满足，任务应立即失败（Fail Fast），而不是执行一半报错。这对于**幂等性**至关重要。

### 2.4 Post (后置条件/验收)
这是任务完成后的验证标准。
*   *例*：
    *   "表 `users` 必须存在。"
    *   "字段 `email` 必须有唯一约束。"
    *   "执行 `INSERT` 测试数据应当成功。"
*   *价值*：这是 **Verify (验证)** 阶段的执行依据。

## 三、Progee中的契约实战

在Progee中，契约被定义为YAML格式：

```yaml
contract:
  name: CreateUserTable
  type: pg_table
  
  # 1. Do: 正向定义
  spec:
    name: users
    columns:
      - name: id
        type: uuid
        default: gen_random_uuid()
      - name: email
        type: varchar(255)
        nullable: false
  
  # 2. Don't: 负向约束
  constraints:
    - no_triggers: true
    - no_foreign_keys: true  # 暂时不建外键
  
  # 3. Pre: 前置检查
  preconditions:
    - check_extension: "pgcrypto"
  
  # 4. Post: 验收标准
  acceptance:
    - verify_table_exists: "users"
    - verify_column: { table: "users", column: "email", unique: true }
```

### 为什么这样设计？

1.  **AI友好**：LLM非常擅长理解这种结构化的约束。当我们明确列出 `Don't` 时，AI生成代码时会自我审查。
2.  **自动化测试**：`Pre` 和 `Post` 部分可以直接被转换为自动化测试脚本。
    *   `preconditions` -> `setup()`
    *   `acceptance` -> `test()`
3.  **责任明确**：如果代码出了问题，我们可以精准定位是"契约没写好"（架构师责任）还是"AI没执行好"（工人责任）。

---

## 四、从 DbC 到 ODD

Bertrand Meyer 在1986年提出了 **DbC (Design by Contract)**，主要用于类的方法设计。ODD将这一思想扩展到了**AI协作**的宏观层面。

在人机协作中，**信任是昂贵的，契约是廉价的**。
与其信任AI"应该知道"，不如用契约白纸黑字地写下来。

*   **Pre** 保护了环境。
*   **Post** 保护了结果。
*   **Don't** 保护了架构的纯洁性。

这就是ODD能让AI产出工业级代码的保障。
