# 03.18 流式大文件处理：64KB分块与多字节边界

## 摘要

处理 100MB 的文件很简单，但处理 10GB 的文件呢？一次性读入内存（ReadAllText）会导致 OutOfMemory。**TransSuccess** 采用 **流式处理 (Streaming)** 架构，以 **64KB** 为单位分块读取。但分块带来了一个致命问题：**多字节字符被切断**。本文揭秘如何优雅地处理这个边界难题。

---

## 一、分块的必要性与风险

为了支持任意大小的文件转码，我们必须使用 `TFileStream` 分块读取。
`Buffer Size = 64 * 1024` (64KB)。

**风险**：
UTF-8 中的汉字 "你" 是 3 个字节 `E4 BD A0`。
如果 64KB 的切分点恰好切在 `E4` 和 `BD` 之间：
*   Block 1 结尾：`... E4`
*   Block 2 开头：`BD A0 ...`

单独解码 Block 1 会产生乱码（），单独解码 Block 2 也会乱码。转码后的文件就会出现损坏。

---

## 二、边界处理算法 (`EncodingConverter_Improved.pas`)

TransSuccess 采用了一种**回溯 (Backtracking)** 策略。

### 2.1 算法逻辑
1.  读取 64KB 到缓冲区。
2.  **检查末尾**：判断缓冲区的最后一个字符是否完整。
    *   对于 UTF-8：检查最后几个字节是否符合 `10xxxxxx` 模式。如果是，说明它是多字节序列的一部分，需要回溯找到该序列的起始字节（`11xxxxxx`）。
    *   对于 GBK/Shift-JIS：检查是否是双字节字符的第一个字节（High Byte）。
3.  **回退 (Rewind)**：如果发现末尾被切断，将切断的这几个字节**退回**给流（Stream.Seek 往回跳），或者暂存到下一个 Block 的头部。
4.  **解码**：只解码完整的部分。

### 2.2 缓冲区池 (Buffer Pool)
为了极致性能，我们使用了 `UtilsBufferPool` 复用内存，避免在循环中频繁分配和释放 64KB 的数组。

## 三、代码实现片段

```pascal
// 伪代码
bytesRead := Stream.Read(Buffer, BufferSize);
incompleteBytes := DetectIncompleteSequence(Buffer, bytesRead, Encoding);

if incompleteBytes > 0 then
begin
  // 回退流指针
  Stream.Seek(-incompleteBytes, soCurrent);
  // 调整本次处理长度
  bytesRead := bytesRead - incompleteBytes;
end;

// 现在 Buffer[0..bytesRead-1] 是安全的，可以转换
ConvertBuffer(Buffer, bytesRead);
```

## 四、总结

大文件处理是检验 IO 功底的试金石。
TransSuccess 的流式架构保证了：无论文件多大，内存占用永远恒定（~几MB）。而精妙的边界处理算法，确保了数据的绝对完整性。这是工业级工具与玩具代码的区别。
