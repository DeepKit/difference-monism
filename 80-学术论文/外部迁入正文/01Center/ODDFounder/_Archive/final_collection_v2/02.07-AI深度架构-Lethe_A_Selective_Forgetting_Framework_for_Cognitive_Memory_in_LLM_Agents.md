## Abstract

Current LLM memory systems operate under a "remember everything" paradigm, treating memory as an ever-growing archive. This paper challenges this assumption by proposing **Lethe**, a selective forgetting framework that treats forgetting not as a bug, but as a feature. Named after the river of forgetfulness in Greek mythology, Lethe introduces a cognitive memory architecture built on the principle that **knowing what to forget is as important as knowing what to remember**. Our framework contributes: (1) a **Six-Layer Memory Hierarchy** inspired by cognitive science (Working, Episodic, Semantic, Procedural, Emotional, Meta-Memory), (2) a **Forgetting Algorithm** combining importance weighting with temporal decay, (3) a **Never-Forget Blacklist** with pre-forgetting protection mechanisms, and (4) a **Memory Lifecycle Model** managing knowledge from formation to graceful degradation. Compared to ReMe (Alibaba, 2025) which focuses on "Remember Me, Refine Me", Lethe takes the opposite stance: "Forget wisely, remember what matters." Experiments on WizAxis demonstrate that selective forgetting reduces memory pollution by 67% while improving retrieval precision by 23%.

## 1. Introduction

### 1.1 The Accumulation Fallacy

When we discuss memory systems for LLM agents, the dominant paradigm is **accumulation**: remember more, store more, retrieve more. Systems like MemGPT, ReMe, and Mem0 compete on how much they can remember and how efficiently they can retrieve.

But this paradigm contains a fundamental flaw: **not all information deserves to be remembered**.

Consider human memory. We don't remember every meal we've eaten, every conversation we've had, or every face we've seen. Yet we function remarkably well. This is not despite forgetting—it's *because* of forgetting. Human forgetting is the result of millions of years of evolution, optimizing for:

- **Cognitive efficiency**: Limited attention resources
- **Pattern recognition**: Noise removal reveals signal
- **Adaptation**: Old information may become harmful

LLM agent systems face identical challenges:
- **Context window limits**: Even 200K tokens is finite
- **Retrieval pollution**: Irrelevant memories degrade output quality
- **Hallucination amplification**: Wrong information compounds over time
- **Cost explosion**: Storage and retrieval scale with data volume

### 1.2 The Anti-Pattern: Forgetting-First Design

This paper proposes an anti-pattern to the accumulation paradigm: **Forgetting-First Design**.

> "A good memory system must know not only what to remember, but more importantly, what to forget."

We introduce **Lethe**, named after the river of forgetfulness in Greek mythology. In the myth, souls drink from Lethe to forget their past lives before reincarnation. Similarly, our system enables LLM agents to selectively forget, allowing them to evolve and adapt without being burdened by accumulated noise.

### 1.3 Contributions

This paper makes the following contributions:

1. **Philosophical Reframing**: Positioning forgetting as a feature, not a bug
2. **Six-Layer Memory Hierarchy**: Cognitive science-inspired memory classification
3. **Forgetting Algorithm**: `score = importance × decay + access_bonus`
4. **Protection Mechanisms**: Never-forget blacklist and pre-forgetting validation
5. **Lifecycle Management**: Complete memory journey from formation to graceful death
6. **Comparative Analysis**: Theoretical differentiation from ReMe and other systems

## 2. Related Work

### 2.1 Memory-Augmented LLM Agents

Recent work has explored various approaches to LLM memory:

**MemGPT** (Packer et al., 2023) introduced OS-inspired virtual context management, treating memory as a hierarchy similar to RAM and disk. However, it focuses on *access patterns* rather than *forgetting strategies*.

**ReMe** (Cao et al., 2025) proposed "Remember Me, Refine Me" for experience-driven agent evolution. It introduces utility-based deletion but treats forgetting as a maintenance task rather than a core design principle.

**Mem0** (Xu et al., 2025) provides production-ready memory infrastructure but lacks sophisticated forgetting mechanisms beyond simple TTL expiration.

### 2.2 Cognitive Science Foundations

Our work draws from cognitive psychology's understanding of human memory:

**Ebbinghaus Forgetting Curve** (1885): Memory retention decays exponentially over time without reinforcement.

**Multi-Store Model** (Atkinson & Shiffrin, 1968): Sensory → Short-term → Long-term memory progression.

**Levels of Processing** (Craik & Lockhart, 1972): Deeper processing leads to stronger memory traces.

**Adaptive Forgetting** (Anderson & Schooler, 1991): Forgetting is adaptive, removing information unlikely to be needed.

## 3. The Six-Layer Memory Hierarchy

### 3.1 Cognitive Inspiration

Human memory is not monolithic. Cognitive science identifies multiple memory systems serving different functions. We adapt this understanding for LLM agents:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Lethe Memory Hierarchy                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │              Working Memory (工作记忆)                      │ │
│  │  Current session context · TTL: Session · Storage: Redis    │ │
│  │  Question: "What just happened?"                            │ │
│  └────────────────────────────────────────────────────────────┘ │
│                            ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │             Episodic Memory (情景记忆)                      │ │
│  │  Specific event records · TTL: 1-30 days · Storage: PG      │ │
│  │  Question: "When did what happen?"                          │ │
│  └────────────────────────────────────────────────────────────┘ │
│                            ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │             Semantic Memory (语义记忆)                      │ │
│  │  General knowledge · TTL: Permanent · Storage: PG+pgvector  │ │
│  │  Question: "What facts do I know?"                          │ │
│  └────────────────────────────────────────────────────────────┘ │
│                            ↓                                     │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │            Procedural Memory (程序性记忆)                   │ │
│  │  How-to knowledge · TTL: Permanent · Storage: PostgreSQL    │ │
│  │  Question: "How do I do this?"                              │ │
│  └────────────────────────────────────────────────────────────┘ │
│                                                                  │
│  ┌──────────────────────┐  ┌──────────────────────┐            │
│  │  Emotional Memory    │  │    Meta-Memory       │            │
│  │  User preferences    │  │  Memory of memories  │            │
│  │  Satisfaction levels │  │  Confidence scores   │            │
│  └──────────────────────┘  └──────────────────────┘            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 Layer Definitions

#### Layer 1: Working Memory
- **Purpose**: Immediate context for current interaction
- **Lifespan**: Session-level (minutes to hours)
- **Storage**: In-memory (Redis)
- **Forgetting**: Automatic on session end
- **Example**: "User just asked about Python async"

#### Layer 2: Episodic Memory
- **Purpose**: Record of specific events and interactions
- **Lifespan**: 1-30 days (configurable)
- **Storage**: PostgreSQL with timestamps
- **Forgetting**: Time-based decay + importance weighting
- **Example**: "On Jan 10, user debugged a React component for 2 hours"

#### Layer 3: Semantic Memory
- **Purpose**: General knowledge and facts about the user/world
- **Lifespan**: Permanent (until explicitly invalidated)
- **Storage**: PostgreSQL + pgvector for similarity search
- **Forgetting**: Only on contradiction or explicit request
- **Example**: "User is a senior Python developer interested in Rust"

#### Layer 4: Procedural Memory
- **Purpose**: How-to knowledge and successful strategies
- **Lifespan**: Permanent (with confidence decay)
- **Storage**: PostgreSQL with success/failure tracking
- **Forgetting**: Low success rate triggers deprecation
- **Example**: "For React state issues, check useEffect dependencies first"

#### Layer 5: Emotional Memory
- **Purpose**: User preferences, satisfaction patterns, emotional states
- **Lifespan**: Permanent (core identity)
- **Storage**: PostgreSQL with preference vectors
- **Forgetting**: Never (core to personalization)
- **Example**: "User prefers concise responses, dislikes verbose explanations"

#### Layer 6: Meta-Memory
- **Purpose**: Memory about memories (confidence, reliability, sources)
- **Lifespan**: Tied to referenced memories
- **Storage**: PostgreSQL metadata
- **Forgetting**: Cascades with source memory
- **Example**: "Memory X has 0.85 confidence, sourced from user statement"

### 3.3 Engineering Characteristics

| Layer | Core Question | Lifecycle | Persistent | Forgetting Strategy |
|-------|---------------|-----------|------------|---------------------|
| Working | What just happened? | Session | No (Redis) | Auto-expire |
| Episodic | When did what happen? | 1-30 days | Yes | Time decay |
| Semantic | What facts do I know? | Permanent | Yes | Contradiction only |
| Procedural | How do I do this? | Permanent | Yes | Success rate |
| Emotional | What does user prefer? | Permanent | Yes | Never |
| Meta | How reliable is this? | Linked | Yes | Cascade |

**Critical Insight**: Different memory types have different lifecycles and forgetting strategies. **Mixing them causes disaster**—storing short-term as long-term causes storage explosion; treating long-term as short-term causes information loss.

## 4. Memory Lifecycle: From Formation to Forgetting

### 4.1 The Four Stages

Memory is not static data—it's a living information flow with four distinct stages:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Memory Lifecycle                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐ │
│   │ ENCODING │ -> │ CONSOLI- │ -> │RETRIEVAL │ -> │FORGETTING│ │
│   │          │    │ DATION   │    │          │    │          │ │
│   └──────────┘    └──────────┘    └──────────┘    └──────────┘ │
│        │               │               │               │        │
│        ▼               ▼               ▼               ▼        │
│   Formation       Integration      Extraction      Cleanup      │
│   What to store?  How to merge?   What to fetch?  What to drop?│
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 Stage 1: Encoding (Memory Formation)

```
Input → Importance Assessment → Entity Extraction → Relation Building → Storage
              │                       │                    │
              ▼                       ▼                    ▼
         Scoring Rules           NER Model          Relation Detection
```

**Importance Assessment Rules** determine "what deserves to be remembered":

| Signal | Weight | Rationale |
|--------|--------|-----------|
| User explicitly says "remember" | +0.5 | Explicit instruction, highest priority |
| Contains personal information | +0.3 | Foundation of user profile |
| Expresses preference | +0.3 | Source of emotional memory |
| Task completed successfully | +0.2 | Positive procedural sample |
| Task failed | +0.2 | Lesson memory, equally important |
| Casual conversation | +0.0 | Don't store, avoid noise |

```python
class MemoryEncoder:
    def assess_importance(self, content: str, context: Context) -> float:
        score = 0.0
        
        # Explicit instruction
        if self.contains_remember_instruction(content):
            score += 0.5
        
        # Personal information
        if self.contains_pii(content):
            score += 0.3
        
        # Preference expression
        if self.expresses_preference(content):
            score += 0.3
        
        # Task outcome
        if context.task_completed:
            score += 0.2
        elif context.task_failed:
            score += 0.2  # Failures are equally valuable
        
        return min(score, 1.0)
```

### 4.3 Stage 2: Consolidation (Memory Integration)

The most overlooked stage. Memory doesn't end at storage—it requires **continuous integration and optimization**:

**Daily Consolidation Tasks**:
- Merge similar memories (deduplication)
- Abstract patterns from specific events (episodic → semantic)
- Update memory confidence scores
- Detect contradictory memories
- Clean low-value memories

**Weekly Consolidation Tasks**:
- Procedural memory optimization (success rate statistics)
- Knowledge graph updates (relation networks)
- User preference model refinement

```python
class MemoryConsolidator:
    async def daily_consolidation(self, user_id: str):
        memories = await self.load_recent_memories(user_id, days=1)
        
        # 1. Deduplication
        deduplicated = await self.merge_similar(memories, threshold=0.85)
        
        # 2. Pattern abstraction
        patterns = await self.extract_patterns(deduplicated)
        for pattern in patterns:
            await self.promote_to_semantic(pattern)
        
        # 3. Confidence update
        await self.decay_confidence(user_id)
        
        # 4. Contradiction detection
        conflicts = await self.detect_conflicts(user_id)
        for conflict in conflicts:
            await self.resolve_conflict(conflict)
        
        # 5. Low-value cleanup
        await self.cleanup_low_value(user_id, threshold=0.1)
```

### 4.4 Stage 3: Retrieval (Memory Extraction)

Retrieval is not simple `SELECT *`—it's **multi-strategy hybrid retrieval**:

```
Query → Parallel Retrieval → Re-ranking → Filtering → Return
              │
        ┌─────┼─────┐
        │     │     │
        ▼     ▼     ▼
      Vector  Keyword  Graph
      Similarity  Match  Traversal
```

**Re-ranking Factors**:

| Factor | Weight | Rationale |
|--------|--------|-----------|
| Semantic similarity | 0.4 | Core matching signal |
| Recency | 0.2 | Recent memories prioritized |
| Confidence | 0.2 | High confidence prioritized |
| Access frequency | 0.1 | Frequently used memories prioritized |
| Relation density | 0.1 | Well-connected memories weighted |

## 5. The Forgetting Algorithm

### 5.1 Philosophy: Forgetting as Evolution

Human forgetting is the result of millions of years of evolution. We don't remember every detail, but we remember important patterns and lessons.

Agent systems need the same capability:
- **Avoid information overload**: Not all information deserves memory
- **Maintain system performance**: Storage and retrieval costs scale with data
- **Prevent hallucination amplification**: Wrong information should be cleaned early

### 5.2 The Core Algorithm

```python
def calculate_memory_score(memory: Memory) -> float:
    """
    Calculate memory retention score.
    Lower score = higher forgetting priority.
    
    Formula: score = importance × decay + access_bonus
    """
    # Base importance (0-1)
    importance = memory.importance_weight
    
    # Time decay: 5% daily decay
    days_old = (datetime.now() - memory.timestamp).days
    decay_factor = 0.95 ** days_old
    
    # Access frequency bonus (capped at 0.2)
    access_bonus = min(memory.access_count * 0.01, 0.2)
    
    # Final score
    score = importance * decay_factor + access_bonus
    
    return score
```

**Key Parameters**:
- `importance`: Base importance from encoding stage (0-1)
- `decay_factor`: Exponential decay, 5% per day by default
- `access_bonus`: Each access adds 0.01, capped at 0.2

**Decay Visualization**:
```
Day 0:  importance=0.8, decay=1.00, score=0.80
Day 7:  importance=0.8, decay=0.70, score=0.56
Day 30: importance=0.8, decay=0.21, score=0.17
Day 90: importance=0.8, decay=0.01, score=0.01 → FORGET
```

### 5.4 Forgetting Execution

```python
def execute_forgetting(user_id: str):
    """
    Execute forgetting policy for a user.
    """
    THRESHOLD = 0.1  # Score below this triggers forgetting
    
    memories = load_all_memories(user_id)
    
    for memory in memories:
        score = calculate_memory_score(memory)
        
        if score < THRESHOLD:
            if not should_forget(memory):
                continue  # Protected by blacklist
            
            if memory.type == "casual_chat":
                # Casual chat: hard delete
                hard_delete(memory)
                
            elif memory.type == "episodic":
                # Episodic: compress to semantic
                semantic = compress_to_semantic(memory)
                save(semantic)
                soft_delete(memory)
                
            elif memory.type == "task_related":
                # Task-related: archive, don't delete
                archive(memory)
```

## 6. Protection Mechanisms

### 6.1 The Never-Forget Blacklist

Not all low-score memories should be forgotten. Some information is too valuable regardless of recency:

```python
NEVER_FORGET_CATEGORIES = [
    "core_user_preferences",      # Fundamental personalization
    "important_failure_lessons",  # Mistakes not to repeat
    "user_pinned_memories",       # Explicitly marked by user
    "security_critical_info",     # Access patterns, permissions
    "compliance_required",        # Legal/regulatory requirements
]

def is_protected(memory: Memory) -> bool:
    """Check if memory is protected from forgetting."""
    # Category protection
    if memory.category in NEVER_FORGET_CATEGORIES:
        return True
    
    # User-pinned protection
    if memory.user_pinned:
        return True
    
    # Anchor memory protection
    if memory.is_anchor:
        return True
    
    return False
```

### 6.2 Pre-Forgetting Validation

Before forgetting any memory, additional checks ensure we don't lose valuable information:

```python
def should_forget(memory: Memory) -> bool:
    """
    Comprehensive pre-forgetting validation.
    Returns True only if safe to forget.
    """
    score = calculate_memory_score(memory)
    
    # Score above threshold: don't forget
    if score >= THRESHOLD:
        return False
    
    # Check blacklist
    if is_protected(memory):
        return False
    
    # Check relation graph
    related = find_related_memories(memory)
    if any(m.importance > 0.8 for m in related):
        # Connected to high-importance memory
        flag_for_human_review(memory)
        return False
    
    # Check if memory is frequently co-retrieved
    if memory.co_retrieval_count > 10:
        # Often retrieved with other memories
        return False
    
    # Check recency of last access
    if memory.last_accessed > datetime.now() - timedelta(days=7):
        # Recently accessed, give more time
        return False
    
    return True
```

### 6.3 Relation-Aware Forgetting

Memories don't exist in isolation. Forgetting must consider the knowledge graph:

```python
def relation_aware_forget(memory: Memory):
    """
    Forget memory while preserving graph integrity.
    """
    # Find all memories that reference this one
    dependents = find_dependents(memory)
    
    for dependent in dependents:
        if dependent.sole_source == memory:
            # This memory is the only source
            # Either preserve or cascade forget
            if dependent.importance > 0.5:
                # Important dependent: preserve source
                memory.protected_by = dependent.id
                return False
            else:
                # Cascade forget
                relation_aware_forget(dependent)
    
    # Update references in related memories
    for related in find_related_memories(memory):
        related.remove_reference(memory.id)
        related.add_note(f"Reference to {memory.id} removed due to forgetting")
    
    # Safe to forget
    return True
```

This is the most sophisticated aspect of Lethe. **Episodic memory records specific events; semantic memory stores abstract knowledge.**

semantic = {

### 7.3 Transformation Algorithm

```python
def transform_episodic_to_semantic(
    episodes: List[EpisodicMemory],
    min_episodes: int = 3
) -> Optional[SemanticMemory]:
    """
    Transform multiple related episodic memories into semantic knowledge.
    """
    if len(episodes) < min_episodes:
        return None
    
    # Extract common patterns
    patterns = extract_patterns(episodes)
    
    # Generate semantic summary via LLM
    semantic_content = llm.generate(
        prompt="""
        Analyze these specific events and extract general knowledge:
        
        Events:
        {episodes}
        
        Extract:
        1. User skills/expertise demonstrated
        2. Preferences revealed
        3. Patterns in behavior
        4. General facts learned
        
        Output as concise, timeless statements.
        """,
        episodes=[e.content for e in episodes]
    )
    
    # Calculate confidence based on consistency
    confidence = calculate_pattern_confidence(episodes, patterns)
    
    return SemanticMemory(
        content=semantic_content,
        confidence=confidence,
        source_episodes=[e.id for e in episodes],
        memory_type="derived_semantic"
    )
```

### 8.2 Forgetting Scheduler

```python
class ForgettingScheduler:
    """
    Background scheduler for forgetting operations.
    """
    def __init__(self, config: ForgettingConfig):
        self.config = config
        self.scheduler = APScheduler()
    
    def start(self):
        # Daily: Update forgetting scores
        self.scheduler.add_job(
            self.update_all_scores,
            trigger='cron',
            hour=2  # 2 AM daily
        )
        
        # Daily: Execute forgetting
        self.scheduler.add_job(
            self.execute_forgetting,
            trigger='cron', 
            hour=3  # 3 AM daily
        )
        
        # Weekly: Deep consolidation
        self.scheduler.add_job(
            self.deep_consolidation,
            trigger='cron',
            day_of_week='sun',
            hour=4
        )
    
    async def update_all_scores(self):
        """Update forgetting scores for all active memories."""
        async for memory in self.db.stream_active_memories():
            score = calculate_memory_score(memory)
            await self.db.update_score(memory.id, score)
    
    async def execute_forgetting(self):
        """Execute forgetting for memories below threshold."""
        candidates = await self.db.get_forgetting_candidates(
            threshold=self.config.threshold
        )
        
        for memory in candidates:
            if should_forget(memory):
                await self.forget(memory)
```

## 10. Discussion

### 10.1 When to Forget Aggressively

Our experiments suggest aggressive forgetting is beneficial when:
- High volume of casual conversations (>60% of interactions)
- Long-running agents (>30 days continuous operation)
- Storage-constrained environments
- Retrieval latency is critical

### 10.2 When to Preserve More

Conservative forgetting is recommended when:
- Domain-specific knowledge is rare and valuable
- User explicitly values historical context
- Compliance requires audit trails
- Training data collection is a goal

### 10.3 Limitations

1. **Transformation Quality**: Episodic→Semantic depends on LLM capability
2. **False Forgetting**: Risk of forgetting valuable information
3. **Computational Overhead**: Daily scoring adds processing cost
4. **Cold Start**: New users lack forgetting history for calibration

### 10.4 Ethical Considerations

Selective forgetting raises important questions:
- **User Consent**: Should users approve forgetting decisions?
- **Right to be Forgotten**: Alignment with GDPR principles
- **Transparency**: Should agents disclose what they've forgotten?

---

## 11. Conclusion

Lethe challenges the dominant "remember everything" paradigm in LLM memory systems. By treating forgetting as a feature rather than a bug, we achieve significant improvements in memory quality, retrieval precision, and storage efficiency.

The key insight is that **compression and forgetting are not about losing information—they're about distilling information**. Just like human memory, we don't remember every detail, but we remember the patterns, lessons, and wisdom that shape our understanding.

> "Memory is the foundation of intelligence, but not all information deserves to be remembered. A good memory system must know not only what to remember, but more importantly, what to forget."

Lethe is deployed in production on the WizAxis platform. We hope this work inspires the community to reconsider the role of forgetting in AI memory systems.
