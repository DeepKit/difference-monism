# 01.16 ODD开发周期：Define → Decompose → Execute → Verify → Seal

> **作者**: Fuyi ( ODDFounder  fuyi.it@live.cn )
> **日期**: 2026-01-11
> **标签**: ODD, 方法论, 开发流程

---

## 摘要

ODD（Output-Driven Development）不是一个抽象的理论，而是一套严丝合缝的工业化流程。本文详解ODD的五个核心步骤：Define（定义）、Decompose（分解）、Execute（执行）、Verify（验证）、Seal（封版）。这是让AI从"玩具"变成"生产力"的标准流水线。

---

## 一、为什么需要标准周期？

在传统开发中，我们习惯了"需求→设计→编码→测试"的瀑布或敏捷流程。但在AI辅助开发（AI-Assisted Development）中，这个流程失效了。

**失效点在于：**
1.  **需求模糊**：我们习惯给AI模糊的自然语言指令（"写个登录功能"），AI只能猜。
2.  **验证困难**：AI生成的代码"看起来是对的"，但往往暗藏Bug。
3.  **无限修改**：因为没有明确的终点，我们很容易陷入与AI的"对话黑洞"，反复修改却得不到想要的结果。

ODD开发周期将软件开发变成了**精密加工**过程。我们不再是"写代码"，而是"定义产出物"。

---

## 二、ODD五步循环详解

ODD的核心是一个闭环系统，每一个功能点的开发都必须经历这五个阶段。

### 2.1 第一步：定义 (Define)

**核心任务**：把"作文题"变成"填空题"。

在让AI写哪怕一行代码之前，你必须先定义清楚**产出物（Artifact）**。这不仅仅是说"我要一个函数"，而是要定义它的**契约（Contract）**。

**定义三要素**：
1.  **Artifact Type**：产出物类型（是 `pg_function`？还是 `python_class`？还是 `api_spec`？）。
2.  **Input/Output Schema**：严格的输入输出定义。
    *   *Input*: `{ user_id: uuid, amount: decimal }`
    *   *Output*: `{ success: boolean, transaction_id: string }`
3.  **Acceptance Criteria**：验收标准（Given-When-Then）。

**实战案例**：
```yaml
# 这里的定义是给AI看的"图纸"
contract:
  name: TransferMoney
  type: pg_function
  input:
    from_id: uuid
    to_id: uuid
    amount: decimal
  output:
    success: boolean
    new_balance: decimal
  acceptance:
    - Case: 余额不足 -> 返回 false
    - Case: 金额为负 -> 抛出异常
    - Case: 正常转账 -> 双方余额更新正确
```

> **ODD原则**：如果没有定义清楚验收标准，就不要开始Execute。

### 2.2 第二步：分解 (Decompose)

**核心任务**：将大契约拆解为原子任务。

AI的工作记忆（Context Window）是有限的，推理能力随着任务复杂度指数级下降。ODD要求我们将复杂契约分解为**原子任务（Atomic Task）**。

**分解原则**：
*   **单一职责**：一个任务只产出一个主要产出物。
*   **依赖明确**：任务A依赖任务B的产出物，形成有向无环图（DAG）。
*   **无歧义**：每个任务的描述必须独立完整。

**例如**：实现"用户注册"功能，分解为：
1.  Task 1: 创建 `users` 数据表（Database Artifact）。
2.  Task 2: 编写 `User` Pydantic模型（Code Artifact）。
3.  Task 3: 实现 `register` API接口（API Artifact）。

### 2.3 第三步：执行 (Execute)

**核心任务**：AI填空。

这是AI真正"干活"的阶段。因为我们在Define阶段已经把题目变成了填空题，AI在这里只需要做翻译工作：**将Schema和Logic翻译成代码**。

**执行模式**：
*   **Context Injection**：系统自动注入相关的上下文（如数据库Schema、项目代码规范）。
*   **Generation**：AI生成代码。
*   **Refinement**：AI自我修正（基于Linter或编译器反馈）。

在ODD中，Execute阶段通常是全自动的。如果Define做得好，Execute就是一瞬间的事。

### 2.4 第四步：验证 (Verify)

**核心任务**：基于结果的客观检验。

这是ODD与传统AI编程最大的区别。我们不相信AI的"我写好了"，我们只相信**验证结果**。

ODD定义了五种验证策略（详见下一篇文章）：
1.  **Compile**：代码能编译/解析吗？
2.  **Execute**：代码能运行吗？
3.  **Test**：通过了所有单元测试吗？
4.  **Effect**：产生的副作用（如写库）正确吗？
5.  **Review**：人工/AI复审（针对文档或设计）。

**自动化验证流**：
```python
# 伪代码
result = ai.generate(code)
if not compile(result):
    return reject("Syntax Error")
if not run_tests(result, acceptance_criteria):
    return reject("Test Failed")
return approve(result)
```

只有通过Verify的代码，才有资格进入下一阶段。

### 2.5 第五步：封版 (Seal)

**核心任务**：固化资产，防止回退。

在AI开发中，很容易出现"改了一个Bug，引入了两个新Bug"的情况。ODD引入了**Seal（封版）**机制。

**封版意味着**：
*   **不可变性**：该产出物被标记为 `Sealed`，AI不得再随意修改其核心逻辑。
*   **版本号**：产出物获得唯一的版本号（Hash）。
*   **基线化**：它成为了后续任务的可靠依赖（Context）。

如果需要修改已封版的产出物，必须发起一个新的**变更契约（Migration Contract）**，重新走一遍五步循环。

## 三、从"手工作坊"到"精密工厂"

传统的AI编程像是一个手工作坊：
> "嘿ChatGPT，帮我写个贪吃蛇。" -> "不对，要这样..." -> "再改改..."

ODD的五步循环将开发变成了精密工厂：
1.  **Define**: 绘制蓝图。
2.  **Decompose**: 安排工序。
3.  **Execute**: 机器加工。
4.  **Verify**: 质检过关。
5.  **Seal**: 入库封存。

这就是ODD能让AI产出企业级高质量代码的秘密。它不是限制了AI的创造力，而是给AI的创造力加上了工程化的导轨。
