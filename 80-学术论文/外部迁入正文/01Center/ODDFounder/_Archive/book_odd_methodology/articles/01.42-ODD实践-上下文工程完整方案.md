# ODD实践：上下文工程完整方案 (Context Engineering Implementation)

> **版本**: v2.0
> **基于**: Progee 2.0 上下文设计文档
> **标签**: 上下文, Token优化, RAG, 知识库

---

## 1. 设计目标：精准与成本的平衡

在 Progee 2.0 中，我们面临一个核心挑战：**如何在有限的 Token 预算（约 4000 tokens）内，让 AI 理解整个项目？**
全量注入是不可能的，RAG 盲搜又太不准确。我们提出了 **"以功能树为中心的动态查询"** 方案。

---

## 2. 17层上下文架构 (The 17-Layer Architecture)

我们将上下文数据存储在 PostgreSQL 的 `context_*` 系列表中。

| 层级 | 类别 | 存储表 | 说明 |
| :--- | :--- | :--- | :--- |
| **L1-L3** | **硬边界** | `context_security/architecture/process_boundaries` | 宪法层，不可逾越。 |
| **L4** | **系统规范** | `context_system_conventions` | 全局命名、错误码规范。 |
| **L7** | **功能树** | `context_function_tree` | 项目的骨架索引。 |
| **L12** | **车间知识** | `context_workshop_knowledge` | 历史经验与 Bug 模式（向量化）。 |
| **L14** | **任务规格** | `context_task_specs` | 当前任务的 ODD 契约。 |

**ODD 实践点**：这 17 层不是平铺直叙的文本，而是**结构化的数据库记录**。AI 不是在"读文档"，而是在"查询数据库"。

---

## 3. 注入策略矩阵 (Injection Matrix)

我们定义了一个 `INJECTION_MATRIX`，决定在什么 **角色 (Role)** 和什么 **阶段 (Stage)** 注入哪些层。

```python
INJECTION_MATRIX = {
    'L1': ('force', ['all'], None),          # 永远强制注入安全规则
    'L12': ('role', ['worker'], 'hot_start'), # 只有工人在热启动时才看知识库
    'L17': ('conditional', ['worker'], 'rework > 0') # 只有返工时才注入错误反馈
}
```

---

## 4. 功能树查询引擎 (Function Tree Query Engine)

这是 Progee 2.0 的核心黑科技。
当 AI 接到一个任务（例如 `ModifyLogin`）时：
1.  **Locate**: 在功能树中定位到 `auth.login` 节点。
2.  **Siblings**: 获取同级的 `auth.register`, `auth.logout` 节点摘要。
3.  **Dependencies**: 查询 L11 表，获取它依赖的 `User` 模型定义。
4.  **Inject**: 只将这些相关代码片段注入 Prompt。

**效果**：上下文信噪比从 20% 提升到了 90% 以上。

---

## 5. Token 控制器

系统内置了一个 `TokenController`。它会根据优先级（Priority）动态裁剪上下文。
*   `System Prompt` (L1-L4): 优先级最高，保留。
*   `Task Spec` (L14): 优先级高，保留。
*   `Related Code` (L7): 优先级中，超长截断。
*   `User Stories` (L6): 优先级低，如果不够就丢弃。

---

## 6. 总结

Progee 2.0 的上下文工程证明了：**AI 的智能不仅取决于模型的大小，更取决于上下文的质量。**
通过数据库化的管理和精细化的注入策略，我们用极低的成本实现了极高的代码生成准确率。
