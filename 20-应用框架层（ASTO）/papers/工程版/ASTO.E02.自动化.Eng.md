## **总则：从控制到无摩擦**

传统治理依赖 **“事后惩罚”** (Deterrence)，导致极高的 **社会摩擦系数 (Social Friction Coefficient)**。
ASTO 自动化旨在实现 **“嵌入式合规” (Embedded Compliance)**：
*   **违规不是被惩罚，而是物理上不可执行** (Violation as Invalid State)。
*   目标不是把人“抹掉”，而是在文明底线之内让**合规路径的阻抗趋近于零**：把信任成本下沉为可审计的结构，把裁决/退出权保留给人。

> **前置概念锚定**：
> 在 ASTO 体系中，**“五态”** 描述了规范的存在形态（自在 → 共识 → 编码 → 物化 → 定向），而 **“动变性”** 是驱动系统演化的因果力量，分为四类：**本律式、涌现式、目标式、建模式**。自动化本质上是对 **本律式/目标式动变性** 的编码与管控。

### **文明守护条款：零摩擦不是最高目标**
> 对齐 ASTO.P04.宣言.Phil.md 与 ASTO.P05a.公理.Phil.md。

*   **优先级**：禁行红线/复数性/不可触达维 > 动变性 > 效率。
*   **AGI 前默认保守**：任何不可逆的自动化必须可审计、可中断、可退出、责任链清晰；否则强制进入黄灯/红灯流程。
*   **反滥用**：若自动化导致拒绝权/退出权失效，即便能带来高效率，也视为文明退化，应触发熔断/降级。

## **第一部分：红绿灯分级标准 (Traffic Light Protocol)**

基于规范的可执行性，我们将系统划分为三个红绿灯区域：

### **2.2 红绿灯协议：三区划分**

在自动化执行前，必须对“指令/契约”本身的**清晰度与风险**进行评估。这是 AI 时代的“准入安检”。

#### **2.2.1 契约清晰度评估 (Clarity Assessment Mechanism)**
*   🔴 **红色（Clarity 0-49%）**：歧义严重，定义模糊（如“优化用户体验”）。**动作**：拒绝执行，退回定义层澄清。
*   🟡 **黄色（Clarity 50-79%）**：存在歧义但风险可控，或需人工确认参数（如“大额订单需审批”——多大？）。**动作**：请求人工确认或补充参数。
*   🟢 **绿色（Clarity 80-100%）**：定义精确，边界清晰，可自动化测试（如“订单金额 > 10000 触发 SMS 通知”）。**动作**：进入执行管道。

#### **2.2.2 执行风险分区**
根据指令通过清晰度评估后的性质，进入不同执行区域：

*   **🟢 绿灯区：确定性执行域 (L1/L2)**
    *   **定义**：逻辑封闭、无歧义的重复性任务。
*   **ASTO 映射**：**物化态 (Materialized)**。
*   **机制**：**嵌入式合规**。系统像物理定律一样运行，不存在“违规”的可能性。
*   **场景**：税收自动扣除、Token 转账、CI/CD 自动部署。
*   **代码特征**：`Hardcoded`, `Immutable`, `Type-Safe`.

### **🟡 黄灯区：模糊裁决域 (L3)**
> **"人机协同，处理边缘情况。"**

*   **定义**：涉及复杂上下文或统计概率的任务。
*   **ASTO 映射**：**共识态 (Consensus)**。
*   **机制**：**异常抛出 (Exception Escalation)**。机器处理 99%，遇到置信度低的情况抛出给人类。
*   **场景**：大额转账风控、内容审核争议、医疗辅助诊断。
*   **代码特征**：`Thresholds`, `Review Workflow`, `Feedback Loops`.

### **🔴 红灯区：创造性禁域 (L4)**
> **"人类专属，机器止步（除非打碎玻璃）。"**

*   **定义**：涉及伦理、终极价值、战略方向的决策。
*   **ASTO 映射**：**定向态 (Oriented)**。
*   **机制**：**主权阻断 (Sovereignty Circuit Breaker)**。严禁自动化介入，必须由人类主体承担责任。
*   **例外**：**玻璃破碎机制 (Break-Glass)**。在极端危机（如人类失联但物理系统即将熔毁）下，允许通过不可篡改的审计通道绕过阻断，但操作者需承担无限责任。
*   **场景**：修改宪法/核心协议、判处死刑、发动战争、定义“什么是美”。
*   **代码特征**：`Manual Only`, `Physical Key Required`.

### **🧪 配套机制：对抗测试与赛马（红绿灯协议的默认免疫）**

> 红绿灯协议回答“能不能自动化”，而这两个机制回答“如何在真实世界里不被打穿”。

*   **绿灯区（L1/L2）**：对抗测试尽量全自动（Fuzz/性质/变异/规则校验），失败即阻断；变更默认走灰度/回滚。
*   **黄灯区（L3）**：对抗测试进入人机协同（红队/滥用演练/故障演练）；赛马用 Shadow/Canary/A-B 等方式并行跑方案，用证据裁决。
*   **红灯区（L4）**：任何自动化只能停留在模拟/证据层；真正的裁决与签名仍归属于人类主权。

> 详见第三部分新增的「对抗测试」「赛马机制」，以及 `./ASTO.E05.工程实践手册.Eng.md`。

## **第二部分：人机对话平台 (Human-Machine Dialogue Platform)**

我们拒绝将代码视为单向的“命令通道”（Pipeline），那意味着指令的单向压迫。
我们构建 **“对话平台”**：这是意图（L4）与实现（L1）之间的**协商空间**。

### **1. 意图层 (The Intent Layer)**
*   **主体**：人类（公民、开发者、管理者）。
*   **语言**：自然语言 (Natural Language)。
*   **动作**：**Proposal (提案)**。人类提出“我们想要什么”（例如：降低通胀）。

### **2. 协商层 (The Negotiation Layer)**
*   **主体**：形式化验证工具 (Formal Methods) + 模拟器 (Simulators) + 对抗测试工具链 (Adversarial Harness) + 沙箱运行时 (Sandbox/Shadow)。
*   **语言**：DSL (Domain Specific Language)。
*   **动作**：
    *   **Simulation (模拟)**：机器反馈：“如果执行这个策略，系统崩溃概率是 30%。确定吗？”
    *   **Adversarial Testing (对抗验证)**：把“最坏场域”提前搬进评估流程，暴露隐性禁行红线与结构裂纹。
    *   **Horse Race (赛马试跑)**：在可回滚、小流量、可观测的条件下并行跑 A/B 方案，用证据替代信仰。
*   **交互**：人类修改提案，机器重新模拟/重跑对抗/重跑赛马。这是一个**双向修正**的过程。

> 延伸：对抗测试与赛马的工程化模板见 `./ASTO.E05.工程实践手册.Eng.md`。

### **3. 执行层 (The Implementation Layer)**
*   **主体**：运行时环境 (Runtime)。
*   **语言**：二进制代码 (Binary / Wasm)。
*   **动作**：**Execution (执行)**。一旦在协商层达成共识（Signed），代码即刻生效，不仅是法律，更是物理。

## **第三部分：混合治理模式 (Hybrid Governance Patterns)**

### **1. 乐观执行，悲观回滚 (Optimistic Execution)**
*   **逻辑**：默认所有操作都在 **绿灯区**（L1），以光速执行。
*   **限制**：**严禁用于不可逆操作**（如数据物理删除、硬件控制、对外资产转移）。此类操作必须强制进入 L3 或增加**执行延迟 (Execution Delay)**。
*   **纠错**：一旦触发 **黄灯** 阈值（如资金流向异常），系统自动**回滚 (Rollback)** 并冻结，转入 **对话平台** 等待人类裁决。

### **2. 策略即对话 (Policy as Dialogue)**
*   不要把规则写死在代码里，也不要写死在文档里。
*   把规则写在 **DSL** 中。当规则需要变更时，不是修改代码，而是发起一次 **RFC (Request for Comments)**。
*   只有经过 **多方签名 (Multi-sig)** 的 RFC，才能被编译器转化为新的 L1 代码。

### **3. 对抗测试：把“最坏场域”提前搬进 CI (Adversarial Testing)**

如果自动化的目标是“违规不可执行”，那么**你必须先知道什么叫违规**。
现实世界会用误用、滥用、对抗、分布漂移来试探你的结构边界——
对抗测试的作用，是把这些“最坏场域”提前搬进 CI/演练，让系统在小脉冲里暴露裂纹。

*   **ASTO定位**：主动制造小型脉冲（六阶），提前显性化隐性禁行红线/隐性基元。
*   **落地形态**（从 L1 到 L4）：
    *   **L1（绿灯）**：可自动化对抗（Fuzz / 性质测试 / 变异测试 / 规则校验）→ 失败即阻断合规路径。
    *   **L3（黄灯）**：红队/滥用演练（需要人工参与）→ 产出“可复现攻击样本（EN）+ 红线说明（NEN）”。
    *   **L4（红灯）**：对抗测试触发的价值冲突裁决（例如：安全 vs 可用性）→ 必须回到人类主权。

> **关键原则**：不要把对抗测试当“额外质量项”，它是自动化治理的**免疫系统**。

### **4. 赛马机制：把变迁从豪赌改成可回滚实验 (Horse Race Mechanism)**

当系统进入“流变”甚至“脉冲”时，最危险的不是“做错”，而是：

*   只有一种方案（单点押注）
*   没有证据（信仰之争）
*   不能回滚（不可逆崩解）

赛马机制的工程定义是：**并行探索 + 证据化裁决 + 可回滚淘汰**。

*   **ASTO定位**：为“变迁”提供可控的过渡结构（桥），用并行小试替代一次性跃迁。
*   **工程抓手**：Shadow / Canary / A-B / Bandit + Feature Flag + 指标与追踪。
*   **红线护栏**：胜负不只比“快/省/转化”，必须同时比较护栏指标（错误率、合规、隐私、偏见）与禁行红线。
*   **反作弊**：Goodhart 定律默认成立——必须引入隐藏集/随机审计/对抗测试作为裁判。

> **赛马的终点不是“更强模型/更快方案”，而是：把胜者固化成新结构，把败者彻底消解（删开关、删代码、归档证据）。**

> 延伸：更完整的工程模板与度量见 `./ASTO.E05.工程实践手册.Eng.md`。

### **5. 封板/解封：把胜者固化为可上线稳态（Seal/Unseal）**

赛马解决“选谁”，对抗解决“抗打”，但如果不封板，胜者仍会在持续变更中被稀释；如果不解封，系统又会僵化。

*   **封板（Seal）**：把某个版本/组合在当下场域中声明为“可上线稳态”，并冻结其契约与关键行为边界；只允许白名单修复（不改变契约、不穿透禁行红线）。
*   **解封（Unseal）**：当需求/契约变化必须发生时，由人类主权发起“可回滚的小升级循环”（解封 → 最小赛马 → 对抗 → 再封板）。

网络/边缘场景下，封板对象不是单一镜像，而是一份 **稳态承诺包（Seal Bundle）**：版本、契约、依赖窗口、发布路径、收敛目标、回滚方案与证据。

> 延伸：封板/解封、Seal Bundle Checklist、Seal Bundle Manifest（SBM）与 CI 门禁见 `./ASTO.E05.工程实践手册.Eng.md`（§2.7、§2.9.5、§4.3、§4.4）。

## **第四部分：工程实现指南**

### **1. 类型系统的伦理边界**
```typescript
// 🔴 红灯区类型：必须包含人类签名
type LethalAction = {
    target: string;
    authorization: HumanSignature; // 只有人类能生成此签名
};

// 🟢 绿灯区类型：自动执行
type Transaction = {
    amount: number;
    execute(): void;
};

// 编译器强制检查：机器无法构造 LethalAction
function autoPilot(action: LethalAction) {
    if (!action.authorization.isValid()) {
         // 🚨 玻璃破碎机制：紧急情况下绕过签名
         if (System.isCriticalMeltdown() && action.auditLog.isImmutable()) {
             System.breakGlass("Emergency Override by AI - Full Liability Logged");
             return action.forceExecute(); 
         }
         throw new Error("Compiler Error: Machines cannot initiate lethal actions.");
    }
    action.execute();
}
```

### **2. 状态机的非法状态**
*   在 L1 系统中，**违规状态 (Invalid State)** 根本不应该在状态机中被定义。
*   例如：不要写 `if (balance < 0) punish()`，而要定义 `uint balance`（无符号整数），使得余额**物理上不可能为负**。

### **3. 把“对抗测试”和“赛马机制”变成可执行规范**

很多团队的失败不是“没有聪明人”，而是：
*   对抗只存在于事故之后（被动挨打）
*   选择只存在于会议之中（信仰争论）

建议把二者写成**结构化的 EN**：

1.  **对抗测试门禁**：关键接口/关键策略变更必须通过对抗测试套件（否则 CI 直接红灯）。
2.  **赛马立项门禁**：涉及核心策略/模型/规则的变更，必须提供最小赛马方案。
3.  **认知活体检测 (Cognitive Liveness Check)**：为了防止人类因“零摩擦”而退化，系统应定期（如每季度）**强制降级**为手动模式（Random Friction），迫使人类重新介入 L1 操作，维持对系统的认知掌控力。
4.  **消解门禁**：赛马到期必须裁决；败者必须删除开关与代码。
5.  **封板门禁（Seal Bundle Gate）**：任何“可上线稳态”的发布必须提交 Seal Bundle Manifest（SBM）。

## **结语：构建理性外骨骼**

ASTO 自动化不是要建造一个关押人类的铁笼，而是要建造一套 **“理性外骨骼” (Exoskeleton of Reason)**。

*   它承担重量（处理 L1/L2 的繁杂）。
*   它保护软弱（通过 L3 过滤风险）。
*   它完全服从于核心（L4 的人类意图）。

通过**人机对话平台**，我们将法律的威严转化为代码的坚固，将官僚的傲慢转化为逻辑的透明。
