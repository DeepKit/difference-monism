# Code is Obligation: The Ontology of Digital Artifacts in the Age of Generative AI

**Target Journal**: *Philosophy & Technology*
**Type**: Research Article
**Status**: Draft v1.0

---

## Abstract

The prevailing metaphor of "Technical Debt" in software engineering implies that code is primarily an asset with an attached financial liability that can be repaid. This paper argues that this metaphor is ontologically misleading. In the age of generative AI, where code production costs approach zero, we propose a radical ontological shift: digital artifacts are not assets but solidified obligations. Drawing on the philosophy of technology and thermodynamics, we define a piece of executable code not by what it *does*, but by what it *demands* from its environment to maintain its executability against entropy. We introduce the concept of "Sealing" as the only mechanism that temporarily transmutes this obligation into an asset. This "Engineering Turn" in understanding digital objects has profound ethical and governance implications: to generate code is to impose a future burden of maintenance, effectively blackmailing the future.

**Keywords**: Philosophy of Software, Technical Debt, Ontology of Maintenance, Generative AI, Sealing, Obligation.

---

## 1. Introduction: The Metaphorical Trap

The software industry is dominated by financial metaphors. We speak of "Technical Debt," "Code Assets," and "ROI." These metaphors structure our understanding of what software *is*. The concept of "Technical Debt," introduced by Ward Cunningham in 1992, suggests that taking shortcuts in code is like taking a financial loan: it speeds up delivery now but requires paying "interest" (extra effort in maintenance) later.

However, this metaphor conceals a deeper ontological reality. A financial debt can be paid off. Once paid, the obligation ceases. Code, however, never ceases to demand. Even "perfect" code requires constant maintenance—updates to dependencies, migration to new hardware, adaptation to changing user requirements.

With the advent of Large Language Models (LLMs), the cost of *producing* code has plummeted. We face a "Code Flood." If we continue to view code as an asset (wealth), we are deluded into thinking we are becoming richer. This paper argues the opposite: every line of code is a liability. It is a promise that demands future labor to be kept.

## 2. The Ontology of Executability

What does it mean for code to "exist"? A digital artifact does not exist like a rock; it exists only insofar as it is *executable*.

### 2.1 Executability as a Demand
Executability is not an intrinsic property of the text file. It is a relational property between the artifact and its environment (compiler, OS, hardware, network).
Therefore, to assert "this code works" is to assert a demand upon the environment: "You must remain in state S for me to function."

### 2.2 The Law of Digital Entropy
Digital systems are fragile. The environment is in constant flux (perturbation). If the code does not change, it breaks. This is the "Red Queen's Race" of software maintenance.
Thus, the "default state" of any digital artifact is not stasis, but decay. Its existence is defined by the energy required to fight this decay.

**Thesis 1**: A digital artifact is an *obligation* to perform continuous maintenance work against environmental entropy.

## 3. Code is Obligation

If existence is maintenance, then ownership is liability.

### 3.1 The "Asset" Illusion
We usually consider a house an asset, even though it needs maintenance. But a house provides shelter even if slightly decayed. Software is binary: a single character change can render a million-line system useless. The ratio of "maintenance cost" to "utility" in software is uniquely high.

### 3.2 The Generative Crisis
Generative AI allows us to generate obligations at the speed of light. When a developer accepts an AI-generated function, they are not acquiring a tool; they are signing a contract of indefinite servitude to that function's dependencies and logic.
We are witnessing the accumulation of "Negative Equity" in global software systems: the total obligation to maintain generated code is growing faster than the human capacity to maintain it.

## 4. Sealing: The Transmutation of Obligation

Is it ever possible for code to be an asset? Yes, but only through a specific operation: **Sealing**.

### 4.1 Definition of Sealing
Sealing is the act of declaring a system "finished" and refusing to accept further perturbations. It is the rejection of new requirements and the freezing of the environment (e.g., embedded systems, retro-computing).
Only when a system is sealed does the maintenance cost drop to zero. Only then does it become a pure tool (like a hammer) rather than a demanding pet.

### 4.2 The Paradox of Value
The software industry values "Agility" (continuous change). But value (utility/cost) is maximized in "Stability" (Sealing).
We propose that the true value of software engineering lies not in *writing* code, but in *sealing* it. The goal of engineering is to create artifacts that *stop* demanding attention.

## 5. Ethical Implications: Blackmailing the Future

If code is obligation, then creating code is an ethical act.
To write a line of code is to impose a burden on one's future self or one's successors.
"Legacy Code" is simply the accumulated obligations of the past that we cannot discharge.

**Conclusion**: We must move from an "Asset-based" ontology to an "Obligation-based" ontology. In the age of AI, the virtuous engineer is not the one who generates the most code, but the one who refuses to generate code that cannot be sealed.
