# Code is Obligation: Re-framing Digital Assets in Socio-Technical Systems

**Target Journal**: *Philosophy & Technology* (Springer)
**Core Thesis**: Executable code does not constitute organizational assets; it introduces open-ended governance obligations. Only sealed operational artifacts can be regarded as asset-bearing entities.

---

## 1. Submission-Ready Thesis & Arguments

### Core Thesis (Submission-grade)
> **Executable code does not constitute organizational assets.**
> **Rather, it introduces open-ended governance obligations.**
> **Only sealed operational artifacts can be regarded as asset-bearing entities in socio-technical systems.**

*(Note: This frames the paper as an ontological re-classification, not just engineering advice.)*

### Expanded Thesis (for Introduction)
This paper argues that treating code as an asset is a category mistake.

From an operational governance perspective, executable code continuously generates maintenance, interpretive, and responsibility obligations. These obligations accumulate regardless of code quality, architectural elegance, or engineering discipline.

By contrast, value accumulation occurs only when operational outcomes are institutionally closed, stabilized, and authorized. Such closure transforms transient execution into durable organizational capital.

We therefore propose a principled distinction between obligation-generating structures and asset-bearing artifacts, and show why confusing the two leads to systemic fragility in modern socio-technical systems.

### Conceptual Reframing (for Philosophy Reviewers)
The prevailing metaphor of "technical debt" presupposes that well-written code may function as an asset, while poorly written code constitutes debt.

This paper challenges that assumption. We argue that debt is not an accidental property of bad code, but an intrinsic feature of executable systems.

Code, insofar as it remains executable, cannot be finalized. Its meaning is perpetually deferred to future environments, interpretations, and failures.

As a result, code is structurally incapable of functioning as a stable bearer of value.

### Key Concept Definition
We define an operational artifact as an outcome whose acceptance conditions have been explicitly specified, evaluated, and institutionally sealed.

Only such artifacts—rather than the executable processes that produced them—can serve as organizational assets, since they delimit responsibility, stabilize meaning, and terminate governance obligations.

### The "One-Liner" (for Conclusion)
> **Code does not accumulate value.**
> **It accumulates obligation.**

### 3. The Ontological Problem of Executable Code (Core Section)
#### 3.1 Executability as Open-Ended Commitment
*   Code derives meaning only through execution.
*   Execution depends on future environments (which are unknown).
*   Every deployment reopens the interpretive space.
*   **Conclusion**: Executability implies indefinite obligation.

#### 3.2 Why Code Cannot Be Finalized
*   No natural closure condition for running code.
*   No stable identity across updates.
*   No termination of responsibility.
*   **Contrast**: Legal documents can be sealed; medical diagnoses can be archived; code cannot, as long as it runs.

### 4. Obligation vs. Asset: A Structural Distinction
#### 4.1 Asset-Bearing Entities
*   Assets require: Bounded identity, stabilized meaning, finite responsibility.

#### 4.2 Obligation-Generating Structures
*   Code generates: Maintenance obligation, interpretive obligation, accountability diffusion.
*   **Core Claim**: Code belongs to the latter category by structure, not by accident.

### 5. Operational Artifacts and Closure
*(Light introduction of ASTO concepts without heavy jargon)*

#### 5.1 Closure as Value Formation
*   Acceptance criteria.
*   Institutional authorization.
*   Responsibility anchoring.

#### 5.2 Sealed Outcomes as Organizational Capital
*   Examples: Certified reports, approved models, audited results, released decisions.
*   **Emphasis**: Value arises from closure, not execution.

### 6. Implications for Governance and AI Systems
*   Governance of AI-generated code.
*   Continuous deployment as "obligation inflation."
*   Why "more automation" often increases governance cost (by increasing the volume of obligations).

### 7. Discussion: Rethinking Digital Value
*   Why digital transformation often fails.
*   Why organizations accumulate complexity but not capital.
*   Limits of the argument: Where code *appears* as an asset (and why that appearance is misleading).

### 8. Conclusion
*   Restate the distinction (Code = Obligation; Artifact = Asset).
*   Implications for philosophy of technology.
*   The need for operational ontologies (ASTO).
*   **Final thought**: Sustainable digital systems require not more code, but clearer conditions of closure.
