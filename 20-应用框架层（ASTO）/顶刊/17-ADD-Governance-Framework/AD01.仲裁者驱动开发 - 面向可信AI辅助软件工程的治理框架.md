# Arbitration-Driven Development: A Governance Framework for Trustworthy AI-Assisted Software Engineering
**仲裁者驱动开发：一种面向可信 AI 辅助软件工程的治理框架**

**Authors**: Yi Fu  (ODDFounder fuyi.it@live.cn)
**Target Venue**: ICSE 2026 / FSE 2026 (Technical Track)

---

## Abstract (摘要)
随着大语言模型（LLM）在软件工程中的广泛应用，生产力的瓶颈已从“代码生成（Generation）”转移至“代码治理（Governance）”。现有的开发范式（如 TDD、ODD）侧重于描述“如何构建”，却未能有效解决“如何确信”的问题。在 AI 生成代码的速度远超人类审查能力的今天，传统的“人机回环（Human-in-the-loop）”模式正因认知过载而失效。
本文提出了 **Arbitration-Driven Development (ADD，仲裁者驱动开发)**。这是一种独立于生成模型的元治理框架，旨在将工程师的角色从“创作者（Author）”重塑为“仲裁者（Arbiter）”。ADD 建立了以**“责任守恒定律”**为核心的哲学基础，并通过**动态风险分级**、**契约对抗生成机制（CAP）**以及**状态机确权体系**，构建了一个零信任的治理环境。通过在 **Progee** 系统中的参考实现以及一个“积分兑换”功能的英雄之旅案例，我们展示了 ADD 如何通过前置冲突（Engineering Conflict）和分级治理，在不牺牲核心安全性的前提下，驾驭 AI 的无限生产力。

## 1. Introduction (引言)

### 1.1 The Paradox of Productivity (生产力悖论)
自 Copilot 等工具问世以来，代码生成的边际成本趋近于零。然而，软件工程领域出现了一个危险的悖论：**生成速度越快，系统熵增（Entropy）越快，信任崩塌得越快**。LLM 固有的“谄媚症（Sycophancy）”倾向于生成语法完美但逻辑有缺陷的代码以取悦用户。当 AI 在几秒钟内生成数百行代码时，人类审查者（Reviewer）往往因缺乏上下文构建能力而沦为“橡皮图章”，导致严重的隐式缺陷（如并发竞态、权限漏洞）逃逸至生产环境。

### 1.2 The Conservation of Accountability (责任守恒定律)
我们提出软件工程中的**“责任守恒定律”**：无论 AI 承担了多少认知劳动，责任的总量不变，且最终必须由人类承担。当前的危机在于**“权责分离”**——人类失去了对细节的掌控权，却仍需对系统的故障负责。为了解决这一伦理与工程困境，我们需要一种新的范式来**重新确权**。

### 1.3 Our Contribution (本文贡献)
本文提出了 ADD 框架，作为对我们前序工作 Output-Driven Development (ODD) [1] 的治理层升级。主要贡献如下：
1.  **理论重构**：定义了“仲裁者”新角色，确立了以“封板（Sealing）”为核心的责任转移仪式。
2.  **治理架构**：提出了包含“风险三色灯”、“L1-L4 动态分级”和“契约对抗协议（CAP）”的治理模型。
3.  **初步案例研究**：通过 Progee 系统中的深度集成，展示了 ADD 在拦截高危逻辑漏洞方面的可行性。

## 2. Theoretical Foundation (理论基础)

### 2.1 From ODD to ADD: The Meta-Framework (从 ODD 到 ADD)
ODD [1] 关注**“生产力”**，通过定义产出物（Artifact）解决了 AI 的“空白画布”问题；ADD 则关注**“生产关系”**。虽然 ADD 在 ODD 环境中表现最佳，但它是一个**元框架（Meta-Framework）**。它的核心原则（仲裁、对抗、分级）同样适用于传统的 Copilot 工作流。如果说 ODD 是强劲的引擎，ADD 就是方向盘与刹车系统，确保 AI 的速度服务于人类的意图。

### 2.2 The Cognitive Decoupling (认知解耦)
传统开发要求工程师同时维持“宏观架构”和“微观语法”的认知。ADD 实现了**认知解耦**：
*   **AI** 负责微观层面的执行与实现。
*   **人类** 仅负责宏观层面的价值判断与风险控制。
这种解耦是人类在 AI 时代保持理智并有效监管的关键。

## 3. The ADD Framework (ADD 框架定义)

### 3.1 Role Redefinition: The Arbiter (角色重塑：仲裁者)
在 ADD 中，工程师不再是代码的父亲（Author），而是价值的法官（Arbiter）。其核心职责转变为：
1.  **Intent Injection (意图注入)**：提供模糊的业务价值主张，而非具体的实现指令。
2.  **Conflict Resolution (冲突裁决)**：当系统内部的红蓝军 AI 发生分歧时，介入做最终的价值权衡（例如：性能 vs 安全）。
3.  **Sealing (封板确权)**：通过不可逆（或高成本可逆）的操作，正式接管 AI 产出物的责任。

### 3.2 Dynamic Governance Model (动态治理模型)
ADD 基于**比例原则（Proportionality）**：治理的成本（算力与人类注意力）应与任务的风险成正比。
*   **L1 (Simple)**: 简单 CRUD。全自动流转，人类仅需事后抽检。
*   **L2 (Standard)**: 常规业务逻辑。增加标准单元测试与覆盖率检查。
*   **L3 (Complex)**: 核心算法/复杂状态。引入变异测试（Mutation Testing），确保测试用例的有效性。
*   **L4 (Critical)**: 资金/安全/隐私相关。强制激活 **CAP 协议** 与 **三方交叉审核**。

### 3.3 CAP: Contract Adversarial Protocol (契约对抗协议)
为了打破 AI 的顺从性，ADD 在契约生成阶段引入了对抗机制：
*   **Agent Blue (The Drafter)**：基于人类意图起草“建设性”的契约。
*   **Agent Red (The Adversary)**：拥有**独立上下文**和**攻击者人格**，加载 OWASP 等攻击向量库，对 Blue 的草案进行黑盒攻击。
*   **机制**：只有当 Blue 修正契约并成功防御 Red 的所有攻击后，系统才会点亮绿灯。如果双方僵持不下（例如 Red 认为存在风险，Blue 认为已防御），系统点亮黄灯，召唤仲裁者介入。

## 4. Reference Implementation: The Progee System (参考实现：Progee)

Progee 是一个为验证 ADD 范式而原生构建的 AI 辅助开发环境，它将 ADD 的理论固化为代码与界面。

### 4.1 Constitutional Context (宪法级上下文)
Progee 构建了 17 层上下文结构。其中 L1-L4 被定义为**“宪法层”**，包含安全硬边界（如“禁止在 L1 层直接调用外部未授权 API”）。这些规则通过 RAG 机制被强制注入到底层 LLM 的 System Prompt 中，且拥有最高优先级，构成了治理的物理围栏。

### 4.2 State Machine as Law (状态机即法律)
Progee 将治理流程固化为一组严格的状态机：  
`pending` -> `in_progress` -> `quality_check` -> `adversarial_test` -> `sealed`

*   **Hard Block**: 在 L4 任务中，若 `adversarial_test` 未通过，状态机物理锁死，禁止流转至 `sealed`。
*   **Audit Trail**: 一旦 `sealed`，代码变为只读。唯有人类授权并填写 `unsealed_reason` 后，方可回退至 `rework`。这确保了每一次责任转移都有据可查。

### 4.3 Tripartite Review System (三方交叉审核)
在 L4 流程的最后阶段，Progee 实现了**三 AI 民主制**：
*   **AI-A (Builder)**: 负责生成最终代码。
*   **AI-B (Reviewer)**: 负责静态扫描与规范检查。
*   **AI-C (Attacker)**: 负责生成动态攻击用例。
系统仅在 AI-B 与 AI-C 产生严重分歧时，通过 UI 召唤人类。这种机制显著过滤低级噪音，让人类主要关注真正的“高风险决策”。

## 5. Case Study: The Hero's Journey (案例研究：英雄之旅)

为了展示 ADD 的实际效能，我们记录了一个典型的高风险场景：**“用户积分兑换商品”**功能的开发全过程。

### 5.1 The Ambiguous Intent (模糊的意图)
*   **人类输入**：“我要一个功能，允许用户用 100 积分兑换一个杯子。”
*   **ADD 响应**：🔴 **清晰度红灯亮起**。
*   **系统分析**：系统识别到“积分扣减”涉及资产操作，且属于高频并发场景，意图清晰度不足。

### 5.2 The Interaction of Clarity (清晰度的交互)
Progee 系统没有要求人类重写 Prompt，而是生成了一个**选择题**：
> *\"系统检测到并发扣减风险。如果用户发起请求时库存充足，但在扣减积分后库存变为 0，该如何处理？\"*  
> *   **A. 强一致性 (Strong Consistency)**：使用数据库锁，确保不超卖，但可能会降低并发性能。*  
> *   **B. 最终一致性 (Eventual Consistency)**：允许短时间超卖，后续通过补货或退款补偿。*
*   **人类仲裁**：点击选择了 **A**。  
*   *分析：这一步将复杂的分布式系统设计问题，降维成了人类可理解的业务价值选择。*

### 5.3 CAP Activation & Conflict (对抗激活与冲突)
*   **系统定级**：自动定级为 **L4 (Critical)**。  
*   **Agent Blue**：草拟了标准的 `check -> deduct -> send` 逻辑。  
*   **Agent Red**：加载了独立的 OWASP 攻击向量库，并被禁止访问 Agent Blue 的思维链（CoT），以确保攻击的独立性。它发起攻击向量 `Race_Condition`。*"模拟两个并发请求，Blue 的逻辑会导致扣 100 积分发两个杯子。"*
*   **结果**：Agent Blue 被迫修正，引入了 `SELECT FOR UPDATE` 悲观锁机制。

### 5.4 Sealing (封板)
*   **系统行为**：修正后的契约通过了 Agent Red 的复查，并通过了三 AI 交叉审核。  
*   **最终状态**：🟢 **绿灯亮起**。人类点击“Seal”按钮，代码生成并锁定。  
*   **结果**：在没有写一行代码的情况下，人类获得了一个达到金融级安全标准的模块。

## 6. Discussion (讨论)

### 6.1 The Cost of Safety (安全的成本)
ADD 诚实地增加了开发成本。L4 任务通常会显著增加 Token 消耗，并且 CAP 协议会引入可感知的延迟（Latency）。为此，Progee 将耗时的变异测试设计为异步执行，仅在最终封板前强制阻塞。然而，在类似“积分兑换”这类资产相关场景中，前置治理所增加的少量算力成本，往往可以避免上线后高代价的生产事故。这种**“治理成本左移”**在软件工程经济学上通常是划算的。

### 6.2 Proportional Governance (比例治理)
ADD 的核心优势在于其动态性。对于修改 UI 颜色等低风险任务（L1），ADD 几乎是隐形的；只有在触及核心资产时（L4），它才会显露其獠牙。这种设计使得 ADD 能够在不扼杀创新速度的前提下保障系统底线。

### 6.3 Future Work (未来工作)
目前的 L1-L4 分级主要基于静态分析。未来我们将探索基于**运行时遥测（Runtime Telemetry）**的动态分级——即在开发过程中，如果发现某模块反复未通过测试（Flaky Tests），系统自动将其风险等级升级，调动更多算力进行监管。

---

## 7. Conclusion (结论)

AI 不仅仅是生产力的倍增器，也是混乱的倍增器。**ADD（仲裁者驱动开发）** 的提出，标志着软件工程范式从关注“如何写代码”向关注“如何管理智能体”的根本性转变。通过将人类智慧从低维的语法构建中解耦，聚焦于高维的价值仲裁，ADD 为构建可信、安全、负责任的 AI 软件系统提供了一张清晰的蓝图。

在这个新范式中，人类不再需要证明自己比 AI 写代码更快，而是证明自己比 AI 更懂得**什么是正确的（Rightness）**。
