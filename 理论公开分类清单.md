# 理论公开分类清单

> 用途：把当前体系中的理论、摘要层、执行层和经营层，区分为三类：
> 1. 必须先确权后发布
> 2. 可直接推广
> 3. 必须保留用于收入
>
> 说明：这是一份发布与商业边界建议，不替代正式法律意见；若进入版权登记、Zenodo、期刊投稿、合同授权或商标使用，以正式法律与平台流程为准。

---

## 一、总原则

1. 越靠上游、越像“法源母本”的理论，越要先确权，再公开全文。
2. 越靠下游、越像“执行方法 / 参数 / 模板 / 案例 / 交付 SOP”的内容，越不应全量公开。
3. 可直接推广的，通常是摘要层、导航层、总图层和白皮书层，不等于把全部底稿直接放出。
4. 真正要保留用于收入的，不是哲学地基本身，而是：
   - 执行法
   - 参数
   - 阈值
   - 模板
   - 样例
   - 案例证据
   - 转化与承接路径

压缩一句话：

**把法源公开，把摘要推广，把执行法收费。**

---

## 二、A 类：必须先确权后发布

这类对象适合公开，但不建议在“未做统一署名、未做确权引用、未定 Canonical 版本”前随意散发全文。

### A1. 主链法源理论

| 对象 | 当前范围 | 建议 |
| --- | --- | --- |
| `DM` | `10-哲学核心层（DM）` | 必须先确权，再作为主理论公开 |
| `ASTO` | `20-应用框架层（ASTO）` | 必须先确权，再作为桥接法源公开 |
| `ECET` | `30-文明演化层（ECET）` | 必须先确权，再作为文明与治理边界理论公开 |
| `TAT` | `40-责任架构层（TAT）` | 必须先确权，再作为责任门槛框架公开 |
| `ODD` | `50-工程方法层（ODD）` | 必须先确权，再作为工程方法主稿公开 |

原因：

- 这些对象构成当前公开法源链：
  `DM -> ASTO -> ECET -> TAT -> ODD`
- 它们是整套体系最容易被外部引用、拆用、转述和“改名拿走”的部分。
- 一旦公开传播，外界通常只记“谁先提出、谁有稳定版本、谁可引用”，因此必须先锁作者、版本和引用入口。

### A2. 需要先确权的次级理论与论文线

| 对象 | 当前范围 | 建议 |
| --- | --- | --- |
| `LMM` 公共理论面 | `52-认知方法层（LMM）/01-03 + 07` | 先确权后有限公开，不宜先全量外放 |
| `WSM` 核心理论面 | `52-认知方法层（LMM）/09.五行察局学/WSM.000-001` | 先确权后公开，不宜先把方法细节放出 |
| `DM-P00 / P04 / P09` | `80-学术论文/投稿申请包` | 先确权、先投稿链，再外发全文 |
| `Progee-System-Paper` | `80-学术论文/投稿申请包/Progee-System-Paper-申请包` | 先确权后公开论文全文 |
| `ExecNorm` | `80-学术论文/投稿申请包/ExecNorm-申请包` | 先确权后公开，不宜先散发母稿 |

原因：

- 这类对象兼具理论性和应用区分度，公开价值高，但也最容易被“摘要拿走、方法换皮、案例被抄”。
- 尤其 `LMM / WSM / Progee / ExecNorm` 已经靠近可交付、可组织、可商业化的边界，更要先锁作者与版本。

### A 类的最小动作

发布前至少完成：

1. 统一作者署名
2. 统一 Canonical 引用入口
3. 统一版本号与发布日期
4. 统一对外摘要页
5. 统一“可宣称 / 不可宣称”边界

## 三、B 类：可直接推广

这类对象的职责不是“承载全部原创法源”，而是：

- 让外部快速理解
- 建立第一认知
- 支撑演讲、短视频、文章、合作沟通
- 为 A 类理论导流

### B1. 根级总纲与摘要件

| 对象 | 当前范围 | 用法 |
| --- | --- | --- |
| 仓库总入口 | `README.md` | 可直接作为体系首页传播 |
| 体系简介 | `付毅的理论体系简介(从一元论到灯塔营销).md` | 可直接用于对外介绍 |
| 体系总述 | `付毅的理论体系（从一元论到灯塔营销）.md` | 可用于长摘要传播 |

| BetterCiv 说明 | `我为什么构建BetterCiv.md` | 可用于解释上下游分工 |

### B2. 导航层与桥接层

| 对象 | 当前范围 | 用法 |
| --- | --- | --- |
| `00-索引与导航` | 术语、命名、送审入口、发布说明 | 可直接传播为导航层 |

### B 类的边界

可以直接推广，不等于可以直接放出全部底稿。

更准确的做法是：

1. 用 `B 类` 先建立理解和权威感
2. 用 `A 类` 承接正式确权与深读
3. 不让 `B 类` 反向暴露 `C 类` 的执行细节

## 四、C 类：必须保留用于收入

这类对象默认不适合做“全量公开”。

如果需要对外展示，建议只展示：

- 原则
- 结构图
- 脱敏样例
- 摘要版接口

不要放出：

- 全文
- 参数
- 阈值
- 模板
- 样例库
- 真实案例证据
- 完整执行链

### C1. 诊断与执行协议层

| 对象 | 当前范围 | 原因 |
| --- | --- | --- |
| `COP` | `51-认知计算层（COP）` 全协议、题组、类型包、权重、阈值、误判库、校准规则 | 这是你的诊断引擎，不是公共福利件 |
| `Harness` | 运行时对象、事件、权限、门禁、冻结、回滚 | 这是执行底盘，直接关系交付能力 |
| `TAT-COP-ODD-Harness` 接口族 | 根目录接口白皮书、转接表、授权编译表、申诉样例包 | 这是整套体系最值钱的执行法之一 |

### C2. `LMM` 的私有层与变现层

| 对象 | 当前范围 | 原因 |
| --- | --- | --- |
| `LMM/04.产品打造` | 自测件、承接件、交付件、产品化封装 | 这是可直接卖的交付层 |
| `LMM/05.理论变现` | 报价、港口承接、回流设计、R=1 / R-Eval 闭环 | 这是收入路径本身 |
| `LMM/06.理论发售(隔离区)` | 理论发售方案 | 这部分一旦公开，别人可直接照抄销售法 |
| `LMM/08.扩展应用` | 自动化部署、场景实验、单人创始人工作流 | 这是操作优势，不宜全量外放 |

| 对象 | 当前范围 | 原因 |
| --- | --- | --- |
| `Progee2` 落地件 | 根目录四份 `Progee2` 文档 | 这是把理论压成产品的最短路径 |

### C4. 案例、参数与真实证据

原则上也应归入收入护城河：

1. 真实客户案例
2. 承接前问诊模板
3. 结构判定口径
4. 报价逻辑
5. 失败补偿逻辑
6. 回款与升级逻辑
7. 样本库、误判库、复审库

这些东西一旦全放，别人复制你的速度会远快于他们复制 `DM`。

## 五、哪些内容默认不应对外

以下对象不建议作为公开材料：

- `99-归档`
- `internal`
- 历史命名阶段残稿
- 本地任务板、缓存稿、草案堆
- 仍含真实经营路径、真实案例、真实收入逻辑的执行文

最短句：

**历史留档可以保留，但不要误当公开主包；执行草案可以存在，但不要误当传播材料。**

## 八、按动作的公开矩阵

| 动作 | A 类 | B 类 | C 类 |
| --- | --- | --- | --- |
| 对外讲解 / 演讲 / 短文介绍 | 可以 | 可以 | 只讲原则，不讲细法 |
| 发布总图 / 一页摘要 / 白皮书摘要 | 可以 | 可以 | 只允许脱敏版 |
| 放出全文 PDF / Markdown 主稿 | 先确权后可以 | 一般没必要全文放出 | 原则上不建议 |
| 上预印本 / 投稿 / 建 DOI | 先做 | 通常作为附带入口 | 不作为主线 |
| 给合作方做体系介绍 | 可以 | 可以 | 只展示边界与效果，不给底稿 |
| 公开模板 / 参数 / 阈值 / 题组 | 不建议 | 不需要 | 禁止全量公开 |
| 公开真实案例证据链 | 只限最小必要案例 | 只限脱敏摘要 | 原则上不公开 |
| 做课程 / 顾问 / 服务收费 | 可以基于理论收费 | 可以作为获客入口 | 这是核心收费区 |
| 授权他人二次使用 | 必须写清署名与边界 | 可做传播合作 | 必须合同控制 |

压缩理解：

- `A 类` 的关键不是藏，而是先锁定作者与正典版本。
- `B 类` 的关键不是深，而是负责解释、传播和导流。
- `C 类` 的关键不是完全不讲，而是不能把可复制执行法白送出去。

## 九、最小发布顺序

如果你现在就要开始对外发布，最稳的顺序是：

1. 先把 `DM / ASTO / ECET / TAT / ODD / LMM 公共理论面 / WSM 核心理论面` 统一作者、版本与引用口径。
2. 再用 `README`、八层总图、体系简介、各层摘要页做第一轮外部传播。
3. 投稿或预印本只优先放 `A 类` 主稿，不把 `COP / Harness / Progee2 / LMM 私有层` 混入主稿。
4. 需要合作转化时，只展示 `C 类` 的原则图和脱敏接口，不给模板、阈值、问诊件和全流程文档。
5. 真正进入服务、顾问、产品和部署时，再启用 `C 类` 内容完成收入闭环。
