# 论文体系总规划（现行版）

> **版本**：v1.0
> **日期**：2026-03-29
> **定位**：作为当前整套理论体系的**现行论文总规划主文件**，统一规定论文主位、文稿分工、仓库边界与后续推进顺序。
> **替代关系**：本文件用于取代历史命名较重的 `80-学术投稿（DM.papers）/RM论文体系.md` 在“现行总规划”上的作用；后者保留作历史溯源。

---

## 一、总原则

当前论文体系先按三层分工固定：

1. **`一元论`** 负责理论法源、理论主稿、预印本、征评稿、送审稿与论文总规划
2. **`.BetterCiv/05_学术型`** 负责案例证据、复核包、匿名化、盲评 SOP、投稿执行与推进状态
3. **`02Business`** 负责项目侧系统论文、历史来源稿、传播拆解稿、白皮书与展示副本

压缩一句话：

**理论主位在 `一元论`，学术执行在 `.BetterCiv`，项目论文与传播副本留在 `02Business`。**

---

## 二、当前最核心的边界规则

### 1. 什么必须放在 `一元论`

- 理论论文母本
- 预印本主稿
- 征评稿 / 送审稿主位
- 论文体系总规划
- 各理论层的正式法源表达

### 2. 什么必须放在 `.BetterCiv/05_学术型`

- 案例证据
- 补证清单
- 复核包
- 盲评 SOP
- 投稿执行文
- 论文推进状态

### 3. 什么可以放在 `02Business`

- Progee 这类系统/工具论文
- 项目展示用学术版副本
- 战略白皮书
- 系列传播拆解稿
- 理论旧稿与历史提纲

### 4. 三条禁止规则

禁止：

- 用项目侧历史稿反向竞争 `一元论` 的理论主稿
- 用投稿执行文冒充论文正文
- 用白皮书或展示副本替代理论法源文本

---

## 三、当前论文主链

整套体系当前论文主链，按以下顺序理解：

`DM -> ASTO -> ECET -> TAT -> ODD -> COP -> LMM -> Progee/System Paper`

其中要区分两件事：

- 前半段主要是**理论主链**
- 后半段开始出现**方法、研究、系统实现与现实验证接口**

## 五、当前已经定死的版本关系

### 1. 理论主稿 vs 发布副本

- 理论主稿负责持续编辑
- 发布副本负责对外展示、征评、预印本与项目发布
- 不允许双头维护

### 2. 理论主稿 vs 历史来源稿

- 历史来源稿保留溯源价值
- 不再作为现行主稿
- 如需复用内容，应回收进入现行主位后再发布

### 3. 理论主稿 vs 白皮书

- 白皮书承担传播、战略说明与高层解释
- 不承担理论法源任务

### 4. 论文正文 vs 投稿执行文

- 论文正文用于论证与发表
- 投稿执行文用于路线、分类、endorsement、会务、期刊筛选与提交操作
- 二者必须物理分开

## 七、当前推进顺序

当前论文体系不应同时四处开火，而应按以下顺序推进：

### 第一阶段：先稳法源主稿

优先稳住：
- DM
- ASTO
- ECET
- TAT
- ODD
- LMM 理论总述

目标：
- 保证每一层都有清楚的理论主位

### 第二阶段：再稳研究执行链

优先推进：
- LMM 案例稿
- LMM 方法稿
- LMM 补证与盲评
- 投稿执行文

目标：
- 形成从理论到证据、从证据到复核、从复核到投稿的闭环

### 第三阶段：再挂系统论文与项目副本

优先挂接：
- Progee System Paper CN/EN
- ODD v7.1 项目发布副本
- Whitepaper 系列

目标：
- 让项目侧表达与理论主链形成稳定映射，而不是版本竞争

## 八、当前最重要的纪律

1. **一稿一主位**：每个主题必须有唯一现行主位
2. **副本不夺权**：发布副本、白皮书、展示稿不能反向替代主稿
3. **执行不冒充正文**：投稿执行文不能再放在论文正文区
4. **项目不反向偷改理论**：`02Business` 中的项目论文不能反向改写 `一元论` 的上位定义
5. **历史稿只作溯源**：历史来源稿保留，但不再回升为现行母稿

---

## 九、下一步最值得继续做的事

1. 给 COP 明确论文主位或占位规划
2. 继续完善 LMM 从理论总述到案例/方法/复核/投稿的闭环
3. 视需要在 `一元论/80-学术投稿（DM.papers）/` 下建立现行导航，把本总规划与各层主稿统一挂接
4. 未来如出现新的系统论文，也按“理论主稿 / 系统论文 / 白皮书 / 执行文”四分法处理
