# 领域迁移：ODD 超越软件的实证

> **作者**: Yi Fu (ODDFounder, fuyi.it@live.cn)
> **版本**: v1.0（中文初稿）
> **目标刊/会**: ASE 2027 或 AAAI 2027

---

## 摘要

产出物驱动开发（ODD）诞生于软件工程领域，但其核心概念——契约定义产出物、验证替代审查、封存保护合法性——是否具有领域无关性？本文通过将 ODD 完整迁移到 **AI 辅助小说创作**领域（kiro-gateway 项目），实证回答这一问题。我们展示了 ODD 七个核心概念在创作域的自然映射、五次方案演进（v1→v5）中的领域适配过程、以及 ASTO 哲学框架（六阶×五态×七序）如何为非软件领域提供契约坐标系。结果表明：ODD 的**内核**（契约→验证→封存循环）可直接复用；ODD 的**外壳**（验证手段与知识表示）需要领域适配。整体结论是：ODD 的本质不是软件工程方法论，而是**人类治理 AI 产出物的通用框架**。

**关键词**: 领域迁移、领域无关性、AI 辅助创作、契约化写作、ASTO

---

## 1. 引言

### 1.1 方法论的领域锁定问题

大多数软件工程方法论与"软件"深度绑定：
- TDD 的"测试"是代码测试；
- Agile 的"迭代"是代码迭代；
- DevOps 的"部署"是服务部署。

这种强绑定导致这些方法论难以迁移到非软件领域。如果 ODD 也只能适用于软件，那么它的理论贡献将受限于软件工程本身。

### 1.2 为什么选择小说创作？

小说创作是软件的对立面：它是主观的、创意驱动的、审美导向的。选择小说创作作为迁移对象有两个意义：

1. **极端测试**：如果 ODD 能在这个最不"工程化"的领域成立，说明它有强烈的领域无关性。
2. **完整实现**：本文不是概念类比，而是在真实项目中完成 ODD 的全流程实现（契约 → 验证 → 封存）。

kiro-gateway 是一个 AI 辅助长篇小说创作系统，目标是将“章节级写作需求”转换为“可被 AI 稳定执行的写作契约”。它为本研究提供了完整的迁移场景。

### 1.3 本文贡献

1. **跨领域映射**：给出 ODD 七个核心概念在小说领域的完整映射。
2. **ASTO 坐标系验证**：展示 ASTO 哲学框架如何在创作域中扮演“契约坐标系”。
3. **迁移过程实证**：基于 v1→v5 的五次方案演进，分析迁移适配的真实过程。
4. **内核-外壳理论**：识别 ODD 的领域无关内核与领域依赖外壳。

## 2. 概念映射：从软件到小说

### 2.1 七个核心概念的映射

| ODD（软件） | kiro-gateway（小说） | 映射难度 |
|-------------|---------------------|---------|
| 契约（Contract） | 场景乐谱（score_json） | 自然 |
| 产出物（Artifact） | 场景段落集 | 自然 |
| 变异测试 | 四层校验 + 七评委评分 | 需适配 |
| 封存（Seal） | 集封版（episode seal） | 直接 |
| CAP 对抗 | 契约三重门禁 | 自然 |
| bug_intention_map | writing_knowledge | 需适配 |
| 17 层上下文 | 12 层上下文（NL1-NL12） | 需裁剪 |

该映射表明：ODD 的多数概念在小说领域具有自然对应物，只有“验证手段”和“知识表示”需要适配。

### 2.2 直接复用的概念（ODD 内核）

- **契约→产出物→封存循环**：完全可复用。小说场景被视为产出物，契约定义“可接受空间”，封存保护最终版本。
- **契约门禁机制**（清晰度评估→CAP 对抗→PK）：可直接复用。写作契约也需要防止模糊与漏洞。
- **审计链**：契约、评审、封存记录可以完整迁移。

这些内容构成 ODD 的领域无关内核：**契约治理产出物的责任结构**。

### 2.3 需要适配的概念（ODD 外壳）

**（1）变异测试 → 四层校验**

软件变异测试依赖代码算子替换（如条件取反、返回值替换），而文本没有等价的算子。kiro-gateway 采用四层校验替代：

- **Layer 0 禁元**：硬约束检查（如已死角色不得出现）
- **Layer 1 基元**：乐谱命中率、叙事关键事件命中
- **Layer 2 指标**：反 AI 味词频、细节密度、节奏一致性
- **Layer 3 人类裁决**：七评委评分系统

这是一种“语义变异测试”的替代形式——不修改文本，而通过多维约束验证语义质量。

**（2）bug_intention_map → writing_knowledge**

软件中的缺陷意图映射在小说领域演化为“写作知识库”。两者的核心机制一致：
- 记录模式
- 设定置信度
- 根据效果衰减或淘汰

区别在于知识内容：从“缺陷模式”转向“叙事模式”。

**（3）17 层上下文 → 12 层上下文**

软件领域的上下文层包含语言/框架/代码层，这些在小说领域不存在。迁移过程中：
- 丢弃 4 层（软件特有）
- 合并 7 组（降低信息密度）
- 拆分 2 层（创作特有，如“叙事指令”和“禁元规则”）

核心保持不变：分层原则与确定性组装。

### 2.4 映射启示：ODD 的本质

- 可直接复用的部分构成**领域无关内核**
- 需要适配的部分构成**领域依赖外壳**

因此，ODD 的本质不是“软件工程方法论”，而是“人类治理 AI 产出物的通用责任结构”。

## 3. ASTO 哲学框架作为契约坐标系

ODD 的关键问题是“契约坐标系”：如何用统一坐标描述产出物期望？在软件中，契约通常使用“接口/约束/性能”描述。在小说领域，ASTO 提供了一个跨领域的坐标系。

### 3.1 ASTO 简介

ASTO 由三组结构构成：
- **六阶**：混沌 → 秩序 → 流变 → 脉冲 → 崩解 → 归元
- **五态**：触感 → 编码 → 解码 → 干预 → 超越
- **七序**：觉醒 → 感知 → 解析 → 干预 → 设计 → 回溯 → 消解

### 3.2 六阶映射：叙事节律

| 六阶 | 叙事特征 | 契约体现 |
|------|---------|----------|
| 混沌(a) | 信息碎片、氛围铺垫 | phase="a"，低信息密度 |
| 秩序(b) | 规则建立、关系明确 | phase="b"，对话推进 |
| 流变(c) | 悬念推进、渐变累积 | phase="c"，线索串联 |
| 脉冲(d) | 高潮、不可逆 | phase="d"，高冲击约束 |
| 崩解(e) | 旧秩序瓦解 | phase="e"，信念崩塌 |
| 归元(f) | 收束、沉淀 | phase="f"，留白约束 |

这使得“场景节奏”可以被量化并嵌入契约。

### 3.3 五态映射：存在质感

五态用于描述场景的存在质感（“AI 应该用什么腔调写”）。在契约中，通过 `mode` 字段注入。

例如：
- 触感态 → 强调感官描写
- 编码态 → 强调结构化信息
- 解码态 → 强调解释与揭示

### 3.4 七序映射：注意力权重

在契约中，`sequence_profile.weights` 直接映射七序权重，控制场景的注意力焦点（比如“解析”占比高意味着该场景以解释和理性分析为主）。

### 3.5 ASTO 的通用坐标系假说

ASTO 不仅适用于小说领域：
- **软件领域**：六阶可映射开发阶段，五态可映射系统状态
- **创作领域**：六阶映射叙事节律，五态映射存在质感

这提示 ASTO 可能是一个**跨领域的契约坐标系**。ODD 在创作领域的成功迁移，为这一假说提供了实证支持。

## 4. 五次方案演进：领域适配的实证过程

kiro-gateway 的 v1→v5 演进记录了完整的领域适配过程。

| 版本 | 方案 | Token量 | 核心问题 | 保留洞察 |
|------|------|---------|---------|----------|
| v1 | Lorebook（借鉴 SillyTavern） | ~4000 | 关键词触发是猜测式 | 递归扫描、Token预算 |
| v2 | 编译器方案 | ~1500 | 只解决数据组织 | 结构化 > 散文 |
| v3 | 状态机 + 后置校验 | ~800 | 信息仍然太多 | 三层分离、后置校验 |
| v4 | 乐谱 + 声音样本 | ~300 | 太极端，缺伏笔 | 乐谱隐喻、范文优先 |
| v5 | Progee2 工业化 | ~500 | 最终方案 | 12层分层 + 规则引擎 |

### 4.1 决策链（D1-D25）的意义

- v1→v2：从“关键词触发”转向“结构化编译”
- v2→v3：加入状态机和后置校验，增强可控性
- v3→v4：范式跳跃，采用“乐谱隐喻”
- v4→v5：工程回归，将工程化规则重新注入

这些决策形成完整的证据链，证明领域适配不是一次性设计，而是持续演化的过程。

### 4.2 迁移的一般模式

我们总结出领域迁移的四阶段模式：

1. **直接套用**（通常失败）：v1 的 Lorebook 方案
2. **发现差异**：v2-v3 发现“文本不是数据”
3. **目标域隐喻**（可能矫枉过正）：v4 的乐谱方案
4. **工程化平衡**：v5 结合工程规则与创作隐喻

这一模式解释了为什么迁移通常不是线性过程，而是“摆动式”收敛。

## 5. 契约化创作的实证效果

### 5.1 契约结构（score_json）

score_json 包含九大属性组：ASTO 坐标、时空锚点、角色集、叙事指令、禁元规则、地板条件、字数范围、验收标准等。它是“场景契约”的核心表达形式。

与软件契约相比：
- 结构上相似（约束 + 依赖 + 验收）
- 内容上不同（叙事约束 + 审美指标）

### 5.2 四层校验效果

- **Layer 0 禁元**：高效阻止硬错误（如已死角色出场）
- **Layer 1 基元**：确保乐谱命中率与关键事件一致
- **Layer 2 指标**：反 AI 味词频、细节密度、节奏一致性
- **Layer 3 人类裁决**：七评委评分作为最终裁决

这形成了“自动化验证 + 人类裁决”的组合，与软件中的“自动化测试 + 人工评审”具有同构性。

### 5.3 自动封版效果

封版策略：当七评委均分 ≥ 7.5，且单项最低 ≥ 7.0，系统自动封版。实证结果显示：
- 自动封版显著减少人类干预频率
- 质量趋势随集数推进保持稳定

### 5.4 与无契约写作的对比

- **一致性**：有契约时跨场景“声音漂移”显著减少
- **信息门控**：有契约时信息泄漏率降低
- **审计性**：有契约时可追溯“为什么这样写”

这些差异表明，契约化写作不仅是“提示词技巧”，而是可治理、可审计的系统化生产方式。

### 5.5 数据表：契约化创作效果摘要

| 指标 | 数值 | 说明 |
|------|------|------|
| Token 演进 | 4000 → 500（-87.5%） | v1→v5
| 自动封版门槛 | 均分 ≥ 7.5；单项 ≥ 7.0 | 七评委规则 |
| 方案演进次数 | 5 次 | v1→v5
| 层数适配 | 17 → 12 | 上下文层裁剪 |
| 关键验证结构 | 四层校验 | 禁元/基元/指标/裁决 |

## 6. 讨论

### 6.1 ODD 的领域无关内核

- 契约治理产出物的责任结构是领域无关的
- 产出物合法性需要验证与封存

### 6.2 领域依赖外壳

- 验证手段必须适配领域特性（变异测试 → 四层校验）
- 知识表示必须适配领域语义（bug_intention_map → writing_knowledge）

### 6.3 创意领域的契约化边界

- 可契约化部分：结构、一致性、信息门控、禁忌、字数
- 难契约化部分：审美共鸣、原创性、情绪感染力

因此，创意领域的契约化必须承认“不可完全形式化”的边界，并通过 Layer 3 人类裁决补足。

### 6.4 与现有 AI 写作工具的对比

现有 AI 写作工具（ChatGPT / NovelAI / Sudowrite）侧重于生成能力，而不是治理结构。ODD 的迁移结果表明：**生成能力可以借用，但治理结构必须自行设计**。这正是 ODD 的独特价值。

### 6.5 威胁与局限

- 单部小说、单作者的案例限制外部效度
- 小说质量评估仍存在主观性
- ASTO 坐标系尚未在更多领域验证

---

## 7. 结论

本文通过 kiro-gateway 的完整实现证明：ODD 的核心机制能够迁移到非软件领域，并在创意写作中建立可治理、可审计的生产流程。关键结论如下：

1. **ODD 的内核具有领域无关性**：契约→验证→封存循环可直接复用。
2. **ODD 的外壳需要领域适配**：验证手段与知识表示必须贴合领域语义。
3. **ASTO 可作为通用契约坐标系**：在创作域中已实证其可用性。
4. **迁移是演化式过程**：从直接套用到工程化平衡，需要多次迭代。

ODD 的本质不在于“写代码”，而在于“治理 AI 产出物”。因此，它有潜力成为跨领域的通用方法论。未来工作将验证教育、设计、法律文书等领域的迁移可能性。
