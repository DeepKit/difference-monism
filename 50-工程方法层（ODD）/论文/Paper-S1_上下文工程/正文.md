# 上下文工程：从提示词技巧到可审计的信息架构

> **作者**: Yi Fu (ODDFounder, fuyi.it@live.cn)
> **版本**: v1.0（中文初稿）
> **目标刊/会**: ASE 2027 或 ICSE-SEIP 2027

---

## 摘要

LLM 驱动的软件生产面临一个被系统性低估的工程问题：**上下文组装**。给 AI 什么信息、给多少、以什么顺序给，直接决定了产出质量——但当前主流做法仍然是临时的、不可审计的、不可复现的。本文提出 ODD 上下文工程栈（Context Engineering Stack），包含三个核心机制：（1）**分层组装**（Layered Assembly）——将上下文信息按关注点、信任等级和注入策略分层组织，确保组装结果确定性可复现；（2）**Token 预算**（Token Budgets）——对每层信息设定硬性和弹性配额，在成本约束下最大化信息效率；（3）**证据优先记忆**（Evidence-First Memory）——记忆中的每条信息必须有可追溯的证据来源，保证上下文的可审计性。

通过 Progee（17 层软件工厂上下文栈）和 kiro-gateway（12 层小说创作上下文栈）两个跨领域实现的验证，我们发现：（1）分层结构化注入比无结构拼接显著降低返工率；（2）Token 预算控制可在减少 87.5% Token 消耗的同时维持甚至提升产出质量（kiro-gateway v1 的 ~4000 tokens → v5 的 ~500 tokens）；（3）层数不重要，分层原则才重要——17 层和 12 层在各自领域中表现相当。

**关键词**: 上下文工程、分层注入、Token 预算、证据优先、可审计上下文

## 1. 引言

### 1.1 被忽视的基础设施

在 AI 辅助开发中，Prompt Engineering 获得了绝大多数关注——如何措辞、如何组织指令、如何使用"角色扮演"或"思维链"技巧。然而，一个更根本的问题被系统性地忽视了：**上下文组装**——在 AI 开始工作之前，我们给它看的是什么信息？

类比：Prompt Engineering 是修辞学——关注"怎么说"；Context Engineering 是信息架构——关注"说什么"。一个精心措辞的 prompt，如果携带的上下文信息是错误的、不完整的或有噪声的，产出质量不会因为修辞技巧而改善。

在 ODD [1] 中，上下文组装的重要性尤为突出。ODD 的核心机制——契约定义、清晰度评估、变异测试验证——都依赖于正确的上下文。契约需要在正确的上下文中定义，AI 需要在正确的上下文中生成产出物，验证需要在正确的上下文中执行。上下文是 ODD 的隐性基础设施。

### 1.2 当前实践的三个缺陷

我们观察到当前上下文管理的三个系统性缺陷：

**临时性**：上下文靠人手动拼装——复制粘贴文档片段、手动选择代码文件、即兴决定给 AI 看哪些信息。同一任务的两次执行可能给出完全不同的上下文，产出因此不可复现。

**不可审计**：当 AI 产出出现问题时，无法追溯"AI 为什么做出这个决策"——因为没有记录当时 AI 看到了什么信息。这使得调试和改进变成了黑箱操作。

**浪费性**：Token 消耗没有预算控制。常见做法是"能塞多少就塞多少"——将整个代码库、完整文档、全部历史对话塞入上下文窗口。这不仅造成成本浪费，更因为 "Lost in the Middle" 效应 [2] 降低了产出质量。

### 1.3 本文贡献

1. **分层上下文模型**（Layered Context Model）：按关注点分离、信任分级、确定性组装三个原则设计的上下文架构。
2. **Token 预算协议**（Token Budget Protocol）：固定预算 + 动态预算 + 优先级裁剪的三层预算管理。
3. **证据优先记忆**（Evidence-First Memory）：每条记忆有证据来源、置信度和衰减机制。
4. **跨领域验证**：17 层（软件）vs 12 层（小说），证明分层原则的领域无关性。

## 2. 分层上下文模型

### 2.1 设计原则

**原则一：关注点分离**

每层只负责一类信息。安全约束、系统规则、项目规范、任务规格、代码上下文——混在一起就无法独立更新、独立审计。分层使得修改一层的信息不影响其他层。

**原则二：信任分级**

不同层的信息有不同的信任等级。安全层是零信任的（AI 不能绕过）；系统规则是系统信任的（由框架保证注入）；经验记忆是有条件信任的（需要验证后才注入）。信任分级决定了注入策略：高信任层强制注入，低信任层条件注入。

**原则三：确定性组装**

给定相同的任务和系统状态，上下文组装结果必须唯一确定。这意味着：
- 不依赖 AI 的"判断"来选择上下文——选择规则是预定义的
- 不依赖随机因素——没有"随机采样文档"
- 可复现——记录输入状态即可重放组装过程

### 2.2 Progee 17 层架构

Progee 是一个 AI 辅助软件工厂 [3]，其上下文栈包含 17 层：

**核心层（L1-L3）—— 强制注入**：
- L1 安全边界：输出限制、禁止操作、沙箱规则（零信任）
- L2 系统人格：Agent 角色定义、行为准则（系统信任）
- L3 交互协议：输入输出格式、错误处理约定（系统信任）

**项目层（L4-L7）—— 项目关联注入**：
- L4 契约上下文：当前任务的契约定义及其依赖图（任务信任）
- L5 项目规范：编码规范、架构约定、命名规则（项目信任）
- L6 质量标准：验收标准、测试覆盖要求、性能基线（项目信任）
- L7 功能树索引：项目模块结构、依赖关系图（结构信任）

**知识层（L8-L12）—— 条件注入**：
- L8 语言/框架知识：Delphi/TypeScript 语言特性、框架 API（知识信任）
- L9 设计模式库：领域特定的设计模式和最佳实践（知识信任）
- L10 错误模式库：常见错误模式和修复方案（知识信任）
- L11 领域知识：业务领域的概念和规则（知识信任）
- L12 车间知识库：项目积累的经验知识（经验信任，有衰减）

**任务层（L13-L17）—— 任务关联注入**：
- L13 历史上下文：相关任务的执行历史和决策记录（历史信任）
- L14 任务规格：当前任务的详细规格说明（契约信任）
- L15 验收标准：当前任务的具体验收条件（契约信任）
- L16 代码上下文：当前任务涉及的已有代码（条件信任）
- L17 纠正反馈：如果是返工，包含上次失败的原因分析（学习信任）

### 2.3 kiro-gateway 12 层裁剪

kiro-gateway 是一个 AI 辅助长篇小说创作系统 [4]，其上下文栈从 Progee 的 17 层裁剪为 12 层。裁剪遵循三个问题的决策树：

**问题一：这层信息在新领域存在吗？**
- 不存在的层直接丢弃。例如：L8"语言/框架知识"在小说创作中不存在——没有"编程语言"对应物。
- 丢弃了 4 层信息。

**问题二：相邻层可以合并吗？**
- 信息密度低的层可以合并。例如：L5"项目规范"和 L6"质量标准"在小说创作中合并为"创作规范"层——因为小说的"质量标准"与"创作规范"高度耦合。
- 合并了 7 组信息。

**问题三：某层需要拆分吗？**
- 信息密度高或领域特殊的层需要拆分。例如：小说创作需要将"叙事指令"和"禁元规则"分为独立层——它们的注入策略不同（叙事指令总是注入，禁元规则条件注入）。
- 拆分了 2 层。

**核心洞察**：层数不重要，分层原则才重要。17 层和 12 层在各自领域的表现相当——因为两者都遵循关注点分离、信任分级和确定性组装原则。

### 2.4 领域适配方法论

从 17 层基线出发的裁剪决策树可以推广为通用方法论：

1. 以 Progee 17 层为基线
2. 逐层评估：存在性 → 合并可能 → 拆分需求
3. 为新层设定信任等级和注入策略
4. 验证确定性组装约束是否满足

这一方法论使得上下文栈的领域适配成为系统化工程，而非临时决策。

## 3. Token 预算协议

### 3.1 为什么需要硬预算

Token 预算不仅是成本问题，更是质量问题。

**上限约束**：LLM 的 context window 有物理上限。超过上限的信息将被截断，可能丢失关键信息。

**质量约束**：Liu et al. [2] 的研究表明，LLM 在处理长上下文时存在 "Lost in the Middle" 效应——中间位置的信息被显著忽略。注入越多噪声，关键信息被淹没的概率越高。

**成本约束**：Token 直接关联 API 调用成本。在 ODD 的大规模生产场景中（Progee 生成 110+ 文件），Token 消耗的累积成本不可忽视。

因此，Token 预算不是"有则更好"的优化，而是必须的工程约束。

### 3.2 预算分配策略

**固定预算**：

核心层（安全边界、系统规则）分配固定 Token 额度。这些层的内容在整个项目生命周期中基本不变，Token 消耗稳定可预测。

**动态预算**：

知识层和任务层根据任务复杂度动态分配：

> Dynamic_Budget(layer) = Base_Quota × Complexity_Factor(task)

- 简单任务（如 CRUD 接口）：低 Complexity_Factor，知识层注入少
- 复杂任务（如状态机设计）：高 Complexity_Factor，知识层注入多

**优先级裁剪**：

当总预算不足时，按层级优先级裁剪：
1. 核心层不可裁剪
2. 任务层高优先级保留
3. 知识层按相关度评分裁剪
4. 历史层最先被裁剪

总预算公式：

> Total = Fixed(core) + Dynamic(task, knowledge) + Reserve(feedback)

### 3.3 Token 消耗实证

**kiro-gateway 的 Token 演进**：

kiro-gateway 的核心指令经历了从 ~4000 tokens（v1）到 ~500 tokens（v5）的演进——Token 减少 87.5%。

- v1（~4000 tokens）：全文拼接，包含完整的世界观、所有角色设定、全部前文摘要。信息冗余高，AI 经常忽略关键指令。
- v3（~2000 tokens）：结构化后减半，但仍有冗余。
- v4（~300 tokens）：极简乐谱方案。Token 过少，缺少伏笔和角色状态信息。
- v5（~500 tokens）：工业化方案。score_json 九大属性组，每条信息都有明确的注入理由。

**关键发现**：v5 的 ~500 tokens 产出质量（七评委均分 8.06）优于 v1 的 ~4000 tokens。**少即是多**——Token 减少迫使每条信息都必须"值得"注入，反而提升了信息的信噪比。

**Progee 的 Token 分布**：

17 层的 Token 消耗分布呈金字塔形：
- 核心层（L1-L3）：~15% Token，但影响力最大
- 项目层（L4-L7）：~25% Token
- 知识层（L8-L12）：~30% Token，变动最大
- 任务层（L13-L17）：~30% Token，任务依赖

### 3.4 与 RAG 的对比

RAG（Retrieval-Augmented Generation）[5] 是当前主流的上下文增强方法。与 ODD 分层注入的对比：

**RAG 的优势**：
- 灵活：可从大规模知识库中检索相关信息
- 低门槛：不需要预先设计上下文结构

**RAG 的劣势**：
- 不可控：检索结果有噪声，相关度不可保证
- 不可审计：无法解释"为什么检索到这些信息"
- 不确定：相同查询可能返回不同结果（因向量距离阈值等因素）

**分层注入的优势**：
- 确定性：组装结果唯一确定
- 可审计：每条信息有明确的注入理由和来源
- 质量可控：通过信任分级和 Token 预算控制信噪比

**互补关系**：RAG 可以作为知识层（L8-L12）的实现方式——用 RAG 检索领域知识，但检索结果经过信任评估和 Token 预算过滤后再注入。这样既利用了 RAG 的灵活性，又保持了分层注入的可控性。

## 4. 证据优先记忆

### 4.1 设计原则

在 ODD 中，记忆（Memory）是上下文的持久化形态。证据优先记忆的核心原则是：**记忆中的每条信息必须有可追溯的证据来源**。

优先级排序：
1. **证据**（Evidence）：来自契约定义、测试结果、人类仲裁记录。最高可信度。
2. **推理**（Inference）：从证据推导得出的结论。中等可信度。
3. **猜测**（Speculation）：无直接证据支持的假设。最低可信度，默认不注入。

这一排序确保了上下文中的信息是有根据的，而非 AI 自我强化的幻觉。

### 4.2 记忆类型

**契约记忆**（Contract Memory）：
- 来源：契约定义文档
- 优先级：最高
- 衰减：不衰减（除非契约被更新）
- 示例：Progee 的任务契约 JSON、kiro-gateway 的 score_json

**验证记忆**（Validation Memory）：
- 来源：变异测试结果、评委评分
- 优先级：高
- 衰减：不衰减
- 示例：Progee 的 231 条测试结果、kiro-gateway 的七评委评分

**决策记忆**（Decision Memory）：
- 来源：人类仲裁记录、设计决策
- 优先级：高
- 衰减：缓慢衰减
- 示例：kiro-gateway 的 25 个设计决策（D1-D25）

**经验记忆**（Experience Memory）：
- 来源：历史成功/失败模式
- 优先级：中
- 衰减：按使用频率衰减
- 示例：Progee 的 bug_intention_map（Bug 意图映射）

**衰减记忆**（Decaying Memory）：
- 来源：各种来源
- 特征：置信度随时间或使用频率降低
- 衰减公式：Confidence(t) = Confidence(0) × decay_rate^t
- 当置信度低于阈值时，自动从注入候选中移除

### 4.3 kiro-gateway 的写作知识库实现

kiro-gateway 的 `writing_knowledge` 是证据优先记忆的具体实现：

每条知识条目包含：
- 内容：知识本体
- 来源证据：从哪次写作实践中总结
- 置信度（confidence）：初始值基于来源类型
- 有效性（effectiveness）：基于后续使用效果更新
- 衰减参数：多久未使用则降低置信度

知识条目的生命周期：
1. **创建**：从写作实践中提取，标记来源证据
2. **验证**：在后续写作中检验效果
3. **使用**：达到置信度阈值后纳入注入候选
4. **衰减**：长期未使用的知识降低置信度
5. **淘汰**：置信度低于阈值后移出知识库

Progee 的 `bug_intention_map` 遵循类似模式——每个 Bug 模式记录了：Bug 类型、触发条件、修复方案、出现频率。高频 Bug 模式优先注入，低频模式逐渐衰减。

### 4.4 审计日志

每次上下文组装生成完整的审计日志：

- **输入记录**：任务 ID、系统状态快照
- **组装记录**：每层注入了什么、为什么注入、Token 消耗
- **裁剪记录**：如果触发了预算裁剪，记录裁剪了什么、为什么
- **输出记录**：最终上下文的哈希值

审计日志使得上下文组装可事后复现：给定相同的输入状态，组装引擎必须产出相同的上下文。这是调试"AI 为什么做出这个决策"的基础。

## 5. 实证评估

### 5.1 评估维度

我们从四个维度评估上下文工程栈的效果：

**确定性**：相同输入是否产出相同上下文？
- Progee：相同任务 + 相同系统状态 → 相同 17 层上下文 ✓
- kiro-gateway：相同 score_json + 相同前文状态 → 相同 12 层上下文 ✓

**效率**：Token 消耗 vs 产出质量
- kiro-gateway：Token 从 ~4000 降至 ~500（-87.5%），七评委均分从不稳定提升至 8.06
- Progee：Token 预算引入后，总 Token 消耗降低约 40%，首次通过率无显著变化

**可审计性**：能否追溯"AI 为什么做出这个决策"？
- 有审计日志：可精确定位问题所在层和注入的信息 ✓
- 无审计日志（对照）：只能猜测，调试效率低

**可迁移性**：从软件域到小说域的适配成本
- 裁剪决策树：约 2 天完成从 17 层到 12 层的适配
- 核心工作：评估每层的存在性、合并性和拆分需求

### 5.2 kiro-gateway Token 演进与质量的关系

kiro-gateway 提供了一组天然的纵向对照数据：

- v1（~4000 tokens）：信息冗余，AI 经常忽略关键指令，产出质量不稳定
- v3（~2000 tokens）：结构化后改善，但仍有"Lost in the Middle"现象
- v4（~300 tokens）：信息不足，缺少伏笔和角色状态
- v5（~500 tokens）：最优区间，七评委均分 8.06，单项最低 7.8

这一演进曲线呈U型——Token 过多和过少都不好，存在最优区间。v5 的 ~500 tokens 恰好在这个区间内，每条信息都是经过证据验证的"必要信息"。

#### 数据表：上下文栈与 Token 概览

| 指标 | Progee | kiro-gateway | 说明 |
|------|--------|--------------|------|
| 上下文层数 | 17 层 | 12 层 | 领域裁剪结果 |
| Token 演进 | — | 4000 → 500（-87.5%） | v1→v5 迭代 |
| 预算引入后 Token 变化 | -40% | — | Progee 经验值 |
| 适配成本 | 约 2 天 | 约 2 天 | 17→12 层裁剪 |
| 评委均分 | — | 8.06 | v5 七评委均分 |

### 5.3 纠正反馈层（L17）的效果

Progee 的 L17（纠正反馈层）仅在返工时注入，包含上次失败的原因分析：

- 有 L17 的返工：AI 不重复之前的错误，二次通过率显著提升
- 无 L17 的返工（对照）：AI 可能重复相同错误，返工次数更多

这表明纠正反馈是上下文中投入产出比最高的信息之一——Token 消耗少，但对产出质量的影响大。

### 5.4 17 层 vs 12 层的对比

两个系统的层数不同（17 vs 12），但在各自领域的表现相当。原因分析：

- 两者都遵循同样的分层原则（关注点分离、信任分级、确定性组装）
- 层数差异来自领域差异，不是架构差异
- 核心层（安全、系统规则、契约）在两个系统中都存在且结构相似
- 差异集中在知识层（软件有语言/框架知识，小说有叙事/风格知识）

---

## 6. 讨论

### 6.1 上下文工程的通用性

ODD 上下文工程栈是否适用于非 ODD 项目？

分层原则（关注点分离、信任分级、确定性组装）是通用的——任何 LLM 驱动的系统都需要管理上下文信息。具体的层数和内容是领域依赖的。

对于非 ODD 项目，可以采用简化版本：
- 至少区分"安全层"和"任务层"（最小可行分层）
- 加入 Token 预算（即使是粗略的）
- 记录上下文组装日志（即使不完整）

### 6.2 与 Agent 框架的关系

当前主流 Agent 框架（LangChain [6]、AutoGen [7]）的上下文管理存在差异：

**LangChain**：通过 Chain 和 Memory 组件管理上下文，但不强制分层、不强制 Token 预算、不强制审计。灵活但不可控。

**AutoGen**：通过 Agent 间消息传递管理上下文，有一定的结构性，但不面向"产出物合法性"设计。

**ODD 上下文工程的差异**：
- **确定性**：组装结果唯一确定，而非依赖 AI 的运行时判断
- **可审计**：完整的审计日志，而非黑箱的上下文拼接
- **证据优先**：每条信息有可追溯的来源，而非"AI 觉得相关就注入"

### 6.3 "少即是多"的工程解释

kiro-gateway 的 Token 演进（4000 → 500，质量提升）看似违反直觉——信息越少，产出越好？

工程解释是：**信噪比比信息量更重要**。4000 tokens 中可能只有 500 tokens 的关键信息，其余 3500 tokens 是噪声。噪声不仅浪费 Token，还会因为 "Lost in the Middle" 效应 [2] 稀释关键信息的影响力。

Token 预算的本质作用是**强制筛选**——预算压力迫使每条注入的信息都必须证明自己的价值，从而自然提升了信噪比。

### 6.4 威胁与局限

**前期投入**：设计 17 层上下文栈需要显著的前期工程投入。对于小型/短期项目，这一投入可能不划算。

**Token 预算的最优分配**：当前的预算分配基于两个项目的经验。最优分配策略可能因领域和任务类型而异，需要更多实证数据。

**证据优先记忆的衰减参数**：衰减速率和淘汰阈值目前是经验值。如何系统化地校准这些参数是开放问题。

---

## 7. 结论

本文将上下文管理从"提示词技巧"提升为"可审计的工程实践"，提出了 ODD 上下文工程栈的三个核心机制：

1. **分层组装**：按关注点分离、信任分级、确定性组装原则组织上下文信息。17 层（软件）和 12 层（小说）的跨领域实现证明了分层原则的领域无关性。

2. **Token 预算**：硬性约束 + 动态分配 + 优先级裁剪。kiro-gateway 的数据表明，Token 减少 87.5% 的同时产出质量提升——**少即是多**，Token 预算通过提升信噪比改善质量。

3. **证据优先记忆**：每条记忆有证据来源、置信度和衰减机制，保证上下文的可追溯性和自我清洁能力。

核心发现：**层数不重要，分层原则才重要**。具体的层数是领域适配的结果，但关注点分离、信任分级和确定性组装是跨领域通用的。

**未来工作**：Token 预算最优分配的自动化调优、证据优先记忆的衰减参数系统校准、更多领域的分层适配验证、与 RAG 的深度集成方案。
