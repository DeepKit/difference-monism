# Paper S1 提纲：上下文工程

---

## 摘要（草案）

LLM 驱动的软件生产面临一个被低估的工程问题：上下文组装。给 AI 什么信息、给多少、以什么顺序给，直接决定了产出质量——但目前主流做法仍然是临时的、不可审计的、不可复现的。本文提出 ODD 上下文工程栈：**分层组装**（Layered Assembly）确保信息的结构化注入，**Token 预算**（Token Budgets）强制成本约束，**证据优先记忆**（Evidence-First Memory）保证上下文的可追溯性。通过 Progee（17 层）和 kiro-gateway（12 层）两个实现，展示上下文工程如何从"提示词技巧"升级为"可审计的工程实践"。

**关键词**: 上下文工程、分层注入、Token 预算、证据优先、可审计上下文

---

## 1. 引言

### 1.1 被忽视的基础设施
- Prompt Engineering 关注"怎么说"
- 上下文工程关注"说什么"——更根本的问题
- 类比：Prompt Engineering 是修辞学，Context Engineering 是信息架构

### 1.2 当前实践的三个缺陷
1. **临时性**：上下文靠人手动拼装，不可复现
2. **不可审计**：无法追溯"AI 为什么做出这个决策"
3. **浪费性**：Token 无预算控制，成本不可控

### 1.3 本文贡献
- 提出分层上下文模型（Layered Context Model）
- 定义 Token 预算协议（Token Budget Protocol）
- 设计证据优先记忆架构（Evidence-First Memory）
- 通过两个跨领域实现验证

---

## 2. 分层上下文模型

### 2.1 设计原则
- **关注点分离**：每层只负责一类信息
- **强制/可选分级**：安全层强制注入，知识层条件注入
- **确定性组装**：给定任务和状态，上下文组装结果唯一确定

### 2.2 Progee 17 层架构

| 层级 | 内容 | 注入策略 | 信任等级 |
|------|------|---------|---------|
| L1 | 安全边界 | 强制 | 零信任 |
| L2-L3 | 系统规则 | 强制 | 系统信任 |
| L4 | 契约上下文 | 角色关联 | 任务信任 |
| L5-L6 | 项目规范 | 项目关联 | 项目信任 |
| L7 | 功能树索引 | 动态查询 | 结构信任 |
| L8-L11 | 领域知识 | 条件注入 | 知识信任 |
| L12 | 车间知识库 | 条件注入 | 经验信任 |
| L13 | 历史上下文 | 按需 | 历史信任 |
| L14-L15 | 任务规格+验收标准 | 任务关联 | 契约信任 |
| L16 | 代码上下文 | 按需 | 条件信任 |
| L17 | 纠正反馈 | 返工时注入 | 学习信任 |

### 2.3 kiro-gateway 12 层裁剪
- 从 17 层到 12 层的映射
- 丢弃的 4 层：为什么不需要（领域差异）
- 合并的 7 组：为什么可以合并（信息密度优化）
- 拆分的 2 层：为什么需要拆分（小说领域特殊性）
- 核心洞察：**层数不重要，分层原则才重要**

### 2.4 领域适配方法论
- 从 17 层基线出发的裁剪决策树
- 三个裁剪问题：这层信息在新领域存在吗？可以合并吗？需要拆分吗？

## 3. Token 预算协议

### 3.1 为什么需要硬预算
- LLM Token 有上限（context window）
- 更重要的：注入越多噪声，生成质量越差（Lost in the Middle 问题）
- 成本约束：Token = 钱

### 3.2 预算分配策略
- **固定预算**：安全层/系统层分配固定 Token 额度
- **动态预算**：知识层/代码层根据任务复杂度动态分配
- **优先级队列**：当总预算不足时，按层级优先级裁剪
- 公式：Total Budget = Fixed + Dynamic(task_complexity) + Reserve

### 3.3 Token 消耗实证
- Progee: 17 层平均 Token 消耗分布
- kiro-gateway: v1(~4000) → v5(~500) 的 Token 演进
- 关键发现：Token 减少 87.5% 的同时，产出质量提升

### 3.4 与 RAG 的对比
- RAG: 检索→注入，不可控、有噪声
- 分层注入: 确定性组装，每条信息有明确来源和理由
- 两者可互补：RAG 作为知识层的实现方式

## 4. 证据优先记忆

### 4.1 设计原则
- 记忆中的每条信息必须有**证据来源**
- 证据 > 推理 > 猜测——注入优先级
- 可追溯性：任何上下文条目可回溯到其数据来源

### 4.2 记忆类型
1. **契约记忆**：来自契约定义（最高优先级）
2. **验证记忆**：来自测试结果和变异分数
3. **决策记忆**：来自人类仲裁记录
4. **经验记忆**：来自历史成功/失败模式
5. **衰减记忆**：置信度随时间/使用频率衰减

### 4.3 与 writing_knowledge 的实现对照
- kiro-gateway 的写作知识库：confidence / effectiveness / 衰减 / 淘汰
- 知识条目的生命周期：创建→验证→使用→衰减→淘汰
- 与 bug_intention_map（Progee）的对照

### 4.4 审计日志
- 每次上下文组装生成完整日志
- 日志内容：任务ID、各层注入内容摘要、Token 消耗、裁剪决策
- 事后可复现：给定相同输入，产出相同上下文

## 5. 实证评估

### 5.1 评估维度
- **确定性**：相同输入是否产出相同上下文？
- **效率**：Token 消耗 vs 产出质量
- **可审计性**：能否追溯"AI 为什么做出这个决策"？
- **可迁移性**：从软件域到小说域的适配成本

### 5.2 对照实验设计（建议）
- 对照组 A：无结构上下文（全文拼接）
- 对照组 B：RAG 检索注入
- 实验组 C：分层结构化注入
- 度量：首次通过率、返工次数、Token 消耗、审计完整度

### 5.3 已有数据分析
- kiro-gateway Token 演进曲线与评委评分的关系
- Progee 纠正反馈（L17）对返工次数的影响
- 17 层 vs 12 层：层数不同但质量相当的原因

---

## 6. 讨论

### 6.1 上下文工程的通用性
- ODD 上下文工程是否适用于非 ODD 项目？
- 分层原则的普适性 vs 具体层数的领域依赖

### 6.2 与 Agent 框架的关系
- LangChain / AutoGen 等框架的上下文管理对比
- ODD 的差异：确定性 + 可审计 + 证据优先

### 6.3 威胁与局限
- 分层设计需要前期投入
- Token 预算的最优分配目前基于经验
- 证据优先记忆的衰减参数需要校准

---

## 7. 结论

- 上下文工程是 AI 辅助开发的被忽视的关键基础设施
- 分层模型将"提示词技巧"升级为"可审计的工程实践"
- Token 预算控制成本的同时改善质量（少即是多）
- 证据优先记忆保证可追溯性
- 跨领域验证证明分层原则的领域无关性
