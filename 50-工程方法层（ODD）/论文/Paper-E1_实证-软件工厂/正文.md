# 一人软件工厂：产出物驱动开发与 AI 协作的实证案例研究

> **作者**: Yi Fu (ODDFounder, fuyi.it@live.cn)
> **版本**: v1.0（中文初稿）
> **目标刊/会**: ICSE-SEIP 2027 或 ESEM 2026

---

## 摘要

AI 辅助开发的生产力提升已有大量定性描述，但缺乏系统的项目级定量案例研究。本文报告了一项探索性案例研究：一名开发者使用产出物驱动开发（ODD）方法论与多 Agent AI 车间体系，在 33 天内从零构建了包含 110+ 源文件、231 个自动化测试的完整桌面端软件工厂系统 Progee。研究围绕四个问题展开：（RQ1）实际开发效率如何？（RQ2）ODD 核心机制在实践中表现如何？（RQ3）缺陷发现的来源分布是什么？（RQ4）主要瓶颈和失败模式是什么？

结果表明：项目经历 16 次冲刺完成 225+ 任务，日均产出约 7 个任务，峰值达 47 任务/天。全面审查发现的 28 个缺陷中，100% 由系统机制发现，0% 依赖人工代码审查，初步支持了 ODD "系统验证替代人类审查"的核心主张。同时，研究揭示了契约粒度是影响 AI 生成质量的关键变量——过粗的契约导致理解偏差，细化到接口级别后显著改善。ODD 并不消除人类参与，而是将其集中在不可还原的决策点：契约设计和架构决策。

**关键词**: 案例研究、AI 辅助开发、生产力测量、缺陷检测、产出物驱动开发、变异测试

## 1. 引言

### 1.1 研究动机

大语言模型（LLM）正在根本性地改变软件的生产方式。GitHub Copilot、ChatGPT、Claude 等工具已从辅助角色转变为软件产出物的主要生产者。围绕这些工具，业界流传着大量生产力提升的声称——"10 倍效率"乃至"100 倍效率"——但这些声称普遍缺乏严谨的实证支持 [1, 2]。

现有的 AI 辅助开发评估主要集中在两个层面：

1. **代码补全层面**：如 HumanEval [3]、MBPP [4] 等基准测试，评估 AI 在单函数级别的代码生成能力。这些测试与真实工程中的复杂性差距显著。

2. **任务层面**：如 SWE-bench [5]，评估 AI 解决特定 Issue 的能力。虽然更接近真实场景，但仍是碎片化的任务，而非完整的项目生命周期。

**缺失的层面是项目级别**：一个完整的软件项目，从零开始，经历需求定义、架构设计、编码实现、测试验证、缺陷修复、文档编写的全过程。本文填补这一空白。

### 1.2 ODD 简介

产出物驱动开发（Output-Driven Development, ODD）[6] 是一种面向 AI 时代的软件开发范式，其核心主张是：

- **产出物（Artifact）**而非代码是软件开发的真正目标
- **契约（Contract）**定义产出物的可接受空间
- **变异测试**替代人类代码审查作为质量关口
- **封存（Seal）**保护已验证的产出物不被意外修改

ODD 重新定义了人类的角色：从代码作者和审查者，转变为契约定义者和最终仲裁者。

### 1.3 研究问题

本研究围绕以下四个问题展开：

- **RQ1**：单人使用 ODD + AI 车间体系的实际开发效率如何？
- **RQ2**：ODD 的核心机制（契约、变异测试、封存）在实践中的表现如何？
- **RQ3**：缺陷发现的来源分布是什么？系统审查能否替代人工代码审查？
- **RQ4**：ODD 的主要瓶颈和失败模式是什么？

### 1.4 贡献

本文的贡献包括：

1. **首个项目级 ODD 实证案例**：覆盖完整的软件开发生命周期，而非单一任务或代码补全。
2. **详细的量化数据**：16 次冲刺、225+ 任务、110+ 源文件、231 个测试、28 个缺陷的完整记录。
3. **ODD 核心主张的初步验证与边界识别**：既展示了有效性，也诚实揭示了局限。

## 2. 研究方法

### 2.1 案例研究设计

本研究采用**探索性单案例研究**方法 [7]。选择案例研究而非受控实验的原因在于：ODD 是一种新方法论，需要先在真实项目中探索其实践表现，再设计受控实验进行对比验证。

- **分析单元**：一个完整软件项目的开发过程
- **嵌入分析单元**：16 次冲刺（每次冲刺作为一个子分析单元）
- **数据来源**：Git 提交历史（含哈希）、结构化任务记录、DUnitX 测试 XML 输出、缺陷报告

### 2.2 项目背景

**系统**：Progee 是一个桌面端软件工厂，实现了 ODD 方法论的核心概念。

| 属性 | 值 |
|------|------|
| 技术栈 | Delphi (Object Pascal) + PostgreSQL |
| 源文件 | 110+ (.pas) |
| 测试文件 | 40+（单元 / 集成 / E2E / 压力 / 回归） |
| SQL 脚本 | 40+（DDL + 迁移 + 种子数据） |
| 自动化测试 | 231 个（DUnitX），全部通过 |
| 开发周期 | 2026-01-10 → 2026-02-11（约 33 天） |
| 开发者 | 1 人（本文作者） |

系统功能模块与 ODD 概念的对应关系：

| ODD 概念 | Progee 实现模块 |
|----------|----------------|
| 契约定义 + 清晰度评估 | CtrlContracts + CtrlContractStateMachine |
| 五步执行循环 | CtrlTaskStateMachine |
| 变异测试（信任基础） | CtrlMutationEngine + CtrlMutationOperators |
| 封存保护 | CtrlCodeVersions + audit_log |
| 对抗测试 | CtrlAdversarialTester + CtrlCrossReviewer |
| 诊断优先升级 | CtrlModelRacing |
| 分层上下文注入 | CtrlContextAssembler + CtrlContextBuilder |
| 多 Agent 并发 | CtrlWorkshopPool + CtrlWorkshopThread |

### 2.3 ODD 配置

**契约系统**：

结构化 JSON 契约，包含 `score_json`（结构化规格）、`depends_on`（依赖链）、`source_section_id`（溯源对话）。任务表包含 33 个字段，涵盖 `task_level`（L1-L4）、`input_output_mapping` 和 `failure_context`。清晰度评估通过 `clarity_detect` + `clarity_question` 提示词实现。

**多 Agent 架构**：

- **盘古**（单架构师）：高层设计讨论
- **军帐**（5 角色）：多角色辩论，产出契约
- **工人池**（开发/测试/审核）：代码生成、测试生成、交叉审核

**17 层上下文注入**（CtrlContextAssembler）：

从 L1（安全边界，强制注入）到 L17（纠正反馈，返工时注入），每层有独立的注入策略和信任等级。

**变异测试**：

三类变异算子——AOR（算术运算符替换）、ROR（关系运算符替换）、LOR（逻辑运算符替换）。设有降级策略：当变异测试无法执行时，按 skip → static_only → block 逐级降级。

**赛马模型**（诊断优先升级）：

生成失败时不直接升级模型，而是先诊断原因：上下文不足→补充上下文；契约模糊→要求人类澄清；校验不合理→调整标准；模型能力不够→才消耗赛马配额升级模型。

### 2.4 数据收集与分析方法

**定量数据**：任务数、源文件数、测试数、缺陷数按冲刺统计；时间数据从 Git 提交时间戳提取。

**定性数据**：对关键决策点进行事后回顾（postmortem），对瓶颈和失败模式进行开放编码分类。

**效度措施**：所有定量数据可通过 Git 提交历史（含哈希）独立验证；本文公开数据来源以供审查。

## 3. 结果

### 3.1 RQ1: 开发效率

#### 3.1.1 总量数据

| 指标 | 数据 | 说明 |
|------|------|------|
| 冲刺总数 | 16 次 | 每次 1~3 天 |
| 累计任务 | 225+ | 不含后续修复与对齐任务 |
| 日均任务 | ~7 个 | 含设计/代码/测试/文档 |
| 峰值吞吐 | 47 任务/天 | 冲刺 10: DeBorn 集成 |
| 源文件 | 110+ | .pas 文件 |
| 自动化测试 | 231 个 | DUnitX，全部通过 |

#### 3.1.2 开发阶段与节奏

项目经历了四个明显的阶段：

**阶段一：密集建设期**（第 1 周，冲刺 1-8）

日均 10-15 任务。冲刺 1（37 任务）建立了情报官与提示词系统的数据库和核心模块。冲刺 2-4（44+10+7 任务）完成了五区 UI、数据关联和 VFS 版本控制。冲刺 5-8（10+10+5+3 任务）打通了端到端流程：对话→契约→任务→执行。

**阶段二：功能深化期**（第 2-3 周，冲刺 9-16）

冲刺 9（25 任务）是关键转折点——实现了变异测试引擎和预警机制，同时处理了一批 P0 阻塞问题（安全/状态机/线程/心跳/上下文）。冲刺 10-16 完成了 DeBorn 集成（47 任务）、生产环境、UX 优化、测试保障、文档和高级功能。

**阶段三：稳定化期**（第 4-5 周）

论文支撑（复现实验 + 评估导出）和质量稳定化（231/231 测试全通过）。

**阶段四：规范对齐期**（第 6 周）

全面审查：Bug 修复（28 个）、规范文档一致性（16 项）、数据库对齐（10 项）、TFDQuery→UniDb 全面迁移（37 文件）。

#### 3.1.3 效率分析

**峰值吞吐的背景**：冲刺 10 的 47 任务/天之所以可能，是因为 DeBorn 集成阶段的任务高度结构化——代码扫描→契约自动生成→多应用支持——每个任务的契约清晰度高，AI 可以高效执行。

**效率瓶颈**：冲刺 9 暴露的 P0 阻塞问题（安全漏洞、状态机缺陷、线程安全、心跳超时、上下文错误）表明，基础设施层面的问题需要人类深度介入，ODD 在此场景下无法自动化解决。

### 3.2 RQ2: ODD 核心机制表现

#### 3.2.1 契约机制

**发现 1：契约粒度是关键变量。**

早期契约过粗（如"实现用户认证模块"），AI 产出的代码偏离预期。后来细化到单接口级别（如"实现 login 接口，输入 username/password，输出 JWT token，失败返回 INVALID_CREDENTIALS"），生成质量显著改善。

**教训**：一个契约应对应一个**可独立验收的功能单元**。粒度过粗导致 AI 理解空间过大；粒度过细则人类定义成本过高。接口级是当前实践中的最优粒度。

**发现 2：依赖声明不可或缺。**

初期未声明 `depends_on` 字段，导致任务执行顺序混乱——依赖其他模块的任务先于其依赖项执行。增加依赖声明后，系统自动进行拓扑排序，问题解决。

**发现 3：task_level 自动判定需要校准。**

L1（简单）→ L4（复杂）的任务等级自动判定在前 3 轮中经常误判，需要人工校准。校准后准确度提升，但新类型的任务仍可能误判。

#### 3.2.2 变异测试

变异测试引擎实现了三类算子（AOR、ROR、LOR），覆盖了基本的代码变异场景。

**降级策略的实际使用**：当待测代码包含 UI 组件、数据库连接或外部依赖时，变异测试无法直接执行。降级策略（skip / static_only / block）在此类场景中提供了灵活性，但也意味着验证覆盖率的下降。

**对抗测试**：CtrlAdversarialTester 实现了四类对抗形态，CtrlCrossReviewer 实现了三 AI 交叉审核。对抗测试在发现边界条件和异常处理缺陷方面表现有效。

#### 3.2.3 封存机制

封存机制（seal/unseal + 审计日志）在实践中起到了两个作用：

1. **防止回归**：已封存的代码不会被后续修改意外破坏。
2. **审计追溯**：每次 unseal 都记录了原因和授权人，形成完整的变更链条。

预警分级（返工超限→通知，变异分数低→告警，超时→自动回收）在冲刺 9 中成功捕获了多个心跳超时和任务超时事件，避免了资源浪费。

### 3.3 RQ3: 缺陷发现来源

2026-02-10 的全面审查覆盖了 22 个源文件，发现 28 个缺陷：

| Bug 类别 | 数量 | 发现方式 | 严重度 |
|----------|------|----------|--------|
| FillChar 内存损坏 | 12 处 / 6 文件 | 系统审查 | 高 |
| SQL 注入 | 3 处 | 系统审查 | 高 |
| 线程安全 | 4 处 | 系统审查 | 高 |
| Schema 前缀缺失 | 9 处 | 系统审查 | 中 |
| UUID 类型/参数化 | 5 处 | 系统审查 | 中 |
| 状态转换/逻辑错误 | 5 处 | 规范对齐 | 中 |

> 注：分类口径为规则命中条目。同一缺陷可能被多条规则命中，因此分类合计可能大于去重后的缺陷总数（28）。

**核心发现：28 个缺陷中，100% 由系统机制发现，0% 依赖人工代码审查。**

具体而言：

- **23 个缺陷**由自动化系统审查（静态分析 + 模式匹配 + 变异测试）发现
- **5 个缺陷**由规范对齐检查（跨文档一致性验证）发现

FillChar 内存损坏是一个值得关注的类别：这是 Delphi 特有的问题——AI 生成的代码使用了 `FillChar(Buf, SizeOf(Buf), 0)` 模式，但在某些数据结构上会导致内存损坏。这类语言方言级别的错误，人工代码审查未必能比系统审查更有效地发现。

此外，同期的规范文档审查发现并修复了 15 份文档中的 16 项跨文档不一致，包括：层级描述过时、角色可见层数错误、评分算法误导、状态机缺少终态、变异测试矛盾、命名规范不统一和错误码缺失。

### 3.4 RQ4: 瓶颈与失败模式

#### 失败模式 1：契约过粗导致 AI 理解偏差

当契约描述为"实现用户认证模块"时，AI 产出的代码在架构风格、接口设计、错误处理等方面均与人类期望有较大偏差。这不是 AI 能力的问题，而是**规格不充分**的问题。

#### 失败模式 2：规范漂移

在快速迭代过程中，15 份规范文档逐渐产生了 16 项跨文档不一致。例如，上下文层级的描述在一份文档中已更新为 17 层，但另一份文档中仍然是 12 层。这种漂移在 ODD 框架内通过定期的规范对齐审查发现，但需要人类判断如何修复。

#### 失败模式 3：AI 生成的语言方言错误

AI（特别是 LLM）对 Delphi/Object Pascal 的训练数据远少于主流语言。FillChar 内存损坏（12 处 / 6 文件）是典型表现——AI 机械地套用了在其他语言中安全但在 Delphi 中危险的模式。

#### 核心瓶颈：契约设计仍需大量人类认知投入

ODD 将人类从代码编写中解放出来，但契约设计本身需要深度的领域知识和系统思维。在本案例中，契约设计占据了人类约 60-70% 的认知投入，其余 20-30% 用于仲裁和决策，不到 10% 用于代码层面的干预。

## 4. 讨论

### 4.1 与现有 AI 开发效率研究的对比

**与 GitHub Copilot 研究的对比**：Peng et al. [1] 发现 Copilot 在 JavaScript 任务中提升了约 55% 的完成速度。然而，该研究关注的是代码补全层面的效率。Progee 案例中，效率提升不仅来自代码生成，更来自契约→任务自动分解、17 层上下文注入和系统化验证。两者不在同一比较维度。

**与 SWE-bench 的对比**：SWE-bench [5] 评估 AI 解决已有 Issue 的能力，关注的是任务级别的正确性。Progee 案例覆盖了完整项目生命周期——从无到有，需要连续的架构决策和不断演进的契约设计。

**本案例的独特性**：据我们所知，这是第一个覆盖完整项目生命周期（33 天、16 冲刺、从零到可运行系统）的 AI 辅助开发案例研究。

### 4.2 ODD 主张的初步验证

根据本案例数据，我们对 ODD 的核心主张进行初步评估：

**✅ 系统审查可替代人类代码审查**：28 个缺陷全部由系统机制发现的事实，为这一主张提供了初步支持。但需注意，本案例中系统审查的覆盖范围和深度可能无法推广到所有项目类型。

**✅ 契约驱动提升 AI 生成质量**：契约粒度从模块级细化到接口级后，AI 首次生成通过率显著提升。这表明契约精度与产出物质量之间存在正相关关系。

**⚠️ 人类可部分退出执行循环**：在代码编写和测试层面，人类确实退出了执行循环。但在架构决策、契约设计和异常处理方面，人类的深度参与不可或缺。ODD 的准确表述应该是"人类退出**例行**执行循环"，而非退出所有执行。

**❌ 变异测试不能完全替代代码审查**：在需要降级的场景（UI 组件、外部依赖），变异测试无法执行，需要人类介入。变异测试是强有力的补充，但不是完全替代。

### 4.3 效度威胁

**内部效度**：本案例的最大威胁是开发者与方法论作者为同一人。这可能导致正向偏见——对方法论的缺陷不够敏感。缓解措施：我们在 RQ4 中诚实报告了失败模式和瓶颈。

**外部效度**：单案例、单人、单技术栈（Delphi/PostgreSQL）的结果不能直接推广。特别是 Delphi 的 LLM 训练数据不足，可能使某些发现（如 FillChar 错误）是技术栈特有的而非方法论通用的。

**构建效度**：部分度量（如"日均任务数"）依赖事后统计，可能存在分类标准的不一致。缓解措施：所有数据可通过 Git 提交历史独立验证。

## 5. 相关工作

### 5.1 AI 辅助开发的实证研究

Peng et al. [1] 通过随机对照实验评估了 GitHub Copilot 对开发效率的影响，发现代码补全任务的完成速度提升约 55%。Vaithilingam et al. [2] 研究了 Copilot 对程序员工作流程的影响，发现虽然效率提升存在，但程序员花费大量时间验证和调试 AI 生成的代码。这些研究均在任务层面进行，未覆盖完整项目生命周期。

### 5.2 变异测试的工业应用

Petrovic & Ivankovic [8] 报告了 Google 内部大规模变异测试的实践经验，证明变异测试在工业环境中是可行的，但需要优化策略（如增量变异、采样）来控制计算成本。Progee 案例中实现了类似的降级策略（skip / static_only / block）来应对变异测试无法执行的场景。

### 5.3 案例研究方法

Yin [7] 定义了案例研究的四种设计类型，本研究属于嵌入式单案例设计（Type 2）。Runeson & Höst [9] 为软件工程中的案例研究提供了方法论指南，本研究遵循其数据收集和分析框架。

### 5.4 代码审查研究

Bacchelli & Bird [10] 对 Microsoft 的代码审查实践进行了大规模研究，发现代码审查的主要价值不在于发现缺陷，而在于知识传播和代码理解。McIntosh et al. [11] 发现代码审查的覆盖率与软件质量之间存在正相关。本文的发现——系统审查可有效发现缺陷——并不否定代码审查在知识传播方面的价值，但指出在 AI 辅助开发场景下，缺陷发现功能可由系统机制承担。

## 6. 结论

本文报告了首个项目级的产出物驱动开发（ODD）实证案例研究，主要发现如下：

1. **开发效率**：单人 + ODD + AI 在 33 天内完成了 225+ 任务、110+ 源文件、231 个测试的软件工厂系统构建。峰值吞吐达 47 任务/天（高结构化任务）。

2. **系统审查替代人类审查**：28 个缺陷中 100% 由系统机制发现，初步支持了 ODD "系统验证替代人类审查"的核心主张。

3. **契约粒度是关键变量**：过粗的契约（模块级）导致 AI 理解偏差；细化到接口级后，生成质量显著改善。契约精度与产出物质量之间存在正相关关系。

4. **ODD 的人类角色**：ODD 不消除人类参与，而是将其集中在不可还原的决策点——契约设计（~60-70% 认知投入）、仲裁（~20-30%）和代码干预（<10%）。

5. **局限性**：单人单项目的外部效度有限；变异测试存在降级场景；规范漂移需要定期人类审查。

**未来工作**：多项目、多人团队、多技术栈的对比研究；契约粒度与 AI 生成质量的受控实验；ODD 与传统方法论（Agile/TDD）的对照实验。
