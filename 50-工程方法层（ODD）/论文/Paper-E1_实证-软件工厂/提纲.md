# Paper E1 提纲：一人软件工厂

---

## 摘要（草案）

AI 辅助开发的生产力提升已有大量定性描述，但缺乏系统的定量案例研究。本文报告了一项受控案例研究：一名开发者使用产出物驱动开发（ODD）方法论与 AI 车间体系，在 **33 天**内从零构建了包含 **110+ 源文件、231 个自动化测试**的完整软件工厂系统（Progee）。我们详细记录了 16 次冲刺、225+ 任务的执行过程，分析了缺陷发现方式（28 个 Bug 100% 由系统审查发现）、契约粒度对生成质量的影响、以及 ODD 四阶段循环在真实工程中的效果与局限。

**关键词**: 案例研究、AI 辅助开发、生产力度量、缺陷检测、产出物驱动开发

---

## 1. 引言

### 1.1 研究动机
- AI 辅助开发的生产力声称（10x, 100x）缺乏严谨实证
- 需要：受控条件下的完整项目级案例研究
- 不同于代码补全基准测试（HumanEval, MBPP），关注**项目级**产出

### 1.2 研究问题
- RQ1: 单人 + ODD + AI 的实际开发效率如何？（量化）
- RQ2: ODD 的核心机制（契约、变异测试、封存）在实践中表现如何？
- RQ3: 缺陷发现的来源分布是什么？系统审查能否替代人工审查？
- RQ4: ODD 的主要瓶颈和失败模式是什么？

### 1.3 贡献
- 首个完整项目级的 ODD 案例研究
- 详细的量化数据和定性分析
- 对 ODD 核心主张的初步验证与边界识别

---

## 2. 研究方法

### 2.1 案例研究设计
- 类型：探索性单案例研究（Yin 2018）
- 分析单元：一个完整软件项目的开发过程
- 数据来源：Git 提交历史、任务记录、测试结果、缺陷报告

### 2.2 项目背景
- 系统：Progee——桌面端软件工厂
- 技术栈：Delphi (Object Pascal) + PostgreSQL
- 开发者：1 人（论文作者）
- 开发周期：2026-01-10 → 2026-02-11

### 2.3 ODD 配置
- 契约系统：结构化 JSON 契约 + 清晰度评估
- 多 Agent 架构：盘古(架构师) + 军帐(5角色) + 工人池(开发/测试/审核)
- 17 层上下文注入
- 变异测试：AOR + ROR + LOR 算子
- 赛马模型：诊断优先升级

### 2.4 数据收集与分析
- 定量：任务数、源文件数、测试数、缺陷数、时间分布
- 定性：关键决策点的回顾、瓶颈分析、失败模式分类

---

## 3. 结果

### 3.1 RQ1: 开发效率

| 指标 | 数据 |
|------|------|
| 总任务 | 225+ |
| 总源文件 | 110+ (.pas) |
| 总测试 | 231 (DUnitX) |
| SQL 脚本 | 40+ |
| 总天数 | ~33 天 |
| 日均任务 | ~7 个 |
| 峰值吞吐 | 47 任务/天（冲刺10: DeBorn 集成） |

- 16 次冲刺的时间线与里程碑
- 冲刺规模分布：最小 3 任务 → 最大 47 任务
- 效率瓶颈：P0 阻塞修复（安全/状态机/线程/心跳/上下文）

### 3.2 RQ2: ODD 核心机制表现

#### 契约机制
- 契约粒度演进：模块级（失败）→ 接口级（成功）
- 依赖声明的必要性：初期遗漏导致执行混乱
- task_level 自动判定的校准需 3+ 轮

#### 变异测试
- 变异算子覆盖率
- 降级策略的使用频率：skip / static_only / block
- 对抗测试（四类对抗形态）的实际效果

#### 封存机制
- seal/unseal 操作频率
- 预警分级的触发率
- 超时回收机制的必要性

### 3.3 RQ3: 缺陷发现来源

| Bug 类别 | 数量 | 发现方式 |
|----------|------|----------|
| FillChar 内存损坏 | 12 处 / 6 文件 | 系统审查 |
| SQL 注入 | 3 处 | 系统审查 |
| 线程安全 | 4 处 | 系统审查 |
| Schema 前缀缺失 | 9 处 | 系统审查 |
| UUID 类型/参数化 | 5 处 | 系统审查 |
| 状态转换/逻辑错误 | 5 处 | 规范对齐 |

**核心发现：28 个 Bug，100% 由系统机制发现，0% 依赖人工 Code Review**

### 3.4 RQ4: 瓶颈与失败模式
- 失败模式 1：契约过粗导致 AI 理解偏差
- 失败模式 2：规范漂移（15 份文档 16 项不一致）
- 失败模式 3：AI 生成的代码存在 Delphi 方言错误（FillChar 误用）
- 瓶颈：契约设计仍需大量人类认知投入

## 4. 讨论

### 4.1 与现有 AI 开发效率研究的对比
- GitHub Copilot 研究：代码补全级别 vs 项目级别
- Cursor / Aider / SWE-bench：任务级别 vs 全周期
- Progee 的差异：完整项目生命周期，含设计/测试/文档

### 4.2 ODD 主张的初步验证
- ✅ "系统审查替代人类审查"：28 Bug 数据支持
- ✅ "契约驱动提升 AI 生成质量"：粒度细化后首次通过率提升
- ⚠️ "人类退出执行循环"：架构决策和契约设计仍需深度人类参与
- ❌ "变异测试完全替代代码审查"：降级场景需人类介入

### 4.3 效度威胁
- **内部效度**：单人项目，开发者 = 方法论作者（偏见风险）
- **外部效度**：单案例、单技术栈、单人
- **构建效度**：部分度量依赖事后统计
- 缓解措施：Git 历史完整可验证、数据公开

## 6. 结论

- 首个项目级 ODD 实证案例：33 天 / 1 人 / 110+ 文件 / 231 测试
- 系统审查替代人类审查的初步证据
- 契约粒度是关键变量：过粗必败，接口级最优
- ODD 不消除人类参与，而是将其集中在不可还原决策点
- 未来工作：多项目、多人团队、多技术栈的对比研究
