## 评审 A：整体评价（IEEE Fellow）

### 总体评价

**评分：6/10** - 有亮点，但需要重大修改

### 主要问题

1. **创新性不足**
   - "Pre-Generation Guard" 概念新意有限，本质是"在 LLM 调用前加一个检查层"
   - 三层约束体系直接借用了 ODD 理论，但 ODD 本身是预印本，未经 peer review
   - **建议**：需要更强的理论贡献，而非系统实现

2. **实验规模不足**
   - 500 个样本太少，且多为合成数据
   - 缺少与生产环境的对比
   - **建议**：至少需要 5000+ 样本，且来源多元化

3. **写作问题**
   - 摘要太长（500+ 字），顶刊要求 150-250 字
   - 参考文献太少（仅 9 篇），顶刊要求 30+ 篇
   - 缺少对比表格（Related Work 部分）

### 裁决

**大修（Major Revision）**

## 评审 B：安全性评价（ACM 安全专家）

### 总体评价

**评分：5.5/10** - 方向正确，但深度不够

### 主要问题

1. **检测能力存疑**
   - 规则匹配的方式容易被绕过
   - 对 "间接 Prompt Injection" 的检测率仅 90%，不够高
   - **质疑**：实际攻击中，攻击者会迭代绕过，90% 的防御率远远不够

2. **绕过风险**
   - ChallengeEngine 依赖 LLM 本身检测风险，形成循环
   - 如果 LLM 本身被攻击，ChallengeEngine 也会失效
   - **建议**：引入独立于 LLM 的检测机制

3. **缺乏对抗性测试**
   - 评测中没有看到对自适应攻击的测试
   - 攻击者会不断迭代尝试绕过，需要持续对抗测试

4. **延迟数据不真实**
   - 150ms 延迟是在本地测试，真实云环境可能更差
   - 缺少 P95、P99 的数据

### 裁决

**大修（Major Revision）**

## 评审 C：软件工程方法论评价（OOPSLA 评审）

### 总体评价

**评分：6.5/10** - 有系统贡献，但理论深度不足

### 主要问题

1. **与 ODD 的关系模糊**
   - 论文说"基于 ODD 理论"，但 ODD 是预印本
   - 三层约束体系是"借用"还是"发展"？定义不清
   - **建议**：需要明确 ODD 与本文的学术关系

2. **方法论贡献不清晰**
   - 本文是"系统论文"还是"方法论文"？
   - 如果是系统论文，需要更强的工程贡献
   - 如果是方法论文，需要更强的理论贡献

3. **实验设计问题**
   - 缺少 ablation study（消融实验）
   - 缺少与基线的详细对比
   - 缺少不同 LLM、不同场景的对比

4. **评估指标不全面**
   - 只有召回率、误报率
   - 缺少用户研究（可用性）
   - 缺少成本分析（企业关心的 ROI）

### 裁决

**大修（Major Revision）**

---

## 评审 D：产业实用性评价（前 NSA 安全专家）

### 总体评价

**评分：7/10** - 最接近实际可用，但有改进空间

### 主要问题

1. **场景不够深**
   - 金融场景和 OA 场景都是通用场景
   - **建议**：选择一个垂直领域深挖（如医疗、金融风控）

2. **缺少运维视角**
   - 没有讨论规则如何更新
   - 没有讨论误报后如何处理
   - 没有讨论如何审计和合规

3. **部署问题**
   - Demo 是单机版，企业需要分布式
   - 没有讨论高可用性
   - 没有讨论容灾

4. **竞品分析不足**
   - 只提了 Llama-Guard 和 NVIDIA Guardrails
   - 缺少 Microsoft Azure AI Content Safety
   - 缺少 Google Cloud AI Platform Pipelines

### 裁决

**小修（Minor Revision）**

4. **添加对抗性测试**：自适应攻击测试
