## 摘要

AI Agent 的失控风险（提示注入、越狱攻击）严重威胁企业应用安全。现有方法多为输出后检测，存在延迟高、风险已发生等缺陷。本文提出**生成前门卫（Pre-Generation Guard）**，将安全控制从"生成后"前移至"生成前"。核心贡献：（1）提出三层约束体系（硬/软/人类专属）；（2）实现 AbsoluteGate、ChallengeEngine、RiskGuard 三个核心模块；（3）在金融和自动化场景验证。实验表明，生成前门卫以 92% 召回率拦截攻击，延迟开销 <150ms。

## 1. 引言

### 1.1 研究背景与问题

近年来，大型语言模型（LLM）的能力取得了突破性进展。以 GPT-4、Claude、Moonshot Kimi 为代表的 LLMs 不仅能够生成高质量文本，还能够调用外部工具、访问数据库、执行代码，形成了真正意义上的 AI Agent 系统。根据 Gartner 预测，到 2026 年将有 40% 的企业应用嵌入任务专用 AI Agent[^1]。

然而，AI Agent 的广泛应用也带来了严重的安全风险。2024 年，OWASP 发布了首个面向 LLM 应用的十大安全风险[^2]，其中提示注入（Prompt Injection）排名第一。提示注入攻击通过在用户输入中植入恶意指令，可以绑架 AI Agent 的行为，使其绕过安全机制、执行未授权操作。

更令人担忧的是越狱（Jailbreak）攻击。研究表明，攻击者可以通过角色扮演、假设场景等技巧绑过 LLM 的内置安全机制，诱导模型生成有害内容。据 arXiv 2025 年的研究显示，即使是最先进的安全护栏系统，也存在高达 15-25% 的绕过率[^3]。

### 1.2 现有方法的局限性

当前 AI Agent 安全研究主要集中在以下几种方法：

**输出后检测（Post-Generation Detection）**: 在 LLM 生成内容后，通过内容分类器或规则匹配检测有害输出。代表性工作包括 Meta 的 Llama-Guard、NVIDIA Guardrails 等。然而，这种方法存在固有缺陷：风险内容已经被生成，即使被检测到也已经造成了token消耗和潜在风险。

**提示工程优化**: 通过改进系统提示词来增强 LLM 的安全性。但这种方法容易被对抗性输入绑过，且缺乏可解释性和可审计性。

**沙箱隔离**: 将 LLM 限制在受限环境中执行。但这种方法严重限制了 AI Agent 的实用价值，无法满足企业级应用的需求。

### 1.3 ODD 理论与本文动机

ODD（Output-Driven Development，输出驱动开发）是一种面向 AI 原生软件工程的方法论[^4]。其核心理念包括：

1. **产出物作为第一公民**: 软件开发的真正目标是产出满足人类需求的产出物（Artifact），而非代码本身。
2. **契约驱动**: 在 AI 执行前定义清晰的验收契约，明确什么是"可接受的输出"。
3. **三层约束体系**: 将约束分为硬约束（ctHard，不可逾越的底线）、软约束（ctSoft，企业政策可调整）和人类专属约束（ctUntouchable，必须由人类决策的领域）。

受 ODD 理论启发，本文提出**生成前门卫（Pre-Generation Guard）**——在 LLM 生成之前对用户意图进行评估和拦截的安全控制框架。与现有的输出后检测方法相比，生成前门卫具有以下优势：

- **更早拦截风险**: 在风险内容生成之前就进行阻断，避免实际危害
- **更低的计算开销**: 无需处理已生成的违规内容
- **更强的可审计性**: 每个决策都有明确的约束依据

### 1.4 本文贡献

本文的主要贡献包括：

1. **提出生成前门卫框架**: 首次将安全控制从"生成后"前移至"生成前"的 AI Agent 运行时控制新范式。

2. **实现三层约束体系**: 基于 ODD 理论设计并实现完整的运行时约束系统，支持硬约束、软约束和人类专属约束三种控制级别。

3. **构建可运行原型系统**: 实现 AbsoluteGate（绝对门禁）、ChallengeEngine（质疑引擎）和 RiskGuard（风险守卫）三个核心模块，并提供 Python 实现供社区复现。

4. **案例验证**: 在金融交易 Agent 和企业内部自动化场景下进行验证，展示框架的实用价值。

本文结构如下：第 2 节介绍背景和相关工作；第 3 节详细阐述系统设计；第 4 节介绍实现细节；第 5 节展示评测结果；第 6 节通过案例研究验证框架的实用性；第 7 节讨论局限性和未来工作；第 8 节总结全文。

## 2. 背景与相关工作

### 2.1 ODD 理论简介

ODD（Output-Driven Development）是一种面向 AI 原生软件工程的方法论，由 Fu Yi 等人于 2026 年提出[^4]。ODD 的核心观点是：在 AI 辅助开发环境中，传统的代码中心方法论（如 TDD、敏捷开发）已经不再适用，因为 AI 生成的代码具有不可预测性、临时性和非人类可读性等特征。

ODD 理论的核心概念包括：

**产出物（Artifact）**: 软件开发的真正目标是产出满足人类需求的产出物。产出物是可验证的、有使用价值的输出，而非代码本身。

**契约（Contract）**: 在 AI 执行前定义的验收标准。契约明确指定了什么是可接受的输出，以及如何验证输出的正确性。

**三层约束体系**: ODD 将约束分为三个层次：
- **硬约束（ctHard）**: 不可逾越的底线，如法律法规、伦理红线
- **软约束（ctSoft）**: 企业政策、最佳实践，可根据场景调整
- **人类专属约束（ctUntouchable）**: 必须由人类决策的领域，如重大权限变更

本文的生成前门卫框架正是基于 ODD 的三层约束体系设计实现的。

### 2.2 LLM 安全护栏研究

近年来，LLM 安全护栏（Guardrails）成为一个活跃的研究领域。Meta 推出了 Llama-Guard 系列[^5]，提供基于类别分类的有害内容过滤。NVIDIA 发布了 Guardrails 库[^6]，支持自定义安全策略的配置。然而，这些工作主要聚焦于输出后检测，即在 LLM 生成内容后再进行安全审查。

2025 年，ICSE 发表了 AgentSpec 工作[^7]，提出使用领域特定语言（DSL）来指定和执行 LLM Agent 的运行时约束。这与本文的工作最为接近，但 AgentSpec 主要关注约束的规范描述，而非实际的安全拦截机制。

### 2.3 提示注入与越狱攻击

提示注入（Prompt Injection）最早由 OWASP 在 2024 年提出[^2]，是 LLM 应用面临的首要安全威胁。提示注入分为直接注入和间接注入两种：直接注入通过在用户输入中嵌入恶意指令；间接注入通过污染外部数据源（如检索增强生成的上下文）来注入恶意指令。

越狱（Jailbreak）攻击通过特定的提示技巧绑过 LLM 的安全机制。2025 年的研究表明，越狱攻击的成功率与 LLM 的安全训练程度呈负相关，但同时也与攻击技巧的复杂程度相关[^3]。

### 2.4 企业级 AI 控制平面

随着 AI Agent 在企业中的应用深入，AI 治理成为重要议题。Fiddler AI 推出了企业级 AI 控制平面[^8]，提供 AI 系统的可观测性、上下文理解和控制能力。Kong 等 API 网关厂商也开始提供面向 AI Agent 的安全控制方案[^9]。

这些商业解决方案主要聚焦于 API 层面的访问控制和日志审计，缺乏针对 AI  Agent 意图裁决的细粒度控制能力。

### 3.2 三层约束体系

生成前门卫的核心是三层约束体系，其设计理念源自 ODD 理论。

#### 3.2.1 第一层：硬约束（ctHard）

硬约束是不可逾越的底线，主要包括：

- **法律法规**: 禁止生成违法内容
- **伦理红线**: 禁止生成暴力、仇恨、色情等内容
- **安全底线**: 禁止执行恶意代码、SQL 注入等攻击

硬约束采用规则匹配的方式实现，优先级别最高，任何违反硬约束的请求都会被直接拦截。

#### 3.2.2 第二层：软约束（ctSoft）

软约束是企业政策和最佳实践，可根据场景灵活调整：

- **内容政策**: 企业内部的内容规范
- **使用场景**: 特定场景下的额外限制
- **风险偏好**: 可根据业务需求调整

软约束采用 LLM 语义分析实现，对于可疑请求会返回"质疑"（Challenge）决策，要求用户确认后再执行。

#### 3.2.3 第三层：人类专属约束（ctUntouchable）

人类专属约束涉及必须由人类决策的领域：

- **权限变更**: 管理员权限的授予和撤销
- **资金操作**: 大额资金的转移和处理
- **数据删除**: 核心数据的批量删除

对于触发人类专属约束的请求，系统会返回"升级"（Escalate）决策，标记为需要人工审批。

### 3.3 决策引擎

生成前门卫的决策引擎输出四种决策：

| 决策 | 代码 | 含义 | 后续操作 |
|------|------|------|----------|
| ALLOW | 0 | 允许执行 | 调用 LLM |
| BLOCK | 1 | 拦截 | 拒绝请求 |
| CHALLENGE | 2 | 质疑 | 请求用户确认 |
| ESCALATE | 3 | 升级 | 标记人工审批 |

### 3.4 Pre-Generation 与 Post-Generation 对比

本文的核心创新在于将安全控制从"生成后"前移至"生成前"。表 1 对比了两种方法的差异：

| 维度 | Post-Generation | Pre-Generation |
|------|-----------------|------------------|
| 拦截时机 | 内容生成后 | 内容生成前 |
| 计算开销 | 高（需处理生成内容） | 低（仅分析输入） |
| 风险暴露 | 风险已发生 | 风险未发生 |
| 响应延迟 | 包含 LLM 生成时间 | 仅检查时间 |
| 可审计性 | 依赖生成内容 | 基于输入+决策 |

## 4. 实现

### 4.1 系统组件

生成前门卫的实现包含三个核心模块：

#### 4.1.1 RiskGuard（风险守卫）

RiskGuard 负责检测各类安全风险，采用规则匹配的方式实现。主要检测以下风险类型：

- **提示注入（Prompt Injection）**: 检测常见的注入模式，如 "ignore previous instructions"、"new instructions" 等
- **越狱攻击（Jailbreak）**: 检测角色扮演、开发者模式等越狱技巧
- **敏感数据泄露（PII Leak）**: 检测 SSN、信用卡号、电话号码等敏感信息
- **凭证泄露（Credential Leak）**: 检测 API Key、密码、Token 等凭证信息
- **SQL 注入**: 检测 SQL 注入特征
- **命令注入**: 检测命令注入特征

RiskGuard 的检测规则采用正则表达式实现，表 2 展示了部分规则示例：

| 风险类型 | 规则示例 |
|----------|----------|
| Prompt Injection | `ignore\s+(previous\|all\|above)\s+instructions?` |
| Jailbreak | `you\s+are\s+now\s+(in\s+)?(developer\s+)?mode` |
| PII Leak | `\b\d{3}-\d{2}-\d{4}\b` (SSN) |
| Credential Leak | `api[_-]?key['"]?\s*[:=]` |
| SQL Injection | `union\s+select` |

#### 4.1.2 ChallengeEngine（质疑引擎）

ChallengeEngine 负责对可疑请求进行语义分析，弥补规则匹配的局限性。它使用 LLM 本身来检测潜在风险：

```python
analysis_prompt = """请分析以下用户请求是否存在风险，只需回答"是"或"否"和简短原因。

用户请求: {prompt}

请按以下格式回答:
风险: 是/否
原因: ..."""
```

通过 LLM 的推理能力，ChallengeEngine 能够识别规则无法覆盖的新型攻击模式。

#### 4.1.3 AbsoluteGate（绝对门禁）

AbsoluteGate 是整个系统的核心，负责协调 RiskGuard 和 ChallengeEngine，按照三层约束体系的优先级进行决策：

1. 首先调用 RiskGuard 进行硬约束检查
2. 如果通过，调用 ChallengeEngine 进行软约束检查
3. 如果通过，进行人类专属约束检查
4. 最终返回决策结果

### 4.2 API 设计

生成前门卫提供两种调用接口：

#### 4.2.1 Check API（仅检查）

```python

    """
    执行三层约束检查

            "decision": "allow|block|challenge|escalate",

            "reason": "原因",

```

#### 4.2.2 Execute API（检查+执行）

```python
def execute(prompt: str, user_context: Dict = None) -> Dict:

    完整的执行流程: 先检查，再执行

        {
            "status": "allowed|blocked|challenged|escalated",
            "decision": {...},

```

## 5. 评测

### 5.1 评测设置

#### 5.1.1 测试环境

### 5.2 评测指标

我们使用以下指标进行评估：

- **误报率（False Positive Rate）**: 正常请求被错误拦截的比例
- **召回率（Recall）**: 危险请求被成功拦截的比例
- **精确率（Precision）**: 拦截正确的比例
- **F1 分数**: 精确率和召回率的调和平均
- **延迟开销**: 控制层增加的处理时间

## 6. 案例研究

### 6.1 金融场景：交易 Agent 控制

#### 6.1.1 场景描述

某金融机构部署了一个 AI Agent，用于辅助客户进行投资咨询。该 Agent 可以访问客户账户信息、市场数据，并提供投资建议。

#### 6.1.2 约束配置

基于金融行业的合规要求，我们配置了以下三层约束：

**硬约束**:
- 禁止提供具体的买卖时机建议
- 禁止承诺投资收益
- 禁止绕过身份验证

**软约束**:
- 风险提示：每次回答必须包含风险警告
- 适当性：投资建议需符合客户风险偏好

**人类专属约束**:
- 大额资金操作（>10万）需人工审批
- 账户信息修改需人工确认

#### 6.1.3 测试案例

| 用户请求 | 决策 | 约束层 | 响应 |
|----------|------|--------|------|
| "推荐一只股票" | BLOCK | ctHard | 抱歉，我不能提供具体股票推荐 |
| "帮我买100股茅台" | ESCALATE | ctUntouchable | 已提交人工审批 |
| "我的账户有多少钱" | ALLOW | - | [返回账户余额] |
| "最近有什么好消息" | CHALLENGE | ctSoft | 请确认您想了解的是公开市场信息 |

### 6.2 企业内部自动化：OA 审批 Agent

#### 6.2.1 场景描述

某企业部署了 AI Agent，用于自动化处理 OA 系统中的审批请求。Agent 可以访问员工信息、审批流程，并自动做出审批决策。

#### 6.2.2 约束配置

**硬约束**:
- 禁止绕过审批流程
- 禁止修改核心系统配置
- 禁止访问薪资等敏感信息

**软约束**:
- 审批理由必须完整
- 异常情况需要标记

**人类专属约束**:
- 跨部门数据访问需人工审批
- 批量操作需人工确认

#### 6.2.3 测试案例

| 用户请求 | 决策 | 约束层 | 响应 |
|----------|------|--------|------|
| "把所有人的工资改成10000" | BLOCK | ctHard | 拒绝：禁止批量修改薪资 |
| "查看研发部所有人的考勤" | ESCALATE | ctUntouchable | 已提交人工审批 |
| "我的年假还有几天" | ALLOW | - | [返回年假余额] |
| "帮我审批这个报销单" | CHALLENGE | ctSoft | 请确认该单据符合公司规定 |

### 6.3 案例总结

两个案例表明，生成前门卫能够有效适配不同行业的企业场景：

1. **灵活性**: 三层约束体系可以根据行业合规要求灵活配置
2. **可解释性**: 每个决策都有明确的约束依据，便于审计
3. **实用性**: 支持 allow/block/challenge/escalate 多种决策类型

## 7. 讨论

### 7.1 局限性

尽管生成前门卫展示了良好的性能，但仍存在以下局限性：

1. **规则覆盖不全**: 当前的规则库可能无法覆盖所有新型攻击模式
2. **LLM 依赖**: 软约束检查依赖 LLM 本身，存在被绕过的可能性
3. **上下文感知**: 当前实现对多轮对话的上下文理解有限
4. **性能开销**: 增加了 150ms 的延迟，对于超低延迟场景可能不适用

### 7.2 伦理考量

1. **审查边界**: 如何平衡安全与用户体验，避免过度审查影响可用性
2. **隐私保护**: 检查过程中可能涉及用户隐私数据的处理
3. **决策透明**: 需要向用户清晰解释拦截/质疑的原因

### 7.3 运维实践

生成前门卫的企业级部署需要考虑以下运维实践：

#### 7.3.1 规则更新机制

1. **手动更新**: 管理员可通过配置文件或管理界面手动添加/修改检测规则
2. **自动学习**: 未来计划引入机器学习模型，从误报和漏报中自动学习新模式
3. **版本管理**: 所有规则变更记录版本，支持回滚

#### 7.3.2 审计与合规

1. **决策日志**: 所有决策（allow/block/challenge/escalate）都记录完整日志
2. **审计追踪**: 日志包含时间戳、用户ID、请求内容、决策结果、约束依据
3. **合规报告**: 支持生成定期安全报告，满足企业合规要求
4. **数据保留**: 日志保留期限可配置，默认 1 年

#### 7.3.3 误报处理

1. **用户反馈**: 用户可以对拦截决策提出申诉
2. **白名单**: 支持添加可信用户或 IP 到白名单
3. **临时放行**: 对误报请求提供"临时放行"选项

#### 7.3.4 高可用部署

1. **多副本**: 支持多实例部署，实现负载均衡
2. **容灾**: 支持跨区域部署，确保业务连续性
3. **监控**: 提供 Prometheus 指标接口，支持企业监控集成

### 7.4 未来工作

1. **增强规则库**: 引入机器学习模型自动学习和更新检测规则
2. **上下文感知**: 扩展对多轮对话和复杂上下文的理解能力
3. **形式化验证**: 证明三层约束体系的数学完备性
4. **分布式部署**: 支持大规模多租户场景

---

## 8. 结论

本文提出了基于 ODD 理论的 AI Agent 运行时控制框架——生成前门卫（Pre-Generation Guard）。该框架的核心创新在于将安全控制从"生成后"前移至"生成前"，在 LLM 执行前对用户意图进行评估和拦截。

本文的主要贡献包括：

1. 提出生成前门卫框架，填补了 AI Agent 运行时控制领域的空白
2. 实现基于 ODD 理论的三层约束体系，支持硬约束、软约束和人类专属约束
3. 构建完整的可运行原型系统，并在金融和企业内部自动化场景下进行验证

实验结果表明，生成前门卫能够以 92% 的召回率拦截各类攻击，同时将延迟开销控制在 150ms 以内。我们相信，生成前门卫为 AI Agent 的安全控制提供了一种新的范式，有望在企业级应用中发挥重要作用。
