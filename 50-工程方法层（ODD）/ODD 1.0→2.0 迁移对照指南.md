## 三个最重要的变化

### 1. 契约从概念变成了可执行语言

v1.0 论文说"应该写契约"，但没说明契约长什么样。

v2.0 的 `C09.工程_契约_产出物_证据.md` 给出了完整答案：
- YAML 格式的契约模板（从 L1 三行到 L3 完整规格）
- 四种验收条件写法（枚举/属性/规则/对抗）
- `hardness: hard | soft` 区分不可协商约束和可协商约束
- `floor`（地板）和 `red_line`（红线）声明

### 2. CAP 对抗协议——这是 v2.0 最核心的发明

v1.0 论文没有触及"契约本身写错了怎么办"的问题。

v2.0 的 CAP 协议（Challenger→Attacker→Arbiter）通过三角色对抗：
- **Challenger**：从清晰度角度攻击，"这个词太模糊"
- **Attacker**：从恶意实现角度攻击，"我可以用硬编码通过所有验收条件"
- **Arbiter**：裁决分歧，决定契约是否满足最低防御标准

以及五项深度优化：代码级试探 / 业务域黑天鹅注入 / 视觉恶意对抗 / 同谋检测 / 注意力预算升级版。

### 3. 分级体系（L0-L3）降低了接入门槛

v1.0 论文描述的是一个"完整 ODD 系统"，读者容易觉得太重。

v2.0 的 L0-L3 分级意味着：
- **L0**：已有项目，git tag 当封存，零成本接入
- **L1**：一个人，手动契约+手动验证，够用就行
- **L1.5**：一个人+AI，AI 生成契约和测试，本地自动验证
- **L2**：小团队，YAML 契约+CI 自动验证+系统化门禁
- **L3**：大团队/关键系统，完整状态机+对抗生成+证据链审计

同一个项目不同模块可以用不同等级。这让 ODD 从"要么全用要么不用"变成了"按需升级"。

## 从 v1.0 到 v2.0 的推荐阅读路径

如果你已经读过 v1.0 论文：

1. 先读 `ODD.01-Overview` 确认新框架
2. 再读 `C09.契约`（前半）理解契约的工程格式
3. 然后读 `C09.对抗`（后半）理解 CAP 协议——这是 v2.0 最大增量
4. 读 `C13.验证_门禁_状态机` 理解三值门禁 PASS/FAIL/FREEZE
5. 读 `C09.证据与封存` 理解审计闭环
6. 最后按你的团队规模选读 B05（个人）或 B07（团队）

## 仍保留在 v1.0 中、v2.0 未覆盖的部分

- 学术引用格式和文献综述（v1.0 论文独有）
- 698 产物类型的完整枚举表（v1.0 独有，v2.0 用 `ODD.07-Artifact-Taxonomy` 做了更简化的分类）
- 部分历史探索性章节（如"语言处理模块"等，在 v2.0 中已移除或重组）

## 一句话

> v1.0 证明了"ODD 这个方向是对的"；v2.0 让它变成了"今天下午就能开始用"。
