# ODD 首轮公开发布准备说明

> **更新日期**：`2026-03-19`
>
> **作者 / Author**：Yi Fu（付毅，ODDFounder，fuyi.it@live.cn）
>
> **当前推荐口径**：`公开征评包 / 工程方法层 / AI 辅助软件工程方法框架`
>
> **不推荐口径**：`完整终稿 / 已完成所有领域经验验证 / 已替代全部传统软件工程方法 / 闭合终极理论`

---

## 1. 当前结论

ODD 现在已经适合公开发布。

但当前更稳的公开方式不是把整个 `50-工程方法层（ODD）` 一次性抛出，而是按：

- 主论文
- 技术白皮书
- 根入口与发布说明
- 少量关键支撑入口

的方式发布。

---

- `README.md`

2. `Paper_01_ODD_Core_English_2026-01-16.md`

---

## 3. 公开定位

当前最稳的公开定位应固定为：

> **ODD 是建立在 DM、ASTO、ECET、TAT 之上的工程方法层。它面向 AI 辅助软件工程与高影响产出物治理，讨论契约、验证、留痕、封存与责任闭环，而不是把代码审查或提示工程继续当作唯一主控制面。**

这句话的边界很重要：

- `它不是独立终极本体论`
- `它不是对所有传统软件工程方法的一次性废止宣言`
- `它不是已经完成全领域大规模经验验证的成熟标准`

### 3.1 统一发布模板

- `适用范围`：AI 辅助软件工程与高影响产出物治理中的契约、验证、留痕、封存与追责闭环
- `不是什么`：全场景唯一工程范式、脱离责任边界的纯技术流程、整个仓库内容的公开主包
- `边界`：哲学地基回引 DM；应用桥接回引 ASTO；治理边界回引 ECET；责任门槛回引 TAT
- `与上游关系`：ODD 是 `DM -> ASTO -> ECET -> TAT -> ODD` 的工程执行层
- `工具层接口`：若需要对外说明对象、门禁、证据、实证边界与失败条件，优先补发 `ODD.P99.1-6`

---

## 4. 当前公开纪律

不建议写成：

- `ODD 已经证明代码审查彻底失效`
- `ODD 在所有场景都优于 TDD / Agile / DevOps`
- `ODD 已经完成跨行业大规模验证`

建议固定写成：

- `AI 辅助软件工程的工程方法框架`
- `在 AI 产出量超过人类审查能力条件下的结构性响应`
- `公开征评版 / 已有主论文 DOI`
- `契约、验证、封存与责任闭环`

---

## 6. 最小上传策略

如果平台只允许一个主文：

1. 上传 `论文/论文-学术版.md`
2. 如需英文对象，附 `Paper_01_ODD_Core_English_2026-01-16.md`
3. 白皮书与说明文档作为附件

如果平台支持主文 + 多附件：

1. 主论文：`论文/论文-学术版.md`
2. 英文主论文：`Paper_01_ODD_Core_English_2026-01-16.md`
3. 白皮书：`论文/白皮书.md`
4. 根入口：`README.md`
5. 方法入口：`RM.ODD.工程方法论.md`
6. 规格入口：`ODD-main/README.md`
7. 快速上手与方法对比：`guide/快速上手.md`、`guide/方法论对比.md`
8. 工具层接口：`ODD.P99.1-6`

其中：

- `ODD.P99.1` 负责术语与对象边界
- `ODD.P99.2 / P99.3` 负责门禁、证据与封存模板
- `ODD.P99.4` 负责案例、指标与实证证据索引
- `ODD.P99.6` 负责失败条件、召回与降级判据
