# ODD实证实验要求文档

## 1. 实验背景

### 1.1 问题陈述
AI编程时代，如何验证AI生成的代码质量？传统方式依赖Code Review，但效率低下。ODD（Object-Driven Development）主张用"契约+验证+证据"代替人工Review。

### 1.2 核心假设
**当人类在AI生成代码前提供明确的验收标准（契约），功能缺陷逃逸率将显著低于AI自行理解需求的情况。**

---

## 2. 实验目标

### 2.1 主要目标
量化证明：Contract-First（契约优先） vs AI-First（AI自行理解）的缺陷逃逸率差异

### 2.2 次要目标
- 验证ODD方法论的可行性
- 积累实证案例支持ODD推广
- 发现AI编程工作流的改进方向

---

## 3. 实验设计

### 3.1 实验组设计

| 组别 | 名称 | 流程 | 说明 |
|------|------|------|------|
| 对照组 | AI-First | 需求 → AI生成代码+测试 → 运行测试 | 模拟当前主流AI编程工作流 |
| 实验组 | Contract-First | 需求 → 人类提供契约 → AI生成代码 → 运行测试 | 模拟ODD工作流 |

### 3.2 任务选择标准

**样本要求：**
- 任务数量：30-50个（当前15个，需扩展）
- 任务类型需覆盖：
  - CRUD类（用户注册、订单管理）
  - 业务逻辑类（优惠券、购物车折扣）
  - 基础设施类（缓存、限流、日志）
  - 集成类（API网关、任务队列）

**筛选条件：**
- 需求描述清晰，无歧义
- 可用pytest自动化验证
- 涉及边界条件（过期、并发、限额）

### 3.3 缺陷定义

**缺陷逃逸（Defect Escape）：**
- AI生成的测试全部通过
- 但实际实现存在功能错误
- 需人工审查或更高质量测试才能发现

**不计入缺陷的情况：**
- 测试本身失败（测试设计问题）
- 运行时异常（代码健壮性问题）
- 性能问题（非功能需求）

---

## 4. 验证指标

### 4.1 核心指标

| 指标 | 计算公式 | 说明 |
|------|---------|------|
| 缺陷逃逸率 | (逃逸缺陷数 / AI测试通过数) × 100% | 核心指标 |
| 测试覆盖率 | 通过的测试用例数 / 总测试用例数 | 辅助指标 |
| 契约准确率 | 契约正确描述需求的比例 | 质量指标 |

### 4.2 统计要求

- **样本量**：≥30个任务
- **置信水平**：95%
- **显著性检验**：卡方检验或Fisher精确检验

---

## 5. 实验流程

### 5.1 阶段一：任务准备（需完善）

- [ ] 扩展任务库至30-50个
- [ ] 每个任务编写标准化需求描述
- [ ] 准备"人类契约"模板

### 5.2 阶段二：对照组实验

1. 提供需求描述给AI
2. AI生成实现代码 + 测试代码
3. 运行测试，记录通过/失败
4. 人工审查通过但有缺陷的代码
5. 记录缺陷详情

### 5.3 阶段三：实验组实验

1. 提供需求描述 + 人类验收契约
2. AI仅生成实现代码
3. 使用对照组同一套测试验证
4. 记录结果

### 5.4 阶段四：分析报告

1. 汇总两组缺陷逃逸率
2. 统计分析显著性
3. 生成可视化图表
4. 撰写结论

## 7. 风险与限制

### 7.1 已知限制

1. **AI模型依赖**：实验仅针对Claude Sonnet 4.5，其他模型结果可能不同
2. **任务代表性**：当前15个任务可能不足以代表所有场景
3. **测试质量**：使用AI生成的测试作为验证基准，存在循环论证风险
4. **人工审查主观性**：缺陷判定依赖人工，可能有偏差

### 7.2 改进方向

- 引入多AI模型对比（GPT-4、Gemini）
- 增加盲审机制减少主观偏差
- 使用更高质量的人类测试作为"黄金标准"
