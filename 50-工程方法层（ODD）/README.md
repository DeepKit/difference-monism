## 当前最稳定位

ODD 当前最适合被公开描述为：

> **一个面向 AI 辅助软件工程与高影响产出物治理的工程方法框架，其核心论文已经公开发布，目录内其余材料更适合作为公开征评、补充说明与实践参考。**

这意味着：

- 可以公开发布、公开比较、公开征评
- 可以作为预印本平台、Zenodo 或技术白皮书的引用对象
- 不应把整层目录中的工具、缓存、脚本、数据库和历史材料一并当成首轮公开主包

## 统一发布模板

### 适用范围

- 用于 AI 辅助软件工程与高影响产出物治理中的定义、验证、留痕、封存与追责闭环
- 用于把 `DM -> ASTO -> ECET -> TAT` 的上游约束落到工程执行层
- 用于白皮书、论文、公开征评和工程实践比较

### 不是什么

- 不是替代所有传统软件工程方法的唯一范式
- 不是可以脱离责任主体与治理边界独立成立的纯技术流程
- 不是把目录中的工具脚本、缓存、数据库和样例代码全部视作公开主包

### 边界

- ODD 负责工程执行、验证、封存与追责闭环，不负责重建哲学本体论或文明治理总论
- 高影响责任判断仍需回引 TAT，治理边界仍需回引 ECET，应用桥接语法仍需回引 ASTO，哲学地基仍需回引 DM
- 已公开 DOI 的主论文可以稳定外发，其余材料应按主文、补充件、研发件分层处理

### 与上游关系

- ODD 建立在 `DM -> ASTO -> ECET -> TAT` 之上，是五层公开主链中的执行层
- 它把上游关于结构、边界、治理和责任的要求压成工程闭环
- 若只公开 ODD 而不说明上游来源，容易被误读成单纯工程技巧集
- `ODD` 在进入契约、验证、放行与回滚前，建议先读取 `ASTO` 的结构门前语法，不把门禁写成纯技术流程

## DM 风险防火墙

### 最小依赖

即使不先接受 `DM` 的强版本，`ODD` 至少仍可建立在以下弱前提上：

1. 高影响产出物需要 `契约`
2. 契约需要 `验证`
3. 验证需要 `证据`
4. 错误放行后需要 `封存 / 回滚 / 召回 / 审计`

### `DM` 受争议时如何阅读本层

若 `DM` 的强形而上学或强规范地基受争议，
`ODD` 当前最稳的降级阅读方式是：

- 先把它读成高影响产出物治理协议
- 先把它读成工程审计与可回滚框架
- 不把它继续写成“只有接受 `DM` 才能接受”的工程方法

### 本层可独立成立的部分

- 契约定义
- 验证门禁
- 证据链
- 封存、解封、回滚、召回与审计

### 本层仍强依赖 `DM` 的部分

- 把 `ODD` 写成 `DM` 唯一必然工程结果的强说法
- 以哲学正确性替代工程可验收性的写法

### 本层防火墙句

> **即使 `DM` 受压，`ODD` 仍可作为高影响产出物的工程治理协议成立；本层首先由契约、验证、证据与回滚能力站住，而不是只由上游本体论背书。**

## 经验主张与证据纪律

`ODD` 当前允许讨论案例、指标与初步实证，

但公开主位必须遵守三条纪律：

1. 规格、术语、门禁与封存接口优先回引 `ODD.P99.1-3 / P99.6`。
2. 案例、数字表达与经验主张优先回引 `ODD.P99.4`。
3. 外部仓库、研发工具与样例实现若未被正式锁定为公开对象，
   只能按参考实现、研发说明或历史语境理解。

因此：

- `172 任务 / 59.5% / p=0.029` 这类数字表达，
  若未统一回引 `ODD.P99.4`，
  就不应在导读主文中裸写成定论。
- `odd-core / odd-hub / ProgeeLite` 这类对象，
  若当前未被锁定为正式公开对象，
  就不应充当默认公共入口。

- 不把 ODD 说成已经替代所有传统软件工程方法。
