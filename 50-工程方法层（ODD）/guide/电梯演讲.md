# ODD：产出物驱动开发

## 一句话

**代码是负债，产出物才是资产。**

Code is liability. Artifacts are assets.

---

## 核心循环

```
约定 → 执行 → 验证 → 封存
 ↑                     │
 └─── 新需求 ←────┘
```

1. **约定**：动手前写清楚“做什么、怎样算做好了”。
2. **执行**：人或 AI 按约定干活，产出具体东西。
3. **验证**：系统拿产出物对照约定，检查是否达标。
4. **封存**：验证通过的产出物锁定归档，不可篡改。

---

## 三句话版

1. **人类写规格，不写代码** — 你定义“登录应该怎么工作”，AI 写实现。
2. **系统验证，不靠人眼** — 变异测试证明测试有效，不是人说“看着没问题”。
3. **验证过的就锁住** — 封存后任何修改都要解封审计，不能悄悄改。

---

## 为什么是现在

AI 能写代码了，但 AI 写的代码不可信——AI 写的测试也不可信。

旧问题变了形：
- 以前担心“程序员写错了” → 现在担心“AI 生成的东西对不对”
- 以前靠 code review → 现在 AI 一天生成几千行，人根本审不过来
- 以前测试是辅助 → 现在验证证据成为更可靠的质量定义核心

**ODD 不试图让 AI 写好代码，而是建立系统让坏代码无法通过。**

---

## 蜂巢式开发

ODD 的产出物是原子化的——每个都是自包含、经过验证的最小单元。

```
原子产出物 ─┐
            ├─ 管道组合 → 新产出物 ─┐
原子产出物 ─┘                       ├─ 管道组合 → 更大的产出物
                原子产出物 ─────────┘
```

- **可复用**：封存过的产出物直接引用，不用重做。
- **可替换**：某个有问题，替换它不影响其他部分。
- **可并行**：多个车间同时开工，各自封存，互不干扰。
- **可追溯**：从任何终态产出物，沿功能树追溯到每一个原子单元。

---

## 三条底线

- **没有契约不开工** — 不清楚要做什么，就不要开始做。
- **没有证据不放行** — 说“做完了”不够，要拿得出证据。
- **没有封存不算完** — 通过验证的东西必须锁定，防止被悄悄改掉。

---

## 和现有方法论的区别

- **TDD**：先写测试 → 人写代码 → 跑测试
- **ODD**：先写契约 → AI 写代码 → 系统验证 → 封存保护

关键差异不是步骤多了，而是**角色分工变了**：人从“写代码的”变成“定规格的”，验证从“人做的事”变成“系统做的事”。

详见 方法论对比.md

---

## 复杂度可选

- **L1 轻量**：一个人写小工具。契约 + 自动测试 + 归档。
- **L2 标准**：团队协作。加人工审查 + 质量评分 + 结构化证据。
- **L3 严格**：关键系统 / AI 协作。加对抗测试 + 赛马 + 不可变审计。

从 L1 起步，需要时再升级。
