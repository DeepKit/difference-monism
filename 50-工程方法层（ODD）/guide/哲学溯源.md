# ODD 的上游关系说明：从理论地基到工程协议

> 定位：`ODD` 的上游关系说明件，不是 `ODD` 的唯一哲学主文。
> 目的：回答“ODD 为什么这样设计”，同时明确“哪些问题不由 ODD 处理”。

## 为什么保留这份文档

`ODD` 可以独立使用。

你不需要先学完 `DM / ASTO / ECET / TAT`，也可以直接使用契约、门禁、证据、封存与回滚模板。

但如果要公开解释三件事，这份文档仍然必要：

1. 为什么 `ODD` 不把“代码生成”当作最终对象
2. 为什么 `ODD` 强调责任接口、证据留存与封存纪律
3. 为什么 `ODD` 不能单独裁定治理合法性与责任正当性

---

## 最小结构图

```text
DM -> ASTO -> ECET -> TAT -> ODD -> 领域实践
地基   桥接     治理     责任     执行协议   项目落地
```

各层最小分工：

- `DM` 给出最小存在论与结构地基。
- `ASTO` 把地基压成可桥接的变迁语法与边界语言。
- `ECET` 处理文明、制度、治理与公开可比性边界。
- `TAT` 处理责任门槛、责任主体与补偿承接结构。
- `ODD` 把上游约束压成工程执行协议。
- 领域实践把这些协议落到具体软件、流程与组织对象上。

最短句：

> `ODD` 不是重新发明上游理论，而是把上游要求压成“什么能进入系统、如何被验证、如何被留痕、何时必须回滚”的工程闭环。

## ODD 只新增什么

在当前总体系里，`ODD` 最稳的新增职责只有五类：

1. `产出物契约协议`
2. `验证门禁协议`
3. `证据留存协议`
4. `封存发布协议`
5. `回滚召回协议`

这也是 `ODD` 与上游层最重要的边界：

- `ODD` 可以规定工程对象如何定义、验证、封存、回滚。
- `ODD` 不能单独规定一个系统在治理上是否正当。
- `ODD` 不能单独裁定责任结构是否足够。
- `ODD` 不能把“有日志、有测试”误当成“责任已解决”。

## 上游各层分别给 ODD 什么

### 1. `DM`：为什么对象必须先被定义

`DM` 提醒 `ODD`：

- 工程系统处理的不是“模糊愿望”，而是被界定后的对象
- 没有对象边界，就没有可判断的责任边界
- 没有失败线，就没有可公开复核的主张

这就是为什么 `ODD` 先要求契约、失败条件与状态迁移，而不是直接讨论“代码写得快不快”。

### 2. `ASTO`：为什么对象会变，协议必须承认变迁

`ASTO` 给 `ODD` 的不是一堆必须背诵的术语，而是几条桥接纪律：

1. 对象是会变的，不能假装一次定义永久有效
2. 规范会从说明进入编码、物化与执行
3. 高风险变更需要小步试探、证据裁决与可逆路径
4. 有些边界一旦突破，会导致结构性崩解

`ODD` 中对应的工程落点是：

- 契约版本化
- 门禁状态机
- 证据留存
- 封存与解封纪律
- 回滚与召回判据

也就是说，`ODD` 借用了 `ASTO` 的桥接逻辑，但不要求使用者先掌握完整哲学词表。

### 3. `ECET`：为什么工程协议不能脱离治理边界

`ECET` 提醒 `ODD`：

- 并非一切“能做”的自动化都“应当做”
- 工程执行要接受更高层的制度约束、文明约束与公开可比性纪律
- 同一套工程方法进入不同治理环境，合法性判断可能不同

因此：

- `ODD` 不能单独宣称自己天然适用于所有文明、行业与治理情境
- 涉及治理正当性、跨文明比较或制度外部性时，必须回引 `ECET`

### 4. `TAT`：为什么可追踪不等于已负责

`TAT` 提醒 `ODD`：

- 有记录不等于有责任
- 有责任人名字不等于责任结构成立
- 有补偿口号不等于补偿机制存在

所以更稳的边界是：

- `TAT` 决定哪些责任结构必须存在
- `ODD` 决定这些结构如何在工程系统中被表达、验证、封存与审计

这就是为什么 `ODD` 可以做责任接口实现，
但不能独自做责任门槛裁决。

## 不必背整套上游术语，只要抓住四个桥点

如果你只想理解 `ODD` 的设计来源，抓住下面四个桥点就够了：

### 1. 对象边界

没有对象边界，就没有契约。

### 2. 变更代价

对象会变，所以要有版本、门禁、封存与回滚。

### 3. 禁止越层

工程协议不能冒充治理判断，执行层不能冒充责任裁决层。

### 4. 失败优先

能公开失败条件，才有资格公开成功叙事。

## ODD 与上游的接口关系

用一句话压缩：

```text
上游回答“为什么必须这样约束”，
ODD 回答“这些约束怎样被工程系统可靠执行”。
```

几个常见误读需要直接排除：

### 误读一：`ODD` 是新的总哲学

不是。

`ODD` 是执行层协议，不是总本体论。

### 误读二：`ODD` 能单独解决责任问题

不能。

`ODD` 只能把责任结构工程化，不能替代 `TAT` 做责任门槛判断。

### 误读三：理解上游哲学是使用 ODD 的前提

也不是。

理解上游会帮助你判断边界，但不理解上游，仍然可以先使用 `ODD` 的模板化工具层。
