# 案例研究：AI 写小说，但人说了算（kiro-gateway）

> **数据来源**: kiro-gateway/story-docs/（05_方案演进与决策记录.md、06_契约与产出物规格详解.md）

---

## 摘要

kiro-gateway 将 ODD 方法论应用到**小说创作**领域，证明 ODD 的核心概念（契约、产出物、验证、封存、对抗）完全**领域无关**。ASTO 哲学的六阶×五态×七序直接成为场景分类坐标，ODD 的契约→验证→封存循环映射为乐谱→校验→封版。系统经历 5 次方案演进（v1→v5）和 25 个关键设计决策，最终形成了一套完整的 AI 小说工厂。

---

## 一、ODD 概念映射

ODD 的每个核心概念都在小说领域找到了自然的对应物：

| ODD（软件） | kiro-gateway（小说） | 映射说明 |
|-------------|---------------------|----------|
| 契约 (contract) | 场景乐谱 (score_json) | 9 大属性组定义“写什么、怎么验收” |
| 产出物 (artifact) | 场景段落集 | 10 大属性组记录实际输出 + 校验结果 |
| 变异测试 | 四层校验 + 七评委评分 | Layer0 禁元→Layer1 基元→Layer2 指标→Layer3 人类裁决 |
| 封存 (seal) | 集封版 (episode seal) | 自动封版 + 审计日志 + 回滚能力 |
| CAP 对抗 | 契约三重门禁 | 清晰度评估→CAP对抗→契约PK |
| bug_intention_map | writing_knowledge | 写作知识库（confidence/effectiveness/衰减/淘汰） |
| 17 层上下文 | 12 层上下文 (NL1-NL12) | 从 17 层裁剪：丢弃 4 层，合并 7 组，拆分 2 层 |
| 产出物标准库 | 场景类型库 (scene_type_library) | 按 phase+mode 自动注入隐式需求 |

## 二、方案演进：从 v1 到 v5 的五次迭代

这是 ODD “演进”阶段最好的体现——每次迭代都是对“契约应该精确到什么程度”的更深理解：

| 版本 | 方案 | 上下文量 | 核心问题 | 保留的洞察 |
|------|------|-----------|----------|------------|
| v1 | Lorebook（借鉴 SillyTavern） | ~4000 tokens | 关键词触发是猜测式的 | 递归扫描、Token 预算 |
| v2 | 编译器方案 | ~1500 tokens | 只解决数据组织，不保证一致性 | 结构化 > 散文 |
| v3 | 状态机 + 后置校验 | ~800 tokens | 给 AI 信息还是太多 | 三层分离、后置校验 |
| v4 | 乐谱 + 声音样本 | ~300 tokens | 太极端，缺伏笔/角色状态 | 乐谱概念、范文 > 规则 |
| **v5** | **Progee2 工业化** | **~500 tokens** | **最终方案** | 12 层分层注入 + 规则引擎 |

**v5 的关键突破**：不是“给 AI 少一点信息”或“多一点信息”，而是用规则引擎**确定性地决定该给什么**。

## 三、完整契约示例：从乐谱到封版

以下是一个实际场景的完整流转：

### 3.1 人类编写契约（score_json）

契约包含 **9 大属性组**，每个都是可校验的：

```json
{
  "scene_key": "v01.e003.s02",
  "scene_name": "默的算式",

  // B. ASTO 存在论坐标
  "phase": "c",          // 六阶：流变（渐变积累、悬念推进）
  "mode": "III",          // 五态：编码（理性、分析、信息处理）
  "sequence_profile": {
    "focus": "解析",
    "weights": {"觉醒": 0.05, "感知": 0.15, "解析": 0.35,
               "干预": 0.20, "设计": 0.10, "回溯": 0.10, "消解": 0.05}
  },

  // C. 时空锚点
  "spacetime": {
    "time_anchor": "第三天 深夜",
    "location": "陈默的出租屋"
  },

  // D. 角色集
  "characters": {
    "pov": "陈默",
    "present": ["陈默"],
    "mentioned": ["罗生", "林小草"]
  },

  // E. 叙事指令
  "narrative": {
    "enter_from": "s01 结尾：陈默关上门，背靠门板站了十秒",
    "must_land": [
      "陈默发现数据里藏着一个不属于他的变量名",
      "他第一次对罗生的动机产生怀疑"
    ],
    "exit_to": {"direction": "悬疑加深", "emotion": "不安"},
    "information_gate": {
      "reveal": ["罗生的硬盘里存在异常变量"],
      "withhold": ["变量的真正含义", "星图计划"]
    }
  },

  // F. 禁元规则
  "taboos": [
    "陈默不使用文学比喻、成语典故",
    "不得透露星图计划的存在"
  ],

  // G. 地板 + H. 字数 + I. 验收
  "floors": ["至少3个自然段", "POV角色必须有一次内心反应"],
  "word_count": {"target": 3500, "min": 2800, "max": 4200},
  "acceptance": {
    "required_checks": ["voice", "timeline", "foreshadow", "word_count"],
    "must_land_strict": true,
    "max_rework": 3
  },

  // J. 预期副作用
  "side_effects": {
    "character_state_diffs": {
      "陈默": {"cognition_stage": "entry → breaking"}
    },
    "foreshadow_ops": [
      {"action": "plant", "name": "异常变量名", "planned_resolve": "v01.e008"}
    ]
  }
}
```

### 3.2 契约三重门禁

```
Step 1 清晰度评估：
  → 规则检测：must_land 无模糊词 ✅
  → AI分析：exit_to 情绪与 phase=c 一致 ✅
  → 评分 85/100 → 进入 CAP 对抗

Step 2 CAP 对抗：
  → 攻击："3500字+2个must_land+1个伏笔，节奏够吗？"
  → 修复：将 target 从 3000 调整为 3500
  → 攻击：构造一段满足字面条件但平淡的文本
  → 加固：“怀疑过程需层层递进”

Step 3 契约 PK：
  → 产出候选 A（默认）与候选 B（备选）
  → 分差 1.2 ≥ M=0.6 → 自动选候选 A
```

### 3.3 写作生成 + 四层校验

系统组装 12 层上下文（~500 tokens）→ LLM 渲染→场景草稿，然后进入四层校验：

| 校验层 | 内容 | 示例 |
|----------|------|------|
| Layer 0 禁元 | 硬拒绝：已死角色出场、时空矛盾 | 无已死角色出场 ✅ |
| Layer 1 基元 | 声音一致性、乐谱命中、伏笔、字数 | voice PASS / must_land 100% |
| Layer 2 指标 | 反 AI 味词频、细节密度、风格评分 | 归一化每1000字 |
| Layer 3 人类裁决 | 七评委评分 + 人类确认 | 去极值均分 ≥ 7.5 |

**七评委评分示例**：

| 评委 | 分数 | 说明 |
|------|------|------|
| Style | 8.1 | 技术意象精准，‘光标’反复使用形成节奏 |
| Character | 8.5 | 怀疑过程层层递进 |
| Narrative | 7.8 | 信息门控精准 |
| Consistency | 8.6 | 与前两集声音完全一致 |
| 均分 | **8.06** | ≥ 7.5 且单项 ≥ 7.0 → 达标 |

### 3.4 自动封版

```
自动封版判定：
  Layer 0/1 全通过 ✅
  ≥ 7 名评委评分 ✅
  去最高最低 ✅
  单项 ≥ 7.0 ✅
  总分 7.97 ≥ 7.5 ✅
  标准差 0.28 ≤ 1.0 ✅
  → 自动封版！

封版操作：
  摘要 Agent 提取 state_diffs → 写库
  产出物 JSON 写入 chapter_evidence
  正文写入 危01_代价的计算/003.md
  chapter_specs.status → 'sealed'
  audit_log → action='auto_seal'
```

## 四、ASTO 哲学框架的创作域映射

这是 ODD 与 ASTO 连接最深的案例——ASTO 的抽象概念直接成为工程参数：

### 4.1 六阶→场景叙事节律

| 六阶 | 叙事特征 | 典型写法 |
|------|----------|----------|
| 混沌(a) | 信息碎片、氛围铺垫 | 意识流、环境白描 |
| 秩序(b) | 规则建立、关系明确 | 对话推进、信息交换 |
| 流变(c) | 渐变积累、悬念推进 | 线索串联、疑点堆叠 |
| 脉冲(d) | 突变、高潮、不可逆转折 | 真相揭露、对抗爆发 |
| 崩解(e) | 旧秩序瓦解、失控 | 信念崩塔、关系破裂 |
| 归元(f) | 沉淀、收束、新平衡 | 独处反思、余韵留白 |

### 4.2 五态→场景存在质感

| 五态 | 场景质感 | 校验含义 |
|------|----------|----------|
| I 自在 | 感性、直觉 | 允许模糊，情绪优先 |
| II 共识 | 互动、对话 | 对话逻辑自洽 |
| III 编码 | 理性、分析 | 推理链完整 |
| IV 物化 | 动作、事件 | 物理逻辑正确 |
| V 定向 | 决策、不可逆 | **必须包含不可逆状态变更** |

### 4.3 七序→场景内部段落权重

每个场景的段落按七序分配篇幅权重：

| 序 | 小说场景含义 |
|----|-------------|
| 觉醒 | 场景开场：角色进入情境 |
| 感知 | 角色观察环境、接收信息 |
| 解析 | 角色内心推理、分析局势 |
| 干预 | 角色采取行动、对话交锋 |
| 设计 | 角色形成计划、做出决策 |
| 回溯 | 角色审视后果、自我反思 |
| 消解 | 场景收束：情绪落位、留白 |

### 4.4 基元与禁元

- **基元**（系统必须满足的最低条件）：角色声音一致性、时间线连贯性、伏笔必须有回收
- **禁元**（绝不能违反的边界）：破坏角色声音、制造时空矛盾、AI 重新定义故事主题

## 五、关键设计决策摘录

kiro-gateway 记录了 25 个正式设计决策（D1-D25），以下是与 ODD 关系最密切的几个：

### D6: 契约三重门禁
- **来源**：Progee2 的清晰度评估 + CAP对抗 + 智能赛马
- **原因**：乐谱有缺陷，散文不可能好。验证乐谱成本（~500 tokens）远低于写完返工（~4000 tokens × N）

### D8: ASTO 属性集驱动场景分类
- **决策**：放弃“场景类型分类”（对话/打斗），采用 ASTO 属性集（六阶×五态 + 七序骨架）
- **关键变更**：乐谱不再是 `{emotion, tempo}`，而是 `{phase, mode, sequence_profile}`；校验由“查模板”变为“属性一致性检查”

### D10: ODD 合规自审
- **发现**：13 项缺口（证据绑定、封存完整性、审计日志、预警分级、执行模型）
- **修复**：`chapter_specs` 增加 `schema_version`/`depends_on`/`seal_hash`/`evidence_refs`；新增 `audit_log` 表；新增执行模型（错误分类/补偿/幂等）

### D13: 自动封版 + 去极值评分
- **判定规则**：Layer 0/1 全通过 + ≥ 7 名评委 + 去最高最低 + 单项 ≥ Y + 总分 ≥ X + σ ≤ 阈值
- **未达标时**：回退人类裁决，保留人类最终决策权

### D25: 反 AI 味词表与风格治理
- **决策**：黑名单/抽象情绪/解释句式等词表落库，按规则集版本化
- **口径**：有效字数（汉字=1，连续英数token=1），归一化每1000字

## 六、契约与产出物的对称设计

这是 ODD “可追溯、可校验、可诊断”的工程落地：

```
契约字段                     产出物字段                   关系
────────────────────────    ────────────────────────    ────────
phase / mode                asto_actual.phase/mode      期望 vs 实际
sequence_profile.weights    asto_actual.sequence_profile 偏差校验
spacetime                   spacetime_actual            时空一致性
characters                  characters_actual           角色一致性
narrative.must_land         validation.score_align      命中率校验
taboos                      validation.layer0_taboo     禁元违反检测
word_count                  content.word_count          字数范围
side_effects                state_diffs                 副作用对照
```

每个契约期望值都有产出物实际值对应，确保可追溯、可校验、可诊断。

## 七、核心发现与经验总结

### 7.1 ODD 领域无关性得到验证

ODD 的每个核心概念在小说领域都找到了自然对应物，且映射过程不需要强行套用。关键证据：
- 契约的 9 大属性组在小说场景中完全适用
- 变异测试的“注入错误，看测试能否发现”逻辑映射为“四层校验 + 多评委评分”
- 封存机制完全复用（seal/unseal/rollback + 审计链）

### 7.2 ASTO 提供了真正的“分类坐标”

简单的场景类型标签（对话/打斗/叙事）无法覆盖小说场景的复杂性。ASTO 的六阶×五态×七序提供了一个真正多维度的坐标系，而且每个维度都有明确的校验含义（例：V定向态必须有不可逆后果）。

### 7.3 “验证乐谱”比“修散文”便宜得多

契约三重门禁的核心洞察：在写作前验证契约的成本（~500 tokens）远低于写完后发现问题再返工（~4000 tokens × N 次）。这与软件工程中“越早发现 Bug 成本越低”的原理完全一致。

### 7.4 人类创作主权的保护

ASTO 的“风险层”在小说系统中体现为：
- AI 不能重新定义故事的主题和价值观
- 当 AI 产出“合格但不动人”的文字时，人有权推翻
- 自动封版未达标时回退人类裁决，保留最终决策权
