# ODD 30 分钟上手指南

> **目标**: 一个开发者，不读任何理论文档，30 分钟内用 ODD 方式完成一个真实任务。
> **等级**: L1（轻量）—— 一个人就能跑起来。

## 前提

- 任意编程语言
- 任意 AI 工具（Claude / ChatGPT / Cursor / Copilot / 其他）
- Git
- 可选：变异测试工具（Stryker / mutmut / Pitest）

## 核心思路

```
你要做的事：写契约（定义要什么）
AI 要做的事：写代码 + 写测试（按契约干活）
系统要做的事：验证（对照契约检查）→ 封存（锁定归档）
```

---

## Step 1（5 分钟）：写契约

在项目根目录创建 `contracts/` 文件夹，写一个 YAML 文件：

```yaml
# contracts/user-login.yaml
contract:
  id: C001
  title: 用户登录
  acceptance_criteria:
    - 正确用户名+密码 → 返回 JWT token，有效期 3600秒
    - 错误密码 → 返回 INVALID_CREDENTIALS
    - 连续 5 次错误 → 账户锁定 30 分钟，返回 ACCOUNT_LOCKED
    - 空用户名或空密码 → 返回 400
  boundary_cases:
    - 用户名长度: 3-20 字符
    - 密码长度: 8-128 字符
    - 密码不得出现在任何日志中
    - 响应时间 < 200ms (p99)
  examples:
    - input: { username: "alice", password: "correct_pw" }
      output: { success: true, token: "<JWT>", expires_in: 3600 }
    - input: { username: "alice", password: "wrong" }
      output: { success: false, error: "INVALID_CREDENTIALS" }
    - input: { username: "", password: "any" }
      output: { status: 400, error: "EMPTY_USERNAME" }
```

这就是契约。它回答两个问题：
- **做什么**：acceptance_criteria + boundary_cases
- **怎样算做好了**：examples（输入输出映射）

> 💡 ODD 底线 #1：**没有契约不开工**。不清楚要做什么，就不要开始做。

### 可选：快速自查契约质量

写完后花 1 分钟过一遍：
- 有没有模糊词（“大量”“尽快”“合理”）？
- 验收条件能不能写成测试用例？
- 有没有“字面合规但实际没用”的实现方式？（比如所有输入都返回硬编码 token）

这是 L1 级别的简易版契约对抗。完整版见 C09.工程_契约_产出物_证据.md。

## Step 2（5 分钟）：AI 生成

把契约文件交给 AI，提示词：

```
根据以下契约生成实现代码和完整测试。

要求：
1. 实现代码和测试代码分离
2. 测试必须覆盖所有 acceptance_criteria 和 boundary_cases
3. 测试必须覆盖所有 examples 中的输入输出映射
4. 密码不得出现在任何日志或错误信息中

[contracts/user-login.yaml 的内容]
```

AI 会生成：
- 实现代码（如 `login.py`）
- 测试代码（如 `test_login.py`）

> 关键思维转变：你不需要审查 AI 写的代码逻辑。你只需要审查它的输出是否符合契约。

---

## Step 3（5 分钟）：验证

### 3a. 跑测试

```bash
# Python 示例
pytest test_login.py -v
```

全通过 → 产出物初步达标。
有失败 → 让 AI 重新生成（同一个契约，重新执行）。

### 3b. 变异测试（可选，强烈推荐）

变异测试回答一个问题：**测试本身是否有效？**

原理：故意在代码里制造小错误（变异体），看测试能不能发现。

```bash
# Python 示例（安装 mutmut）
pip install mutmut
mutmut run --paths-to-mutate=login.py --tests-dir=.
mutmut results
```

得到一个变异分数（mutation score）：
- **> 80%**：测试质量良好，可以封存。
- **60-80%**：测试有盲区，让 AI 补充测试后重跑。
- **< 60%**：测试基本无效，重新审查契约和测试。

> 💡 “测试过了”只说明代码能跑通。“变异测试过了”才说明测试真的在检查代码。

> 💡 ODD 底线 #2：**没有证据不放行**。测试报告和变异分数就是证据。

---

## Step 4（5 分钟）：封存

验证通过后，封存产出物：

```bash
# 方法 1：最简单——git tag
git add .
git commit -m "feat(login): implement user login per contract C001

mutation_score=85%
all_tests_pass=true
evidence: test_login.py (14 cases, 14 pass)"

git tag -a C001-v1.0-sealed -m "sealed: mutation_score=85%, 14/14 tests pass"
```

```bash
# 方法 2：结构化封存记录（推荐）
# 创建 seals/C001.json
{
  "contract_id": "C001",
  "artifact": "login.py",
  "version": "1.0",
  "sealed_at": "2026-02-12T12:00:00Z",
  "evidence": {
    "test_report": "14/14 pass",
    "mutation_score": "85%",
    "test_file": "test_login.py"
  },
  "git_tag": "C001-v1.0-sealed"
}
```

封存的意义：
- 这个产出物已经经过验证，可以被其他模块安全依赖。
- 如果以后要修改，必须解封（记录原因）→ 修改 → 重新验证 → 重新封存。不能悄悄改。

> 💡 ODD 底线 #3：**没有封存不算完**。通过验证的东西必须锁定，防止被悄悄改掉。

## Step 5（10 分钟）：感受差异

回顾你刚才做的事：

**你做了什么：**
1. 写了一个 YAML 文件（契约）
2. 把它交给 AI
3. 跑了测试和变异测试
4. 打了一个 tag

**你没做什么：**
- 没有写一行实现代码
- 没有逐行审查 AI 的代码
- 没有手动测试

**但你得到了：**
- 一个有明确规格的功能实现
- 可量化的质量证据（mutation score 85%）
- 不可篡改的封存记录
- 完整的契约 → 执行 → 证据 → 封存链条

## 下一步：从 L1 到 L2

你刚才体验的是 ODD L1（轻量级）。当项目变复杂时，可以按需升级：

### L2 增加的能力

**契约质量评分**：系统自动评估契约清晰度（0-100 分），低于 60 分禁止激活。检测模糊词（“大量”“尽快”“合理”）、边界缺失、依赖缺失、逻辑冲突。

**结构化证据**：证据不再是纯文本，而是带类型、哈希、时间戳的结构化记录，可查询、可审计。

**依赖图**：系统自动维护产出物之间的依赖关系。当某个产出物被修改时，自动标记下游依赖为 stale。

### L3 增加的能力

**契约对抗生成（CAP）**：多角色主动攻击契约——提案者负责写，挑战者找逻辑漏洞，攻击者找“字面合规但实际没用”的实现，裁决者判定攻击是否有效。契约在对抗中越来越健壮。

**赛马机制**：不直接选最强的 AI 模型，而是简单任务用低成本模型，失败后先诊断原因（是上下文不足？契约模糊？还是模型能力不够？），再决定是否升级。

**封存不可变**：seal_hash 绑定全部证据，状态不可逆，任何篡改可检测。

**车间模型**：多个 AI 并行执行不同契约，资源隔离，故障可恢复，交叉审核（A 车间造的东西，B 车间来检）。

## 完整项目结构示例

```
my-project/
├── contracts/
│   ├── C001-user-login.yaml
│   ├── C002-user-register.yaml
│   └── C003-password-reset.yaml
├── src/
│   ├── login.py
│   ├── register.py
│   └── password_reset.py
├── tests/
│   ├── test_login.py
│   ├── test_register.py
│   └── test_password_reset.py
└── seals/
    ├── C001.json                  # 封存记录
    └── C002.json
```

每个契约是一个独立的工作单元。每个产出物独立封存。多个产出物可以通过管道组合成更大的产出物。
