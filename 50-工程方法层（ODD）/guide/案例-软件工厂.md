# 案例研究：一个人的软件工厂（Progee）

> **数据来源**: progee2/history.md 开发历史记录

---

## 摘要

Progee 是 ODD 方法论的**参考实现**——一个人使用 ODD + AI 车间体系，在 **~33 天**内从零构建了一个包含 **110+ 源文件、231 个自动化测试**的完整软件工厂系统。本案例展示 ODD 核心循环（契约→生成→验证→封存）在真实工程中的落地方式、量化数据以及经验教训。

## 三、ODD 四阶段循环的实践落地

### 3.1 契约定义（人类负责“做什么”）

人类通过“军帐”（对话界面）与 AI 讨论需求，系统自动生成结构化契约，经人工确认后进入“参谋部”。

**实际使用的契约结构**：
- 契约表 `contracts`：含 `score_json`（结构化规格）、`depends_on`（依赖链）、`source_section_id`（溯源对话）
- 任务表 `tasks`：33 个字段，含 `task_level`（L1-L4）、`input_output_mapping`、`failure_context`
- 清晰度检测：`clarity_detect` + `clarity_question` 提示词

**关键设计决策**：
- 契约粒度：一个契约对应一个可独立验收的功能单元
- 依赖处理：契约声明 `depends_on`，系统自动排序执行
- 增量导入：DeBorn 代码扫描→契约自动生成

### 3.2 AI 生成（AI 负责“怎么做”）

多车间并发执行，每个车间是一个独立 AI 工人线程。

**多 Agent 架构**：
- **盘古**（单架构师）→ 高层设计讨论
- **军帐 5 角色**→ 多角色辩论，产出契约
- **开发/测试/审核工人**→ 代码生成、测试生成、交叉审核

**17 层上下文注入**（CtrlContextAssembler）：

| 层级 | 内容 | 注入策略 |
|----|------|----------|
| L1 | 安全边界 | 强制 |
| L2-L3 | 系统规则 | 强制 |
| L4 | 契约上下文 | 角色关联 |
| L7 | 功能树索引 | 动态查询 |
| L12 | 车间知识库 | 条件注入 |
| L14-L15 | 任务规格 + 验收标准 | 任务关联 |
| L16 | 代码上下文 | 按需 |
| L17 | 纠正反馈 | 返工时注入 |

**赛马模型**（CtrlModelRacing）——诊断优先升级：
1. 失败后先诊断原因（上下文不足 / 契约模糊 / 校验不合理 / 模型能力不够）
2. 只有“模型能力不够”才消耗赛马配额升级模型
3. 其他原因就地修复

### 3.3 变异测试验证（系统负责“对不对”）

**变异算子**（CtrlMutationOperators）：
- AOR：算术运算符替换（+ ↔ - ↔ * ↔ /）
- ROR：关系运算符替换（> ↔ >= ↔ < ↔ <=）
- LOR：逻辑运算符替换（and ↔ or ↔ not）

**降级策略**（当变异测试无法执行时）：
- `skip`：跳过变异，仅静态校验
- `static_only`：仅编译 + 静态分析
- `block`：阻塞任务，等待人类决策

**对抗测试**（CtrlAdversarialTester）：
- 四类对抗形态（来源：ASTO.E05）
- 三 AI 交叉审核（CtrlCrossReviewer）

### 3.4 封存保护（系统负责“锁住”）

**封存机制**（CtrlCodeVersions）：
- `seal`：标记代码为不可修改，写入 `seal_hash` + 审计日志
- `unseal`：需要授权，记录解封原因，版本号递增
- `rollback`：一键回滚到任意历史版本

**预警分级**：
- 返工次数超限 → 通知
- 变异分数低于阈值 → 告警
- 心跳超时(90s)、任务超时(30min)、契约超时(4h) → 自动回收

## 四、量化数据

### 4.1 开发效率

| 指标 | 数据 | 说明 |
|------|------|------|
| 总任务数 | 225+ | 16 次冲刺 |
| 日均任务 | ~7 个 | 含设计/代码/测试/文档 |
| 峰值吞吐 | 47 任务/天 | 冲刺10: DeBorn 集成 |
| Bug 发现与修复 | 28 个 / 一次冲刺 | 2026-02-10 全面审查 |

### 4.2 Bug 发现方式分布

2026-02-10 全面审查（22 个源文件）发现 28 个 Bug：

| Bug 类别 | 数量 | 发现方式 |
|----------|------|----------|
| FillChar 内存损坏 | 12 处 / 6 文件 | 系统审查 |
| SQL 注入 | 3 处 | 系统审查 |
| 线程安全 | 4 处 | 系统审查 |
| Schema 前缀缺失 | 9 处 | 系统审查 |
| UUID 类型/参数化 | 5 处 | 系统审查 |
| 状态转换/逻辑错误 | 5 处 | 规范对齐 |

**关键发现**：绝大多数 Bug 由**系统审查**发现，而非人工 Code Review。这与 ODD “系统验证替代人类审查”的核心主张一致。

### 4.3 规范文档一致性

同期审查 15 份规范文档，发现并修复 16 项跨文档不一致：层级描述过时、角色可见层数错误、评分算法误导、状态机缺少终态、变异测试矛盾、命名规范不统一、错误码缺失等。

## 五、经验总结

### 5.1 ODD 效果最好的地方

1. **契约→任务的自动分解**：人类只需定义“做什么”，AI 自动分解为可执行任务
2. **17 层上下文注入**：精确控制 AI 接收的信息，显著提高生成质量
3. **返工时的纠正反馈（L17）**：注入上次失败原因，避免 AI 重复犯同一个错误
4. **系统审查替代人工**：28 个 Bug 几乎全部由系统发现

### 5.2 契约需要迭代的地方

1. **粒度问题**：早期契约过粗（“实现用户认证模块”），后来细化到单接口级别才高效
2. **依赖声明**：初期忘记声明导致执行顺序混乱，后增加 `depends_on` 字段
3. **task_level 判定**：L1→L4 的自动判定需要多轮校准
4. **规范漂移**：15 份文档随迭代产生不一致，需定期审查同步

### 5.3 开发模式的演变

```
第1周（冲刺1-8）：密集建设期    → 日均 10-15 任务，搭建骨架，流程打通
第2-3周（冲刺9-16）：功能深化期  → 变异测试、对抗测试、多应用集成
第4-5周：论文+稳定化           → 复现实验、231/231 测试全通过
第6周：规范对齐+全面迁移       → Bug修复(28)、文档一致性(16)、UniDb迁移(37)
```
