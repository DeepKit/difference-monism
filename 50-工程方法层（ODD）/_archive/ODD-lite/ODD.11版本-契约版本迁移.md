# 契约版本迁移

## 意向
契约会演进，但消费者可能还在用旧版本。
需要一套版本策略来定义版本、保持向后兼容、规划废弃路径、设计迁移方案。

## 规范

### 语义化版本
契约版本 MUST 使用 `MAJOR.MINOR.PATCH` 格式：

| 段 | 变更类型 | 示例 |
|----|----------|------|
| MAJOR | 破坏性变更（删除字段、改变类型） | 1.1.1 → 2.0.0 |
| MINOR | 向后兼容的新增（新增可选字段） | 1.0.0 → 1.1.0 |
| PATCH | Bug 修复（不改变契约结构） | 1.1.0 → 1.1.1 |

### 兼容性规则

**向后兼容变更（安全）**：
- ✅ 新增可选输入字段（MUST 有默认值）
- ✅ 新增输出字段
- ✅ 放宽输入约束
- ✅ 新增可选错误码

**破坏性变更（危险）**：
- ❌ 删除输入/输出字段
- ❌ 改变字段类型
- ❌ 新增必填输入字段
- ❌ 收紧输入约束
- ❌ 改变字段语义（如单位从"分"变"元"）

破坏性变更 MUST 递增 MAJOR 版本。

### 版本生命周期
```
CURRENT → DEPRECATED → SUNSET → REMOVED
  │           │           │          │
正常使用    发出警告    拒绝请求    代码删除
```

- DEPRECATED 期 SHOULD ≥ 3 个月（关键系统 ≥ 6 个月）。
- SUNSET 后 MUST 返回明确错误，指引升级路径。

## 机制

### L1 · 轻量

**无正式版本管理。** 直接修改契约，手动通知受影响方。

适用于一人项目，契约变更少且无外部消费者。

---

### L2 · 标准

**契约带版本号 + 兼容性检查。**

契约增量字段：
```yaml
contract:
  version: "1.0.0"
  deprecated: false
  sunset_date: null
```

任务关联契约版本：
```yaml
task:
  contract_version: "1.0.0"
```

#### 兼容性检查
创建新版本契约时，系统 MUST 自动检查兼容性：
- 对比新旧版本的 input_spec / output_spec。
- 删除字段、改变类型、新增必填字段 → 标记为破坏性变更。
- 破坏性变更 MUST 人工确认后才能提交。

#### 三种迁移模式

**适配器模式**（差异小，可自动转换）：
```
旧契约调用 → 适配器转换 → 新契约执行 → 适配器转换 → 旧格式响应
```
- 输入转换：为新增字段填默认值。
- 输出转换：移除旧版本不认识的字段。

**扩展-收缩模式**（字段重命名、结构重组）：
```
阶段1: 扩展 — 同时支持新旧字段（MINOR 版本）
阶段2: 标记废弃 — 旧字段标记 deprecated
阶段3: 收缩 — 移除旧字段（MAJOR 版本）
```

**版本路由模式**（差异大，无法简单适配）：
```
请求携带版本号 → 路由到对应版本的实现
```
- 版本号不存在时，路由到最近的兼容版本。

---

### L3 · 严格

**在 L2 基础上增加：废弃中间件 + 迁移追踪 + DB 层版本管理。**

#### 废弃中间件
- DEPRECATED 版本：请求正常处理，但响应头增加 `Deprecation: true` + `Sunset: <date>`。
- SUNSET 版本：拒绝请求，返回 410 Gone + 升级指引。
- 所有使用废弃版本的调用者 MUST 被记录和通知。

#### 迁移追踪
```yaml
migration_tracking:
  version: "1.0.0"
  status: current | deprecated | sunset | removed
  deprecated_at: datetime?
  sunset_at: datetime?
  active_consumers: int        # 仍在使用的调用者数
  migration_guide_ref: string  # 迁移指南引用
```

- 系统 MUST 统计各版本使用情况。
- SUNSET 前 MUST 确认 active_consumers = 0（或人工确认强制下线）。

#### DB 层版本管理
- 存储过程/函数 SHOULD 带版本后缀（如 `create_user_v1`, `create_user_v2`）。
- Schema 迁移遵循扩展-收缩模式：先 ADD COLUMN，后 DROP COLUMN。
- DROP COLUMN MUST 在确认无调用后执行。

## 实践

### 快速选型
- **小项目** → L1，直接改，手动通知。
- **团队项目** → L2，带版本号 + 兼容性检查 + 三种迁移模式。
- **关键系统** → L3，废弃中间件 + 迁移追踪 + DB 层版本管理。

### 核心原则
- **设计时考虑演进**：字段命名有前瞻性，用可扩展结构。
- **变更前评估影响**：运行兼容性检查，分析调用者。
- **渐进式迁移**：先扩展后收缩，给足废弃期。
- **主动通知消费者**：不要等消费者踩坑，主动推送迁移信息。
