# 上下文工程

## 意向
给执行者（人或 AI）恰到好处的信息：不多不少，来源可查。
多了浪费 token 和注意力，少了做出错误决策。

## 规范

### 上下文原则
- 最小必要注入：只给当前任务需要的信息。
- 证据不内联：长内容下沉为证据对象，用 evidence_ref 引用。
- 来源可查：每段上下文 MUST 标注来源（哪个契约/任务/证据）。
- 返工强制注入：rework 时 MUST 注入 failure_context（含 evidence_refs）。

---

## 机制

### L1 · 轻量

**上下文 = 手动准备。**

执行任务前，手动确认以下信息可用：
- 契约的验收条件
- 任务的输入/输出规格
- 依赖的产出物在哪里

无自动化、无 token 控制。适用于人工开发。

---

### L2 · 标准

**上下文 = 功能树查询 + 自动注入 + token 控制。**

**功能树查询**：
从功能树出发，精确定位当前任务相关的产出物、契约、证据，而不是塞给执行者整个项目。

**注入分层**：
- **强制注入**：安全规则 + 架构约束 + 任务规格 + 验收条件。
- **按需注入**：相关代码片段 + Bug 意向图 + 最佳实践（详见 ODD.0E法宝-Bug意向图与最佳实践）。
- **返工注入**：failure_context + 历史尝试记录（避免重复同样的错误）。

**Token 控制**：
- 系统 SHOULD 有 token 预算，按优先级裁剪低价值内容。
- 优先级：安全规则 > 任务规格 > 验收条件 > 相关代码 > 最佳实践。

---

### L3 · 严格

**上下文 = 多层注入 + 车间知识库 + 双流记忆。**

**多层上下文模型**：
- 硬边界层：安全规则、架构约束、流程规范——强制注入，不可穿透。
- 功能树层：索引查询，定位相关产出物。
- 依赖图层：查询上下游关系。
- 知识库层：车间封存时提炼的语义知识（Bug 意向图、最佳实践、隐式需求；详见 ODD.0E法宝-Bug意向图与最佳实践）。
- 任务执行层：任务规格、验收、返工反馈。

**情报官（Intelligence Officer）**：
- 角色：在调用模型前，统一查询数据库并组装上下文，避免“多次调用+重复上下文”。
- 输入：角色、阶段、task_id、contract_id、module_id。
- 输出：组装好的上下文 + 来源元数据 + token 估算。

**情报官流程**：
```
任务下达 → 情报官接收 → 查询数据库（按角色/阶段）
  → 过滤裁剪（Token预算）→ 分层组装 → 一次性注入模型
```

**17层上下文模型（简版）**：
- L1 安全硬边界（P0）
- L2 架构硬边界（P0）
- L3 流程硬边界（P0）
- L4 系统级规范（P1）
- L12 车间知识库：Bug意向图/最佳实践（P1）
- L14 任务规格（P0）
- L15 验收与测试（P0）
- L16 执行上下文/相关代码（P2）
- L17 纠正反馈（P0，rework时）
其余层按项目扩展，原则是“必要优先”。

**角色视野矩阵（简化）**：
- 架构师：全局架构、契约列表、功能树全貌；不看具体实现细节。
- 经理：契约状态、任务调度、资源分配；不看代码细节。
- 车间：本模块上下文/规格/执行信息；不看其他模块代码。
- 工人：当前任务规格/验收/执行上下文；不看其他任务。

**阶段注入矩阵（简化）**：
- 契约生成：功能树全貌 + 相关契约列表。
- 清晰度检测：契约详情 + 模糊词库 + 历史模糊问题。
- 任务生成：契约详情 + 依赖关系 + 已封版产出物。
- 开发/测试/审核：任务规格 + 验收标准 + 相关代码 + Bug历史 + 最佳实践。

**提示词模板骨架**：
```
## 硬边界（P0）
{L1_security_boundary}
{L2_architecture_boundary}
{L3_process_boundary}

## 专家身份
{expert_identity}

## 任务规格（P0）
{L14_task_spec}
{L15_acceptance_criteria}

## 知识库（P1）
{L12_bug_patterns}
{L12_best_practices}

## 纠正反馈（rework时）
{L17_failure_context}

## 相关代码（P2）
{L16_code_context}
```

**Token预算建议**：
- 契约约束 200、任务规格 300、验收标准 200（P0）。
- Bug历史 300、最佳实践 300（P1）。
- 相关代码 500（P2）。
超额时按优先级裁剪。

**预置参数组**（示例）：
- full_injection（测试阶段，全量）
- minimal（省Token）
- expert_focus（强化专家身份）
- bug_prevention（强化Bug历史）
- rework_focus（强化纠正反馈）
- balanced（正式环境）

**提示词存库规则**：
- 提示词 MUST 存数据库，按版本管理，可回滚与统计。
- 记录使用情况：成功率、返工率、Token消耗。

**车间知识库**：
- 每个车间封存时 SHOULD 提炼知识到知识库（情景记忆 + 语义记忆）。
- 新车间启动时 SHOULD 加载相关知识（热启动）。
- 热启动 MUST 校验知识版本与当前产出物版本一致。

**双流记忆**：
- 情景记忆（短期）：本次执行过程中的关键决策、遇到的问题。
- 语义记忆（长期）：跨任务沉淀的模式、规律、最佳实践。

**监控指标**（SHOULD）：
- tokens_injected / tokens_saved
- layers_included / layers_skipped
- cache_hit_rate
- task_success / rework_count

## 实践

### 快速选型
- **人工开发、小项目** → L1，手动确认上下文即可。
- **团队协作、AI 辅助** → L2，功能树查询 + token 控制。
- **AI 多角色协作** → L3，多层注入 + 车间知识库 + 双流记忆。

### 核心原则
- **最小必要**：给够用的信息，不给多余的。
- **来源可查**：每段上下文都能追到出处。
- **返工必注入**：失败信息不能丢，否则会重复同样的错误。
