# 分级预警与红绿灯

## 意向
系统在运行中会产生各种信号——任务返工、测试失败、门禁阻塞。
问题是：哪些信号需要人类关注，哪些可以自动处理？
分级预警回答这个问题：用灯状态让人一眼看到全局，用升级规则让系统自动处理能自动处理的，只在真正需要时才打扰人类。

## 规范

### 灯状态定义
所有可监控的阶段 MUST 使用统一的灯状态：
- **白色**：未到达该阶段。
- **绿闪**（呼吸闪烁）：正在进行中，正常。
- **红闪**（呼吸闪烁）：进行中但有问题，需人类关注。
- **长绿**（静止）：该阶段已完成。

### 预警分级原则
- 系统 MUST 按严重程度分级响应，不得对所有问题一律告警。
- 低级预警 MUST 自动处理，不打扰人类。
- 红灯上报 MUST 阻塞任务，等待人类决策。
- 人类决策 MUST 以选择题形式提供选项，降低认知负担。

### 红闪触发条件
任何阶段出现以下情况 MUST 转为红闪：
- 阻塞状态（blocked）。
- 返工次数达到告警阈值。
- 门禁检查失败。
- 变异测试分数低于阈值。

---

## 机制

### L1 · 轻量

无正式灯状态。任务状态（pending / in_progress / done / rework）即为信号。
失败后由人工判断是否继续。

---

### L2 · 标准

**灯状态可视化**：每个任务的关键阶段显示灯状态。

阶段与灯的映射：
```
准备 → 开发中 → 质检中 → 验收中 → 封存
 ○       ○        ○        ○       ○
```

各阶段灯色规则：
- 当前阶段 = 绿闪（正常进行）或红闪（有问题）。
- 已完成阶段 = 长绿。
- 未到达阶段 = 白色。
- rework 状态 = 开发中列红闪。

**返工预警**（四级）：
- **正常**（0-1 次）：绿闪，正常执行。
- **黄灯预警**（2 次）：记录日志，诊断失败原因。不打扰人类。
- **橙灯预警**（4 次）：通知人类，但不阻塞任务。
- **红灯上报**（≥6 次）：阻塞任务，等待人类决策。

**变异测试预警**（三级）：
- **正常**（首次 < 阈值）：自动返工补充测试。
- **黄灯预警**（2 次 < 阈值）：记录日志，继续尝试。
- **红灯上报**（3 次 < 阈值）：阻塞任务，等待人类决策。

**人类决策菜单**（红灯触发时提供）：
- 修改契约描述 — 契约不清晰导致执行者无法正确实现。
- 降低门禁阈值 — 临时降低变异测试阈值。
- 人工补充 — 人类补充关键测试用例或代码片段。
- 取消任务 — 放弃该任务。
- 继续尝试 — 重置计数，忽略预警继续执行。

---

### L3 · 严格

**灯状态增强**：增加变异测试、对抗测试、交叉审查阶段的灯列。

```
准备 → 开发中 → 质检中 → 变测中 → 对抗中 → 交叉审查 → 验收中 → 封存
 ○       ○        ○        ○        ○         ○         ○       ○
```

各阶段的红闪触发条件：
- **准备**：blocked 状态（依赖阻塞）。
- **开发中**：rework_count ≥ 2（预警）；rework_count ≥ 6（必须人类介入）。
- **质检中**：测试失败。
- **变测中**：mutation_score < 阈值。
- **对抗中**：发现高危漏洞。
- **交叉审查**：审查方拒绝通过。
- **验收中**：人工验收拒绝。

**预警与赛马联动**：
- 黄灯预警 → 触发智能赛马的诊断流程（见赛马文档）。
- 诊断结果为"执行能力不足" → 自动升级执行者等级。
- 诊断结果为"契约模糊" → 触发清晰度重评估（见 CAP 文档）。
- 红灯上报 → 阻塞，同时冻结赛马升级，等待人类决策。

**数据结构**：
```yaml
task_warning:
  task_id: string
  current_stage: string
  lamp_state: white | green_blink | red_blink | green_solid
  warning_level: normal | yellow | orange | red
  rework_count: int
  mutation_attempts: int
  human_decision:
    decision: modify_contract | lower_threshold | manual_assist | cancel | retry
    decided_by: string
    decided_at: datetime
    note: string
```

**审计**：
- 所有预警升级 MUST 记录在审计日志中。
- 人类决策 MUST 记录选择的选项、决策者和时间。
- 红灯上报的频率和处理时间 SHOULD 作为系统健康度指标。

## 实践

### 快速选型
- **小项目** → L1，无灯状态，人工判断。
- **团队项目** → L2，灯状态 + 四级返工预警 + 人类决策菜单。
- **AI 多角色协作** → L3，全阶段灯状态 + 预警与赛马联动 + 审计。

### 核心原则
- **不要什么都告警**：低级问题自动处理，只在真正需要时打扰人类。
- **灯状态是导航**：一眼看到全局状态，红闪 = 需要你，绿闪 = 不需要你。
- **人类决策要简单**：提供选择题菜单，不让人类从零开始分析问题。
- **预警驱动改进**：红灯频率高 = 契约质量差或任务分级不合理，需要系统性修正。
