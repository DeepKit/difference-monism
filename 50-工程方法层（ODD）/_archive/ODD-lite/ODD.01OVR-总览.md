# 总览

## 意向
ODD（Outcome-Driven Development，产出物驱动开发）是一套工程方法论。

核心认知：**代码是负债，产出物才是资产。** 只有经过约定、验证、封存的产出物，才是可信赖、可复用的资产。

核心闭环：**约定 → 执行 → 验证 → 封存。**

核心结构：**蜂巢式开发。** 原子产出物通过管道组合擢升为新产出物，形成由功能树管理的活的网状结构。每个格子独立封装、可并行建造、整体自然生长。

目标：让“坏输出无法通过”，而不是寄希望于“生产者足够聪明”。

## 规范

### 五大支柱
1. **契约（Contract）**：定义“做什么、怎样算做好”。
2. **产出物（Artifact）**：可验证的具体交付物。
3. **门禁（Gate）**：状态迁移的检查点，不达标不放行。
4. **证据（Evidence）**：门禁检查的可查证记录。
5. **封存（Seal）**：产出物 + 证据锁定归档，不可篡改。

### 结构支柱
- **功能树（Functional Tree）**：业务功能索引，定位产出物、追溯依赖、分析影响。
- **依赖图（Dependency Graph）**：产出物之间的有向无环图，显式记录谁依赖谁。
- **管道（Pipeline）**：有契约的组装线，把已封存的产出物组合擢升为新产出物。

### 规范语言
- MUST：必须
- SHOULD：建议
- MAY：可选

### 复杂度等级
所有机制文档均按三级复杂度分别说明：
- **L1 · 轻量**：个人/小项目/原型。
- **L2 · 标准**：团队协作/中等复杂度。
- **L3 · 严格**：关键系统/AI 多角色协作。

从 L1 起步，按需升级。同一项目可按模块混用等级。

### 核心范式：AI 不可信，测试结果才可信

ODD 不试图让 AI 写“好代码”，而是建立系统让“坏代码”无法通过。

| 旧思维 | 新思维（TD-AI） |
|--------|------------------|
| 优化 AI 输入，期望好输出 | 验证 AI 输出，不管过程 |
| AI 写代码，然后写测试 | 先定义测试，AI 填空 |
| 测试是附属品 | 测试是质量的定义 |
| 失败后升级模型 | 失败后检查测试 |

详见 `ODD.10验证-ODD原生验证.md`。

## 机制总览

### L1 · 轻量
最小可用集，一个人就能跑起来。
- 契约 = 验收条件 + 边界用例。
- 状态机 = 4 个状态（pending → in_progress → review → done）。
- 门禁 = 自动测试通过。
- 证据 = 测试报告。
- 封存 = 产出物归档。
- 依赖 = 手动记录 depends_on。

### L2 · 标准
团队常规工作流。
- 契约增加质量评分、语义检查。
- 状态机增加 blocked、quality_check、acceptance、返工环路。
- 门禁 = 自动检查 + 人工审查两道。
- 证据 = 结构化记录 + 审计日志。
- 封存 = 产出物 + 证据集合绑定。
- 依赖 = 系统维护 DAG，自动标记 stale。
- 上下文 = 功能树查询 + token 控制。

### L3 · 严格
关键系统 / AI 多角色协作的完整体系。
- 契约经过对抗生成（CAP）。
- 状态机按风险等级动态插入门禁链（变异测试、对抗测试、交叉审查、人工审查）。
- 证据 = 不可变审计 + 证据链传递。
- 封存 = seal_hash 绑定全部 evidence_ref，状态不可逆。
- 依赖 = 级联重验 + 自动并行编排。
- 上下文 = 多层注入 + 车间知识库 + 双流记忆。
- 车间模型 = 并行执行、资源隔离、故障可恢复、交叉审核。
