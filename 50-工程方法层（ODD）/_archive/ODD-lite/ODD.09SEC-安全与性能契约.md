# 安全与性能契约

## 意向
安全和性能不是“做完再说”的事，它们是契约的一部分。
写在契约里，就能通过门禁来验证；不写，就永远没人查。

## 规范

### 安全原则
- 敏感字段 MUST 脱敏（输出过滤）。
- 长输出 MUST 证据化（evidence_ref），不得明文流入上下文。
- 安全规则属于上下文硬边界，MUST 强制注入。

### 性能原则
- 性能要求 SHOULD 写在契约中，用可测试的指标表达。
- 性能验证 SHOULD 通过门禁（测试/监控）实施。

---

## 机制

### L1 · 轻量

**安全**：基本意识即可。
- 不硬编码密码/密钥。
- 输入做基本校验。
- SQL 用参数化查询。

**性能**：无正式性能契约。“能用就行”。

---

### L2 · 标准

**安全分层**：
- **API 层**：输入格式校验 + 认证（JWT/API Key）。
- **管道层**：授权检查（RBAC）+ 业务前置条件。
- **DB 层**：约束（CHECK/FK/触发器）作为最后防线。

**性能契约**：写在契约中，用可测试的指标。
```yaml
performance:
  latency:
    p50: 100ms
    p95: 500ms
    p99: 1s
  throughput:
    target: 100 req/s
  timeout: 5s
  resources:
    max_memory: 512MB
    max_rows_scanned: 10000
```

**性能验证**：通过性能测试作为门禁证据。

---

### L3 · 严格

**安全增强**：
- 增加 RLS（行级安全）。
- 敏感数据脱敏策略写入契约。
- 安全相关产出物自动升级 task_level（触发更严格的门禁链）。

**性能增强**：
- 基准测试（benchmark）作为封存证据的一部分。
- 运行时监控持续验证性能契约。
- 定义降级策略（超时/过载时的回退方案）。
```yaml
performance:
  # ...L2 字段...
  benchmark:
    dataset_size: "100万行"
    expected_latency_p95: 200ms
  degradation:
    strategy: "circuit_breaker"
    fallback: "返回缓存结果"
    threshold: "p99 > 2s 连续 5 次"
```

---

## 实践

### 快速选型
- **小项目** → L1，基本安全意识，无性能契约。
- **团队项目** → L2，三层安全 + 可测试性能指标。
- **关键系统** → L3，RLS + 基准测试 + 降级策略。

### 核心原则
- **安全写进契约**：不写就没人查。
- **性能可测试**：“快”不是指标，“p95 < 500ms”才是。
- **DB 是最后防线**：上层可以有 bug，DB 约束保证数据不会坏。
