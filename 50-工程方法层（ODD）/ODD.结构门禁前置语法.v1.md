# ODD.结构门禁前置语法 v1

> **定位**：本文把 `ODD` 进入契约、验证、放行、封存与回滚之前，应接收的 `ASTO` 前置结构语法压成最小门前协议。  
> **用途**：防止 `ODD` 被误读成只处理工程执行，而没有上游结构成熟度与节律判断。

---

## 一、为什么 ODD 需要门前语法

`ODD` 负责契约、验证、证据、封存与回滚。

但工程门禁不是抽象发生的。

在进入 `ODD` 之前，
至少要知道三件事：

1. 当前对象是否已经成熟到可被 `物化`
2. 当前节律是否允许大规模放行
3. 当前动作位于 `设计 / 物化 / 回溯 / 消解` 的哪一步

这正是 `ASTO` 的桥接职责。

---

## 二、最小前置字段

| 字段 | ODD 用途 |
|:---|:---|
| `state_form` | 判断是否适合进入正式契约与门禁 |
| `dynamic_stage` | 判断当前适合扩放、限试还是冻结 |
| `action_step` | 判断当前证据重点应放在设计、实现还是回溯 |
| `boundary_status` | 判断接口是否清晰、不可绕过 |
| `exception_flag` | 判断是否必须保留人工红灯 |
| `reversibility` | 判断回滚成本与召回纪律 |

---

## 三、最小映射规则

### 1. `state_form -> 门禁成熟度`

- `自在 / 共识`：不得直接外放为正式高影响产出物
- `编码`：可进入规格设计与沙箱验证
- `物化`：可进入严格门禁、留痕与条件放行
- `定向`：应额外检查路径依赖、规模外部化与默认扩张

### 2. `dynamic_stage -> 放行节律`

- `秩序`：允许按正常门禁推进
- `流变`：要求短窗口复审与更密集回溯
- `脉冲 / 崩解`：默认限试、冻结或回滚

### 3. `action_step -> 证据重点`

- `设计`：看规格、边界、责任接口
- `物化`：看验证、门禁、不可绕过性
- `回溯`：看日志、证据封存、召回与修复
- `消解`：看退役、交接、历史追溯能力

## 四、最小输出卡

```text
对象：
state_form：
dynamic_stage：
action_step：
boundary_status：
exception_flag：
reversibility：

ODD 默认门禁：
- ALLOW / LIMITED_PILOT / FREEZE / ROLLBACK
```

## 五、最短句

> **`ODD` 不是只在工程末端接锅；它在门前就需要一套结构语法，回答“这个东西现在到底适不适合被物化和外放”。**
