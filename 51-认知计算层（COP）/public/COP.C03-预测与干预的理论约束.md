# COP.C03 · 预测与干预的理论约束

> 定位：说明 COP 从"诊断"到"预测"再到"干预"的升级路径——不是功能 roadmap，而是每一步的结构性约束条件。
> 作者：Yi Fu（付毅，ODDFounder）

---

## 一、诊断、预测、干预的区别

这三个概念经常被混用。COP 对它们做严格区分：

| 概念 | 回答的问题 | 时间指向 | COP 的职责 |
|------|-----------|---------|-----------|
| **诊断** | 当前是什么状态？ | 现在 | 判定分流（P3） |
| **预测** | 如果不干预，会怎样？ | 未来 | 基于诊断结论的趋势推测（P3 + P5 反馈） |
| **干预** | 我们应该做什么？ | 从现在到未来 | 转介或有限建议（P4），不得越权 |

---

## 二、预测的结构性约束

### 约束 1：预测必须基于诊断，不能跳过诊断

不能从原始信号直接跳到"三个月后会出问题"——中间必须经过 P1→P2→P3。跳过诊断的预测等于跳过证据链的判决。

### 约束 2：预测必须标注假设

"基于当前信号，如果结构不改变，趋势指向 X"——这是有假设的预测。
"三个月后会出问题"——这是伪装成预测的断言。

COP 的预测默认格式：

```yaml
prediction:
  based_on_diagnosis: ""      # 必须引用一个诊断结论
  assumes_no_intervention: true
  trend_direction: ""         # 恶化/稳定/改善
  time_horizon: ""            # 预测的时间窗口
  confidence: 0.0-1.0
  key_uncertainties: []       # 最关键的不确定性
  invalidated_if: ""          # 什么信号会推翻这个预测
```

### 约束 3：预测置信度不得高于诊断置信度

如果诊断本身的置信度是 0.55（MIXED），预测的置信度不能超过 0.55——因为预测建立在诊断之上。一条建立在沙地上的预测不能说"我很有把握"。

---

## 三、干预的越界防护

COP 从诊断到干预有一个不可跨越的鸿沟：

> **COP 诊断 + 预测 → 建议。COP 不得 → 执行。**

具体来说：

| COP 可以做 | COP 不可以做 |
|-----------|-------------|
| 建议转介到 TAT | 自己启动 TAT 的责任裁决 |
| 建议 FREEZE | 自己执行 FREEZE（那是 ODD/Harness 的事） |
| 建议补充什么类型的信号 | 自己生成补充信号或替代缺失信息 |
| 基于趋势给出"建议关注" | 给出"必须执行"的强制动作 |

## 四、预测失败的处理

预测失败不是系统错误——它是 P5 校准的最重要输入。

### 预测失败的三种类型

| 类型 | 含义 | 校准动作 |
|------|------|---------|
| 方向错 | 预测恶化，实际改善（或反之） | 检查编码是否漏了关键信号 |
| 时间错 | 方向对了，但发生得比预测的快/慢很多 | 调整时间窗口参数 |
| 假设错 | 预测的假设被打破（如"如果不干预"但实际发生了干预） | 不算预测失败——标注为"假设变更" |

### 纪律

预测失败不得被隐藏。每次预测失败都是一次 P5 校准机会——它揭示了诊断协议的哪一部分在当前信号模式下存在盲区。
