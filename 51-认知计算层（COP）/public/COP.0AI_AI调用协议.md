## 一、一句话定义

> COP（承接协议理论）：通过触发用户的认知相变——将其对自身处境的解释权从旧框架迁移到新框架——使有效用户主动完成入泊的理论与操作体系。

---

## 二、适用条件

以下三个条件必须同时成立，COP才可启动：

```
条件A：激活主体
  定义：目标用户已存在属集张力（现有认知框架无法解释其正在经历的现实）
  判断标志：用户表现出困惑、卡住、持续的不对劲感、反复失败的自我解释尝试
  不满足时的处理：上游灯塔层问题，COP不介入

条件B：承接锚点
  定义：承接方能提供一个真实的新框架，用户能在其中定位自己
  判断标志：新框架能解释用户旧框架无法解释的经历
  不满足时的处理：新框架无效，COP执行后入泊失败

条件C：跨越动作
  定义：目标是用户主动迁移，非被推入或被诱导
  判断标志：设计意图是触发用户的主动行为，而非推销
  不满足时的处理：可执行，但入泊后信任基础薄弱
```

## 三、核心变量（完全枚举）

### 变量1：张力类型

用户属集张力的来源方向。从以下三项中选择一项或多项：

```
TYPE_1：外部压力型
  特征：压力来自外部结构，用户是被动承压方
  识别语言：用户说"我没做错什么，但结果就是这样"
  触发器方向：揭示用户无法控制的外部机制

TYPE_2：内部矛盾型
  特征：用户内部存在两个互相冲突的价值观或目标
  识别语言：用户说"我自己也不知道该怎么做才对"
  触发器方向：命名两个冲突的属集，帮用户区分

TYPE_3：认知缺口型
  特征：用户的解释能力出现断裂，无法理解发生了什么
  识别语言：用户说"我完全不明白为什么会这样"
  触发器方向：直接提供解释框架，填入认知空白

多类型并存：选主要类型作为主触发方向，次要类型作为辅助
```

### 变量2：旧框架形态

用户当前使用的解释框架的责任归因方向。从以下三项中选一项：

```
FRAME_A：归因于自己
  特征：用户把困境原因指向自身（能力、性格、努力程度）
  识别语言："是我的问题"、"是我不够好"、"我需要提升"
  新框架进入：把责任从个人移回结构，不否定其付出

FRAME_B：归因于他人
  特征：用户把困境原因指向特定人或群体
  识别语言："是他们的问题"、"他们不对"
  新框架进入：揭示双方都被困在同一结构中，没有真正的加害者

FRAME_C：归因于环境
  特征：用户把困境原因指向运气、时机、大环境
  识别语言："时候不对"、"运气差"、"大环境不好"
  新框架进入：指出环境是表层，结构性原因在环境之下
```

### 变量3：命名缺口

用户困境被命名的程度。从以下四级中选一级：

```
GAP_4：完全未命名
  特征：用户感受到困境但完全无法描述
  触发力：最强
  操作：直接命名，停在命名处，不解释

GAP_3：模糊命名
  特征：用户有粗略标签，但不精准
  触发力：强
  操作：精准化命名，替代模糊标签

GAP_2：已被命名
  特征：用户已有清晰标签，且在其群体中普遍流通
  触发力：弱
  操作：提供更深层命名，打破已有标签

GAP_1：过度命名
  特征：市场上充斥大量同类命名，用户已免疫
  触发力：极弱
  操作：反向操作——拆解现有命名，然后重建

优先选择：GAP_4或GAP_3，触发力最强
```

## 四、五节点触发器序列

触发器必须按以下顺序执行，不可跳过，不可乱序：

```
NODE_1：场景还原
  输入：用户经历过的具体场景（含时间/地点/动作）
  操作：用具体细节描述那个场景，不使用概念性词语
  输出：用户被拉入情境，张力从休眠变为激活
  质量标准：描述是否让用户能想起一个具体的时间、地点和他当时的身体姿势
  禁止：泛化描述（"职场人经常感到委屈"→禁止）

NODE_2：行为命名
  输入：用户在该场景中做了什么
  操作：客观还原用户的具体应对动作，不加任何判断
  输出：用户感到"你看到了我的行为"
  质量标准：只描述动作，不含"对""错""应该"等判断词
  禁止：任何形式的批判或赞扬

NODE_3：结构揭示（核心节点）
  输入：新框架 + 用户的旧框架形态
  操作：用新框架解释用户的处境，同时将责任归因从用户（或他人/环境）移回结构
  输出：用户旧框架被替代，解释权开始迁移
  质量标准：用户能用这个解释理解"为什么换个人来，结果还是一样"
  禁止：直接否定旧框架（"你之前的理解是错的"→禁止）

NODE_4：串联验证
  输入：新框架 + 用户过去的其他经历
  操作：引导用户用新框架重新解释过去多个经历（不需要逐一列举，引导其自行搜索）
  输出：用户主动在记忆中搜索，发现新框架可以解释多个困惑
  质量标准：用户开始自发说出"还有那次……也是这样"
  禁止：替用户列举经历（应引导，不应代劳）

NODE_5：命名锚定
  输入：新框架的名称
  操作：给新框架一个简洁的名字，然后停住
  输出：用户带着"我要知道这个名字意味着什么"的主动性走进来
  质量标准：停住之后不再解释，让用户主动深入
  禁止：在命名之后继续解释框架内容（会破坏入泊的主动性）
```

## 五、两阶段共鸣判断

```
STAGE_1：冷共鸣（认出自己）
  触发条件：NODE_1的细节精度足够
  用户信号：停止快速扫描，注意力集中
  性质：必要条件，不充分条件
  判断：用户是否能从"一类人"变成"这个人"

STAGE_2：热共鸣（被理解）
  触发条件：NODE_2不含批判，NODE_3解除了错误归因
  用户信号：情感反应（沉默、重新阅读、转发给特定的人）
  性质：相变发生的时刻
  判断：用户是否感到"终于有人懂了"而非只是"说得对"

两阶段均需达成，相变才算完成
```

## 六、相变可观测信号

用于判断入泊是否完成：

```
信号_强（入泊高度可能）：
  - 用户主动转发给特定的人并加上"这说的就是我"
  - 用户在评论区描述自己的具体经历
  - 用户主动追问"这种情况怎么解决"
  - 用户主动向下深入浏览（落地页滚动至底部）

信号_中（入泊可能）：
  - 用户主动收藏
  - 用户停留时长超出预期
  - 用户点开相关链接

信号_弱（不代表入泊）：
  - 泛点赞（社交行为）
  - 礼貌性评论"说得好"
  - 短暂停留后离开
```

## 七、典型失败模式

```
FAIL_1：碰不到
  表现：流量到达但有效用户比例极低
  根本原因：入口触发了好奇心而非痛点识别
  修复：改写入口，使用陈述句结构而非问句或标题党

FAIL_2：碰到不停
  表现：有效用户到达但快速离开，停留时间极短
  根本原因：描述泛化，指向一类人而非这个人
  修复：增加细节精度（添加时间、地点、身体动作等具体信息）

FAIL_3：停了不共鸣
  表现：用户停留但没有主动深入行为
  根本原因A：只触发了冷共鸣，NODE_3未完成结构揭示
  根本原因B：NODE_2或NODE_3中含有隐性批判，防御被触发
  修复A：检查NODE_3是否真正完成了旧框架替换
  修复B：检查文本中是否含有"应该""不对""错误"等评判词

FAIL_4：共鸣了不入泊
  表现：用户反应强烈但没有主动深入行为
  根本原因：NODE_5之后继续解释，破坏了用户的主动性
  修复：NODE_5后严格停住，不再输出任何解释内容
```

## 八、不适用场景（硬边界）

以下场景COP不适用，强行应用将失效：

```
BOUNDARY_1：用户没有属集张力
  描述：用户处于满足状态，旧框架尚能解释现实
  判断：用户没有表现出任何困惑或卡住的信号
  处理：不适用COP，回到灯塔层重新筛选

BOUNDARY_2：用户解法开放度极低
  描述：用户相信没有任何外部方案能帮助自己
  判断：用户完成冷共鸣后立刻拒绝（"说的是我，但这没用"）
  处理：不适用COP，应在灯塔层过滤

BOUNDARY_3：承接方没有真实新框架
  描述：承接方提供的是口号、安慰或空洞承诺，而非真实框架
  判断：新框架无法解释"换个人来结果还是一样"
  处理：不适用COP，应先完善新框架

BOUNDARY_4：强制推入场景
  描述：设计意图是将用户强制推进系统，而非触发主动迁移
  判断：存在硬性CTA、强制弹窗、限时强迫等设计
  处理：COP可执行，但入泊后信任极低，建议重新设计
```

## 九、调用示例

**示例输入：**
```
场景：知乎长文承接《五行察局》产品
用户画像：职场夹心层，有被背锅经历，归因于自己能力不足
张力类型：TYPE_1（外部压力型）+ TYPE_3（认知缺口型）
旧框架形态：FRAME_A（归因于自己）
命名缺口：GAP_4（完全未命名）
```

**对应触发器方向：**
```
NODE_1：还原一个具体的被背锅场景
  → 不是"你曾经被背锅"，而是"那次会议结束后，你走楼梯下去，没等电梯"

NODE_2：还原用户当时的选择
  → "你选择了不解释"（客观，不含判断）

NODE_3：结构揭示
  → "那不是你能力的问题。那个位置上，解释和不解释，结果是一样的。
      这是一个结构性问题，不是你个人的问题。"

NODE_4：串联验证
  → "你想想，不只是那一次。你能找到三次吗？同样的结构，不同的时间。"

NODE_5：命名锚定
  → "这个结构有一个名字。"（停住）
```
