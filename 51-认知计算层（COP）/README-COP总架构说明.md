# COP 认知操作系统 · 总架构说明

> **原始文件**：`认知操作系统协议（Cognitive OS Protocol, COP）.md`（已归档于本目录根）
>
> **核心身份**：主链理论接触复杂现实后的 `诊断压缩 + 风险分流 + 转介协议层`
>
> **与产品的关系**：`asto.cc/diag`、`goodmem.cn/009` 可以调用 COP，但前台产品不等于 COP 本身
>
> **边界**：COP 不单独完成治理合法性裁决，不绕过 `TAT / ODD` 直接触发强制性高风险干预

---

## 一句话定位

> **COP 不是孤立问卷，也不是总能回答的自动判断器，而是一套有界的结构诊断、状态编码、风险分流与转介协议。**

---

## 固定边界

1. `DM` 提供本体地基与最小结构承诺，COP 不重写 DM。
2. `ASTO` 提供桥接语法与状态语言，COP 不重新发明上游语言。
3. `ECET` 处理治理正当性与文明尺度判断，COP 不接管这类裁决。
4. `TAT` 处理责任门槛、同意、冻结、申诉、补偿与责任承接，COP 不能越权替代。
5. `ODD` 处理工程实现、证据留存、版本审计与回滚，COP 只提出协议要求。
6. `LMM` 处理前线显影、接引与现实承接，COP 不等于营销或产品战役手册。

COP 当前只新增六类东西：

- 输入采样
- 状态编码
- 结构诊断
- 风险分流
- 有界预测
- 有条件干预建议

因此：

> `COP` 的正式分流前，建议先补一张 `ASTO` 结构编码卡，再给出 `RESOLVE / MIXED / FREEZE / UNKNOWN / REFER`。

## DM 风险防火墙

### 最小依赖

即使不先接受 `DM` 的强版本，`COP` 至少仍可建立在以下弱前提上：

1. 现实诊断会遇到 `未知`
2. 未知不能被伪装成已知
3. 误判具有不对称成本
4. 高风险诊断必须允许 `冻结 / 转介 / 复核`

### `DM` 受争议时如何阅读本层

若 `DM` 的强本体论或强分类学解释受争议，
`COP` 当前最稳的降级阅读方式是：

- 先把它读成诊断分流协议
- 先把它读成误判成本管理系统
- 不把它继续写成只有接受 `DM` 才能成立的认知本体

### 本层可独立成立的部分

- 输入采样
- 状态编码
- 结构诊断
- `FREEZE / UNKNOWN / REFER` 分流状态
- 校准、误判、版本回滚纪律

### 本层仍强依赖 `DM` 的部分

- 把当前分类学写成唯一正确的本体划分
- 用强形而上学解释替代协议输出与误判审计的写法

### 本层防火墙句

> **即使 `DM` 受压，`COP` 仍可作为有界的诊断分流协议成立；本层首先由能否承认未知、控制误判、及时冻结与转介来站住。**

当前 `COP` 不只是一份主协议，

还已配套三份正式工具文：

- `COP.校准样本协议.v1.md`

它们分别负责：

- 什么样的样本有资格进入校准池
- 误判如何留档、归因与回看
- 协议、权重与阈值更新如何发布、冻结与回滚

因此：

1. `COP.诊断协议.v1.md` 负责说清主合同
2. 这三份工具文负责把主合同压成日常维护纪律

### 当前场景适配包说明

当前 `Phase 1` 里使用的：

- `A-D` 四类型
- `Q1-Q9` 九题问卷

只代表一条已经压出来的场景适配包，

不代表 `COP` 的全部状态本体。

更稳的理解是：

1. `COP` 先是诊断协议层
2. `A-D + 9题` 是当前可上线的首个采样包
3. 后续若换场景，可以更换题组、状态标签与权重矩阵，
   但仍应服从同一套 `输入 -> 输出 -> 冻结 -> 转介 -> 学习 -> 回滚` 协议
4. 若要进入正式高风险分流，建议同步保留 `ASTO` 的 `五态 / 六阶 / 七序 / 边界 / 责任流向模式` 前置字段

## 输出合同

从本轮起，COP 的最小标准输出不再是“一个类型 + 一个风险值”，而至少包含以下字段：

1. `primary_type`
当前最强候选类型；若进入 `FREEZE / UNKNOWN`，允许为空。

2. `secondary_type`
当前第二候选类型；只在存在实质竞争时保留。

3. `primary_break_order`
当前主断裂行动位点；若场景包不使用七序，可为空。

4. `secondary_break_order`
当前次断裂行动位点；只在存在实质竞争时保留。

5. `evidence_gap`
当前判断仍缺失的关键证据或样本。

6. `classification_confidence`
分类集中度，回答“当前分类有多集中”，不再冒充风险等级。

7. `structural_risk`
结构风险等级，回答“当前状态有多危险”，不再和分类集中度混用。

8. `triage_status`
当前分流状态，固定使用：

- `RESOLVE`
- `MIXED`
- `FREEZE`
- `UNKNOWN`
- `REFER`

9. `tags`
本次诊断命中的关键结构标签，如 `结构循环依赖`、`信息失真`、`无阈值控制`。

10. `refer_to`
当结果不应自动闭环时，明确回引：

- `HUMAN_REVIEW`
- `TAT_REVIEW`
- `ODD_AUDIT`
- `ECET_ESCALATION`

最短判断：

1. `分类集中度` 不是 `结构风险`。
2. `高风险` 不等于 `高置信度`。
3. `主断裂位点` 不等于 `责任终裁`。
4. `诊断协议必须允许自己说“我不知道”或“我现在不该继续自动判断”。`

## 五种分流状态

### `RESOLVE`

当前证据足够、候选类型清晰、风险不需要立即上提，允许给出有限动作建议。

### `MIXED`

两个或更多候选类型高度接近，允许输出候选组合与差异标签，但不应强压成单一主型。

### `FREEZE`

当前冲突过高、硬规则互撞、或高风险与低可分辨性并存。
此时应暂停自动处置，只输出冻结、补采样与升级动作。

### `UNKNOWN`

当前输入不足、激活信号过低、或超出模型覆盖域。
此时不得伪装成已有明确判断。

### `REFER`

当前风险、责任或工程影响已越过自动协议的安全边界。
此时可以保留候选类型，但默认转人工、转 TAT 或转 ODD。

## 协议层与产品适配层

COP 当前至少可以被两个前台调用：

| | `goodmem.cn/009` | `asto.cc/diag` |
|:---|:---|:---|
| 目标对象 | 个人困境场景 | 组织诊断场景 |
| 包装语言 | 个体任务与认知阻滞 | 组织摩擦与决策失效 |
| 前台 CTA | 阅读、学习、导引 | 诊断、转介、责任承接 |
| 后端引擎 | 共用 COP 核心协议 | 共用 COP 核心协议 |

但这里必须区分：

1. `产品包装` 不是 `协议定义`
2. `前台 CTA` 不是 `诊断输出字段`
3. `商业漏斗` 不是 `COP 的学术或工程主位`

---

## 当前 Phase 1 的正确目标

当前不应再把 `V1-V3` 描述为“整个 COP 已经完成”，而应固定为：

> **一套可上线、但必须允许冻结、未知、混合与转介的诊断协议 MVP。**

Phase 1 至少要做到：

1. 输出 `classification_confidence` 与 `structural_risk` 两条不同变量。
2. 输出 `triage_status`，并允许 `FREEZE / UNKNOWN / MIXED / REFER`。
3. 所有 `HIGH structural_risk` 的结果默认带 `HUMAN_REVIEW`，必要时带 `TAT_REVIEW / ODD_AUDIT`。
4. 未单独建立估算模块前，不把 `失血量 / 摩擦税` 写成默认必出字段。

## 最短压缩句

1. `COP 的价值，不在于它总能回答，而在于它知道何时该暂停自动回答。`
