## 一、前台合法链条映射

LMM 当前主线的最小合法链条是：

`择场 -> 定客 -> 配品 -> 灯塔内容 -> 自测 -> 港口承接 -> 责任闭合`

对应的实现类型如下：

| 链条阶段 | 理论法源 | 对应实现类型 | 核心作用 |
|:---|:---|:---|:---|
| **择场** | 支柱③ 相变机制 + 02 场路人动力学 | 场域记分卡、蓝海探测卡、风险筛选表 | 判断哪里值得点灯，哪里不应误入。 |
| **定客** | 支柱② 网树降维 | 人群切片、角色判别表、守门角色识别卡 | 锁定谁是真正承受损耗的人。 |
| **配品** | 支柱② 网树降维 + 支柱① 责任边界 | 解决路径模板、范围定义表、承接能力校核表 | 确定能给出的不是“万金油”，而是有边界的单步解。 |
| **灯塔内容** | 支柱③ 相变机制 | 四拍扰动内容、断口显影短文、问题命名页 | 让真实损耗第一次进入目标注意力系统。 |
| **自测** | 支柱③ 相变机制 + 协议层弱测量 | 自测表、样本提交页、轻诊断入口、预约动作 | 进行弱测量，筛出已进入流变阶者。 |
| **港口承接** | 支柱① 责任闭合边界 | 首诊入口、确认单、责任附件、范围锁定说明 | 让已靠近者能够正式靠岸，而非继续漂浮。 |
| **责任闭合** | 支柱① 责任闭合边界 | 承接协议、失败判定、回流条款、必要时 R-Eval | 明确谁负责、负责到哪里、如何验证偏航。 |

## 三、协议层到实现层的映射

`LMM.理论.007` 已将协议层明确为：

`P1 显影 -> P2 弱测量 -> P3 单步引导 -> P4 港口承接`

其在实现层中的合法落点如下：

| 协议层 | 对应实现类型 | 不可越位事项 |
|:---|:---|:---|
| **P1 定向扰动脉冲** | 灯塔文章、问题命名页、结构损耗短视频、断口图示 | 不得在 P1 直接塞报价、确认单或强成交按钮。 |
| **P2 弱测量探针** | 自测表、样本提交、日志上传、低阻预约 | 不得把自测伪装成销售表单。 |
| **P3 单步引导矢量** | 单一 CTA、单一路径说明、唯一下一步入口 | 不得同时给多个港口、多套方案并行分流。 |
| **P4 港口承接与责任闭合** | 首诊沟通、确认单、责任附件、R-Eval、报价说明 | 不得在未出现物理接近动作时提前启动。 |

## 四、几个高频实现类型的法源解释

### 1. 为什么“自测”必须独立成层？

- **法源依据**：支柱③ 相变机制 + 协议层 P2 弱测量。
- **逻辑**：主体在完成初步显影后，仍需一个低压、可回退、可自识别的过滤层。若跳过这一步，前台会把未成熟主体误送入港口，导致承接噪音急剧上升。

### 2. 为什么“港口承接”不能前置？

- **法源依据**：支柱① 责任闭合边界。
- **逻辑**：港口不是任何流量的收集器，而是对已完成自识别、已产生物理接近动作的主体开放的靠岸接口。前置港口，会把承接器退化成推销器。

### 3. 为什么“确认单/责任附件”仍是核心文书？

- **法源依据**：支柱① 责任闭合边界。
- **逻辑**：没有边界确认，就没有真正的 R=1。确认单的作用不是制造仪式，而是把责任范围、失败判定与退出机制写成可核验的物理界面。

### 4. 为什么 `R-Eval` 不能泛化成普遍动作？

- **法源依据**：支柱① 责任闭合边界。
- **逻辑**：`R-Eval` 是高风险、高争议、高客单承接中的客观校验器。其作用是降低争议，不是制造额外门槛。若在所有情形强制上 `R-Eval`，会把承接系统拖入过度摩擦。

### 5. 为什么灯塔内容不能退化成“服务介绍页”？

- **法源依据**：支柱③ 相变机制。
- **逻辑**：灯塔的任务是结构显影，而不是产品宣传。它首先要让主体看见问题，而不是先看见卖方。
