# LMM.产品.011 — 责任滑逸CT首诊模板与责任接驳桩边界

> **版本**：v1.1 | 日期：2026-04-16
> **属性**：04.产品打造 — `LMM.产品.007` 的单病种正式承接总门
> **定位**：把“组织责任滑逸”压成可提交、可复核、可分流的正式首诊模板，并吸收第一段接驳桩边界与自动化最小架构
> **关系**：主规格见 `LMM.产品.007`

---

## 一、一句话定位

> **责任滑逸CT不是总咨询，而是把“责任正在从哪里滑走”做成一张可复核的结构病历。**

---

## 二、单病种定义

`组织责任滑逸` 指：

> 在高影响协作链中，决策、执行与后果承担错位，导致任务继续推进，但责任未被稳定承接，组织靠返工、补位、澄清与事后追责维持运转。

它不同于一般低效。

低效可以只是慢、杂、重；

责任滑逸一定包含：

- 名义负责与实际承后果脱钩
- 关键节点没有稳定叫停权、预算否决权或回滚权
- 失败后责任链难以回溯

---

## 三、适用与禁入

### 适用

- 组织规模 `30+`
- 存在多角色交接或多头汇报
- 近期已出现返工、赔付、升级滞后或客户追责
- 决策者愿意出席并接受责任边界被挑战

### 禁入

- 只想拿背书，不接受边界澄清
- 当前问题核心是诉讼、刑责或重大合规事件
- 决策者完全不在场，只让代理人承担判断

---

## 四、首诊第一页固定字段

以下字段必须在第一页收齐：

| 字段 | 说明 | 最低纪律 |
|:---|:---|:---|
| `case_title` | 本次最危险动作链名称 | 不能写成泛泛主题 |
| `case_scope` | 本次只看哪一条链路 | 必须单链，不接总诊断 |
| `decision_owner` | 谁在做关键决定 | 需写角色与姓名 |
| `execution_owner` | 谁在推动执行 | 需写角色与姓名 |
| `consequence_owner` | 谁在承担失败后果 | 若为空，默认高风险 |
| `latest_break_event` | 最近一次掉球/返工/赔付/追责事件 | 必须写事实，不写感受 |
| `stop_right_status` | 是否存在可独立触发的叫停/否决/回滚权 | 不得空缺 |
| `loss_signal` | 当前可见损耗 | 至少写工时、金额、延迟、客户影响之一 |
| `artifact_status` | 是否已有责任图/流程图/合同条款/复盘记录 | 不得用“大家都知道”替代 |
| `in_scope` | 本次纳入范围 | 必填 |
| `out_of_scope` | 本次不纳入范围 | 必填 |

---

## 五、首诊第一页核心问题

前台固定提这八问：

1. 当前最危险的动作链是什么？
2. 谁在做决定，谁在执行，谁承担最终后果？
3. 最近一次滑逸事件具体发生在哪里？
4. 当时谁有权叫停、否决预算或触发回滚？
5. 现在已经发生了哪些损耗？
6. 这些责任边界写进了哪些文档或系统？
7. 上一次类似问题是谁补位、谁背锅、谁没有留下痕迹？
8. 本次你只希望我们看清什么，不希望我们越过什么？

---

## 六、材料包最低要求

正式首诊前，至少补以下材料中的三类：

1. 责任分工文档、合同、交付里程碑或流程图
2. 最近一次异常、返工、赔付、客户投诉或升级记录
3. 会议纪要、工单、邮件、聊天记录中的责任指令痕迹
4. 复盘、申诉、回滚或补偿记录
5. 损耗测算基础数据

若材料长期补不齐，

默认不进入强判断，

直接走 `triage_status = UNKNOWN / REFER`。

---

## 七、标准输出模板

每次 `责任滑逸CT` 至少输出以下字段：

| 字段 | 含义 |
|:---|:---|
| `diagnostic_summary` | 一句话说明滑逸发生在何处 |
| `responsibility_breakpoint` | 最危险责任断点 |
| `primary_break_order` | 主断裂七序 |
| `secondary_break_order` | 次断裂七序 |
| `seven_order_view` | 七序断裂说明 |
| `loss_hours` | 已观察到的工时损耗 |
| `loss_cost` | 已观察到的金额损耗 |
| `frequency` | 滑逸重复频次 |
| `scope_affected` | 受影响团队/客户/链路 |
| `recovery_time` | 每次补救需要付出的恢复成本 |
| `evidence_index` | 已采到的证据索引 |
| `evidence_gap` | 仍缺的关键证据 |
| `triage_status` | `RESOLVE / MIXED / FREEZE / UNKNOWN / REFER` |
| `prohibited_actions` | 当前禁止继续做的动作 |
| `next_safe_actions` | 当前只允许做的安全动作 |
| `upgrade_trigger` | 何时必须升到 `TAT / ODD` |

### 推荐输出版式

```text
责任滑逸CT首诊结果

1. diagnostic_summary:
2. responsibility_breakpoint:
3. primary_break_order / secondary_break_order:
4. seven_order_view:
5. loss_hours / loss_cost / frequency / scope_affected / recovery_time:
6. evidence_index:
7. evidence_gap:
8. triage_status:
9. prohibited_actions:
10. next_safe_actions:
11. upgrade_trigger:
```

## 八、七序断裂的固定读法

| 七序 | 首诊问法 | 常见滑逸表现 |
|:---|:---|:---|
| `感知` | 谁首先看见问题？ | 只见汇报，不见现场 |
| `解析` | 谁能把责任链说清？ | 人人有理，没人指出主断点 |
| `干预` | 谁能叫停或升级？ | 只会催办，不会止血 |
| `设计` | 谁能重画边界？ | 知道有问题，但接口不改 |
| `物化` | 谁把边界写进去？ | 决定停在会议里 |
| `回溯` | 谁留下证据与误差？ | 事后靠叙事洗白 |
| `消解` | 谁退旧机制？ | 老制度永不下线 |

## 九、分流纪律

### `RESOLVE`

证据够、断点清、风险可控。

允许给出第一段闭环建议，

但仍不替代处方件。

### `MIXED`

存在两个以上竞争断点或七序断裂位。

允许并列给出候选，

不得强压单一病灶。

### `FREEZE`

已触及高风险动作、责任争议或不可逆后果，

但当前证据不足。

默认暂停继续推进。

### `UNKNOWN`

材料不足或信号过弱。

不得伪装成已看清。

### `REFER`

一旦碰到以下情形，必须上提：

- 高影响责任裁决
- 叫停权、否决权、回滚权争议
- 合同、制度、流程、代码级物化
- 补偿、申诉、冻结、召回

## 十、最短句

> **首诊第一页只做一件事：把“谁在决定、谁在执行、谁在承后果”钉到同一页上；一旦钉不上，就先承认责任在滑。**

## 十一、责任接驳桩边界

以下内容吸收原 `LMM.产品.013`，作为本文件的第二部分。

它不再单独占一个产品真本，

而作为 `责任滑逸CT` 之后的第一段有界处方前门。

### 11.1 一句话定位

> **责任接驳桩不是总改造，而是给最危险责任断链先立一根不再继续滑的桩。**

### 11.2 本产品必须做什么

1. 只接 `责任滑逸CT` 已确认的首要断点
2. 明确第一责任承接人是谁
3. 明确其是否拥有叫停权、预算否决权、回滚触发权
4. 明确第一段闭环的触发条件、禁止动作、升级条件
5. 把新边界最小物化到文档、流程、权限、审计对象或系统字段中
6. 给出复审入口与回滚接口

### 11.3 本产品明确不做什么

1. 不承诺全量组织改革
2. 不替代 CEO 全面重画组织架构
3. 不替代 `TAT` 做责任终裁、补偿、申诉与冻结授权
4. 不替代 `ODD` 完成正式契约编译、验证、封存与回滚
5. 不把一段接驳件伪装成企业长期治理总方案

### 11.4 何时必须升级到 `TAT / ODD`

出现以下任一情形，`责任接驳桩` 必须停止单独推进：

1. 责任承接人与实际权力错位
2. 需要冻结、申诉、补偿、召回或正式否决
3. 需要把新边界写入合同、制度、代码、审计流程
4. 继续推进将造成高不可逆损害
5. `triage_status` 为 `FREEZE / REFER`

最短纪律：

> **凡是碰到“谁有权决定最终责任”与“如何把新边界正式写进去”，都不再只靠接驳桩本身解决。**

### 11.5 最小交付清单

每次 `责任接驳桩` 至少应交付：

| 交付物 | 用途 |
|:---|:---|
| `first_closure_target` | 第一段要补的责任闭环 |
| `responsible_owner` | 第一责任承接人 |
| `stop_right_map` | 叫停、否决、回滚权映射 |
| `boundary_patch` | 新边界最小补丁 |
| `prohibited_actions` | 接驳完成前禁止继续的动作 |
| `review_entry` | 复审入口 |
| `rollback_entry` | 回滚入口 |
| `upgrade_route` | 升级到 `TAT / ODD / Harness` 的接口 |

### 11.6 自动化最小架构

#### A. 可自动化

1. 灯塔内容页
2. 弱测量与自测问答
3. 首诊第一页采样表
4. 材料提交、校验、脱敏提醒
5. 损耗粗测、结构字段预填
6. `COP` 预编码草稿
7. 首诊报告模板生成
8. 事件日志、证据索引、版本归档

#### B. 必须人工守门

1. 责任最终确认
2. `FREEZE / REFER` 放行
3. 是否进入 `TAT / ODD`
4. 接驳件签发
5. 回滚、补偿、申诉、冻结与否决
6. 高影响收费与正式承接确认

#### C. 最小流水线

```text
灯塔显影
-> 弱测量
-> 责任滑逸CT第一页
-> 材料提交
-> COP预编码
-> 人工复核
-> 责任滑逸CT报告
-> 是否进入责任接驳桩
-> TAT/ODD/Harness
```

### 11.7 网站最小页面集合

若后续做全自动前门，最小页面只需以下六类：

1. `问题显影页`
2. `弱测量自测页`
3. `责任滑逸CT第一页`
4. `材料提交页`
5. `首诊结果页`
6. `接驳申请页`

原则只有一条：

> **先把责任滑逸看清，再决定是否值得下单接驳；不允许用网站把高风险裁决自动卖出去。**

### 11.8 压缩句

> **责任滑逸CT负责看清“哪里在滑”，责任接驳桩负责立住“第一根不再滑的责任桩”。**
