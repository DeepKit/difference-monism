# 战略-005-首批样本获取路径与首发灯塔内容圆桌纪要

> **版本**：v1.0 | 日期：2026-04-03
> **性质**：专题圆桌纪要 / 工作底稿
> **状态**：老板线首发咨询补档；用于衔接 `战略-003 / 战略-004 / LMM.产品.011 / LMM.产品.013`，不构成公共法源主文

---

## 一、本轮要锁什么

本轮只回答五个问题：

1. 第一批真实样本，优先从哪里拿？
2. 第一批灯塔内容，先打什么题？
3. 第一批试点，先做哪一种单链路病灶？
4. 第一次接触老板，先怎么开口？
5. 哪些看起来像机会，实际上不该接？

---

## 二、主持人总裁决

### 1. 样本不是先从公开流量池拿，而是先从连接层交付关系里拿

本轮固定判断：

> **首批样本的最佳入口，不是泛行业撒网，而是沿着“正在交付、已经掉球、老板能感到疼”的连接层关系链切进去。**

优先顺序固定为：

| 优先级 | 样本来源 | 建议联系人 | 主要价值 | 主要风险 |
|:---|:---|:---|:---|:---|
| `P1` | `机器人 / 工业自动化系统集成` 交付链 | 项目总监、交付经理 | 最容易碰到停线、延期、返工、赔付，责任滑逸证据最直观 | 对方担心责任界定影响既有合作 |
| `P2` | `专用设备制造 / 产线升级` 实施与售后链 | 工程管理、运营保障、售后负责人 | 容易拿到升级过程中的交接失真与售后补位样本 | 客户或制造商可能要求过度技术交底 |
| `P3` | `工业软件 / 企业AI集成` 实施链 | 实施负责人、客户成功负责人 | 容易显影“系统很多，结果没闭上”的责任黑箱 | 跨系统数据合规与日志权限会拖慢节奏 |
| `P4` | `信息技术服务 / 系统实施` 咨询链 | 项目顾问、解决方案架构师 | 可作为补充样本来源，便于看到责任真空如何被流程话语掩盖 | 客户容易把项目当一般咨询，不愿给证据 |

裁决很简单：

- 首批样本优先拿 `P1`
- 第二批补 `P2`
- `P3 / P4` 作为扩展样本池，不作为第一枪

### 2. 第一批试点不能做“大组织诊断”，必须做单链路病灶

本轮固定判断：

> **首个样板案例，锁定“接口绕行”。**

原因：

1. 单链路最清楚
2. 责任断点最容易被复核
3. 不需要等全组织改革，先看一条高危输出链是否绕过审批/限额/审计即可
4. 在机器人、自动化和集成场里更容易快速拿到样板

当前首批三种单链路病灶排序如下：

1. `接口绕行`
2. `日志黑箱`
3. `停止假权`

### 3. 第一批灯塔不能讲抽象转型，只能打老板的责任别扭感

本轮固定判断：

> **灯塔内容要先把“钱花了、系统上了、结果却没人负责”的别扭感打亮，而不是先讲宏大数字化叙事。**

首发建议优先只打三题：

1. `机器人系统交付后的“没人负责”危机`
2. `专用设备制造商如何避免责任滑逸`
3. `工业软件 + 企业AI：谁在为业务落地负责？`

这三题分别对应：

- `机器人 / 自动化系统集成`
- `专用设备制造 / 产线升级`
- `工业软件 / 企业AI集成`

### 4. 第一次接触不要先卖方案，要先钉事实

首轮开场只做一件事：

> **让对方把最近一次停线、延期、返工、赔付或责任互推说出来。**

本轮固定的稳妥开场问法如下：

1. 最近一次停线、延期、返工或赔付，发生在哪条交付链？
2. 那条链上，谁在做决定，谁在执行，谁在承担最终后果？
3. 如果今天只能补一个责任空洞，你最担心先爆的是哪一段？

### 5. 不接“假机会”

以下三类对象，当前统一判定为不宜首发：

1. `只想多买系统，不想重看责任边界`
2. `只想听概念，不愿给事实和证据`
3. `一上来就想做全面组织改造，却压不出单切口`

## 三、四位专家意见摘要

### `Sagan`：先沿交付关系拿样本，不要先沿行业标签拿样本

他强调：

- 最稳的首批样本来自已在交付链中的接口人
- 优先顺序应是 `系统集成商交付 -> 专用设备实施/售后 -> 工业软件实施 -> IT 服务咨询`
- 首发样本的关键不是“公司大不大”，而是“责任滑逸是否已经显影且能拿到最小证据”

### `Dirac`：灯塔必须照见老板的别扭，不要照概念热词

他强调：

- 不能先讲 AI、数字化、组织升级这些抽象词
- 要先讲“花钱后为什么没人对结果负责”
- 第一批题目必须分别穿透自动化交付、专用设备交接、工业软件落地这三种责任滑逸高发场

### `Turing`：样板必须从单链路、单接口、单断点开始

他强调：

- `接口绕行` 最适合做首个样板
- `日志黑箱` 和 `停止假权` 适合做后续扩样
- 每个试点都必须配最小证据包，而不是先输出大报告

### `Ramanujan`：老板先说出来的，不是需求，而是抱怨

他强调：

- 老板真正会说的是：
  - `系统买了，怎么还是拖？`
  - `每个人都说自己负责，为什么事情还是掉？`
  - `钱花了很多，结果没出来，到底是哪段在漏？`
- 第一次接触不该先讲模型，而该先让这些抱怨落到一条事实链上

## 四、首批灯塔内容矩阵

| 编号 | 题目 | 对应场域 | 打中的老板别扭感 | 建议对应的 `CT` 入口 |
|:---|:---|:---|:---|:---|
| `L1` | 机器人系统交付后的“没人负责”危机 | 机器人 / 自动化集成 | 上了系统却还停线、返工、赔付 | `接口绕行` |
| `L2` | 自动化设备上线后业绩沉默的真实原因 | 自动化 / 产线升级 | 投入很多，但结果沉默 | `接口绕行 / 停止假权` |
| `L3` | 专用设备制造商如何避免责任滑逸 | 专用设备制造 | 工程、实施、售后责任互推 | `停止假权` |
| `L4` | 工业软件 + 企业AI：谁在为业务落地负责？ | 工业软件 / 企业AI集成 | 系统并在，但结果没人闭合 | `日志黑箱` |
| `L5` | 控制室看板背后的“我都负责”黑洞 | 工业软件 / 信息服务 | 数据很多，责任很空 | `日志黑箱` |
| `L6` | 钱花了很多，结果没出来：怎样用责任证据打破空转循环 | 跨场域通用 | 老板只见概念，不见证据 | `CT 首诊总入口` |

固定纪律：

- `L1 / L3 / L4` 作为首发三题
- `L2 / L5 / L6` 作为补光题

## 五、首批试点的最小证据包

### 1. `接口绕行`

最低材料：

1. 接口定义或控制链说明
2. 审批矩阵、变更记录或权限分配表
3. 实际执行日志、抓包记录或控制器事件记录
4. 审批被绕过的指令痕迹

### 2. `日志黑箱`

最低材料：

1. 数据采集规范
2. 关键输入/输出或推理链日志
3. 第三方存储、哈希或审计留痕说明
4. 一组“有记录 vs 无记录”的对照样本

### 3. `停止假权`

最低材料：

1. 停止权或叫停权分布表
2. 现场操作手册或应急流程
3. 停止指令发送到执行的时间线
4. 无响应、迟响应或弃权说明

## 六、老板线第一次接触的固定脚本

### 1. 开场目标

不是证明我们懂理论，

而是先判断：

> **这家公司是否真的存在一个能被量化、能被复核、能被单点切开的责任滑逸。**

### 2. 固定三问

1. 最近一次最让你火大的停线、延期、返工或赔付，发生在哪条链？
2. 当时谁拍板、谁执行、谁背最后结果？这三者现在是不是同一个位置？
3. 如果只准先补一个洞，哪一段一旦继续滑下去，代价最大？

### 3. 一旦出现以下回答，可继续推进

1. 能说出最近一次具体事件
2. 能指出至少两个责任角色的错位
3. 愿意提供至少两类最小证据

### 4. 一旦出现以下回答，应立即降级或跳过

1. 只谈理念，不谈事件
2. 只讲“大家都知道”，不给任何材料
3. 只能由中层接待，老板或后果承担者完全不在场

---

## 七、第一批样板的最小航线

当前最稳的咨询航线不是大项目制，

而是：

`灯塔显影 -> 首轮筛查 -> 样本申请 -> CT首诊 -> 判断是否进入接驳桩 -> 形成样板 -> 再反哺灯塔`

压缩成六步：

1. 用 `L1 / L3 / L4` 三篇灯塔内容照亮目标场域
2. 用 `战略-004` 做 `S1 / S2 / S3 / S4` 首发筛查
3. 由 `P1 / P2` 路径拿第一批样本
4. 用 `责任滑逸CT` 先只打单链路病灶
5. 若断点清楚，再判断是否值得进入 `责任接驳桩`
6. 把样板案例反写成下一轮灯塔与筛查问题
