## 0. 使用原则

本清单只判定一件事：

`该用什么承接器进入。`

这里的“产品”可以是：

- 咨询包
- 诊断服务
- 顾问包
- 软件
- 工具包
- 训练营
- 白皮书

原则只有一句：

`产品形态服从问题场，不让问题场服从既有产品。`

补充纪律：

- 配品时必须区分 `灯塔内容层 / 自测层 / 港口承接层`。
- 前台还未完成照明，就不应把港口承接器推到第一屏。
- 任何配品都应服从 `弱测量 -> 物理接近 -> 责任闭合` 的节奏。

---

## 2. 成立层检查项

### 2.1 形态成立

- [ ] `单一主任务成立`：产品只解决一个主问题，不一品多职
- [ ] `客体匹配成立`：产品直接对应首要客体与首切场景
- [ ] `价值表达成立`：客户能理解它带来的结果，而非只理解功能
- [ ] `位阶归属成立`：该形态清楚属于灯塔、自测或港口中的哪一层

### 2.2 交付成立

- [ ] `交付边界清楚`：哪些做，哪些不做，写得清楚
- [ ] `承诺边界清楚`：承诺没有超出责任闭合能力
- [ ] `失败判定清楚`：可以定义什么叫失败、何时失败
- [ ] `补偿与退出路径存在`：失败后能补偿、降级或退出
- [ ] `首单可完成`：能在当前资源下交得起第一单

### 2.3 节奏成立

- [ ] `前后步骤清楚`：能说清它前面接什么、后面接什么
- [ ] `弱测量友好`：若它是前台或中间层，不要求对象一开始做过高承诺
- [ ] `物理接近动作清楚`：能说清主体通过什么动作进入下一步

### 2.4 商业成立

- [ ] `价格点可接受`：至少存在一个被真实接受的价格点
- [ ] `支付动作匹配`：付款方式与客户当前认知阶段匹配
- [ ] `可标准化`：首单之后能沉淀流程、案例、模板或升级路径
- [ ] `可回流总盘`：成交后能回流为案例、信任、复购或其他线索

---

## 3. 最低通过条件

满足下面六条，才算通过：

- `主产品形态` 已定义
- `单一主任务` 已写清
- `成功交付` 可定义
- `失败判定` 可定义
- 至少有一个客户愿意按该形态和价格真实付费
- 无一票否决项触发

若该形态属于前台或中间层，

建议再加两条：

- 已写清 `它属于哪一层`
- 已写清 `上一步 / 下一步`

建议补充判断：

- 若首单无法轻量完成，先缩小产品
- 若客户只认同价值、不愿意付钱，回到 `LMM.实践.017-定客检查清单v2`
- 若产品承诺大于交付能力，直接回炉
- 若前台形态承担了过多承接任务，拆层重配

---

## 4. 一票否决项

出现以下任一情况，直接停止：

- [ ] 先有产品，再反向编造痛点
- [ ] 产品承担多个主任务，导致边界失控
- [ ] 承诺超出责任承接能力
- [ ] 无失败判定、无补偿设计、无退出路径
- [ ] 一上来就是重开发、重投入、重交付
- [ ] 产品不能形成首单闭环，只能制造热闹
- [ ] 灯塔内容、自测、港口承接三层混写成一个页面或一个产品
- [ ] 尚未出现弱测量信号，就要求对象直接预约、付费或签约

## 7. 最短结论

把本清单压成一句话：

`配品不是决定你想卖什么，而是决定用什么分层承接器，在正确节奏上把问题场和首要客体接住。`
