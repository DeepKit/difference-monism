# ECET.B07 闭环演化动力学

> 前置阅读：B03（三大公理）、B05（偏差与创造力）

---

## 一、核心命题

单次决策不是ECET的分析单元。

> ECET的分析单元是**循环**：认知→行动→反馈→认知更新→再行动。

一个系统的质量不由单次输出决定，而由它的闭环能力决定——它能多快、多准确地从反馈中更新自己的认知模型。

---

## 一·五、结构稳定性的操作性定义

> 本节定义"结构稳定"的判定标准，是后续"漂移"概念的前提。

**定义（结构稳定状态）：**

系统在时刻 $t$ 处于**结构稳定状态**，当且仅当同时满足以下三个条件：

1. **约束可行性**：三大约束（资源约束 $R$、适应选择约束 $A$、准备约束 $P$）均处于可行域内，即 $R(t) \geq R_{min}$，$A(t) \in A_{feasible}$，$P(t) \geq P_{threshold}$。

2. **输出一致性**：系统在连续 $n$ 个执行周期内，输出偏差 $\delta(t)$ 满足 $|\delta(t)| \leq \delta_{stable}$（偏差预算上界，见 B05）。

3. **回溯收敛性**：最近一次回溯操作的修正幅度 $\Delta_{retro}$ 小于修正阈值 $\epsilon_{retro}$，即系统不需要大幅修正即可维持运行。

**结构漂移**定义为：系统从稳定状态退出，即上述任一条件被违反。漂移的严重程度由违反条件的数量和幅度共同决定。

| 违反条件数 | 漂移等级 | 建议响应 |
|-----------|---------|---------|
| 1 | 轻度漂移 | 执行层自动修正 |
| 2 | 中度漂移 | 触发学习层介入 |
| 3 | 重度漂移 | 触发演化层重构或 FREEZE |

---

### FREEZE 状态机：退出条件

FREEZE 不是终态。系统进入 FREEZE 后，必须通过以下三条路径之一退出：

```
              ┌─────────────────────────────────────┐
              │           FREEZE 状态                │
              │  执行暂停 / 等待外部条件              │
              └──────┬──────────┬──────────┬─────────┘
                     │          │          │
          ┌──────────▼──┐  ┌────▼──────┐  ┌▼──────────────┐
          │  路径 F→P   │  │ 路径 F→F  │  │  路径 F→X     │
          │  → PASS     │  │ → FAIL    │  │  → 强制终止   │
          └──────────┬──┘  └────┬──────┘  └┬──────────────┘
                     │          │           │
          条件满足：  │  超时或   │  超时阈值  │
          约束重新    │  条件恶化  │  T_max    │
          进入可行域  │           │  耗尽     │
```

**退出条件定义：**

| 路径 | 触发条件 | 结果 |
|------|---------|------|
| F→PASS | 阻塞约束解除（资源补充 / 外部授权到达 / 准备度恢复至阈值以上） | 恢复执行 |
| F→FAIL | FREEZE 持续时间超过 `T_freeze_max`，或监测到约束进一步恶化 | 标记失败，触发回溯 |
| F→终止 | 系统级超时 `T_system_max` 耗尽，或收到强制终止信号 | 记录状态快照后退出 |

**参数默认值（可由部署层覆盖）：**
- `T_freeze_max`：单次 FREEZE 最大等待时间，默认 = 执行层心跳周期 × 10
- `T_system_max`：系统级总超时，默认 = `T_freeze_max` × 5

> 注：FREEZE 期间系统必须持续记录心跳日志，以便回溯层重建状态序列。


## 二、三层循环结构

ECET将闭环演化动力学分为三层，每层对应一个时间尺度和一个约束：

```
┌─────────────────────────────────────────┐
│  第三层：演化层（长期）                    │
│  结构本身的演化：规则、制度、框架的更新     │
│  时间尺度：月/年    约束：适应选择约束      │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  第二层：学习层（中期）                    │
│  模型的更新：从反馈中修正认知              │
│  时间尺度：小时/天  约束：不完备约束        │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│  第一层：执行层（短期）                    │
│  单次决策的生成与验证                      │
│  时间尺度：毫秒/秒  约束：能量约束          │
└─────────────────────────────────────────┘
```

三层不是独立的。执行层的反馈积累成学习层的更新，学习层的模式积累成演化层的结构变化。

---

## 三、第一层：执行层

### 结构

执行层是单次决策的完整循环：

```
感知（输入）
    ↓
解析（意图识别 + 风险分类）
    ↓
生成（候选输出）
    ↓
验证（独立门禁检查）
    ↓
PASS → 输出 → 记录反馈
FAIL → 回溯 → 重新生成
FREEZE → 暂停 → 请求人类介入
```

### 关键设计原则

**验证必须独立于生成**

这是适应选择约束在执行层的直接要求。筛选机制不能由被筛选的主体自己执行。

**FREEZE是功能，不是失败**

当系统无法判定PASS或FAIL时，FREEZE是正确的响应。一个从不FREEZE的系统在假装完备——这违反不完备约束，比承认不确定性更危险。

**反馈必须被记录**

每次执行的结果（PASS/FAIL/FREEZE及原因）必须被记录，作为学习层的输入。没有记录的反馈等于没有反馈。

### 能量约束在执行层的表现

执行层受能量约束最直接：token预算、API调用限制、响应时间要求。这些约束决定了单次决策的质量上限。

## 四、第二层：学习层

### 结构

学习层处理执行层积累的反馈，更新认知模型：

```
执行层反馈积累
    ↓
模式识别（哪类输入导致FAIL？哪类导致FREEZE？）
    ↓
模型更新（调整生成策略、更新风险分类）
    ↓
验证（更新后的模型是否改善了执行层表现？）
    ↓
部署（将更新后的模型推送到执行层）
```

### 关键设计原则

**学习必须有边界**

无限制的学习会导致模型漂移——系统在适应新数据的过程中遗忘了原有的约束。学习层必须有明确的边界条件：哪些可以更新，哪些不能更新。

规约层（Hard Constraints）不能通过学习被修改。只有映射层（Soft Constraints）可以通过学习调整。

**反事实学习**

不只从成功案例学习，也从FAIL和FREEZE案例学习。FAIL告诉系统什么不能做，FREEZE告诉系统什么是边界情况。这两类反馈往往比PASS更有信息量。

**不完备约束在学习层的表现**

学习层永远无法完备地更新模型——新数据总是有限的，环境总是在变化。这意味着学习层必须保持谦逊：更新是改善，不是完备化。

---

## 五、第三层：演化层

### 结构

演化层处理学习层积累的模式，更新系统的结构性规则：

```
学习层模式积累
    ↓
结构性问题识别（规则本身是否需要修改？）
    ↓
Challenge（对现有规则提出质疑）
    ↓
评估（修改规则的成本与收益）
    ↓
决策（保留/修改/替换规则）
    ↓
部署（将新规则推送到执行层和学习层）
```

### 关键设计原则

**演化层必须有人类参与**

规则的修改是高风险操作。演化层的决策不能完全自动化——必须有人类主体参与评估和授权。这是责任对称原则在演化层的体现。

**Challenge机制**

系统必须有能力对自己的规则提出质疑（Challenge），而不是在规则自相矛盾时死循环或崩溃。Challenge是演化层的核心功能。

但Challenge本身也必须有约束：
- Challenge必须有明确的触发条件（不是随意质疑）
- Challenge必须经过人类审核
- Challenge的结果必须被记录

**反终极化在演化层的体现**

演化层的存在本身就是反终极化原则的工程实现——任何规则都可以被质疑和修改，没有永久有效的结构。

**适应选择约束在演化层的表现**

演化层是适应选择约束作用最直接的地方：不适应环境变化的规则会在演化层被识别和替换。这是系统长期存续的机制。

## 六、三层循环的失效模式

理解失效模式和理解正常运作同样重要：

| 层级 | 失效模式 | 原因 | 后果 |
|------|---------|------|------|
| 执行层 | 从不FREEZE | 假装完备 | 幻觉输出被直接采用 |
| 执行层 | 验证不独立 | 自我验证 | 毁灭性偏差无法被筛选 |
| 学习层 | 无边界学习 | 模型漂移 | 规约层被侵蚀 |
| 学习层 | 只从成功学习 | 忽略FAIL/FREEZE | 边界情况处理能力退化 |
| 演化层 | 无Challenge机制 | 规则僵化 | 系统无法适应环境变化 |
| 演化层 | 无人类参与 | 自主演化 | 责任主体消失 |
| 跨层 | 反馈不记录 | 循环断裂 | 三层变成三个独立系统 |

---

## 七、与影子主权的关系

影子主权结构（C13）本质上是闭环演化动力学的失效：

- **执行层失效**：决策发生，但反馈不被记录（无审计）
- **学习层失效**：反馈存在，但不更新模型（无问责）
- **演化层失效**：问题被识别，但规则不被修改（无改革）

当三层同时失效时，系统继续运作，但责任结构完全断裂——这就是影子主权的工程描述。

## 八、操作性度量

闭环演化动力学的健康程度可以通过以下指标度量：

| 指标 | 含义 | 健康范围 |
|------|------|---------|
| FREEZE率 | 系统承认不确定性的频率 | 非零，但不过高 |
| FAIL→修正周期 | 从FAIL到模型更新的时间 | 越短越好 |
| Challenge频率 | 规则被质疑的频率 | 非零，有人类参与 |
| 反馈记录完整率 | 执行结果被记录的比例 | 趋近100% |
| 规则更新周期 | 演化层更新规则的频率 | 与环境变化速度匹配 |

## 补充节：漂移的规范性分类

> 本节回应一个核心哲学问题：ECET 的稳定性偏好是否系统性地惩罚正当变革？

### 问题的本质

历史上最重要的社会进步——废奴、民权运动、民主化——在纯粹的结构动力学框架下都表现为"结构漂移"，都会触发约束违反信号。如果 ECET 不区分漂移的性质，它就具有内在的保守主义偏向：奖励现状，惩罚变革。

### 漂移的规范性分类

| 类型 | 定义 | 判断标准 | ECET 响应 |
|------|------|---------|---------|
| **病理性漂移** | 系统退化、腐败、权力集中导致的约束违反 | 漂移方向与系统声明目标相悖；受损方为系统内弱势群体 | 触发 FREEZE/FAIL，需干预 |
| **进步性漂移** | 制度革新、范式转换、正义修复导致的约束变化 | 漂移方向与更广泛的价值共识一致；受益方为之前被排除的群体 | 保护漂移空间，调整 ODD 边界 |
| **不确定性漂移** | 无法立即判断性质的约束变化 | 两种判断标准均不明确 | 进入 FREEZE，启动外部评估程序 |

### 关键原则：判断标准不能由系统自身生成

漂移的规范性分类**必须来自外部价值框架**，不能由 ECET 系统内部的结构判断决定。原因：

- 一个系统无法客观判断自身的漂移是否"进步"——这需要外部视角
- 历史上，"系统健康"的定义往往由掌权者制定，系统性地将挑战权力的行为归类为"病理性漂移"
- ECET 的 ODD 边界本身可能需要被"进步性漂移"所修正

### 操作性建议

当检测到漂移时，在触发 FREEZE/FAIL 之前，增加一个**规范性评估步骤**：

```
检测到漂移
    ↓
漂移幅度是否超过 δ_budget？
    ↓ 是
启动规范性评估（外部利益相关方参与）
    ↓
病理性漂移 → FREEZE/FAIL
进步性漂移 → 保护 + ODD 重新评估
不确定性漂移 → FREEZE + 独立评估
```

### 与 ASTO 的关系

本节继承自 ASTO P09 的 Fork/Merge 治理模型：进步性漂移对应"合法的 fork"，病理性漂移对应"恶意的 fork"。ASTO P09 的核心洞见是：**异见需要制度化的表达渠道**，而不是被结构性地压制。ECET 的漂移分类机制是这一洞见的操作化实现。
